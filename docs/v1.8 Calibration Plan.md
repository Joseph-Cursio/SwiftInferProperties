# v1.8 Calibration Plan

Working doc for the SwiftInferProperties v1.8 cut. v1.8 is the **fifth calibration cycle** — closes cycle-5 priority #1 from `docs/calibration-cycle-4-findings.md` (round-trip template coverage-candidate tightening) and runs the §17.3 loop a fifth time against the cycle-1+2+3+4 corpora. Mirrors the v1.5 + v1.6 + v1.7 plans' V1.X.0–V1.X.7 cadence.

> **v1.8 is a calibration cut, not a feature cut.** Like v1.5 / v1.6 / v1.7 it doesn't extend PRD §7.8 — it tightens V1.5.2's `RoundTripTemplate.protocolCoverageVeto(...)` so the `[codableRoundTrip]` veto fires only when the pair's forward/reverse functions actually match a Codable encoder/decoder *shape* (`(T) -> Data` ↔ `(Data) -> T` or `(T) -> String` ↔ `(String) -> T`). User-defined inverse pairs on Codable carriers (e.g., `minimumCapacity(forScale:) ↔ scale(forCapacity:)` on `(Int) -> Int`) fall through unsuppressed because they're *not* Codable round-trips by intent. The fix unwinds V1.7.1's over-suppression cleanly.

**Ephemeral** — archive to `docs/archive/v1.8 Calibration Plan.md` once the tag is pushed and the §13 + §16 release-blocking criteria still hold.

## Context: cycle-4 surfaced an inherited V1.5.2 design question

Per `docs/calibration-cycle-4-findings.md`: V1.7.1's stdlib bake-in worked exactly as designed — extending V1.5.2's coverage veto reach to stdlib-typed carriers. That extension surfaced 22 of 23 V1.7.1 suppressions on the round-trip template via `Int: Codable` / `UInt64: Codable` / `Double: Codable` reach. Sampling those suppressions revealed they were **user-defined inverse pairs**, not Codable round-trips:

```
Suppressed in cycle-4 (false positives by intent):
  minimumCapacity(forScale:) (Int) -> Int  ↔  scale(forCapacity:) (Int) -> Int
  index(after:) (Int) -> Int                ↔  index(before:) (Int) -> Int
  word(after:) (Int) -> Int                 ↔  word(before:) (Int) -> Int
  ...19 more (Int) -> Int / (UInt64) -> Int? pairs...
```

None of these are Codable encode/decode round-trips. The kit's `checkCodablePropertyLaws(for:)` doesn't verify the user-defined inverse claim. V1.5.2's design — vetoing *any* round-trip suggestion on a Codable-typed carrier — was over-broad; V1.7.1's bake-in just made the over-breadth empirically visible.

## What v1.8 ships

Two deliverables:

1. **Shape-gated `RoundTripTemplate.protocolCoverageVeto(...)`.** Replace the current "fire whenever forward type is Codable" logic with a gate on the pair's actual encoder/decoder shape:
   - Forward = encoder `(T) -> Codec`, reverse = decoder `(Codec) -> T` ⇒ veto candidate is the `T` carrier.
   - Forward = decoder `(Codec) -> T`, reverse = encoder `(T) -> Codec` ⇒ veto candidate is the `T` carrier.
   - Otherwise (e.g., `(T) -> T` user inverse, `(T) -> U` non-encode/decode) ⇒ no veto.

   Curated `Codec` set for v1.8: `{Data, String}` — the two canonical Swift Codable wire formats. `[UInt8]` (byte array), `Foundation.URL`-encoded, custom typealiases for codecs deferred to cycle-6 if a sampling pass finds them.

   Mechanism reuses `ProtocolCoverageMap.coverageVetoSignal(...)` directly. No `Signal.Kind` changes; no `KnownProperty` changes; no template-side scoring changes outside `RoundTripTemplate`. The other five algebraic templates (idempotence / commutativity / associativity / inverse-pair / identity-element) are untouched — their candidate sets are op-class-mapped, not type-shape-mapped, so they don't have the same over-breadth.

2. **Cycle-5 calibration run + findings writeup.** Re-run `swift-infer discover --include-possible` against the four cycle-1+2+3+4 corpora at the v1.8 commit; capture post-tightening snapshots to `docs/calibration-cycle-5-data/`. Compare suppression deltas vs the cycle-4 post-bakein baseline (326 total `--include-possible` hits across the four corpora). Findings doc at `docs/calibration-cycle-5-findings.md` documents per-corpus delta + the headline-effect demonstration (the 22 V1.7.1 suppressions on OrderedCollections should re-emerge as Possible-tier surface) + cycle-6 hypotheses.

PRD §19 sets the long-term acceptance-rate target at ≥ 70% after 6 months of dogfooding; v1.8 is cycle 5 of that loop.

### v1.8-only mechanical changes the cycle didn't cover

- **No new `Signal.Kind` additions, no new `KnownProperty` cases.** v1.8's mechanism is a per-template helper rewrite that consults the same V1.5.2 catalog with stricter input criteria.
- **`RoundTripTemplate.suggest(...)` API contract is unchanged.** Same signature, same return type. The shape gate is a new private helper consumed by the existing `protocolCoverageVeto(...)` private helper.
- **Other five algebraic templates untouched.** Their `protocolCoverageVeto(...)` helpers stay as-is — the cycle-4 over-suppression was specific to round-trip's broader candidate set.

## Out of scope

- **Splitting RoundTripTemplate into KitCodable / UserDefinedInverse templates.** This was option (b) in cycle-4 priority #1. More invasive. Open-decision #1 chooses option (a) — the shape gate — for v1.8 because: (i) it's localized to one helper (`protocolCoverageVeto`); (ii) it preserves the M1.4 round-trip identity hash (suggestion deduplication / Decisions-record continuity for any pre-v1.8 accepted suggestions); (iii) it doesn't introduce a new templateName ("user-defined-inverse-pair") that downstream metrics would need to handle separately. If cycle-6 sampling reveals that user-defined inverse pairs need different scoring weights than Codable round-trips, a v1.9 split is the natural next step.
- **Cycle-5 priority #2 — approximate-equality template arm for FP types.** Real `KitFloatingPointTemplate`. Sequenced as a v1.9 candidate.
- **Cycle-5 priority #3 — Possible-tier sampling on the post-v1.8 surface.** Empirical-only follow-up. Best run after V1.8.2 captures the post-tightening baseline.
- **Cycle-5 priority #4 — `surfacedAt` plumbing.** Independent. Sequenced as a v1.9 mechanical addition.
- **Cycle-5 priority #5 — math-library op extension to non-identity templates.** ~1 hour, but bundling with v1.8 muddies attribution. v1.9 candidate.
- **Extended `Codec` set.** v1.8 includes `Data` and `String` only. `[UInt8]` / custom typealiases / domain-specific codec types deferred until cycle-6 sampling reveals a corpus example.
- **Conditional conformance modelling.** Same v1 limitation as v1.5 / v1.7. v1.1 constraint-engine territory (PRD §20.2).
- **PRD §20 v1.1+ trajectory items.** SemanticIndex, IDE integration, `swift-infer apply` stay deferred. v1.8 closes cycle-5 priority #1 only.
- **Kit-side coordination.** `Package.swift` stays at `from: "2.0.0"`. v1.8 doesn't touch the kit.

## Open decisions

1. **Shape gate vs template split.** Two choices: (a) shape gate on existing `RoundTripTemplate.protocolCoverageVeto(...)` helper (this plan); (b) split `RoundTripTemplate` into `KitCodableRoundTripTemplate` (kit-blessed, narrow Codable-only) and `UserDefinedInversePairTemplate` (no Codable veto). Default proposal: **(a) shape gate.** Rationale: localized change to one helper; preserves M1.4 identity-hash continuity; no new templateName for downstream metrics to handle. Cycle-6 can promote to (b) if sampling shows the two semantics need distinct scoring.

2. **Codec format set.** Two choices: (a) narrow `{Data, String}` (this plan); (b) broader set including `[UInt8]`, `Substring`, etc. Default proposal: **(a) narrow.** Rationale: textual-match precision matters more than recall for a coverage veto — v1.8's goal is tightening, not loosening. Cycle-6 sampling will reveal if a corpus example warrants broadening.

3. **Where the helper lives.** Two choices: (a) private static func on `RoundTripTemplate` (this plan); (b) shared utility on `FunctionPair` or a new `CodableShapeDetector` module. Default proposal: **(a) private helper.** Rationale: only `RoundTripTemplate.protocolCoverageVeto(...)` consumes it; sharing it without a second consumer adds an indirection without separation of concerns.

4. **Whether to commit cycle-5 decisions data.** Two choices: (a) commit `docs/calibration-cycle-5-data/` mirroring cycle-1+2+3+4 (this plan); (b) keep cycle-5 data in working copies only. Default proposal: **(a) commit them.** Rationale: cycle-5 captures the *post-tightening surface*. The surface diff vs cycle-4 is the empirical record. Files are small.

5. **Whether to bundle cycle-5 priority #5 (math-library op extension to non-identity templates) into v1.8.** Two choices: (a) ship priority #1 only (this plan); (b) bundle priorities #1 + #5. Default proposal: **(a) priority #1 only.** Rationale: same "one rule per release" precedent as v1.5 / v1.6 / v1.7. Saving #5 for v1.9 keeps v1.8's empirical effect attributable to one mechanism.

6. **Re-run cadence after the tightening lands.** Two choices: (a) capture only post-tightening snapshots (cycle-5 has post-bakein-of-cycle-4 + post-tightening-of-cycle-5); (b) re-capture pre-tightening-of-cycle-5 baselines too. Default proposal: **(a) post-tightening only.** Rationale: cycle-4's post-bakein snapshots are already committed at the v1.7.0 commit and serve as the cycle-5 pre-tightening baseline.

7. **Which file the new tests live in.** Two choices: (a) extend the existing `Tests/SwiftInferTemplatesTests/ProtocolCoverageVetoTests.swift` — natural home given the function lives on `RoundTripTemplate`; (b) new `RoundTripCodableShapeGateTests.swift` — shape-gated concerns deserve their own file. Default proposal: **(a) extend existing file if it stays under 400 lines; otherwise (b).** Following V1.6.1's IdentityElementPairingFilterTests / V1.7.1's StdlibBakeIn split precedent.

## v1.8 trajectory framing

PRD v1.0 §17.3 names calibration as "the operationalization of high precision, low recall — without it, the philosophy is just a slogan." v1.8 is cycle 5 of that loop and is the **first cycle to *increase* the surface count** — every prior cycle decreased it (1167 → 326 over cycles 1–4). This is intentional and informative: cycle 4 surfaced an over-suppression in V1.5.2's design; cycle 5 fixes it. The trajectory framing isn't "more is bad" — it's "the suppression mechanism should match the semantic claim." After v1.8 ships:

- The four prior cycles' structural rules stay in-tree, each guarding a structurally distinct cause-of-noise class. v1.8 doesn't undo any of them; it just narrows V1.5.2's round-trip veto to the cases where the kit law actually applies.
- The cycle-5 post-tightening snapshots in `docs/calibration-cycle-5-data/` are the diff target for cycle 6.
- The cycle-5 findings doc captures the V1.5.2 design question now closed.

The natural cadence stays at one cycle per release. Cycle 6 (~v1.9) likely picks up cycle-5 priority #2 (approximate-equality FP template arm) + #3 (Possible-tier sampling triage) + #4 (`surfacedAt` plumbing) + #5 (math-library op extension to non-identity templates) as paired mechanical additions. After two more cycles, the §19 acceptance-rate target should be measurable.

After the calibration loop reaches steady state, the natural next contributions return to the §20 v1.1+ trajectory:

- **PRD §20 SemanticIndex** — biggest v1.x lift. The v1.8 shape gate is the textual approximation of one slice of what SemanticIndex would deliver authoritatively (via `lookupConformance` and signature-shape querying); cycle-6+ may re-frame the curated codec set as a `SemanticIndex` consumer once that engine ships.
- **PRD §20 IDE integration** — depends on SemanticIndex.
- **PRD §20 `swift-infer apply`** — moves from "all output is opt-in via `--interactive`" to "single command applies a baseline set of accepted suggestions."
- **Multi-predicate equivalence classes** (M13 axis 3) — SemanticIndex-blocked.
- **Cross-target enum coverage** (M14 deferred bit) — SemanticIndex-blocked.
- **Cross-test data-flow correlation** (M10 + M16 deferred) — natural sequencing post-SemanticIndex.

No active milestone plan opens with the v1.8 commit. Subsequent work picks one of the above and opens a fresh `docs/<plan>.md`.
