# Verified-only DocC generation (`swift-infer docc`, V1.142)

Emits DocC documentation for the properties in `.swiftinfer/index.json`, gated
to the **verified** subset: a row is documented only when it has a
`measured-bothPass` record in `.swiftinfer/verify-evidence.json` (joined by
identity hash, `0x`-stripped).

## Why verified-only

Published documentation is trusted as authoritative, so it must be **provable**,
not inferred. A `.likely` / `.possible` property is a reviewed *guess*; putting
it in DocC makes a guess read as a guarantee — worse than silence. So the
generator documents exactly the subset the measured-verify pipeline confirmed,
and nothing weaker. (Inferred-but-unverified rows remain visible through
`swift-infer query` / `discover`, which are understood as candidate surfaces.)

Two consequences of that principle, baked in:
- **Jargon → behavior.** `monoid` / `semilattice` labels are translated to the
  consequence a caller cares about ("argument order doesn't change the result"),
  not the algebra term.
- **No cross-type / refactor insights in DocC.** "You have three monoids,
  consider unifying them" is author-facing design feedback, not reader
  documentation — it stays in the tool's output, out of the published docs.

## CLI

```
swift-infer docc [--directory <root>] [--index-path <p>] [--evidence-path <p>]
                 [--output <dir>] [--dry-run]
```

- Reads the index + verify-evidence, groups verified properties by containing
  type, and writes one DocC **symbol-extension** `.md` per type
  (`# ``Type`` ` + `@DocumentationExtension(mergeBehavior: append)`) plus a
  single `VerifiedProperties.md` **article** for top-level functions.
- **Non-invasive default:** writes to `<root>/.swiftinfer/docc/` (tool-owned),
  never the source tree. Point `--output` at a real `<Target>.docc/Extensions/`
  directory to feed the pages straight into a DocC catalog.
- `--dry-run` lists the pages without writing.
- No verified evidence → nothing is written (with a hint to run
  `swift-infer verify --all-from-index` first).

## Example output

A reducer with a verified `.dismiss` idempotence invariant yields
`.swiftinfer/docc/NavFeature.md`:

```markdown
# ``NavFeature``

@Metadata {
    @DocumentationExtension(mergeBehavior: append)
}

## Verified Properties

Behavioral properties SwiftInfer verified by executed property tests.

- **Idempotent action** — Dispatching the action twice leaves state as if it were dispatched once. `.dismiss`

- Note: Generated by `swift-infer docc` from measured verify evidence — every
  property here is backed by a passing property test. Regenerate after
  `swift-infer verify`; do not edit by hand.
```

## Deliberate omissions

- **Name-redundancy suppression** (skipping a note the symbol name already
  implies, e.g. idempotence on `normalize`) is *not* implemented. For the
  verified subset it's low-value — the note is provably true either way — and
  the heuristic risks suppressing genuine information; left as a possible future
  refinement, not a correctness gap.
- **Source-injected `///` comments** are not written; the non-invasive
  extension-file model is preferred (the tool never mutates source, per §16).
