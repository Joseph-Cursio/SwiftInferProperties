# v1.13 Hoist Plan

Working doc for the SwiftInferProperties v1.13 cut. v1.13 is the **zero-behavior-change refactor** queued from the v1.11 + v1.12 plans' open-decision-#2 commitments — hoist the curated `directionLabels` set out of `IdempotenceTemplate` (where it landed in V1.10.1 as the first-consumer-by-default) into a shared `SwiftInferCore.DirectionLabels` namespace, now that V1.10.1 / V1.11.1 / V1.12.1 have made it a three-template shared utility.

> **v1.13 closes the four-cycle abstraction-development cadence: introduce (cycle 7) → replicate (cycle 8) → complete the family (cycle 9) → hoist (cycle 10, this release).** Same posture as V1.6.1's three-orthogonal-cleanup release between mechanism cycles: pure refactor for site-of-truth cleanup; doesn't affect surface counts; lands as an attribution-clean release between mechanism cycles.

**Ephemeral** — archive to `docs/archive/v1.13 Hoist Plan.md` once the tag is pushed and the §13 + §16 release-blocking criteria still hold.

## Context: the open-decision-#2 commitment from v1.11 + v1.12

Per `docs/archive/v1.11 Calibration Plan.md` open decision #2 ("Where the curated `directionLabels` set lives"):

> **(a) reuse `IdempotenceTemplate.directionLabels` directly (this plan)** … Hoisting (option b) is premature — wait until cycle-9's `RoundTripTemplate` becomes the third consumer, then refactor to a shared namespace as a v1.13 atomic move.

Per `docs/archive/v1.12 Calibration Plan.md` open decision #2:

> **(a) continue reusing `IdempotenceTemplate.directionLabels` directly (this plan)** … Rationale: the v1.11 open-decision-#2 commitment was "hoist as a v1.13 atomic move when round-trip becomes the third consumer." That commitment stands — *bundling* the hoist with v1.12 would muddy attribution between (i) the structural change (counter-signal) and (ii) the refactor (hoist).

v1.13 executes that commitment.

### Current shape (post-v1.12)

```
SwiftInferCore.Signal.Kind.directionLabel    ← already in core (V1.10.1)

SwiftInferTemplates.IdempotenceTemplate.directionLabels: Set<String>   ← lives here
                                                                          (10-element curated set)

SwiftInferTemplates.IdempotenceTemplate    consumes directionLabels (V1.10.1)
SwiftInferTemplates.InversePairTemplate    consumes IdempotenceTemplate.directionLabels (V1.11.1)
SwiftInferTemplates.RoundTripTemplate      consumes IdempotenceTemplate.directionLabels (V1.12.1)
```

Three consumers reach across to a static let on a type that owns the set "by historical accident" — V1.10.1 was the first consumer, so the set landed there. Now that the set is template-agnostic (validated empirically across `+25`/`+30`/`+30` baselines in cycles 7-9), the cross-template reach is the wrong shape.

### Target shape (post-v1.13)

```
SwiftInferCore.Signal.Kind.directionLabel    ← unchanged (already in core)

SwiftInferCore.DirectionLabels.curated: Set<String>   ← new canonical home
                                                       (10-element curated set verbatim)

SwiftInferTemplates.IdempotenceTemplate    consumes DirectionLabels.curated
SwiftInferTemplates.InversePairTemplate    consumes DirectionLabels.curated
SwiftInferTemplates.RoundTripTemplate      consumes DirectionLabels.curated
```

All three templates reach across to the same shared utility in core, mirroring how `Signal.Kind.directionLabel` is already factored.

## Open decisions

1. **Backwards compatibility: clean cut vs deprecated re-export.** Two choices: (a) **clean cut (this plan's default)** — remove `IdempotenceTemplate.directionLabels` entirely, update all in-tree consumers; (b) deprecated re-export — leave `@available(*, deprecated, message: "Use SwiftInferCore.DirectionLabels.curated") public static let directionLabels` as a forwarding alias. Default proposal: **(a) clean cut.** Rationale: the field is internal-mechanism territory (a calibration knob exposed via `public` for cross-template reuse, not a documented external API). The package's user-facing surface is the `swift-infer` CLI, not the SwiftInferTemplates Swift API. The CHANGELOG records the move. CLAUDE.md global guidance ("If you are certain that something is unused, you can delete it completely") applies. v1.x SemVer is loose for internal-mechanism API in this project (per the v1.4-v1.12 minor-version cadence).

2. **New name: `DirectionLabels.curated` vs `DirectionLabels.all` vs `DirectionLabels.labels`.** Default proposal: **`.curated`.** Rationale: matches the existing v1.x calibration vocabulary (`RoundTripTemplate.curatedInversePairs`, `IdempotenceTemplate.curatedVerbs`). Conveys "this is the curated subset, not exhaustive." Alternatives `.all` / `.labels` are less precise — `.all` implies completeness which is wrong (cycle-10's stride-style label extension would add a separate set), and `.labels` adds no information beyond the namespace name.

3. **Whether to update doc-comment cross-references in the three source files.** Default proposal: **yes.** All three source files have doc comments referencing `IdempotenceTemplate.directionLabels` in their textual descriptions; updating to `DirectionLabels.curated` is part of the refactor's site-of-truth-cleanup intent. Cost is small (~6 doc-comment lines across 3 files).

4. **Where to put the cycle-6 motivation comment.** Default: **migrate to `DirectionLabels.curated`'s doc comment.** The comment explains *why* this set has these 10 elements (cycle-6's 0/10 idempotence acceptance rate measurement); it should travel with the data.

5. **Whether the byte-stable verification needs all four corpora or one.** Default: **one corpus (Algorithms — largest direction-counter consumer at 90.0% of round-trip surface eliminated in cycle-9).** Rationale: the hoist is purely structural; if it changes any corpus's output, it changes all of them identically. One corpus's byte-identical comparison is sufficient evidence for zero-behavior-change. (If paranoid, all four corpora can be checked; the plan defaults to one.)

6. **Whether to track this as cycle 10.** Default: **yes — formally cycle 10 of the §17.3 empirical-tuning loop, but with a refactor character rather than a mechanism character.** Per the four-cycle abstraction-development cadence (introduce → replicate → complete → hoist), cycle 10 is the hoist phase. The cumulative-trajectory chart in CLAUDE.md gets a row showing "Surface 257 → 257 (0)" for cycle 10, recording the no-op surface posture. Alternative ("treat as v1.6.1-style maintenance patch, not a numbered cycle") under-counts the strategic role of the abstraction-completion step.

## v1.13 trajectory framing

PRD v1.0 §17.3 names calibration as "the operationalization of high precision, low recall."

Cycle 7 (v1.10) introduced the mechanism on `IdempotenceTemplate`.
Cycle 8 (v1.11) replicated it on `InversePairTemplate`.
Cycle 9 (v1.12) completed the family on `RoundTripTemplate`.
Cycle 10 (v1.13, this release) hoists the abstraction.

The four-cycle abstraction-development cadence is now confirmed as a useful design pattern for the calibration loop. Future mechanisms (domain-mismatch detection on idempotence + inverse-pair + round-trip simultaneously, SetAlgebra-shape coverage candidates, FP approximate-equality template arm) will likely follow the same shape — possibly compressed into fewer cycles when the mechanism shape is well-understood at introduce-time, possibly extended when more templates need replication.

After v1.13 ships:
- The `directionLabels` set lives at its canonical home in core, factored alongside `Signal.Kind.directionLabel`.
- The cycle-10 priority list rotates: SetAlgebra-shape detection on inverse-pair → priority #1 (was #2); domain-mismatch family → priority #2 (was #3 elevated); stride-style label extension → priority #3 (was #4); Possible-tier re-sampling on the 257-surface → priority #4.
- The next mechanism cycle (v1.14) picks one of the rotated priorities and opens a fresh `docs/v1.14 Calibration Plan.md`.

The natural cadence stays at one mechanism per release, with refactor releases interspersed when the abstraction-development cadence calls for them. v1.6.1 (three orthogonal cleanups) and v1.13 (the hoist) are the two refactor releases shipped to date; both follow mechanism cycles whose work either accumulated cleanup debt (v1.6.1) or completed an abstraction's third-consumer threshold (v1.13).

After the calibration loop reaches steady state, the natural next contributions return to the §20 v1.1+ trajectory.

No active milestone plan opens with the v1.13 commit. Subsequent work picks one of the §20 items, the cycle-10 priority list (post-rotation), or a fresh refactor target and opens a fresh `docs/<plan>.md`.
