# TestLifter M4 Execution Plan

Working doc for the TestLifter M4 milestone defined in `SwiftInferProperties PRD v0.4.md` §7.9 + §7.4. Decomposes M4 into six sub-milestones so progress is checkable session-by-session. **Ephemeral** — archive to `docs/archive/TestLifter M4 Plan.md` once M4 ships and the §7.9 + §7.4 acceptance bars are met (mirroring TestLifter M1 + M2 + M3 + TemplateEngine M1–M8).

> **TestLifter M4 closes the type-recovery + generator-synthesis gap M3 left open.** TestLifter M3 promoted lifted suggestions into the visible `discover` stream with one strict fallback: when `LiftedSuggestionRecovery`'s FunctionSummary lookup misses, the promoted Suggestion ships with `?`-sentinel evidence and a `.todo<?>()` accept-flow stub. M4 widens recovery on **two axes**, both grounded in PRD §7.4: **(i) syntactic type recovery** from the sliced test body's setup region (`let x: T = ...` typed bindings + `let x = T(...)` constructor expressions feed type names into `LiftedSuggestionRecovery` as a second-tier fallback below FunctionSummary lookup); **(ii) mock-based generator synthesis** from observed test construction (the §7.4 "inferred from observed test construction" rung between `memberwiseArbitrary` and `.todo` — when ≥3 test sites construct `T` via the same initializer with the same argument-label shape, synthesize a `Gen<T> { T(label: <Gen for kind>.run()) }` and route it through the existing `GeneratorSelection` pass with `.inferredFromTests` source + `.low` confidence). The case enum entry — `GeneratorMetadata.Source.inferredFromTests` at `Sources/SwiftInferCore/Suggestion.swift:98` — is **already reserved** in M1's placeholder shape; M4 wires the path, doesn't grow the type. After M4, `.todo<?>()` stubs survive only when neither FunctionSummary lookup, setup-region annotation, nor a corpus-wide ≥3-site construction record matches — i.e., genuinely-novel types the test corpus has never built.

## What M4 ships (PRD v0.4 §7.9 + §7.4)

> **M4 (TestLifter).** Mock-based generator synthesis from observed test construction. *(Verbatim from PRD §7.9.)*

> **§7.4 fallback ladder (verbatim).** CaseIterable → RawRepresentable → memberwise (delegated to shared `DerivationStrategist`) → mock-inferred from observed test construction → Codable round-trip → `.todo`. Generator confidence (`high` / `medium` / `low`) flows into the explainability block.

> **§13 acceptance numerical target (verbatim).** Mock-based generator synthesis produces a valid `Gen<T>` for at least 50% of the types where ≥3 test sites construct via the same initializer.

Five concrete deliverables, all additive on top of M1 + M2 + M3's library:

1. **`SetupRegionTypeAnnotationScanner` — syntactic type recovery from `SlicedTestBody.setup`.** New `Sources/SwiftInferTestLifter/SetupRegionTypeAnnotationScanner.swift`. Public entry: `static func annotations(in slice: SlicedTestBody) -> [String: String]` returns a `[bindingName: typeName]` map covering both `let x: T = ...` typed bindings (annotation explicit) and `let x = T(arg: 42)` constructor expressions (type read from the called expression's identifier). Pure-syntactic; no FunctionSummary needed; no corpus-wide threading. Per-test-method scope. Setup-region only — the property region's bindings are already covered by M3's FunctionSummary lookup against the callees they reference.

2. **`SetupRegionConstructionScanner` — corpus-level construction-record aggregation.** New `Sources/SwiftInferTestLifter/SetupRegionConstructionScanner.swift`. Public entry: `static func record(over testFiles: [TestSuiteSummary]) -> ConstructionRecord` walks every test method's setup *and* property region for `T(label1: literal1, label2: literal2)` shapes and aggregates them into a `ConstructionRecord` keyed by `(typeName, argumentShape)` where `argumentShape` is `[(label: String?, kind: ParameterizedValue.Kind)]` (labels positional → `label: nil`; reuses M1's `ParameterizedValue.Kind` enum so no new vocab). Each record entry tracks `siteCount: Int` plus the observed literal texts per argument slot (kept for OD #5 — finite-set vs by-kind generator). The record is built once per `discover` run, threaded into `LiftedSuggestionPipeline.promote(...)` alongside the existing FunctionSummary index.

3. **`LiftedSuggestionRecovery` annotation fallback.** Extend `Sources/SwiftInferTestLifter/LiftedSuggestionRecovery.swift` with a third recovery tier. Current ladder: FunctionSummary match → recovered type; otherwise `nil`. M4 ladder: FunctionSummary match → recovered type; *else* setup-region annotation match by binding name → recovered type; *else* `nil`. The annotation lookup uses M4.0's `[bindingName: typeName]` map per the lifted suggestion's binding name (round-trip's `inputBindingName`, idempotence's input variable, commutativity's two operand bindings). Failures still drop through to `.todo<?>()` — annotation recovery is **strict syntactic** (no type-checking, no generic resolution), so unrecognized bindings still produce `nil`.

4. **`MockGeneratorSynthesizer` — the §7.4 mock rung.** New `Sources/SwiftInferTestLifter/MockGeneratorSynthesizer.swift`. Public entry: `static func synthesize(typeName: String, record: ConstructionRecord) -> MockGenerator?` returns a `MockGenerator(typeName: String, argumentSpec: [(label: String?, kind: ParameterizedValue.Kind, observedLiterals: [String])])` when the record has a single dominant `argumentShape` for `typeName` with `siteCount ≥ 3` (the §13 threshold). Multiple distinct argument shapes for the same type → `nil` (the test corpus is ambiguous about which constructor to mock; conservative posture per PRD §3.5). Single-site / two-site shapes → `nil` (under the §13 threshold). The synthesized `MockGenerator` value is plumbed through a new `GeneratorSelection.applyMockInferredFallback(...)` pass that runs **after** the existing `DerivationStrategist`-based pass — so corpus-side memberwise derivation still wins when both apply, and mock inference only fires when memberwise can't produce a generator (the type's not in `corpus.typeDecls`, or has stored properties whose shapes the strategist can't memberwise-derive).

5. **Accept-flow rendering for `.inferredFromTests` source.** Extend `Sources/SwiftInferCLI/InteractiveTriage+Accept.swift`'s lifted-stub renderer to recognize `GeneratorMetadata.Source.inferredFromTests` and emit a `Gen<T> { T(label: <Gen.kind>.run(), ...) }` body using the M4.3 `MockGenerator.argumentSpec`. Adds a per-stub provenance comment: `// Mock-inferred from N construction sites in test bodies (low confidence — verify the generator covers your domain)`. The existing `.todo<?>()` path stays load-bearing as the final fallback when even mock synthesis can't fire.

### Important scope clarifications

- **No kit-side coordination commit needed.** `GeneratorMetadata.Source.inferredFromTests` is already in the `Sources/SwiftInferCore/Suggestion.swift:91-100` enum (reserved at M1, untouched through M3). The mock-inferred metadata is fully describable in the existing enum + confidence shape (`.low` confidence per PRD §7.4); no new fields, no new variants. SwiftProtocolLaws is one-way downstream; mock synthesis is a SwiftInferProperties-side concept that the kit's `DerivationStrategist` doesn't see and doesn't need to. **`Package.swift` stays at `from: "1.9.0"`** for M4. No kit PR, no version bump.

- **Mock synthesis runs *after* the `DerivationStrategist` pass, not in place of it.** §7.4's fallback ladder is ordered: CaseIterable → RawRepresentable → memberwise → mock → Codable round-trip → `.todo`. M4.3 wires this ordering by running `GeneratorSelection.apply(...)` first (the existing M3 path — strategist-driven), then `GeneratorSelection.applyMockInferredFallback(...)` second on suggestions whose `generator.source == .notYetComputed` after the first pass. Suggestions whose memberwise derivation succeeded never see mock synthesis, even if their type also appears in the construction record. The corpus-side strategist always wins.

- **Codable round-trip generator (rung 5 of §7.4) is out of scope for M4.** The PRD §7.4 ladder lists Codable round-trip as the rung between mock-inferred and `.todo`. M4 ships rung 4 (mock-inferred) only; rung 5 is straightforward (when `T: Codable` per a `FunctionSummary`-side conformance scan, synthesize `Gen<T>` that round-trips through encode/decode of a fixed seed object) but slots more naturally with TestLifter M5 (ordering / count-change / reduce-equivalence) since both touch the same FunctionSummary conformance index. Open decision #6 below covers the ratchet.

- **Mock synthesis is corpus-wide, not per-test-file.** §13's threshold is "≥3 test sites construct via the same initializer" — sites can live in different test methods or different test files. M4.1's `ConstructionRecord` aggregates over the *entire* `[TestSuiteSummary]` array `TestLifter.discover` already builds, so the threshold count reflects corpus-level construction frequency. Per-test-file aggregation would miss the common case of three CodecTests files each constructing one fixture.

- **Construction-shape fingerprinting is by argument-label + literal-kind, not by exact value.** Three sites that all construct `Doc(title: "x", count: 3)`, `Doc(title: "y", count: 5)`, `Doc(title: "z", count: 7)` share the same fingerprint `[(title, .string), (count, .integer)]` and aggregate to one record entry with `siteCount: 3`. Three sites with different label sets (`Doc(title:)`, `Doc(name:)`, `Doc(title:author:)`) produce three distinct record entries each at `siteCount: 1` — none clear the threshold, mock synthesis returns `nil`. Open decision #5 below covers a tighter alternative (strict-equality literal aggregation).

- **Setup-region annotation walking is bounded — typed `let` bindings + bare `T(...)` constructor expressions only.** No type inference, no expression-result walking, no method-chain inspection. `let a: Doc = makeDoc()` recovers `a → Doc` via the type annotation; `let a = Doc(...)` recovers `a → Doc` via the constructor's identifier; `let a = makeDoc()` does **not** recover (no annotation, no bare constructor — the type lives only in `makeDoc`'s return type which we'd need a FunctionSummary lookup to recover, and that's the M3 path that already failed). The same posture as M3's "FunctionSummary lookup or `?`": one syntactic step, no ambitious inference.

- **Receiver-type recovery for instance-method patterns is in scope and reuses the same annotation-fallback path.** `AssertSymmetryDetector`'s `a.merge(b)` shape needs `a`'s type. M3 recovered it via the callee's `containingTypeName` from FunctionSummary; when that misses, M4.2's annotation fallback looks up `a` in the setup-region map. Same path as the parameter-type recovery for free functions — no per-detector special-casing.

- **Lifted suggestions accepted with mock-inferred generators write to the same `Tests/Generated/SwiftInfer/<template>/<TestMethodName>_lifted_<TemplateName>.swift` location as M3.** No new directory layout. The provenance comment header from M3.3 ("Lifted from `Tests/FooTests/CodecTests.swift:42` `testRoundTrip()`") gets a sibling line ("Mock-inferred generator from N construction sites — low confidence"). Hard-guarantee §16 #1 (writeouts go ONLY under `Tests/Generated/SwiftInfer/`) preserved.

- **Mock-inferred generators ship at `.low` confidence per PRD §7.4.** The §4.5 explainability block surfaces the confidence + the construction-site count + the fingerprint shape. PRD §3.5 conservative-engine bias means `.low` is genuinely meant — users should treat mock-inferred generators as starting points, not as load-bearing test infrastructure. The `// Mock-inferred...verify the generator covers your domain` comment in the writeout is the user-facing reinforcement.

- **No counter-signal scanning for mock synthesis in M4.** A test that constructs `Doc(title: "")` once with an empty-string title doesn't suppress mock synthesis the way an asymmetric assertion would suppress a symmetry suggestion (TestLifter M7's row). Mock synthesis is permissive on argument-value range — that's why it's `.low` confidence. Open decision #7 below covers a future tighter posture.

- **No `// swiftinfer: skip` honoring for mock-inferred suggestions in M4.** TestLifter M6 (still bundles the test-side `// swiftinfer: skip` recognition + decisions.json persistence). M4 produces accept-flow writeouts but doesn't persist mock-inferred lifted-suggestion identities to `.swiftinfer/decisions.json`. Same posture as M3's M6 deferral.

- **Mock-inferred generator metadata for *TemplateEngine* suggestions (not just lifted ones) is in scope but bounded.** When a TemplateEngine-side suggestion's type isn't in `corpus.typeDecls` but appears in the `ConstructionRecord` with ≥3 sites, the M4.3 fallback pass also fires for that suggestion. This is the right behavior — the construction record describes the test corpus, not the lifted-suggestion population. Open decision #2 below covers the alternative (lifted-only fallback).

- **§13 perf budget compliance with M4 active.** The 100-test-file budget already includes the slicer + three detectors + M3 promotion + GeneratorSelection. M4 adds (a) one extra walk per test method body for setup-region annotations (linear in body size), (b) one extra walk per test method body for construction-record aggregation (linear in body size), (c) the synthesizer is O(record_entries × suggestions) which is bounded by the suggestion count. Targeted overhead: < 200ms on the synthetic 100-file corpus. The §13 row stays at `< 3s` wall.

## M4 acceptance bar

Mirroring PRD §7.9 + §7.4 + §13's numerical acceptance row + the v0.4 §5.8 acceptance-bar pattern + the M1 / M2 / M3 / TemplateEngine M1–M8 cadence, M4 is not done until:

a. **`SetupRegionTypeAnnotationScanner.annotations(in:)` returns a `[bindingName: typeName]` map covering both `let x: T = ...` typed bindings and `let x = T(...)` bare-constructor bindings** in the slice's setup + property regions. Tuple patterns and method-call-result bindings are silently skipped.

b. **`SetupRegionConstructionScanner.record(over:)` returns a corpus-wide `ConstructionRecord` keyed by `(typeName, argumentShape)`** with per-entry `siteCount` and observed-literal rows. Constructor calls with non-literal arguments are skipped (can't fingerprint).

c. **`LiftedSuggestionRecovery` falls back to setup-region annotations when FunctionSummary lookup misses.** Precedence: FunctionSummary > annotation > nil. The `Discover+Pipeline.LiftedSuggestionPipeline.promote(...)` call computes the per-test-method annotation map and threads it through.

d. **`MockGeneratorSynthesizer.synthesize(typeName:record:)` returns a `MockGenerator?`** under the §13 threshold (single dominant shape with `siteCount ≥ 3`); ambiguous-shape and under-threshold inputs return `nil`.

e. **`GeneratorSelection.applyMockInferredFallback(...)` runs after the existing strategist-driven pass** and only rebuilds Suggestions whose `generator.source == .notYetComputed`. Strategist-derived generators are never overwritten.

f. **`InteractiveTriage+Accept` renders `Gen<T> { T(label: <Gen for kind>.run(), ...) }` stubs** for accepted lifted suggestions with `.inferredFromTests` source, with a "Mock-inferred from N construction sites — low confidence" provenance line above the existing M3 "Lifted from..." line.

g. **PRD §13 numerical acceptance — ≥50% of types with ≥3 same-shape construction sites get a valid `Gen<T>`** — verified by `MockSynthesisCoverageTests` (M4.5 (c)).

h. **§13 performance budget for `TestLifter parse of 100 test files` (`< 3s` wall) holds** with all of M4.0–M4.4 active. Regression test fails if exceeded.

i. **§16 #1 hard guarantee preserved** — M4's mock-inferred accept-flow writeouts go ONLY to `Tests/Generated/SwiftInfer/`. Verified by extending `TestLifterHardGuaranteeTests`.

j. **§16 #4 hard guarantee preserved** — `.todo<?>()` survivors (types with no FunctionSummary, no annotation, <3 construction sites) still produce stubs that don't compile until the user resolves them. Verified by extending `TodoFallbackHardGuaranteeTests`.

## Out of scope for M4 (re-stated for clarity)

- **Codable round-trip generator (rung 5 of §7.4's ladder)** — TestLifter M5 (paired with the ordering / count-change / reduce-equivalence pattern detection that also touches the FunctionSummary conformance index).
- **Ordering, count-change, reduce-equivalence pattern detection** — TestLifter M5.
- **TestLifter-side `decisions.json` persistence** for mock-inferred suggestions — TestLifter M6.
- **`// swiftinfer: skip` honoring** on the test side — TestLifter M6.
- **`--test-dir` override** — TestLifter M6.
- **Counter-signal scanning** for mock synthesis (e.g. a single asymmetric construction `Doc(title: nil)` suppressing a non-nil mock) — TestLifter M7.
- **`swift-infer convert-counterexample`** — TestLifter M8.

## Open decisions to make in-flight

1. **Site-count threshold for mock synthesis — ≥3 (PRD §13 default) or relax to ≥2 / tighten to ≥4?**
   - **(a) ≥3.** Matches PRD §13's stated bar verbatim. The numerical acceptance test in §13 is calibrated against this threshold.
   - **(b) ≥2.** Catches more types; doubles the false-positive risk (a two-site fixture pattern that happens to share a shape doesn't necessarily generalize).
   - **(c) ≥4.** Tighter precision, lower recall. Misses the PRD §13 numerical-acceptance row (which is calibrated at ≥3).
   - **Default unless reason emerges:** **(a) ≥3**. PRD §13 spells the threshold out explicitly. M4 doesn't relitigate the calibration.

2. **Mock-inferred fallback for *TemplateEngine* suggestions (not just lifted ones) — fire on both, or lifted-only?**
   - **(a) Fire on both.** TemplateEngine suggestions whose type isn't in `corpus.typeDecls` (e.g. a function operating on a type defined outside the scanned root) but appears in the construction record get the mock-inferred generator too. The construction record describes the test corpus; it's relevant to any suggestion whose type lives in that corpus, regardless of which contribution surfaced the suggestion.
   - **(b) Lifted-only.** Mock synthesis only fires on Suggestions with non-nil `liftedOrigin`. Cleaner separation but artificially narrows mock synthesis's reach.
   - **Default unless reason emerges:** **(a) fire on both.** The construction record is corpus-scoped fact; gating on lifted-origin-ness would be incidental.

3. **Multi-shape ambiguity policy — return `nil` (conservative) or pick the dominant shape and emit a warning?**
   - **(a) Return `nil`.** Multi-shape types stay `.todo`. PRD §3.5 conservative-engine bias — when in doubt, fewer suggestions.
   - **(b) Pick the dominant shape (highest siteCount), emit a diagnostic.** More mock generators land; the diagnostic surfaces the ambiguity for user review.
   - **Default unless reason emerges:** **(a) nil**. Mock generators are already `.low` confidence; layering "and we picked one of multiple shapes for you" on top doubles the user-trust load. The PRD §13 50% target is calibrated against single-shape types.

4. **Mock-generator carry through `Suggestion` — new field on `GeneratorMetadata`, opaque on `Suggestion`, or via `Signal.detail` parsing?**
   - **(a) Add `GeneratorMetadata.mockGenerator: MockGenerator?` field.** Direct, explicit, structured. Bloats `GeneratorMetadata`'s storage with a TestLifter-specific concept that's `nil` for every other source.
   - **(b) Add `Suggestion.mockGenerator: MockGenerator?` opaque field** alongside the M3 `liftedOrigin`. Same shape as the M3 decision — pure additive optional field on `Suggestion`. Cleaner than (a) (the `nil`-for-non-mock case lives at the call-site rather than in `GeneratorMetadata`'s shape).
   - **(c) Parse from the `Signal(kind: .testBodyPattern)`'s detail string.** Zero new fields. Fragile string-parsing contract.
   - **Default unless reason emerges:** **(b) `Suggestion.mockGenerator: MockGenerator?` field**. Mirrors the M3 `liftedOrigin` shape exactly, so the additive surface is one-line cost. Renderer + accept-flow read the field directly.

5. **Generator-argument literal source — sample-from-observed (`Gen.element(of: ["x", "y", "z"])`) or by-kind (`Gen<String>`)?**
   - **(a) By-kind `Gen<T>`.** Mock generator covers the full `T` domain — `Gen<String>` produces arbitrary strings, not just the three observed literal values. Maximizes property-test value (the property runs against unconstrained inputs); minimizes "you're testing your test data" risk.
   - **(b) Sample-from-observed `Gen.element(of: ...)`.** Mock generator's domain is exactly the observed test values. Honest about what the test corpus has actually exercised; PRD §3.5 conservative-bias-friendly. Risks turning the property test into a glorified table test.
   - **Default unless reason emerges:** **(a) by-kind `Gen<T>`.** PRD §3.5 conservative bias is about *not making confident claims*, not about *narrowing the input domain*. A property that only holds against three specific values shouldn't be claimed as a property at all. The `.low` confidence + provenance comment cover the user-trust load; the generator itself should give the property test a fair shot at finding counterexamples. The observed-literal data still lives in `MockGenerator.argumentSpec.observedLiterals` for OD-9-shaped future use (e.g. M9 inferred-domain detection).

6. **Codable round-trip generator (§7.4 rung 5) — slot in M4 or defer to M5?**
   - **(a) Defer to M5.** Pairs naturally with M5's reduce-equivalence patterns which also touch the FunctionSummary conformance index (`T: Codable`, `T: Sequence`, etc.). M4 stays focused on the mock-synthesis rung.
   - **(b) Slot in M4.** Closes both intermediate rungs at once. Doubles the M4 surface area; requires a FunctionSummary conformance scan that M3 didn't ship.
   - **Default unless reason emerges:** **(a) defer to M5**. Conformance scanning is a discrete chunk of work. M4's mock-synthesis surface is already 4-deliverable-wide; folding Codable round-trip in adds pipeline complexity without sharpening the mock-synthesis story. M5's conformance-scan touchpoint is the natural home.

7. **Mock generator suppression on test-corpus contradictions — gate on M7-style counter-signal scan, or ship M4 unconditionally?**
   - **(a) Ship M4 unconditionally.** Mock synthesis fires on ≥3 same-shape sites regardless of whether other sites use a different shape (those produce a separate ambiguous record entry — M4.3 returns `nil` for ambiguous types per OD #3). No counter-signal pass needed; the multi-shape ambiguity already filters.
   - **(b) Gate on M7-style asymmetric-construction scan.** A test that constructs `Doc(title: "")` once with an empty title alongside three `Doc(title: "x")` calls would *suppress* mock synthesis (the empty-title case is a counter-signal even though the shape matches). Folds M7-style work into M4.
   - **Default unless reason emerges:** **(a) ship M4 unconditionally**. The `.low` confidence + by-kind generators (OD #5 default) already give the property test fair shake — an empty-string observation isn't a counter-signal in the by-kind regime, it's just one point in `Gen<String>`'s natural domain. M7's counter-signal scan is the right home for the *symmetric/asymmetric assertion* asymmetry it's designed for; bolting mock-synthesis suppression onto it conflates two different concepts.
