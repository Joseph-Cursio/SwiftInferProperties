## Context: the post-v1.14 OC domain-marker survivors

Per `docs/calibration-cycle-11-data/post-setalgebra-veto-swift-collections-OrderedCollections.discover.txt`, the OC round-trip + idempotence Possible-tier survivors with first-parameter labels in a candidate semantic-intent set fit a single pattern:

```
round-trip claims surviving cycles 7-11 on OC with domain-marker labels (n=10):

  both-sides labeled (n=9):
    minimumCapacity(forScale:)   ↔ maximumCapacity(forScale:)
    minimumCapacity(forScale:)   ↔ scale(forCapacity:)
    minimumCapacity(forScale:)   ↔ wordCount(forScale:)
    maximumCapacity(forScale:)   ↔ scale(forCapacity:)
    maximumCapacity(forScale:)   ↔ wordCount(forScale:)
    scale(forCapacity:)          ↔ wordCount(forScale:)
    _minimumCapacity(forScale:)  ↔ _maximumCapacity(forScale:)
    _minimumCapacity(forScale:)  ↔ _scale(forCapacity:)
    _maximumCapacity(forScale:)  ↔ _scale(forCapacity:)

  one-side labeled (n=1):
    _value(forBucketContents:)   ↔ _bucketContents(for:)
    (the asymmetric `forBucketContents` ↔ `for` pair — possibly a true-positive
     round-trip pair; "for:" is the unlabeled-domain "given X" carrier, while
     the forward-side `forBucketContents` is the explicit semantic-intent label.
     Open decision #2 below considers either-side vs both-sides detection.)

idempotence claims surviving cycles 7-11 on OC with domain-marker labels (n=7):

  minimumCapacity(forScale:)
  maximumCapacity(forScale:)
  scale(forCapacity:)
  wordCount(forScale:)
  _minimumCapacity(forScale:)
  _maximumCapacity(forScale:)
  _scale(forCapacity:)

inverse-pair claims surviving cycles 7-11 on OC with domain-marker labels (n=0).
(Post-v1.14, OC inverse-pair surface is at 0; the SetAlgebra-shape veto from
 V1.14.1 cleared the 6 pre-cycle-11 candidates, leaving no domain-marker candidates.)
```

These are HashTable internals that map between three logically distinct *domains* — scale, capacity, and wordCount/bucketContents — even though the carrier type is `Int -> Int` on every signature. The Swift parameter label `forScale:` / `forCapacity:` / `forBucketContents:` is the semantic-intent marker that tells the reader "this function takes the X domain". A pair of these is a cross-domain conversion, not a forward-reverse round-trip; running one twice doesn't return to the original because the function crosses domains.

The mathematical truth: **`min(forScale: x) ↔ max(forScale: x)` is two functions over the *same* input domain (scale), not a forward+reverse pair**; **`min(forScale: x) ↔ scale(forCapacity: x)` is a domain-mismatch chain (scale-domain producing capacity-domain in forward, then capacity-domain producing scale-domain in reverse); the carrier is `Int -> Int` on both sides but the *semantic codomain* is incompatible**. These are not round-trip pairs even though the type-symmetry signature passes (`Int -> Int ↔ Int -> Int`). Idempotence claims have the same structural problem — applying `min(forScale:)` twice means feeding capacity-domain output into a scale-domain input slot, which is structurally wrong.

### Why the existing direction-label counter (cycles 7-9) misses these

Per `Sources/SwiftInferCore/DirectionLabels.swift`:

```swift
public static let curated: Set<String> = [
    "after", "before", "next", "prev", "previous",
    "advance", "succ", "pred", "successor", "predecessor"
]
```

The curated direction-label set is *spatial-sequence* labels (positional iteration over an ordered space). HashTable conversion labels (`forScale`, `forCapacity`, `forBucketContents`) are *semantic-intent* labels (named domain markers). The two classes are disjoint by intent and don't overlap textually. Adding HashTable terms to `DirectionLabels.curated` would break the v1.10/v1.11/v1.12 mechanism's narrow charter (sequence-iteration suppression).

A separate curated set + counter in the same parameter-label-counter mechanism class is the right factoring.

### Why a single mechanism applied to three templates is correct here

The semantic-intent argument "any function with parameter label `forScale:` produces output in a *different* domain than its input" holds:

- **For idempotence**: `f(f(x))` requires `f`'s output type to match `f`'s input domain; `forScale:` declares the input is scale-domain, but the output (typed `Int`, semantically capacity-domain) isn't scale-domain.
- **For round-trip pairs with both sides labeled**: each side crosses domains; the round-trip property requires the reverse to undo the forward, but cross-domain functions form chains, not inverses.
- **For inverse-pair (no current candidates, but defensive)**: if a future pair like `x.foo(forScale:) ↔ y.bar(forCapacity:)` arises, the same logic suppresses.

This is the **first cycle to ship a single mechanism applied to three templates simultaneously**, paralleling the direction-counter family's three-cycle cadence (V1.10.1 → V1.11.1 → V1.12.1) but compressed into a single release. The compression is justified because:
1. The mechanism is uniform across templates (same curated set, same weight, same first-param-label check).
2. The cycles 7-9 pattern is now established — three template integrations are mechanical given the v1.13 hoist precedent.
3. The cycle-11 findings deferred this from cycle-9 because the mechanism class is distinct (semantic-intent vs sequence-iteration); now that the new class is identified, deploying across three templates in one release is the natural shape.

## Open decisions

1. **Counter weight: -15 (this plan's default) vs per-template tuning.** Two choices:
   - (a) **Uniform -15 across all three templates** (this plan's default). Rationale: matches the v1.10/v1.12 direction-counter weight precisely; produces clean Suppressed for round-trip/idempotence (`+30 - 15 = +15`) and for inverse-pair (`+25 - 15 = +10`); calibrated on the most established mechanism's weight.
   - (b) Per-template tuning (-15 round-trip/idempotence, -10 inverse-pair to match V1.11.1's curated direction-counter weight). Rationale: matches each template's existing direction-counter weight verbatim; fewer cross-template surprises if the curated direction set ever expands.
   - (c) Stronger -25 across all three. Rationale: deeper Suppressed margin; eliminates the inverse-pair `+25 + 10 - 25 = +10` Suppressed boundary case.

   **Default proposal: (a) uniform -15.** Minimizes per-template surprise; matches the most-replicated weight in the calibration trajectory; the -10 difference for inverse-pair is small and the boundary case (`+25 + curated +10 - 15 = +20` Possible) is unlikely to fire in practice (curated names like `parse/format` don't coincide with HashTable domain labels).

2. **Detection mode: both-sides vs either-side for round-trip + inverse-pair.** Two choices:
   - (a) **Both-sides labeled** (this plan's default). Rationale: preserves the 1 asymmetric OC candidate (`_value(forBucketContents:) ↔ _bucketContents(for:)`) which may be a true-positive round-trip pair (encoding/decoding bucket contents); high-precision posture per PRD §3.5.
   - (b) Either-side labeled. Rationale: matches v1.10/v1.11/v1.12 direction-counter detection mode (either-side); 1 additional suppression at the cost of suppressing 1 likely true-positive; net suppression delta from -16 → -17.

   **Default proposal: (a) both-sides.** PRD §3.5 conservative posture; the asymmetric OC candidate's likely-true-positive status outweighs the marginal +1 suppression. The user can re-evaluate in the cycle-13 priority-list rotation if the asymmetric candidate turns out to be a false-positive on triage.

3. **Curated set scope: 3 elements (witnessed) vs broader pre-emptive.** Two choices:
   - (a) **3 witnessed elements: `{forScale, forCapacity, forBucketContents}`** (this plan's default). Rationale: matches the V1.10.1 introduction posture (curated set started small at 10 elements, all witnessed in cycle-7 calibration data); avoid speculative broadening that adds noise without empirical justification; the cycle-13 priority list can extend the set when new corpora surface candidates.
   - (b) Broader pre-emptive: add `forSlot`, `forIndex`, `forBucket`, `forKey`, `forValue`, `forIdentifier`, `forElement`, `forNode`. Rationale: HashTable internals are not unique — other corpora may have similar domain-marker labels; pre-emptive coverage saves a future calibration cycle.
   - (c) Allow-list of label-pair shapes (`forX ↔ forY`) rather than individual labels. Rationale: distinguishes "round-trip with same label both sides" (`forScale ↔ forScale`, possibly true-positive on certain shapes) from "cross-domain mismatch" (`forScale ↔ forCapacity`).

   **Default proposal: (a) 3 witnessed elements.** Empirical-justification posture; no false-positive risk from speculative additions; v1.16+ can extend.

4. **Whether to add a dedicated `Signal.Kind.domainMismatchLabel` case.** Two choices:
   - (a) **Reuse `Signal.Kind.directionLabel`** (this plan's default). Rationale: existing semantics — "first-parameter-label-based suppression on a curated set" — fit; the explainability detail string can fully describe the structural argument ("Pair shape: `forScale: ↔ forCapacity:` cross-domain conversion; not a round-trip"); `Signal.Kind` granularity isn't user-facing in the rendered output. Mirrors the cycle-7 → cycle-8 → cycle-9 reuse of `Signal.Kind.directionLabel`.
   - (b) Add a dedicated `Signal.Kind.domainMismatchLabel` case. Rationale: explicit class for explainability clarity; future template integrations (e.g., a hypothetical commutativity domain-marker arm) consume the right enum case.

   **Default proposal: (a) reuse.** Mirrors v1.14's open decision #4 reuse posture; the explainability text is the user-facing surface, not the enum case. Future cycles can split if classes diverge meaningfully.

5. **File split: 3 separate files vs 1 combined file vs inline.** Default proposal: **3 separate files** mirroring V1.10.1/V1.11.1/V1.12.1 + V1.14.1 file-split precedent. Each template's domain-marker counter lives in its own file (`Idempotence` / `RoundTrip` / `InversePair` `DomainMarkerCounter.swift`). Keeps each calibration mechanism's per-template integration self-contained for attribution clarity.

6. **Single-cycle delivery vs three-cycle cadence.** Default proposal: **single-cycle delivery** (V1.15.1 wires all three templates simultaneously). Rationale: mechanism is uniform (same curated set, weight, detection); cycles 7-9 already established the per-template integration shape; the v1.13 hoist precedent removes the cross-template-reach-during-introduction concern. Splitting into three cycles (v1.15 idempotence, v1.16 round-trip, v1.17 inverse-pair) would mirror cycles 7-9's cadence but add no value here. Defensive note: if V1.15.1 turns out architecturally unwieldy mid-implementation, a clean fallback is to ship V1.15.1a (idempotence + curated set) and defer round-trip + inverse-pair to V1.15.1b/c — but the plan defaults to single-shot.

7. **Whether to commit cycle-12 decisions data.** Default: **yes** — `docs/calibration-cycle-12-data/` mirroring cycles 7-9 + 11.

8. **Cycle numbering: v1.15 = cycle 12.** Contiguous with v1.14 = cycle 11; v1.13's hoist refactor was cycle 10 (no-op surface delta but a real cycle in the trajectory chart).

## v1.15 trajectory framing

Cycle 7 (v1.10) introduced direction-label counter on `IdempotenceTemplate`.
Cycle 8 (v1.11) replicated on `InversePairTemplate`.
Cycle 9 (v1.12) completed the family on `RoundTripTemplate`.
Cycle 10 (v1.13) hoisted the abstraction.
Cycle 11 (v1.14) introduced function-name + type-shape composite mechanism.
Cycle 12 (v1.15, this release) ships the **first single mechanism applied to three templates simultaneously**.

Mechanism class: parameter-label counter (semantic-intent variant) — extends the cycles 7-9 family with non-directional curated labels for cross-domain conversions. Distinct from V1.10.1's spatial-sequence labels (`after`/`before`/`next`/...) by intent: domain markers describe *named domains* (scale, capacity, bucket-contents), not *positions in an ordered sequence*.

- Cycle-11 priority-#1 candidate set is fully accounted-for: 9 (or 10) round-trip + 7 idempotence OC suppressions.
