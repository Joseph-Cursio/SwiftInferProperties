# v1.7 Calibration Plan

Working doc for the SwiftInferProperties v1.7 cut. v1.7 is the **fourth calibration cycle** — closes cycle-4 priority #1 from `docs/calibration-cycle-3-findings.md` (curated stdlib-conformance bake-in) and runs the §17.3 loop a fourth time against the cycle-1+2+3 corpora. Mirrors the v1.5 + v1.6 plans' V1.X.0–V1.X.7 cadence with two parallel work streams: (a) code work for the bake-in, (b) empirical work re-running discover on the four corpora to measure suppression delta vs cycle-3's post-filter baseline.

> **v1.7 is a calibration cut, not a feature cut.** Like v1.5 and v1.6 it doesn't extend PRD §7.8 — it extends V1.5.2's coverage veto's *reach* by baking curated stdlib types' conformances into `inheritedTypesIndex(...)`'s output. The mechanism is the *complementary* mechanism to v1.5's coverage veto (which was textual-only and missed stdlib-typed candidates) and v1.6's pair-formation filter (which targeted identity-element noise specifically). Combined with v1.5 + v1.6, suggestions on stdlib-typed (`Int`, `Double`, etc.) carriers in OrderedCollections / Algorithms / PropertyLawKit should now reach the coverage veto.

**Ephemeral** — archive to `docs/archive/v1.7 Calibration Plan.md` once the tag is pushed and the §13 + §16 release-blocking criteria still hold.

## What v1.7 ships

Two deliverables:

1. **Curated stdlib-conformance bake-in on `ProtocolCoverageMap.inheritedTypesIndex(...)`.** New `ProtocolCoverageMap.stdlibConformances: [String: Set<String>]` static let — a curated table of stdlib types whose conformances are unconditional and well-known:
   - **Integer family:** `Int`, `Int8`, `Int16`, `Int32`, `Int64`, `UInt`, `UInt8`, `UInt16`, `UInt32`, `UInt64`. Conformances: `BinaryInteger`, `FixedWidthInteger`, `Numeric`, `AdditiveArithmetic`, `Comparable`, `Hashable`, `Codable`, `Equatable`. Signed integers add `SignedNumeric`; unsigned add `UnsignedInteger`.
   - **Floating-point family:** `Float`, `Double`. Conformances: `BinaryFloatingPoint`, `FloatingPoint`, `Numeric`, `AdditiveArithmetic`, `SignedNumeric`, `Comparable`, `Hashable`, `Codable`, `Equatable`.
   - **Bool, String:** `Bool` → `Equatable`, `Hashable`, `Codable`. `String` → `Comparable`, `Hashable`, `Codable`, `Equatable`.

   `inheritedTypesIndex(from:)` is extended to fold these stdlib entries into the result alongside the corpus typeDecls, so a corpus `extension Int: SomeProto` *unions* with the curated set rather than replacing it. Per-key set-union semantics preserved.

   Mirrors `EquatableResolver.curatedEquatableStdlib`'s posture: a hand-curated set of stdlib types whose conformances are both stable Swift idioms and unconditional (no generic-parameter conditioning). Expressly excluded: `Optional<T>`, `Array<T>`, `Set<T>`, `Dictionary<K,V>`, tuples — all conditional on element/key types and a v1.1 constraint-engine concern.

2. **Cycle-4 calibration run + findings writeup.** Re-run `swift-infer discover --include-possible` against the four cycle-1+2+3 corpora at the v1.7 commit; capture post-bake-in snapshots to `docs/calibration-cycle-4-data/`. Compare suppression deltas vs the cycle-3 post-filter baseline (350 total `--include-possible` hits across the four corpora). Findings doc at `docs/calibration-cycle-4-findings.md` documents per-corpus suppression counts + the headline-effect demonstration on the three corpora that showed 0-delta in cycles 2-3 (OrderedCollections / Algorithms / PropertyLawKit) + cycle-5 hypotheses.

PRD §19 sets the long-term acceptance-rate target at ≥ 70% after 6 months of dogfooding; v1.7 is cycle 4 of that loop.

### v1.7-only mechanical changes the cycle didn't cover

- **No new `Signal.Kind` additions, no new `KnownProperty` cases.** v1.7's mechanism is a curated-data extension of an existing helper. The v1.5 catalog stays exactly as-is.
- **`inheritedTypesIndex(from:)` API contract is unchanged.** Same signature, same return type, same caller wiring at `SwiftInferTemplates.SwiftInferTemplates.swift:79`. The merge-in is purely additive: existing keys union with stdlib entries; new keys (Int, Double, etc.) appear in the result whenever the corpus contains code referencing those types.
- **`coverageVetoSignal(...)` lookup behavior is unchanged.** It still does `inheritedTypesByName[strippingGenericParameters(typeText)]`. The bake-in just makes more keys hit.

## Out of scope

- **Cycle-4 priority #3 — approximate-equality template arm for FP types.** Real `KitFloatingPointTemplate` emitting `checkFloatingPointPropertyLaws(for: T.self, using: gen)` stubs. Sequenced as a v1.8 candidate. Synergy with v1.7: once the bake-in suppresses Numeric-covered FP suggestions, FP-conforming-but-not-kit-checked types are the natural target.
- **Cycle-4 priority #4 — Possible-tier sampling on the post-v1.7 surface.** Empirical-only follow-up. Best run after V1.7.2 captures the post-bake-in baseline.
- **Cycle-4 priority #5 — `surfacedAt` plumbing.** Independent of the bake-in. Unblocks PRD §17.2's time-to-adoption metric. Sequenced as a v1.8 mechanical addition.
- **Conditional conformance modelling.** `Array<T>: Equatable where T: Equatable` is intentionally out of scope; the textual coverage scan can't tell whether a generic argument satisfies a constraint without semantic resolution. Mirrors `EquatableResolver`'s same v1 limitation. v1.1 constraint-engine territory (PRD §20.2).
- **Module-prefixed protocol names.** `: Swift.Numeric` written out fully won't match the bare `"Numeric"` key in `protocolCoverage`. Cycle-2 findings already noted this; deferred until SemanticIndex reframes the table as a constraint-engine consumer.
- **User-defined protocols inheriting from a curated stdlib type.** `MyAlgebra: Numeric` — `Int` having `Numeric` in its bake-in means `let x: Int` resolves the way it should, but `let x: MyType` where `MyType: MyAlgebra` doesn't get reach via the bake-in (the textual scan would still need `MyType` to also list `Numeric` in its inheritance clause). Same v1 limitation as v1.5; documented in `ProtocolCoverageMap`'s type-level docs.
- **PRD §20 v1.1+ trajectory items.** SemanticIndex, IDE integration, `swift-infer apply` stay deferred. v1.7 closes cycle-4 priority #1 only.
- **Kit-side coordination.** `Package.swift` stays at `from: "2.0.0"`. v1.7 doesn't touch the kit.

## Open decisions

1. **Where the curated table lives.** Two choices: (a) `ProtocolCoverageMap.stdlibConformances` static let (this plan) — co-located with `protocolCoverage` because both are curated tables consumed by the same helpers; (b) new `StdlibConformances.swift` file. Default proposal: **(a) `ProtocolCoverageMap.stdlibConformances`.** Rationale: the table is the textual-protocol-name *publisher* side of `protocolCoverage`'s consumer side. Co-locating documents that the two are coupled (adding a key to one requires considering the other). Splitting adds an import hop without separating concerns.

2. **Whether to include `Float80` / `Float16`.** Two choices: (a) skip — platform-conditional (Float80 is x86_64-only; Float16 is ARM-only on Apple, 5.4+); (b) include with a comment. Default proposal: **(a) skip.** Rationale: textual-match doesn't care about platform availability, but adding platform-conditional types makes the curated table harder to reason about. Most user code on these types is rare; defer to cycle-5 if a corpus example surfaces.

3. **Whether to merge stdlib entries into `inheritedTypesIndex`'s output by default.** Two choices: (a) always merge (this plan) — the helper's name says "inherited types"; stdlib types' textual conformances are part of that scope; (b) gate behind a parameter `includeStdlib: Bool = true`. Default proposal: **(a) always merge.** Rationale: callers want suppression on stdlib types; the merge is cheap (~14 entries); back-compat-breakage is minimal because the only other downstream behavior is `coverageVetoSignal(...)`'s lookup, which already returns `nil` cleanly when a key isn't in the table.

4. **Whether to commit cycle-4 decisions data.** Two choices: (a) commit `docs/calibration-cycle-4-data/` mirroring cycle-1+2+3; (b) keep cycle-4 data in working copies only. Default proposal: **(a) commit them.** Rationale: cycle-4 captures the *post-bake-in surface*. The surface diff vs cycle-3 is the empirical record. Files are small.

5. **Whether to bundle cycle-4 priority #5 (`surfacedAt` plumbing) into v1.7.** Two choices: (a) ship priority #1 only (this plan); (b) bundle priorities #1 + #5. Default proposal: **(a) priority #1 only.** Rationale: same "one rule per release" precedent as v1.5 + v1.6. Priority #5 is a half-day orthogonal addition; saving it for v1.8 keeps v1.7's empirical effect attributable to one mechanism. (Cycle-4 priorities #2, #6, #7 already shipped in v1.6.1 patch series, so v1.7 only has #1 as the structural deliverable; bundling #5 would muddy the calibration attribution.)

6. **Re-run cadence after the bake-in lands.** Two choices: (a) capture only post-bake-in snapshots (cycle-4 has post-filter-of-cycle-3 + post-bake-in-of-cycle-4); (b) re-capture pre-bake-in-of-cycle-4 baselines too. Default proposal: **(a) post-bake-in only.** Rationale: cycle-3's post-filter snapshots are already committed and serve as the cycle-4 pre-bake-in baseline. Re-running pre-bake-in would just re-capture the same numbers.

7. **Which file the new tests live in.** Two choices: (a) extend the existing `Tests/SwiftInferCoreTests/ProtocolCoverageMapTests.swift` — natural home given the function lives on `ProtocolCoverageMap`; (b) new `ProtocolCoverageMapStdlibBakeInTests.swift` — bake-in-shaped concerns deserve their own file. Default proposal: **(a) extend existing file if it stays under 400 lines; otherwise (b).** Following V1.6.1's IdentityElementPairingFilterTests precedent, split when over.

## v1.7 trajectory framing

PRD v1.0 §17.3 names calibration as "the operationalization of high precision, low recall — without it, the philosophy is just a slogan." v1.7 is cycle 4 of that loop. After v1.7 ships:

- The three complementary algebraic-noise rules are in-tree: v1.5's coverage veto (textual, kit-covered) + v1.6's pair-formation filter (structural, identity-element only) + v1.7's bake-in (textual reach extension to stdlib carriers). Cycle-5 can extend any without a refactor.
- The cycle-4 post-bake-in snapshots in `docs/calibration-cycle-4-data/` are the diff target for cycle 5.
- The cycle-4 findings doc captures the v1.5 + v1.7 composition demonstration on the three previously-0-delta corpora (OrderedCollections / Algorithms / PropertyLawKit).

The natural cadence stays at one cycle per release. Cycle 5 (~v1.8) likely picks up cycle-4 priority #3 (approximate-equality FP template arm) + #4 (Possible-tier sampling triage) + #5 (`surfacedAt` plumbing) as paired mechanical additions. After two more cycles, the §19 acceptance-rate target should be measurable on the meaningfully-narrowed surface.

After the calibration loop reaches steady state, the natural next contributions return to the §20 v1.1+ trajectory:

- **PRD §20 SemanticIndex** — biggest v1.x lift. The cycle-4 stdlib-conformance bake-in is the textual approximation of one slice of what SemanticIndex would deliver authoritatively (via `lookupConformance`); cycle-5+ may re-frame the curated stdlib table as a `SemanticIndex` consumer once that engine ships.
- **PRD §20 IDE integration** — depends on SemanticIndex.
- **PRD §20 `swift-infer apply`** — moves from "all output is opt-in via `--interactive`" to "single command applies a baseline set of accepted suggestions."
- **Multi-predicate equivalence classes** (M13 axis 3) — SemanticIndex-blocked.
- **Cross-target enum coverage** (M14 deferred bit) — SemanticIndex-blocked.
- **Cross-test data-flow correlation** (M10 + M16 deferred) — natural sequencing post-SemanticIndex.

No active milestone plan opens with the v1.7 commit. Subsequent work picks one of the above and opens a fresh `docs/<plan>.md`.
