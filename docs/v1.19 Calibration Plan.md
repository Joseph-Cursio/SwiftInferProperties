# v1.19 Calibration Plan — Workstream B (mutating-method lift)

**Status:** Draft / proposal — pre-implementation, framing only.
**Date:** 2026-05-09.
**Origin:** Workstream B from `docs/archive/v1.18 Calibration Plan.md` §2; ChatGPT conversation `_3_Mutating API to Property Tests.md` step 5 onwards.

Working doc for the SwiftInferProperties v1.19 cut. v1.19 is the **sixteenth calibration cycle** and the **single-workstream follow-on to v1.18** that ships the largest behavioral change since M8.5: re-admitting the entire `mutating func` surface to the algebraic-property scoring pipeline via a metadata-only "lift" transformation. v1.20 will be the empirical-only measurement cycle that re-runs the harness against the post-v1.19 surface (cycle-15 + cycle-16 mechanisms together) and reports per-template + per-corpus acceptance-rate movement vs cycle-6 (26.7%) and cycle-14 (34.8%).

> v1.19 is a **single-workstream mechanism cycle** (Workstream B from the v1.18 plan). v1.20 ships the empirical-only re-measurement; v1.21+ rotates to the cycle-16 carry-forward priorities (math-library forward-function counter, fixed-point-name positive signal on idempotence) once the post-v1.19 acceptance-rate data is in.

**Ephemeral** — archive to `docs/archive/v1.19 Calibration Plan.md` once the v1.19 tag is pushed.

## 1. What this cycle is

Every template in the v1.18 surface still gates on `!summary.isMutating` (`IdempotenceTemplate.swift:124`, `RoundTripTemplate` via `FunctionPairing.swift:69`, `AssociativityTemplate.swift:113`, `CommutativityTemplate.swift:119`, `MonotonicityTemplate.swift:127`, `IdentityElementPairing.swift:154`, `InverseElementPairing.swift:103`). Workstream A (v1.18) added the structural carrier-kind signal that is the **necessary precondition** for re-admitting `mutating func`s; Workstream C added the dual-style consistency template that pairs them with non-mutating siblings. Workstream B closes the loop: the entire mutating surface is re-evaluated against the algebraic templates by *lifting* each mutating method into a pure shadow form.

The lift is the load-bearing move from `_3_Mutating API to Property Tests.md` step 2:

> Treat a mutating method as if it were a pure function from `(oldValue, input)` → `newValue`.

```swift
// Source:
mutating func increment(by amount: Int) { value += amount }

// Lifted shadow (metadata only — no codegen, no source rewrite):
func increment(_ self: Counter, by amount: Int) -> Counter {
    var c = self
    c.increment(by: amount)
    return c
}
```

The lift is **sound only on value-semantic carriers** — without value semantics on `T`, the `var c = self` copy aliases shared state and the laws don't hold. v1.18's `valueSemanticCarrier` signal is the structural precondition; v1.19 consumes it as an admission gate.

**Why high-leverage:** every `mutating func` in stdlib (`Array.append`, `Array.removeAll`, `Set.formUnion`, `Set.removeAll`, `Dictionary.merge`, `String.removeAll`, `String.append`) and every domain mutating method gets evaluated for free. Conservative estimate from hand-counting `mutating func` declarations across the four cycle-1..14 corpora: **80–120 new candidates** entering the scoring pipeline. The biggest single-cycle surface expansion since M8.5.

## 2. What v1.19 ships

Five deliverables (mirroring v1.18's mechanism-cycle shape, minus the third workstream since v1.19 is single-workstream):

### Deliverable 1 — `LiftedTransformation` summary type

**Location:** `Sources/SwiftInferCore/LiftedTransformation.swift`.

**Shape:**

```swift
public struct LiftedTransformation: Sendable, Equatable {
    /// The original mutating method this lift was derived from. Always
    /// satisfies `originalSummary.isMutating == true` and
    /// `originalSummary.containingTypeName != nil`.
    public let originalSummary: FunctionSummary

    /// The carrier type (`originalSummary.containingTypeName`). Captured
    /// here for direct access without the optional unwrap.
    public let carrier: String

    /// The lift's pure-function shape — `(carrier, params...) -> carrier`.
    /// `liftedReturnType == carrier`; `liftedParameters[0].typeText ==
    /// carrier` (the `_ self:` binding); subsequent entries mirror
    /// `originalSummary.parameters` verbatim.
    public let liftedParameters: [Parameter]

    /// Always `carrier` — the lift returns the post-mutation value.
    public let liftedReturnType: String

    /// Why the templates can trust the lift. Renders as a single rationale
    /// line in the §4.5 explainability block:
    /// `"Lifted from `mutating func <Carrier>.<name>(...)`. Property holds
    /// iff `<Carrier>` has value semantics — `var copy = original;
    /// copy.<name>(...)` does not aliase original's state."`
    public let rationale: String
}
```

**Derivation:** new `LiftedTransformationBuilder` (or static method on `LiftedTransformation` itself — open decision #1) takes `[FunctionSummary]` + `CarrierKindResolver` and emits one `LiftedTransformation` per `(isMutating == true && containingTypeName != nil && CarrierKindResolver.classify(typeName: containingTypeName) == .valueSemantic)` candidate.

**Strict admission gate per v1.18 plan open decision #2 lean:** `valueSemanticCarrier` signal must be **positive** (i.e. `CarrierKind.valueSemantic`). `.mixed` and `.unknown` carriers do NOT admit the lift in v1.19. v1.20 measurement may relax to permissive (struct-only filter, demote downstream) if recall is too low; that's a v1.21 candidate.

### Deliverable 2 — Template fan-out

**Four template integration sites:**

#### 2a — `IdempotenceTemplate` — admits the lift in two shapes

| Original mutating shape | Lifted shape | Idempotence test |
|---|---|---|
| `mutating func op() -> Void` | `(T) -> T` | `op'(op'(s)) == op'(s)` for all `s: T` |
| `mutating func op(x: T)` (param type matches carrier) | `(T, T) -> T` (binary, but x-curried becomes `(T) -> T` for any fixed `x`) | `op'(op'(s, x), x) == op'(s, x)` for all `(s, x): (T, T)` |

Examples:
- **No-param case:** `Set.removeAll` lifts to `(Set<E>) -> Set<E>`; idempotent (`removeAll · removeAll == removeAll`).
- **Param-matches-carrier case:** `Set<E>.formUnion(_:Self)` lifts to `(Set<E>, Set<E>) -> Set<E>`; x-curried idempotence reads `s.formUnion(s.formUnion(x)).contains == s.formUnion(x).contains` for all `s, x` — **the canonical SetAlgebra idempotent-union law**.

Single-parameter case where `x: X` does NOT match the carrier (`mutating func increment(by: Int)` on `struct Counter`) is **not** an idempotence candidate by itself — `op'(op'(s, x), x) == op'(s, x)` doesn't hold for `increment` (it doubles the increment). That case is handled by the IdentityElement / Composition templates below.

**Score posture:** lifted suggestions inherit IdempotenceTemplate's existing scoring stack (type-symmetry +30, curated verb +40, self-composition signal, value-semantic carrier +5 from Workstream A) PLUS a new **`Signal.Kind.liftedFromMutation`** signal at `+10` rendering "Lifted from mutating method `<Carrier>.<name>(...)`". Totals match the non-lifted Strong-tier baseline modulo the +10 and the value-semantic carrier (which is required to admit the lift, so always fires).

#### 2b — `IdentityElementPairing` — admits the lift for "no-op param value" case

For `mutating func op(by: X)` (X ≠ T or X == T with curry-disqualified shape), look for an `X` value `e` such that `op'(s, e) == s` for all `s`. The ChatGPT example: `Counter.increment(by: 0)` — `incremented(c, by: 0) == c`.

Pairing source: existing `IdentityElementPairing` candidates extended to consider lifted transformations alongside non-mutating binary ops. The pairer scans for a same-corpus `IdentityCandidate` whose `typeText == X` (the increment-amount type), and the template fires the standard identity-element scoring stack with the `liftedFromMutation` `+10` boost.

**Score posture:** existing identity-element scoring (type-shape +30, identity-naming +40, optional empty-seed +20, value-semantic carrier +5) + lifted-from-mutation +10 = 85+ → Strong by construction on canonical numeric-identity cases.

#### 2c — NEW `CompositionTemplate`

For `mutating func op(by: X)` where X is in a curated additive-monoid set, check `op'(op'(s, a), b) == op'(s, a + b)`.

**Curated additive-monoid set (open decision #3 lean: numeric-only for v1.19):**
```
Int, Int8, Int16, Int32, Int64, UInt, UInt8, UInt16, UInt32, UInt64,
Float, Float16, Float32, Float64, Float80, Double, Decimal, CGFloat,
Duration
```

Generic-monoid extension (`combine` / `+` op identifying `T` as monoidal) is a v1.21+ candidate per v1.18 plan open decision #3.

**File:** new `Sources/SwiftInferTemplates/CompositionTemplate.swift`. Score signals:
- type-shape signal (`+30`): the lifted shape `(T, X) -> T` with X in curated additive set
- canonical naming signal (`+40`): the function name matches the curated additive-action verb list (`increment`, `add`, `advance`, `accumulate`, `accrue`, `step`, `extend`, `expand`, `shift`, `offset`, `bump`, ...) — open decision #4 covers the curated list
- value-semantic carrier signal (`+5` via Workstream A — required to admit the lift)
- lifted-from-mutation signal (`+10`)
- non-deterministic veto (standard)
- domain-marker counter / direction-label counter / FP-storage counter (carry forward from existing template scoring stacks)

Total: `30 + 40 + 5 + 10 = 85` → Strong by construction on canonical accumulate-style mutating methods.

**Property body:**
```swift
let original = s
var stepA = original
stepA.<op>(by: a)
var stepAB = stepA
stepAB.<op>(by: b)

var combined = original
combined.<op>(by: a + b)

return stepAB == combined
```

#### 2d — `InversePairTemplate` — admits dual-mutating add/remove pairs

For two lifted transformations `add(_:T)` and `remove(_:T)` on the same carrier, propose `add(remove(s, x), x) == s` and `remove(add(s, x), x) == s`.

Pairing source: new `InverseLiftedPairing` (parallel to `FunctionPairing` for non-mutating pairs, and `IdentityElementPairing` for binary-op + identity-constant pairs). Pairs lifted-mutating methods by name pairs (curated set: same as `RoundTripTemplate.curatedInversePairs` — `{insert/remove, push/pop, add/remove, ...}`) AND matching parameter types.

**Score posture:** inherits existing InversePair scoring (type-shape +25, curated naming +10, value-semantic carrier +5) + lifted-from-mutation +10 = 50 → Likely. With matching @Discoverable group: +35 lifts to 85 → Strong.

Re-measured against the v1.19 working copy. Expected delta: **≤+10% wall time** vs v1.18 (the v1.18 plan §6 v1.19 release-blocking criterion). The lift admission adds:

- One per-mutating-summary classification call (carrier-kind resolver lookup — already cached) + one `LiftedTransformation` construction. O(M) where M = mutating-func count per corpus.
- Per-template suggest call now also iterates the lifted transformations (additive). O(M) per template per discover.

Expected memory delta: ≤+5% (lifted transformations are small structs; M is bounded by the corpus's mutating-func count, typically < 100 for the cycle-1..14 corpora).

## 4. Open decisions

1. **`LiftedTransformation` derivation site.** Options:
   - (a) Eager — derived once during `TemplateRegistry.discover` alongside `EquatableResolver` and `inheritedTypesByName` and `CarrierKindResolver`. Threaded into `collectSuggestions` as `liftedTransformations: [LiftedTransformation]`.
   - (b) Lazy — each template that needs lifts derives them on-the-fly from `(summaries, carrierKindResolver)`.
   
   **Lean: (a) eager.** Mirrors the existing per-discover-resolver pattern; one allocation pass; downstream templates iterate a single shared vector. Avoids recomputing the carrier-kind classification per template.

2. **Lift conservatism (carry-forward from v1.18 plan #2).** Strict (require `valueSemanticCarrier` positive) for v1.19. **Confirmed lean:** strict; revisit at v1.20 if recall is too low.

3. **CompositionTemplate scope (carry-forward from v1.18 plan #3).** Numeric-only curated set (`AdditiveArithmetic` curated subset above). **Confirmed lean:** numeric-only for v1.19; monoid-shaped extension is a v1.21+ candidate after the numeric-only acceptance rate is measured.

4. **Curated additive-action verb list for CompositionTemplate naming signal.** v1.0 list:
   ```
   increment, add, accumulate, accrue, advance, step, extend, expand,
   shift, offset, bump, grow, augment, append, push, pop, deposit,
   withdraw
   ```
   Project extension via `Vocabulary.compositionVerbs: [String]` (mirrors `idempotenceVerbs`/`commutativityVerbs` schema). **Lean:** ship the list as drafted; project extension via `Vocabulary` slot. v1.20 measurement may prune false-positive-prone names (`pop` / `withdraw` are ambiguous on signed amounts).

5. **`Signal.Kind.liftedFromMutation` weight.** Options:
   - (a) `+10` — modest positive; rewards the structural soundness check that admitting the lift requires (since strict gate filters most noise).
   - (b) `+5` — same magnitude as `valueSemanticCarrier`; signals "this is a lifted form" without compounding the value-semantic positive.
   - (c) `+0` — informational only; the lift admission itself is the load-bearing structural check, scoring uses the inherited template signals only.
   
   **Lean: (a) +10.** The v1.18 plan §2 Workstream B suggestion-rendering specification carries this magnitude implicitly via the example total `30 + 40 + 5 + 10 = 85` for CompositionTemplate. Re-baseline at v1.20 if the lifted-suggestion acceptance rate is high enough that the +10 over-promotes.

6. **`InverseLiftedPairing` independence from `FunctionPairing`.** Options:
   - (a) New separate pairing pass over `[LiftedTransformation]` paralleling `FunctionPairing` — matches the v1.18 `DualStylePairing` pattern.
   - (b) Extend `FunctionPairing` to also consider lifted transformations.
   
   **Lean: (a) separate pass.** Cleaner separation; the pairing rules differ (curated mutating add/remove names vs the wider non-mutating curated inverse list); easier to reason about per-template surfaces in cycle-16 findings.

## 6. Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Lift admission generates suggestions on `mutating func`s that have unstated preconditions (e.g. `Array.append(_:)` only valid when `count < capacity`) | Medium | Suggestions ship with the explicit "requires value semantics on T" rationale; the kit's existing precondition-hint signal already gates these via `PreconditionHint`. |
| Strict gate (open decision #2 lean) over-filters — many real value-semantic struct carriers with stored properties whose types aren't in the curated allow-list end up classified as `.unknown` | Medium | The cycle-16 findings doc reports lift-admission counts per corpus + a diagnostic line "value-semantic struct carriers excluded due to unresolved member type". v1.20 empirical cycle informs whether to relax to permissive at v1.21. |
| CompositionTemplate fires on naming-pair coincidences where the additive law doesn't hold (e.g. `mutating func extend(by: Int)` for "extend the array by Int new elements" — not additive in the algebraic sense) | Medium | The naming verb list (open decision #4) is curated to favor algebraic-additive intent (`increment`, `accumulate`, `accrue`); `extend` is a known noisy example flagged in the v1.20 triage rubric. |
| `InverseLiftedPairing` over-pairs: e.g. corpus has `mutating func unsafePush` + `mutating func popUnchecked` that satisfy the curated `push/pop` rule but have semantic divergence | Low | Explainability caveat names the pair explicitly + a per-pair @Discoverable hint can override; expected-rare in stdlib + the cycle-1..14 corpora. |
| Perf budget exceeded on the cycle-1..14 corpora | Low-Medium | Lift derivation is single-pass O(M) over mutating funcs; per-template iteration adds O(M) per discover. Memory cost is bounded by lifted-transformation struct size × M. |
| Double-counting: a `mutating func formUnion(_:Self)` is admitted by both `IdempotenceTemplate` (param-matches-carrier curry case) AND `DualStyleConsistencyTemplate` from v1.18 (formUnion/union pair) | Medium | Both templates emit; the cross-validation key already deduplicates suggestion identity per template. The two suggestions express different properties (`formUnion` is x-curried-idempotent vs. `formUnion` and `union` agree) so dual emission is correct — but the cycle-16 findings doc should explicitly call this out as expected. |

## 7. What this plan is *not*

- **Not** a kit-side change. SwiftPropertyLaws stays at `from: "2.0.0"`. Future v1.21+ M-VS-2/3/4 milestones from `docs/ideas/ValueSemantic Kit Proposal.md` remain deferred.
- **Not** a TestLifter change. M-VS-4 from the kit proposal stays out of scope.
- **Not** a SemanticIndex prerequisite. The textual approximations Workstream B uses (curated value-type allow-list, curated additive-monoid set, curated additive-action verb list) are good enough for the cycle-1..14 corpora; SemanticIndex (PRD §20.1) would tighten them but isn't blocking.
- **Not** an IDE-integration change. PRD §20 v1.1+ work stays deferred.
- **Not** a stateful-mutation / sequenced-`[Op]` template (ChatGPT `_3` step 6). That's workstream D in v1.21+ once Workstream B is in production and the kit's `Gen<[Op]>` story matures.
