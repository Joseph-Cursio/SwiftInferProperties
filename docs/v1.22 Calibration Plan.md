## 2. The four workstreams

Each workstream is independently mergeable. v1.22 ships A + B + C + D in four focused commits; sub-commit ordering is small-to-large by code complexity (B then C then A then D — B and C are tiny extensions, A is the new positive signal, D is the most code).

### Workstream A — Fixed-point-name positive signal on non-lifted idempotence

**What:** New `+10` curated-name signal on `IdempotenceTemplate.suggest(for:)` that fires when `summary.name ∈ FixedPointNames.curated`. Curated set:

```
{normalize, canonicalize, dedupe, simplify, flatten, clamp,
 truncate, sanitize, standardize}
```

**Why these names:** all signal "transform input into a canonical form whose application is idempotent" (calling twice yields the same result as once). The curated list is intentionally small + lexically distinctive; project extension via the existing `Vocabulary.idempotenceVerbs` slot already covers project-specific names.

**Score posture:** signal fires at `+10`. Composes with the existing curated-verb signal (+40) — but the curated-verb list ALREADY contains some of these names per `IdempotenceTemplate.curatedVerbs` (V1.4.1 list). Per V1.4.1 design, the curated-verb signal's detail says "Curated idempotent verb"; this new signal's detail says "Curated fixed-point name". **Open decision #1 below** explores whether to merge into the existing list (V1.21.B-style: extend existing curated set without new signal kind) vs add as a separate curated set with new detail string.

**Mechanism class:** new sub-class of class 7 (function-name + type-shape composite) — a positive variant. Or alternatively a recall-positive extension of class 6 (parameter-label semantic-intent). Open decision #2.

**Implementation site:** new file `Sources/SwiftInferTemplates/IdempotenceTemplate+FixedPointNames.swift` extending `IdempotenceTemplate` with `fixedPointNameSignal(for:)` static helper. Wired into `IdempotenceTemplate.suggest(for:)` alongside existing `nameSignal(for:vocabulary:)`. Curated set in new `Sources/SwiftInferCore/FixedPointNames.swift`.

**Cost:** ~half day. New core file ~30 LOC, new template extension ~50 LOC, ~10 unit tests.

**Plan-vs-actual projection:** **introduces ~5-10 new accepts** on the v1.21 surface — recall-positive movement. Cycle-19 picks status: 1 OC `_description(type:)` formatter (cycle-17 #16 reject) is NOT a curated fixed-point-name and stays surfaced (good); names like `normalize`, `canonicalize`, `dedupe` would surface NEW accepts where they currently don't fire because the curated-verb list misses them.

### Workstream B — BucketIterator name extension on V1.21.A

**What:** Extend V1.21.A's name fallback rule to also match carriers whose name has the `Iterator` suffix without requiring the `.Iterator` (dot-prefixed) form. The cycle-18 finding identified 3 surviving OC `_HashTable.BucketIterator` picks (`advance()`, `findNext()`, `advanceToNextUnoccupiedBucket()`) — carrier ends in `.BucketIterator`, the V1.21.A rule checks `*.Iterator` suffix only.

**Implementation:** modify `IdempotenceTemplate.iteratorProtocolCarrierVeto(for:inheritedTypesByName:)`'s name-fallback condition from:

```swift
let isIteratorNamed = carrier == "Iterator" || carrier.hasSuffix(".Iterator")
```

to:

```swift
let isIteratorNamed = carrier == "Iterator"
    || carrier.hasSuffix(".Iterator")
    || carrier.hasSuffix("Iterator")  // captures BucketIterator, MyArrayIterator, etc.
```

AND extend the curated `iteratorMethodNames` set with `findNext` + `advanceToNextUnoccupiedBucket` (the two non-curated method names cycle-18 identified):

```swift
static let iteratorMethodNames: Set<String> = [
    "next", "advance", "nextState", "step",
    "findNext", "advanceToNextUnoccupiedBucket"
]
```

**Cost:** ~1 hour. Two-line file edit + ~5 unit tests covering the new fallback shapes (BucketIterator + non-curated method names).

**Open decision #3:** whether the carrier-name extension is too aggressive. A user-defined `MyArrayIterator` that doesn't actually advance state (e.g., a wrapper that exposes a snapshot) would be falsely vetoed. **Lean: ship the extension with the curated method-name JOINT match retained** — `MyArrayIterator` only vetoes if the method name is also in `iteratorMethodNames`, which conservatively limits the false-positive surface.

**Plan-vs-actual projection:** **closes 3 OC candidates** (`BucketIterator.advance()`, `BucketIterator.findNext()`, `BucketIterator.advanceToNextUnoccupiedBucket()`).

### Workstream C — OC `index(after:) × index(before:)` direction-pair full-veto extension on V1.12.1

**What:** Extend V1.12.1's direction-label counter (currently fires at `-15` when **either** pair side has a direction-labeled first parameter) to fire at `-25` (full veto magnitude) when **both** pair sides have direction-labeled parameters. The cycle-18 finding identified 17 OC round-trip survivors at v1.21, mostly `index(after:) × index(before:)` cross-cell pairs across `OrderedDictionary.Elements.SubSequence`, `OrderedDictionary.Elements`, `OrderedDictionary.Values`, `OrderedSet+RandomAccessCollection`, `OrderedSet+SubSequence`. V1.18.A carrier-kind `+5` lifted these above visibility post-v1.18.

**Implementation:** modify `RoundTripTemplate.directionLabelCounterSignal(for:)` (V1.12.1) to detect both-sides direction labels and emit at `-25` instead of `-15`. The current rule:

```swift
let forwardLabel = pair.forward.parameters.first?.label
let reverseLabel = pair.reverse.parameters.first?.label
let forwardIsDirectional = forwardLabel.map { DirectionLabels.curated.contains($0) } ?? false
let reverseIsDirectional = reverseLabel.map { DirectionLabels.curated.contains($0) } ?? false
guard forwardIsDirectional || reverseIsDirectional else { return nil }
return Signal(kind: .directionLabel, weight: -15, ...)
```

becomes:

```swift
guard forwardIsDirectional || reverseIsDirectional else { return nil }
let weight = (forwardIsDirectional && reverseIsDirectional) ? -25 : -15
let detail = (forwardIsDirectional && reverseIsDirectional)
    ? "Both pair sides direction-labeled — paired index advance, not a round-trip codec"
    : "Direction-labeled parameter on \(forwardIsDirectional ? "forward" : "reverse") side"
return Signal(kind: .directionLabel, weight: weight, detail: detail)
```

**Score posture:** `index(after:) × index(before:)` round-trip score becomes 30 type-symmetry + 5 carrier - 25 direction-counter = 10 → Suppressed. Closes the candidate.

**Mechanism class:** extension of existing class 6 (parameter-label direction-counter). Magnitude bump from -15 to -25 in the both-sides case; new detail string distinguishes the two firing modes.

**Cost:** ~1-2 hours. Single-method change + ~5 unit tests covering both-sides veto, single-side preserved at -15, no-sides preserved at no-fire.

**Plan-vs-actual projection:** **closes ~12 OC candidates** (the `index(after:) × index(before:)` cross-cell pairs identified in cycle-18 findings).

### Workstream D — Stride-style label extension

**What:** Suppress the lone Algo `endOfChunk(startingAt:) ↔ startOfChunk(endingAt:)` round-trip + inverse-pair + idempotence triple via a stride-style label counter on round-trip + inverse-pair templates. Cycle-14 demotion target (cycle-14 priority #1 was demoted to v1.18; not shipped in cycles 15-18; cycle-18 priority #6).

**Why:** the suppression target is **emission usability** (auto-emitted property tests need chunk-boundary generators which don't fit standard `Gen<Int>` template), not correctness — cycle-14 #19 / #29 / #49 + cycle-17 #15 / #32 all measured **accept** verdicts on this triple. The cycle-15+ rationale for ongoing non-suppression has been "correctness-positive picks shouldn't suppress just for usability"; cycle-18's accumulated user-experience signal is now sufficient to demote them out of Strong-tier visibility (where they'd otherwise become emit-target candidates).

**Implementation:** new `StrideStyleLabels.curated` in `Sources/SwiftInferCore/`:

```
{startingAt, endingAt, fromIndex, toIndex, startingFrom, endingAt:fromOffset:}
```

Consumed by:

- `RoundTripTemplate.strideStyleLabelCounter(for:)` — fires at `-25` when **both** pair sides have first-parameter labels in the curated set. Mirror's V1.12.1's both-sides posture.
- `InversePairTemplate.strideStyleLabelCounter(for:)` — same, but on inverse-pair.

**Score posture:** `endOfChunk(startingAt:) × startOfChunk(endingAt:)` round-trip + inverse-pair scores both demote 30+5 = 35 + name signal → 30+5+10-25 = 20 → Suppressed. Closes both round-trip and inverse-pair on the same Algo function pair.

**Out-of-scope refinement:** the v1.22 stride-style extension does NOT touch the lone Algo `endOfChunk(startingAt:)` non-lifted idempotence pick (cycle-14 #29 also accept). That's a separate fire on the single-function template, which doesn't pair with `startingAt:` ↔ `endingAt:` directly — would need its own gate. **Defer to v1.23+ if cycle-19 measurement shows the lone Algo idempotence pick still surfacing.**

**Cost:** ~half day. New core file ~25 LOC, two template extensions ~80 LOC each, ~10 unit tests.

**Open decision #4:** whether to also extend to the lone Algo idempotence pick. **Lean: defer** (v1.22 closes round-trip + inverse-pair only; idempotence remains surfaced; cycle-19 measurement on a fresh sample triages whether the surviving idempotence pick warrants its own mechanism).

**Plan-vs-actual projection:** **closes 2 candidates** (1 Algo round-trip + 1 Algo inverse-pair, on the same function pair).

## 5. Open decisions

1. **Workstream A — extend curated verbs (V1.4.1 list) vs separate curated set.** Default proposal: **(b) separate curated set + new signal kind.** Rationale: the V1.4.1 curated-verb list is `+40`; fixed-point names should be `+10` (smaller signal, additional detail). Mixing magnitudes in the same curated set complicates the rendered detail line. Trade-off: two curated sets to maintain vs one. Acceptable.

2. **Workstream A — new class 14 vs extend class 7.** Default proposal: **(a) new class 14.** Rationale: this is the FIRST positive (recall-positive) signal in the post-V1.4.3 era; previous mechanism cycles have all been precision-positive (suppression-only). Documenting this taxonomic shift is itself the value of the new class. Trade-off: taxonomy size grows.

3. **Workstream B — extend `hasSuffix("Iterator")` carrier rule.** Default proposal: **(a) extend, with the curated method-name JOINT match retained** (per the workstream description above). Rationale: the joint match (`isIteratorNamed && iteratorMethodNames.contains(methodName)`) preserves the false-positive guardrail; the extension only catches `BucketIterator`-style carriers that have a curated Iterator-pattern method name.

4. **Workstream D — stride-style extension to lone Algo idempotence pick.** Default proposal: **(b) defer to v1.23+.** Rationale: v1.22 closes round-trip + inverse-pair on the Algo `endOfChunk × startOfChunk` pair (-2 candidates); the lone idempotence pick at the same site is structurally separate (different template; doesn't pair with `endingAt:`). Cycle-19 measurement on a fresh sample triages whether the surviving pick warrants its own mechanism.

5. **Whether v1.22 should re-measure perf at V1.22.E.** Default proposal: **(a) yes, re-measure.** v1.22 ships four Sources/ changes; mirrors v1.21 plan §"Open decisions" #6. Expected delta vs v1.21 baseline: ≤+5% wall time (the new vetoes + positive signal are O(1) hash-set lookups per template-suggest call; the workstream A positive signal is recall-positive which doesn't short-circuit the suggestion pipeline, so the wall delta is at the additive-cost end of the band).

6. **Whether v1.22 should re-run the cycle-1..14 corpora at V1.22.E.** Default proposal: **(a) yes, re-run all four** (mirrors V1.21.D's plan-vs-actual posture). Captures the ~17-22-candidate closure (~3 + 12 + 2 + 0 to -10 from workstream A's recall-positive movement) into `docs/calibration-cycle-19-data/`. Cycle-20 (v1.23 empirical re-measurement) inherits the v1.22 surface for its sampling.

7. **Cycle numbering: cycle 19 = v1.22.** Contiguous with v1.21 = cycle 18, v1.20 = cycle 17 (empirical), v1.19 = cycle 16, v1.18 = cycle 15.
