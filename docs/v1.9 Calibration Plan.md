# v1.9 Calibration Plan

Working doc for the SwiftInferProperties v1.9 cut. v1.9 is the **sixth calibration cycle** — closes cycle-6 priority #1 from `docs/calibration-cycle-5-findings.md` (Possible-tier sampling on the 349-surface) and runs the §17.3 loop a sixth time. **Empirical-only release** — no Sources/ changes, no new structural rules; the headline deliverable is per-template acceptance-rate data computed from a stratified ~50-decision triage sample.

> **v1.9 is the calibration cycle that has been deferred since cycle 2.** Cycles 1-5 each shipped one structural rule; cycle 6 is the empirical loop-closing round. The 349-surface (post-V1.8.1) no longer contains a known false-suppression bug, so the per-template Possible-tier acceptance rates this cycle produces are *trustworthy* signal for cycle-7+ scoring weight tuning.

**Ephemeral** — archive to `docs/archive/v1.9 Calibration Plan.md` once the tag is pushed and the §13 + §16 release-blocking criteria still hold.

## Why this cycle is different

Cycles 1-5 each shipped one structural mechanism. Cycle 6 ships zero structural mechanisms — the deliverable is a triage rubric + sample + findings.

The cycles-1-5 trajectory established that:
- **Surface-reduction mechanisms work as designed.** V1.4.3's counter-signals, V1.5.2's coverage veto, V1.6.1's pair-formation filter, V1.7.1's stdlib bake-in, and V1.8.1's shape gate each demonstrably suppressed a specific cause-of-noise class with attribution-clean empirical evidence.
- **The remaining 349-surface is genuinely uncertain.** Each prior cycle's findings doc deferred the empirical question — "are these 349 actually correct property suggestions?" — to a future Possible-tier sampling pass. Without that pass, scoring weight tuning is conjecture.

V1.8.1's V1.5.2-narrowing in particular handed back 23 user-defined inverse pairs that were previously suppressed. Those 23 are a high-priority sample because we now have the empirical question: "should the round-trip template surface them at Possible tier, or is something deeper going on with how round-trip is scored?"

## What v1.9 ships

Three deliverables. None are code:

1. **A triage rubric.** New `docs/cycle-6-triage-rubric.md` defines accept/reject/suppress criteria per template. Mirrors PRD §4 + §5 contracts. Per-template definitions of what "the property holds" means semantically (e.g., for round-trip: `g(f(x)) == x` for all `x` in the function's effective domain), what counts as "ship it" vs "false positive," and how to handle ambiguity (flag `unknown` rather than forcing a binary call).

2. **A stratified ~50-decision triage sample.** Sample selection committed to `docs/calibration-cycle-6-data/sample-manifest.md` listing the 50 (suggestion-identity, source-location, decision, rationale) tuples. Stratified across:
   - Templates: round-trip (~15) / idempotence (~12) / commutativity+associativity (~8) / monotonicity (~8) / inverse-pair (~5) / identity-element (~2)
   - Corpora: OrderedCollections / ComplexModule / Algorithms / PropertyLawKit
   - Score tiers: all Possible (Score ~30) — Strong-tier triage was cycle-1's scope and isn't relevant here

3. **A findings doc with per-template acceptance rates + scoring tuning recommendations.** New `docs/calibration-cycle-6-findings.md` documents: per-template / per-corpus accept-vs-reject counts, the patterns that emerged (e.g., "round-trip on `(T) -> T` user-inverse pairs accepted at X% — suggests Score 30 is calibrated correctly" or "commutativity on user-named ops rejected at Y% — suggests scoring tightening"), explicit single-runner-triage caveat (one rater, public-API only, no test runs), and the cycle-7 priority list driven by what the rates reveal.

PRD §19 sets the long-term acceptance-rate target at ≥ 70% after 6 months of dogfooding; v1.9 is cycle 6 of that loop and the first cycle to *measure* the rate empirically rather than infer it from suppression deltas.

### v1.9-only mechanical changes the cycle didn't cover

- **No code changes.** `swift-infer --version` will report `1.9.0` after V1.9.5's bump but the binary's behavior is byte-identical to v1.8.0. Tests / lint / perf budgets all carry forward unchanged.
- **No new templates, no new signals, no new vetoes.** v1.9's mechanism is *measurement*.
- **No corpus changes.** Cycle-6 reuses cycle-5's post-tightening snapshots as the input — every triaged suggestion is at the v1.8.0 commit.

## Open decisions

1. **Sample size.** Two choices: (a) ~50 decisions across all 6 templates × 4 corpora (this plan); (b) ~100 decisions for tighter per-cell rates. Default proposal: **(a) 50.** Rationale: single-runner triage scales linearly with rater fatigue; 50 is the largest size that maintains decision quality from one rater in one session. Future multi-rater cycles can stratify-replicate.

2. **Sample stratification.** Two choices: (a) stratify by (template × corpus) cells (this plan); (b) random uniform. Default proposal: **(a) stratify.** Rationale: random uniform produces ~70% round-trip cells (because round-trip is 187/349 of the surface); stratified ensures each template gets enough samples to compute a rate. Per-cell minimum is 1, per-template minimum is 5.

3. **Where the rubric lives.** Two choices: (a) `docs/cycle-6-triage-rubric.md` (this plan); (b) merge into `docs/calibration-cycle-6-findings.md`. Default proposal: **(a) separate file.** Rationale: the rubric is a methodology artifact reusable for cycle-7+ triage runs; the findings doc is cycle-specific. Separating documents the methodology / data distinction.

4. **Decisions JSON schema.** Two choices: (a) match `.swiftinfer/decisions.json` shape exactly so cycle-6 decisions could in principle be replayed against the v1.8 binary (this plan); (b) custom schema optimized for triage analysis. Default proposal: **(a) match the shape.** Rationale: forensic reusability — future Decisions-replay scenarios can include cycle-6 data without translation. Per-decision rationale fields can be additive (the existing schema has `note` already).

5. **What "acceptance rate" means in cycle 6.** Two choices: (a) accept / (accept + reject) — excludes `unknown` from the denominator (this plan); (b) accept / total — counts `unknown` as fail. Default proposal: **(a) exclude unknowns.** Rationale: the §19 long-term target ("≥ 70% acceptance after 6 months of dogfooding") implicitly assumes triage decisions are *made*; a cycle-6 sample where 30% are flagged unknown shouldn't drag the visible rate below 49% just because the rater wasn't sure. The `unknown` count surfaces as a separate metric ("triage uncertainty rate").

6. **Whether to bundle cycle-6 priorities #2-#6 (FP template arm, math-lib op extension, surfacedAt plumbing, codec broadening, identity continuity fixture) into v1.9.** Two choices: (a) ship empirical-only (this plan); (b) bundle 1-2 small structural items. Default proposal: **(a) empirical-only.** Rationale: same "one mechanism per release" precedent. The empirical baseline this cycle produces is *the* mechanism; bundling structural changes confuses attribution downstream.

## v1.9 trajectory framing

PRD v1.0 §17.3 names calibration as "the operationalization of high precision, low recall — without it, the philosophy is just a slogan." Five cycles have shipped suppression mechanisms; cycle 6 is the missing input — what the user actually thinks of the surfaces those mechanisms produce.

After v1.9 ships:

- The five prior cycles' structural rules + the new empirical baseline are in-tree. Each prior structural change has a cycle-1+ post-rule snapshot in `docs/calibration-cycle-N-data/`; v1.9 adds the *human accept/reject decisions* layer on top.
- Cycle-7 has data-driven priority candidates rather than purely conjecture-driven ones.
- The §19 acceptance-rate target becomes measurable for the first time.

The natural cadence stays at one cycle per release. Cycle 7 (~v1.10) likely picks up cycle-6 priorities #2-#6 as paired structural additions, weighted by what the v1.9 triage reveals.

After the calibration loop reaches steady state, the natural next contributions return to the §20 v1.1+ trajectory:

- **PRD §20 SemanticIndex** — biggest v1.x lift. The five prior cycles' textual approximations + cycle-6's empirical baseline collectively define what SemanticIndex would replace authoritatively.
- **PRD §20 IDE integration** — depends on SemanticIndex.
- **PRD §20 `swift-infer apply`** — depends on a high-confidence acceptance rate, which cycle-6 starts to establish.
- **Multi-predicate equivalence classes** (M13 axis 3) — SemanticIndex-blocked.
- **Cross-target enum coverage** (M14 deferred bit) — SemanticIndex-blocked.
- **Cross-test data-flow correlation** (M10 + M16 deferred) — natural sequencing post-SemanticIndex.

No active milestone plan opens with the v1.9 commit. Subsequent work picks one of the above and opens a fresh `docs/<plan>.md`.
