## Why this work

Cycle-27 measured 4 CM canonical math round-trip pairs at Accept (`exp/log` × 2 orientations, `sinh/asinh`, `tanh/atanh`, `tan/atan`). Cycle-25 measured 4 more at Accept (`exp/log`, `cosh/acosh`, `cos/acos`, `sin/asin`). Combined: 8 of 8 sampled CM math forward-inverse pairs ACCEPT.

But the current `LiftedTestEmitter.roundTrip` emits a strict-equality assertion:
```swift
let property = "\(inverseName)(\(forwardName)(value)) == value"
```

For FP types (`Complex<Double>`, `Double`, `Float`), this assertion fails under IEEE 754 rounding even on canonical inverse pairs. The property is *mathematically correct* but the *emitted test* is broken.

The fix: detect FP types and emit `isApproximatelyEqual(to:)` (a uniform method available on swift-numerics' `AlgebraicField` protocol, which covers both `Real` types — `Double`, `Float`, `Float16`, `Float80` — and `Complex<Real>` types).

## V1.31.A — `FloatingPointEquatableTypes` curated set

New file `Sources/SwiftInferCore/FloatingPointEquatableTypes.swift`. Mirrors the `MathForwardFunctions` / `IdentityOperatorAlgebra` curated-set pattern.

**Type set** (canonical Swift FP + swift-numerics extensions):
- Real types: `Double`, `Float`, `Float16`, `Float80`, `CGFloat`
- Complex types: `Complex`, `Complex<Double>`, `Complex<Float>`, `Complex<RealType>`, `ComplexModule.Complex`, `ComplexModule.Complex<Double>`, etc.

**Detector** (`isFloatingPointEquatable(typeText:)`): consults the curated set with generic-parameter stripping (so `Complex<Double>` and `Complex<RealType>` both match the `Complex` entry). Returns `Bool`.

**Tests**: `FloatingPointEquatableTypesTests.swift` — verify set membership, generic-stripping behavior, and detector edge cases (e.g., `Int`, `String`, custom user types all return `false`).

## V1.31.B — `EqualityKind` enum + approximate-equality emit

**New enum** in `LiftedTestEmitter.swift` (or a sibling file):

```swift
public enum EqualityKind {
    case strict
    case approximate
}
```

**Emit rule:** for `case .strict`, current behavior unchanged (`lhs == rhs`). For `case .approximate`, emit `lhs.isApproximatelyEqual(to: rhs)`.

**API additions** (optional parameter; default `.strict`):
- `LiftedTestEmitter.roundTrip(forwardName:, inverseName:, seed:, generator:, equalityKind: EqualityKind = .strict)`
- `LiftedTestEmitter.idempotent(funcName:, typeName:, seed:, generator:, equalityKind: EqualityKind = .strict)`
- `LiftedTestEmitter.inversePair(forwardName:, inverseName:, typeName:, seed:, generator:, equalityKind: EqualityKind = .strict)`

Backward compatibility: all existing call sites compile unchanged (pass nothing → `.strict` → current behavior).

**Tests**: extend `LiftedTestEmitterTests.swift` with approximate-equality cases for the three modified arms. Verify the golden output mentions `isApproximatelyEqual`.

## V1.31.C — Dispatch wiring

`InteractiveTriage+Accept` currently dispatches to `LiftedTestEmitter` via three sites (round-trip, idempotent, inverse-pair). Each dispatch needs to consult `FloatingPointEquatableTypes.isFloatingPointEquatable(typeText:)` on the suggestion's carrier (or forward parameter type) and pass the resulting `EqualityKind`.

`CheckPropertyMacro` has two dispatch sites (idempotent + round-trip); same wiring approach.

**Tests**: extend the integration tests in `InteractiveTriageTests.swift` / `CheckPropertyMacroTests.swift` to verify that an FP-typed suggestion produces approximate-equality output and a non-FP-typed suggestion produces strict-equality output.

## Scope boundaries

- **In scope**: round-trip, idempotent, inverse-pair (the three unary-property emit arms; round-trip is the primary motivation).
- **Out of scope this cycle**: commutativity, associativity, monotonicity, identity-element, dual-style-consistency. These arms are less FP-affected (FP `_relaxedAdd/_relaxedMul` are commutative bit-for-bit; monotonicity uses `<=` which is exact). v1.32 may extend if motivated.
- **Out of scope this cycle**: Counter-example regression emitters (`*Regression` arms in `LiftedTestEmitter+Regression.swift`) — they have a different shape (concrete value, not generator-driven).
- **Out of scope this cycle**: Module imports in emitted test files — assume the test file already imports `Numerics`. If the emitter writes imports separately, those stay as-is.

## Risk + mitigation

- **Risk**: `isApproximatelyEqual(to:)` requires the import. **Mitigation**: documented in the explainability block ("Test file must import Numerics for FP approximate-equality").
- **Risk**: `isApproximatelyEqual(to:)` defaults may be too tight/loose for specific algorithms. **Mitigation**: emit with default tolerance for v1.31; future cycle may add a `tolerance:` parameter override.
- **Risk**: A user type that wraps FP values (e.g., `struct Money { var value: Double }`) won't match the curated set but should use approximate equality. **Mitigation**: out of scope; the curated set covers stdlib + swift-numerics common cases. Project-vocabulary extension is the supported path in future cycles.
