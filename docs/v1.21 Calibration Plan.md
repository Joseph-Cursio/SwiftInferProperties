## 1. Why this cycle is different

PRD §17.3's empirical-tuning loop has produced three measurement points (cycles 6 + 14 + 17 = 26.7% → 34.8% → 52.3%). The cycle-14 → cycle-17 delta (+17.5pp / 3 mechanism cycles) was **larger than expected** under the v1.20 plan's outcome-B framing and landed in **outcome A** territory ("Aggregate ≥ 50% — suppression + new-class introduction is paying off"). The cycle-17 findings doc identified the per-mechanism contribution:

- **Workstream C (V1.18.C dual-style)**: 5/5 = 100% — by-construction precision; +6.4pp aggregate contribution.
- **Workstream B (V1.19.B-D lifted-mutation)**: 2/7 = 28.6% — mixed; -0.8pp at current admission gate. Two specific reject classes:
  - **Iterator-shape `mutating func next()` / `advance()` lifted to `(Iterator) -> Iterator`** — 4/4 reject in the cycle-17 sample. The V1.19.B no-param admission on `IteratorProtocol`-conforming carriers is structurally over-broad: `Iterator.next()` advances state per call, so the lifted shadow `(Iterator) -> Iterator` is NOT idempotent. Affects 20 Algorithms candidates + 4 OC candidates = ~24 of 44 idempotence-lifted v1.19 surface (54%).
  - **Monotone-bounded `mutating func advance(until: Int)` / `seek(to:)` / `bound(at:)` lifted to `(T, Int) -> T`** — 1/1 reject. The V1.19.C curated additive-action verb gate (`'advance' ∈ compositionVerbs`) is over-broad on parameter labels that signal monotone-bounded semantics (`until:`, `to:`, `at:`), where `op(s, a).op(s, b)` produces `max(a, b)`-bounded state, not `op(s, a + b)`-additive. Affects 1 of 1 v1.19 composition-lifted candidate (100%).
- **Workstream A (V1.18.A carrier-kind)**: precision-modulator (~0pp aggregate effect at the current sample size).

**The two cycle-17 reject classes are addressable with focused, low-risk mechanisms.** Both fit the "function-name + type-shape composite" (mechanism class 7, V1.14.1 / V1.16.1 lineage) and "parameter-label semantic-intent counter" (mechanism class 8, V1.15.1) shape — established mechanism classes with three-cycle precedent and ~95% plan-vs-actual exact-match rate.

The third workstream — **Math-library forward-function counter** — has been carried forward across cycles 15 + 16 + 17 with three-cycle justification:

- Cycle-15 priority #2: identified after V1.18 didn't address it.
- Cycle-16 priority #2: re-confirmed after V1.19 didn't address it.
- **Cycle-17 measurement directly confirms**: 0/3 = 0% non-lifted idempotence rate on CM elementary functions (`exp`, `log`, `sqrt` picks #18-#20 all reject). The CM elementary-functions noise class is the **dominant survivor** in the non-lifted idempotence pool (17 of 25 v1.19 = 68%). Suppressing this class via a curated `MathForwardFunctions` set + `(T) -> T` shape gate would close all 17 with zero false-negatives (these functions are by-construction not idempotent: `exp(exp(z)) ≠ exp(z)` is a mathematical identity).

## 2. The three workstreams

Each workstream is independently mergeable. v1.21 ships A + B + C in three focused commits; sub-commit ordering matches dependency order (A before B since both touch lifted-template gates; C is independent of A/B).

### Workstream A — IteratorProtocol carrier veto on `idempotence-lifted`

**What:** New veto signal on `IdempotenceTemplate.suggest(forLifted:)` that fires when:

1. The lift's carrier (`lifted.carrier`) appears in the corpus-wide `inheritedTypesByName` index (the V1.5.2-built map of type name → set of inherited type names) with `IteratorProtocol` in its inheritance set, OR
2. The carrier's name matches the curated set `IteratorProtocolCarriers.curated` of stdlib + common Iterator names (`Iterator`, `IteratorProtocol` — for protocol-itself usage; conformance check is the primary path).

**Mechanism class:** existing **class 7** (function-name + type-shape composite), but the gate is on the **carrier's protocol conformance** rather than the function's name + type-shape — call it a "carrier-protocol-conformance veto" sub-class. Same scoring posture as V1.14.1: full veto (-25 weight, equivalent to `protocolCoveredProperty` magnitude) on a structural argument (Iterator.next() / advance() advances state by Iterator protocol semantics).

**Implementation site:** new file `Sources/SwiftInferTemplates/IdempotenceTemplate+IteratorVeto.swift` extending `IdempotenceTemplate` with a `iteratorProtocolCarrierVeto(for:inheritedTypesByName:)` static helper. Wired into `IdempotenceTemplate.suggest(forLifted:)` alongside the existing `protocolCoverageVeto(...)` and `setAlgebraShapeVeto(...)` calls.

**Score posture:** veto fires → score collapses to Suppressed (filtered from `--include-possible` output? Yes — Suppressed-tier suggestions are explicitly filtered; consistent with V1.14.1 setAlgebra-shape veto behavior). Calibration record preserved per V1.5.2 rationale ("the suggestion still scores; it just lands in Suppressed and gets filtered" — cycle-19+ metrics can introspect "how many lifted-idempotence suggestions did Iterator-veto suppress?").

**Detail string:** `"Carrier '\(carrier)' conforms to IteratorProtocol — `next()` / `advance()` advance state per call by protocol semantics, not idempotent on lifted shadow"`.

**Magnitude estimate (cycle-19 projection):** closes 20 Algorithms `*Iterator.next()` + 4 OC `_HashTable.BucketIterator.advance()` = ~24 v1.19 candidates. Lifts the lifted-idempotence acceptance rate from 33% (2/6 cycle-17 sample) to projected ~67% on the post-v1.21 surface at sample size 6 (the 2 accept-class internal-CoW picks remain; the 4 reject-class Iterator picks are suppressed; new sampling would weight more heavily toward the surviving accept class).

**Cost:** ~half day. New file ~80 LOC; ~10 unit tests covering protocol-conformance-match (Iterator types in stdlib + corpus-defined Iterator structs), name-fallback (when conformance lookup unavailable), interaction with existing veto stack (carrier-protocol veto fires before / after setAlgebra veto deterministically).

**Plan-vs-actual projection:** **closes 24 ± 1 candidates** (24 from the cycle-17 surface; ±1 for cross-corpus drift between cycle-17 capture at HEAD `19e45ab` / `0b43769` and v1.21 release-time HEAD).

### Workstream B — `composition-lifted` monotone-bounded suppression

**What:** New parameter-label counter-signal on `CompositionTemplate.suggest(forLifted:)` that fires at **-25 weight (full veto-equivalent)** when the lift's first non-self parameter's label is in a curated `MonotoneBoundedLabels.curated` set:

```
{until, to, at, upTo, before, through}
```

**Why these labels:** they signal "advance / move state up to a target", not "add a quantity additively." The composition property `op(op(s, a), b) == op(s, a + b)` requires the parameter to contribute additively; monotone-bounded semantics produce `max(a, b)`-bounded state instead.

**Mechanism class:** existing **class 8** (parameter-label semantic-intent counter, V1.15.1 domain-marker lineage). The curated label set is small + lexically distinctive; the gate is one hash-set lookup per call. Mirror's V1.15.1's `DomainMarkerLabels.curated` shape exactly.

**Implementation site:** add `MonotoneBoundedLabels.curated` to `Sources/SwiftInferCore/CompositionTemplate.swift`'s curated-set section (alongside existing `curatedAdditiveTypes` and `curatedVerbs`). Add a `monotoneBoundedLabelVeto(for:)` private helper to the `CompositionTemplate` enum that returns a `Signal?` at -25 weight when the first non-self parameter's label matches.

**Score posture:** veto fires → 30 + 40 + 5 + 10 - 25 = 60 → Likely. (Not full Suppressed because the additive-action verb signal still fires; the calibration goal is to demote `advance(until:)` from Strong to Likely, putting it below the default-visible threshold while preserving the calibration record.) Open decision #2 below explores -40 (full Suppressed) as alternative.

**Detail string:** `"Monotone-bounded parameter label '\(label)' — `op(s, a).op(s, b) = max(a, b)`-bounded state, not additive composition"`.

**Magnitude estimate (cycle-19 projection):** closes the 1 v1.19 composition-lifted candidate (`_HashTable.BucketIterator.advance(until: Int)`). Future-corpora projection: small but non-zero — `seek(to:)` patterns on file-handle-like APIs are common. Cycle-19 measurement on a broader corpus (or cycle-17 sample re-extension) will quantify.

**Cost:** ~1-2 hours. Tiny code change (one curated set + one helper + one signal-pipeline insertion); ~5 unit tests covering label-match veto, label-mismatch no-veto, multi-param interaction (counter applies to first non-self only), interaction with curated-verb (verb fires +40, label fires -25, net -25 demotion to Likely).

**Plan-vs-actual projection:** **closes 1 candidate** (the lone v1.19 `advance(until:)` pick).

### Workstream C — Math-library forward-function counter on idempotence + round-trip

**What:** New curated `MathForwardFunctions` set in `Sources/SwiftInferCore/`:

```
{exp, log, log2, log10, sin, cos, tan, asin, acos, atan,
 sinh, cosh, tanh, asinh, acosh, atanh, sqrt, cbrt,
 exp2, expMinusOne, log1p, hypot, atan2}
```

Consumed by `IdempotenceTemplate.suggest(...)` (non-lifted path) and `RoundTripTemplate.suggest(...)` (non-lifted path) via new helper methods:

- **`IdempotenceTemplate.mathForwardFunctionVeto(for:)`** — fires veto (-25) when `summary.name ∈ MathForwardFunctions.curated` AND `summary.parameters.count == 1` AND `summary.parameters[0].typeText == summary.returnTypeText` (i.e., `(T) -> T` shape on math name).
- **`RoundTripTemplate.mathForwardFunctionPairVeto(for:)`** — fires veto (-25) when **both** pair sides have name in `MathForwardFunctions.curated` AND both have `(T) -> T` shape, BUT the pair is **not** in the curated forward-inverse list `{(exp, log), (sin, asin), (cos, acos), (tan, atan), (sinh, asinh), (cosh, acosh), (tanh, atanh), (sqrt, /), (cbrt, /)}`. Forward-forward (`exp × cosh`) and forward-mismatch (`exp × sqrt`) cross-products veto; forward-inverse pairs preserve. Asymmetric: `(exp, log)` is allowlist; `(log, exp)` matches by orientation-insensitive lookup.

**Mechanism class:** existing **class 7** (function-name + type-shape composite, V1.14.1 / V1.16.1 lineage). Same shape as V1.16.1's SetAlgebra-shape veto extension to round-trip + idempotence — third-template extension of an established mechanism class.

**Implementation sites:**
- `Sources/SwiftInferCore/MathForwardFunctions.swift` — new file. Public `MathForwardFunctions.curated: Set<String>` + public `MathForwardFunctions.canonicalInversePairs: [(forward: String, inverse: String)]` (the forward-inverse allowlist for round-trip's pair-aware veto).
- `Sources/SwiftInferTemplates/IdempotenceTemplate+MathForwardVeto.swift` — `IdempotenceTemplate.mathForwardFunctionVeto(for:)` extension.
- `Sources/SwiftInferTemplates/RoundTripTemplate+MathForwardVeto.swift` — `RoundTripTemplate.mathForwardFunctionPairVeto(for:)` extension.

**Score posture:** uniform -25 veto weight across both templates (matches V1.14.1 / V1.16.1 calibration). Idempotence: `30 + 40 - 25 = 45` Likely if curated verb match (none of the math names are in the curated idempotence verbs); bare `30 - 25 = 5` Suppressed. Round-trip: `30 + 40 - 25 = 45` Likely if curated inverse-pair match; bare `30 - 25 = 5` Suppressed. **Forward-inverse allowlist preserves the genuine inverse pairs** that cycle-17 measured at 7/7 = 100% accept (`exp × log`, `cos × acos`, `sin × asin`, etc. all stay surfaced).

**Detail string** (idempotence): `"Math-forward function '\(name)' — `\(name)(\(name)(z))` is not idempotent (e.g., exp(exp(z)) = e^(e^z) ≠ exp(z))"`.

**Detail string** (round-trip pair veto): `"Math-forward cross-product '\(forward.name) × \(reverse.name)' — not in canonical inverse-pair allowlist (`{(exp, log), (sin, asin), ...}`); both forward, not inverses"`.

**Magnitude estimate (cycle-19 projection):**
- **Idempotence:** closes 17 CM elementary-functions picks (the cycle-17 surface had 17 CM idempotence candidates of which 6 are MathForwardFunctions members visible at the sample-manifest level; full v1.19 surface projection: ~17 of 25 CM idempotence = 68%).
- **Round-trip:** closes the CM cross-product noise class = ~120 of 136 CM round-trip pairs (the 7 canonical-inverse pairs cycle-17 sampled at 100% accept survive; the 129 cross-products veto). **Largest single mechanism in v1.21.**

**Cost:** ~half day for the curated set + two extension methods + ~20 unit tests covering: math name + (T) -> T shape match veto (idempotence); pair-shape match veto with forward-inverse allowlist preserve (round-trip); composition with V1.16.1 SetAlgebra veto (orthogonal — different curated sets); composition with V1.18.A carrier-kind signal (orthogonal — score-additive); cross-corpus stability (CM-only effects on the four cycle-1..14 corpora).

**Plan-vs-actual projection:** **closes ~146 candidates** (17 idempotence + 129 round-trip), reducing the v1.19 335-surface to ~189 — **back below the cycle-13 229 surface count, restoring the descending trend** at the post-v1.21 measurement (cycle 19 = v1.23 if cadence holds).

## 5. Open decisions

1. **Workstream A IteratorProtocol detection: textual-conformance-only (cycle-17 finding default) vs textual-conformance + name-fallback.** Default proposal: **(b) textual-conformance + name-fallback.** The cycle-17 finding identifies all 4 picks via direct corpus inspection (the carrier types are textually-defined Iterator structs); textual conformance via `inheritedTypesByName` will catch them. Name-fallback (`carrier.hasSuffix("Iterator")` AND no conformance match) catches edge cases where the corpus defines an Iterator-shape struct without explicit conformance (rare in stdlib but seen in user code). Trade-off: name-fallback adds ~5 picks of false-positive risk on user-code Iterator-named types that don't actually advance state (cycle-17 sample has zero such picks; future corpora may differ).

2. **Workstream B veto magnitude: -25 (Likely demotion) vs -40 (full Suppressed).** Default proposal: **(a) -25 (Likely demotion).** Rationale: the cycle-17 sample is n=1 on composition-lifted; demotion to Likely preserves the calibration record while filtering from the default-visible Strong tier. Promoting to -40 would hide the candidate entirely; cycle-19 measurement on broader corpora may reveal `until:` / `to:` / `at:` labels with genuinely-additive semantics in some domain (stride-based numerical algorithms, e.g.). The -25 magnitude is reversible at cycle-19 if data warrants.

3. **Workstream C MathForwardFunctions curated set: stdlib-only (`Foundation` + `Numerics` math families) vs include `swift-numerics`-extended (`expMinusOne`, `log1p`, `hypot`, `atan2`).** Default proposal: **(b) include the swift-numerics extensions.** The cycle-17 sample includes CM `expMinusOne(_:)` (Algo `log(_:)` × `log(onePlus:)` is the corresponding Algo two-overload reject pair from cycle-14 #20). Including `expMinusOne` + `log1p` etc. closes the numerical-variant noise class. Trade-off: the curated set is engine-internal; rebasing if swift-numerics adds new variants (`logAccurate`, etc.) requires a one-line patch. Acceptable maintenance cost.

4. **Workstream C MathForwardFunctions canonical-inverse-pair allowlist: literal pairs (default) vs orientation-insensitive matching.** Default proposal: **(b) orientation-insensitive matching.** A cycle-17 sample showed both `exp × log` and `log × exp` orientations; `(exp, log)` and `(log, exp)` should both preserve. Implementation: the `RoundTripTemplate.mathForwardFunctionPairVeto(for:)` helper checks `(forward, reverse) ∈ canonicalInversePairs ∨ (reverse, forward) ∈ canonicalInversePairs`. Mirrors V1.18.C `DualStylePairing`'s orientation-insensitive matching.

5. **Cycle numbering: cycle 18 = v1.21.** Contiguous with v1.20 = cycle 17 (empirical), v1.19 = cycle 16, v1.18 = cycle 15, v1.17 = cycle 14 (empirical), v1.16 = cycle 13.

6. **Whether to re-measure perf at V1.21.D.** Default proposal: **yes, re-measure.** v1.21 ships three Sources/ changes (one new file in SwiftInferCore + four new files in SwiftInferTemplates + per-template integration); the V1.20.E carry-forward posture explicitly does not apply. Expected delta vs v1.19 baseline: ≤+5% wall time (the new vetoes are O(1) hash-set lookups per template-suggest call; the IteratorProtocol veto adds one per-summary `inheritedTypesByName` lookup which is already cached).

7. **Whether to re-run the cycle-17 four corpora at V1.21.D for plan-vs-actual closure.** Default proposal: **yes, re-run all four** (mirrors V1.13's cycle-10 hoist re-run + V1.16's cycle-13 mechanism re-run). Captures the projected ~146-candidate closure (Workstream C) + ~24-candidate closure (Workstream A) + 1-candidate closure (Workstream B) into `docs/calibration-cycle-18-data/` with per-corpus deltas vs the cycle-17 surface counts. Cycle-19 (v1.22 mechanism release) inherits the v1.21 surface for its priority rotation; cycle-20 (v1.23 empirical re-measurement) inherits both.
