# TestLifter M3 Execution Plan

Working doc for the TestLifter M3 milestone defined in `SwiftInferProperties PRD v0.4.md` §7.9. Decomposes M3 into five sub-milestones so progress is checkable session-by-session. **Ephemeral** — archive to `docs/archive/TestLifter M3 Plan.md` once M3 ships and the §7.9 acceptance bar is met (mirroring TestLifter M1 + M2 + TemplateEngine M1–M8).

> **TestLifter M3 closes the lifted-suggestion stream-entry loop.** TestLifter M1 + M2 lit up cross-validation `+20` for round-trip / idempotence / commutativity. M3 ships **generator inference** (PRD §7.9 M3 row) and lands the **stream-entry + writeouts + accept flow** that the M2 plan's open decision #1 explicitly deferred to here ("M3 brings generator inference and at that point the stream-entry + LiftedTestEmitter-extension + `Tests/Generated/SwiftInfer/` writeouts + `InteractiveTriage+Accept` extension all justify their own milestone-shaped scope, AND the lifted stubs would actually carry inferred generators (rather than `.todo` placeholders) by the time they surface to the user"). After M3, a test-only `roundTrip`/`idempotence`/`commutativity` body whose callee has *no* matching TemplateEngine signature-side suggestion still surfaces in `swift-infer discover`'s suggestion stream — accepted lifted suggestions write to `Tests/Generated/SwiftInfer/<template>/<file>.swift` with an inferred generator from the shared `DerivationStrategist` (or `.todo` when type recovery fails).

## What M3 ships (PRD v0.4 §7.9 + M2 plan deferred work)

> **M3 (TestLifter).** Generator inference: stdlib derivation strategies + `.todo` stub pattern; consume shared `DerivationStrategist`. *(Verbatim from PRD §7.9.)*

Plus the M2-plan-deferred chunk (M2 plan open decision #1):

> **Stream-entry + writeouts + accept flow.** TestLifter's `LiftedSuggestion` entries enter the main `discover` suggestion stream; `Tests/Generated/SwiftInfer/<template>/<file>.swift` writeouts are produced when a lifted suggestion is accepted; `InteractiveTriage+Accept` dispatches on the lifted-suggestion variant the same way it dispatches on TemplateEngine-originated suggestions today.

Four concrete deliverables, all additive on top of M1 + M2's library:

1. **`LiftedSuggestion → Suggestion` promotion adapter.** New `Sources/SwiftInferTestLifter/LiftedSuggestionPromotion.swift`. Public entry: `LiftedSuggestion.toSuggestion(typeName: String?, returnType: String?, receiverTypeName: String?) -> Suggestion`. Builds a `Suggestion` whose `templateName` matches the lifted suggestion's; whose `evidence` carries one synthetic `Evidence` per detection (carrying the callee name, signature-shaped from the recovered types, and a `displayName` derived from the test source location); whose `score` includes a `Signal(kind: .testBodyPattern, weight: 50, detail: "TestLifter — \(detection.kind)")` per PRD §4.1's "+50 test-body pattern" row; whose `generator` is the M3-default `.notYetComputed` placeholder so the existing `GeneratorSelection` pass overwrites it with the inferred metadata. The promotion is the bridge that lets the existing renderer + accept flow consume lifted suggestions without per-call-site special casing.

2. **TestLifter-side type recovery.** Extend `Discover+Pipeline.collectVisibleSuggestions` with a per-lifted-suggestion type-recovery pass that walks the `[FunctionSummary]` the FunctionScanner already produces (and which TemplateEngine consumes) and resolves each lifted callee name to its parameter / return / receiver types. When the callee name is matched, the recovered types feed `LiftedSuggestion.toSuggestion(...)` and the resulting `Suggestion` flows through `GeneratorSelection` like any TemplateEngine suggestion. When the callee name is **not** matched (test-only function the scanner didn't pick up), the promotion is invoked with `typeName: nil` and `GeneratorSelection` produces `.todo` for the missing type — the lifted suggestion still surfaces, but its accept-flow stub will compile-error on the `.todo` until the user resolves it.

3. **Cross-validation suppression.** Lifted suggestions whose `crossValidationKey` already matches a TemplateEngine-side `Suggestion` are dropped from stream-entry. The TemplateEngine suggestion already carries the +20 cross-validation signal (M1.5); double-emitting a parallel lifted suggestion would be confusing surface area. The suppression is a pure post-promotion filter inside `Discover+Pipeline`, ordered after `GeneratorSelection` so generator metadata is consistent on whichever record survives.

4. **Accept-flow extension for lifted suggestions.** `InteractiveTriage+Accept.handleAccept(suggestion:context:)` already covers all eight TemplateEngine arms (`idempotence` / `round-trip` / `monotonicity` / `invariant-preservation` / `commutativity` / `associativity` / `identity-element` / `inverse-pair`) via the `liftedTestStub(for:)` switch. M3.4 adds (i) a `liftedSuggestionMetadata: LiftedSuggestionMetadata?` field on `Suggestion` (or carries the lifted origin via a `Signal` detail / `Explainability` block — see open decision #4) so the writeout layer can recover the originating test-method name + sliced source location for file naming + provenance comments; (ii) a `lifted_<TestMethodName>` filename infix that distinguishes lifted writeouts from template-only writeouts within the same `<template>/` subdirectory.

### Important scope clarifications

- **No kit-side coordination commit needed.** The PRD §11 + §21 OQ #4 prerequisite ("promote `DerivationStrategist`, `DerivationStrategy`, `TypeShape`, `MemberwiseEmitter` from `package` to `public`") was **already met** in the SwiftProtocolLaws v1.7 K-prep-M1 cluster (see `Sources/ProtoLawCore/MemberwiseEmitter.swift:22` "Promoted from `package` to `public` in the v1.7 K-prep-M1 cluster"). All four target types are already `public` in `ProtoLawCore` as of the kit's v1.7 line; SwiftInferProperties tracks `from: "1.9.0"` which is well past that point. **`Package.swift` stays at `from: "1.9.0"` for M3.** No kit PR, no version bump.

- **No new generator-inference machinery on the SwiftInferProperties side.** The TemplateEngine consumer of `DerivationStrategist` already exists at `Sources/SwiftInferTemplates/GeneratorSelection.swift:42` and ships the calibration table per the M4 plan. M3's job is **promote LiftedSuggestion → Suggestion before the GeneratorSelection pass**, not duplicate the calibration table. Reuse, not re-implementation.

- **Type recovery in M3 is "FunctionSummary lookup OR `.todo`" — no setup-region parsing.** The setup region of a sliced test body sometimes carries `let x: T = ...` annotations or `let x = T(...)` constructor calls that recover the type even when no production-side function matches the callee. Walking the setup region for these is real syntactic work and a natural M4 deliverable (alongside the M4 PRD-row "mock-based generator synthesis from observed test construction"). M3's type-recovery pass is strict: FunctionSummary match → recovered type; otherwise `nil` → `.todo`. Open decision #2 below covers the framing.

- **Receiver-type recovery for instance-method patterns is in scope but bounded.** `AssertSymmetryDetector` (M2.2) recognizes `a.f(b)` instance shapes; `AssertAfterTransformDetector` (M1.3) recognizes `a.encode().b.decode()` chains. For the writeout to compile, the lifted stub's generator must produce `(T, T)` (commutativity instance shape) or `T` (round-trip instance shape) where `T` is the receiver type. M3 extends the detection structs (`DetectedRoundTrip`, `DetectedIdempotence`, `DetectedCommutativity`) with an optional `receiverTypeName: String?` populated by detector lookup against `[FunctionSummary]` (free function → `nil`; method on a known type → that type's name). Same FunctionSummary lookup path as the parameter / return type recovery — no setup-region walking.

- **Lifted suggestions enter the visible stream at the `~ Likely` tier by default.** PRD §4.1 has the "+50 test-body pattern" row but the per-pattern base scores (+30 round-trip / +25 idempotence / +25 commutativity, see the M1 + M2 plans) accrue from TemplateEngine's structural analysis, not from the test body. A pure-TestLifter suggestion (no production-side match) gets the +50 test-body signal but no structural base — that lands it at ~50 points, which is `~ Likely` per the §4.4 tier table. Promotion does not synthesize structural signals on top of the +50 — the lifted suggestion claims only what TestLifter actually saw. Open decision #5 below covers the alternative ("synthesize a +25 base for the matching template — promotes to `✓ Strong`").

- **Cross-validation suppression is pure drop, not "hide and merge".** The dropped lifted suggestion's payload (slice source location, test method name) is **not** threaded into the surviving TemplateEngine suggestion's explainability block. M3 keeps the suppression pass simple — drop on key match, nothing else. Open decision #3 below covers the alternative ("merge slice-location pointer into TemplateEngine suggestion's explainability for richer 'why suggested'").

- **Writeout file naming uses `<TestMethodName>_lifted_<TemplateName>.swift` within the same `Tests/Generated/SwiftInfer/<template>/` subdirectory.** Disambiguates lifted writeouts from TemplateEngine-accepted writeouts for the same callee. Keeps PRD §3.6 step 3 + §16 #1 directory layout intact (no new sibling `Tests/Generated/SwiftInfer/lifted/` subtree). Open decision #4 below covers two alternatives.

- **`.swiftinfer/decisions.json` persistence still gated on TestLifter M6.** Lifted suggestions accepted in M3 produce a writeout file but are **not** persisted to `.swiftinfer/decisions.json` — that's TestLifter M6 (mirror of TemplateEngine M6's persistence wiring). The TemplateEngine M6 persistence wiring already exists; M3 just doesn't extend it to TestLifter records. Same posture as the M1.5 ship: cross-validation works without persistence.

- **No `// swiftinfer: skip` honoring on the test side in M3.** TestLifter M6 (also defers to the bundled CLI surface widening per the M2 plan).

- **No counter-signal scanning in M3.** PRD §4.1's "asymmetric assertion" `-25` row is TestLifter M7. M3 still recognizes only the *positive* shapes M1 + M2 ship.

- **No kit-side LiftedSuggestion changes.** SwiftProtocolLaws is one-way downstream from SwiftInferProperties; the kit doesn't know LiftedSuggestion exists. The `DerivationStrategist` consumption is read-only.

- **Cross-validation matching stays by `CrossValidationKey`.** No identity-shape changes in M3 — same M1.4 key shape (`templateName + sortedCalleeNames`), same matching semantics.

- **No new SwiftPM dependencies.** TestLifter stays on `swift-syntax` + `Foundation` + `SwiftInferCore` + (new) `SwiftInferTemplates` for the `Suggestion` / `Evidence` / `GeneratorMetadata` types the promotion adapter constructs. SwiftProtocolLaws stays at `from: "1.9.0"`.

## M3 acceptance bar

Mirroring PRD §7.9 + the v0.4 §5.8 acceptance-bar pattern + the M1 / M2 / TemplateEngine M1–M8 cadence, M3 is not done until:

a. **`LiftedSuggestion.toSuggestion(typeName:returnType:receiverTypeName:)` is the promotion adapter from `LiftedSuggestion` to `Suggestion`.** Returns a `Suggestion` whose `templateName` matches, whose `evidence` array carries one synthetic `Evidence` per detection callee with the recovered (or `?`-sentinel) types, whose `score` includes a +50 test-body-pattern signal, and whose `generator` defaults to `.notYetComputed` so `GeneratorSelection` overwrites it.

b. **`Discover+Pipeline.collectVisibleSuggestions` recovers callee types from `[FunctionSummary]` for each lifted suggestion and feeds the recovered types into the M3.0 adapter.** Type recovery is strict — only matches against the existing FunctionScanner index, no setup-region parsing. Unrecovered types route to `.todo` via `GeneratorSelection`.

c. **Lifted-promoted Suggestions whose `crossValidationKey` matches a TemplateEngine-side Suggestion are dropped from stream-entry.** No double-counting; the TemplateEngine suggestion's existing +20 cross-validation signal carries the corroboration.

d. **`InteractiveTriage+Accept.handleAccept(suggestion:context:)` writes lifted-promoted suggestions to `Tests/Generated/SwiftInfer/<template>/<TestMethodName>_lifted_<TemplateName>.swift`** with a provenance comment header pointing at the originating test method's source location.

e. **The visible `discover` suggestion stream surfaces lifted suggestions whose callee has no matching production-side function** — verified by integration tests for round-trip, idempotence, and commutativity. The lifted suggestions tier at `~ Likely` (or higher per open decision #5) and carry the +50 test-body-pattern signal in their explainability.

f. **`.todo` stubs never silently compile-pass** — verified by the M3.4 `TodoFallbackHardGuaranteeTests` extension. Lifted suggestions with unrecovered types produce stubs that compile-error on the `.todo<?>()` placeholder until the user resolves it (PRD §16 #4 invariant).

g. **§13 performance budget for `TestLifter parse of 100 test files` (`< 3s` wall) holds** with all of M3.0–M3.3 active (promotion + recovery + suppression + accept-flow stub generation). Regression test fails if exceeded.

h. **§16 #1 hard guarantee preserved** — M3's accept-flow extension does not write to source files, only to `Tests/Generated/SwiftInfer/`. Verified by extending `TestLifterHardGuaranteeTests`.

## Open decisions to make in-flight

1. **Should the `LiftedSuggestion → Suggestion` promotion go through `Suggestion` synthesis (single downstream stream) or via a parallel `LiftedSuggestion`-typed renderer arm (two-stream output)?**
   - **(a) Synthesize `Suggestion` records (single stream).** The renderer + tier filter + `GeneratorSelection` + accept-flow + drift mode + baseline snapshot all consume a uniform `[Suggestion]`. M3 adds zero new pipeline branches; the lifted origin is carried as a field on `Suggestion`. Downside: `Suggestion`'s shape was designed for TemplateEngine-originated records, so the synthetic `Evidence` records have to fake a `signature` / `displayName` from the test-body context.
   - **(b) Parallel `LiftedSuggestion`-typed renderer arm.** Renderer prints two sections ("From signatures: ..." + "From tests: ..."). The accept-flow gets a parallel `handleAcceptLifted(_:context:)` arm that takes `LiftedSuggestion` directly. Cleaner semantic separation. Downside: doubles the surface area of every downstream consumer (renderer, drift, baseline, accept-flow); contradicts PRD §3.6's "one suggestion stream the developer reviews."
   - **Default unless reason emerges:** **(a) synthesize `Suggestion` records**. The PRD's developer workflow §3.6 is built on the assumption of one unified stream; a parallel TestLifter section would force the user to context-switch between two lists per `discover` run. The synthetic-Evidence cost is real but bounded — TestLifter only has three patterns to fake signatures for through M3 (round-trip, idempotence, commutativity) and the `?` sentinel is honest about what TestLifter actually knows.

2. **Should M3's type recovery walk the test body's setup region for `let x: T = ...` annotations / `let x = T(...)` constructor calls when FunctionSummary lookup fails?**
   - **(a) Strict — FunctionSummary lookup only.** When the callee isn't in the production-side scanner output, fall back to `.todo`. Setup-region walking is genuinely M4-scoped work that the PRD pairs with mock-based generator synthesis ("inferred from observed test construction" — §7.4).
   - **(b) Loose — also try setup-region annotations.** Catches the "test-only callee with annotated bindings" case in M3 instead of M4. Doubles M3.1's syntactic surface area; collides scope with the M4 PRD-row deliverable.
   - **Default unless reason emerges:** **(a) strict FunctionSummary lookup**. Matches the PRD §7.9 row separation — M3 is "stdlib derivation strategies + `.todo` stub pattern", M4 is "mock-based synthesis from observed test construction". Setup-region walking is the M4 work; M3's `.todo` fallback is the explicit M3-row deliverable.

3. **When a lifted suggestion is suppressed (its key matches a TemplateEngine suggestion), should the lifted slice's source location be threaded into the surviving TemplateEngine suggestion's explainability?**
   - **(a) Pure drop — no merging.** Suppression is a single-line `Set<CrossValidationKey>` membership filter. The TemplateEngine suggestion already carries a `Signal(kind: .crossValidation, weight: 20, detail: "Cross-validated by TestLifter")`; that's enough to communicate "TestLifter saw this too". Adding the slice source location is nice-to-have but not load-bearing.
   - **(b) Merge slice-location pointer into the TemplateEngine suggestion's explainability.** The "why suggested" block becomes more concrete ("...corroborated by `Tests/FooTests/CodecTests.swift:42` `testRoundTrip()`"). Adds a one-way data flow from the suppression pass back into the surviving suggestion, which means the suppression filter is no longer a pure drop.
   - **Default unless reason emerges:** **(a) pure drop**. Keeps the suppression pass as a one-liner Set filter. The PRD §4.5 "explainability is first-class" target is met by the existing `+20 cross-validation` signal; richer per-slice provenance is a polish item that can land in a later TestLifter milestone if user research surfaces a need. Trying to do (b) in M3 conflates suppression scope with explainability scope.

4. **Lifted-origin tracking on the `Suggestion` record — opaque metadata field, tagged Signal detail, or new Suggestion variant?**
   - **(a) `Suggestion.liftedOrigin: LiftedOrigin?` opaque field.** New struct carries `(testMethodName, sourceLocation)`. Accept-flow + renderer read the field directly. Bloats `Suggestion`'s storage shape with a TestLifter-specific concept the rest of the system doesn't need.
   - **(b) Tagged `Signal.detail` — encode the test-method name + source location into the +50 test-body-pattern signal's detail string.** Zero new fields on `Suggestion`. Accept-flow parses the detail string when it needs the file-naming infix. Fragile string-parsing contract.
   - **(c) New `Suggestion` variant (enum-ify the struct).** Heaviest refactor; touches every consumer of `Suggestion`. Out of scope for M3.
   - **Default unless reason emerges:** **(a) `Suggestion.liftedOrigin: LiftedOrigin?` field**. The cost of one optional field on `Suggestion` is small; the alternatives either fragile-string-parse or balloon the M3 surface. The field is `nil` for TemplateEngine-originated suggestions, so existing code paths are unchanged.

5. **Tier of lifted suggestions with no production-side TemplateEngine corroboration — accept the natural ~50-point landing (`~ Likely`), or synthesize a structural-base signal that promotes to `✓ Strong`?**
   - **(a) Natural landing — `~ Likely` from the +50 test-body signal alone.** Honest about what TestLifter saw: a test body with the right shape but no signature-side structural evidence. Per PRD §3.5 "high precision, low recall" — when in doubt, fewer suggestions / lower tier.
   - **(b) Synthesize a +25 structural base signal so the lifted-only suggestion lands at ~75 → `✓ Strong`.** Gives lifted-only patterns the same surface visibility as TemplateEngine-strong suggestions. Risks the PRD §3.5 "Daikon trap" (too many high-tier suggestions desensitize the user to the strong tier).
   - **Default unless reason emerges:** **(a) natural ~50-point `~ Likely` landing**. PRD §3.5 conservative-engine philosophy. Lifted-only suggestions *should* be quieter than dual-source suggestions — that's the whole point of cross-validation. Users who want lifted-only at higher visibility can `--include-possible` (which doesn't apply at the `~ Likely` tier — `~ Likely` is already visible by default per the §4.4 tier table). The +50 signal alone clears the visibility threshold.

6. **Writeout file naming — `<TestMethodName>_lifted_<TemplateName>.swift` (default), `<TestMethodName>_lifted.swift`, `lifted_<TestMethodName>.swift`, or sibling subdirectory `Tests/Generated/SwiftInfer/lifted/<template>/<file>.swift`?**
   - **(a) `<TestMethodName>_lifted_<TemplateName>.swift` in the existing `<template>/` subdirectory.** Disambiguates from TemplateEngine writeouts (which use `<callee>.swift`) AND keeps the template grouping intact. Test method origin visible at a glance. Filename gets long but readable.
   - **(b) `<TestMethodName>_lifted.swift` (no template infix).** Shorter. Risks collision when one test method body lifts multiple patterns (rare but possible — a single test could combine round-trip + idempotence assertions).
   - **(c) Sibling subdirectory `Tests/Generated/SwiftInfer/lifted/<template>/<file>.swift`.** Cleanest separation. Doubles directory-tree depth; introduces a "lifted" namespace not present in PRD §3.6 / §10.5 today.
   - **Default unless reason emerges:** **(a) `<TestMethodName>_lifted_<TemplateName>.swift`**. Disambiguates without restructuring the directory layout the PRD already specifies. The longer filename is a worthwhile cost for the readable provenance.
