# v1.141 Calibration Plan — Minimal-Counterexample & Replay-Corpus Phase (shrink → auto-regression → replay corpus)

Opens the **minimal-counterexample & replay-corpus epic** (cycles 152–155). First
cross-repo epic: pairs a SwiftPropertyLaws **v2.6** (Backend Shrinking) prerequisite
with four SwiftInfer cycles and one opt-in SwiftIdempotency add-on.

Builds directly on the verify pipeline (v1.42–v1.51) and `Tier.verified` (V1.65) /
`Signal.verifyBothPass`+`verifyDisproven` (V1.66): **verify already runs the
property and moves the tier.** This epic makes the discovered counterexample
*minimal* (shrinking), *durable* (auto-emitted regression in the adopter's own
idiom), and *replayable* (seed + environment corpus, re-checked first on every run).

## Why now — the three residual holes in the verify loop

| Hole | Current reality (v1.140) | This epic |
|---|---|---|
| No shrinking anywhere | `SwiftPropertyBasedBackend.check()` returns `failingInput` verbatim ("Shrinking is NOT YET exercised (M5+ scope)"). Verify stubs don't even call the backend — they emit hand-rolled `for trial in 0..<100 { … exit(1) }` loops printing the *first* failing value via `VERIFY_DEFAULT_INPUT`. | v2.6 + V1.141.x |
| Broken seam verify → regression | `verify` prints the counterexample; `convert-counterexample` is a *separate* command needing a hand-typed `--counterexample`. The failure is found, then dropped. | V1.142.x |
| No replay corpus | `VerifyEvidence` stores the *outcome* but not the seed or counterexample. SwiftPropertyLaws' `Seed` (base64 Xoshiro) + `Environment` fingerprint + `ReplayEnvironmentMismatch` guard are entirely unused by SwiftInfer. | V1.143.x |

## Decisions

- **D1 — shrinking lands in SwiftPropertyLaws v2.6 as a value-level closure, not in stub emitters and not by threading `Generator`.** swift-property-based's `Shrink` namespace (`FixedWidthInteger.shrink(towards:)`, `Shrink.shrinkTuple(_:shrinkers:)`, `Shrink.None`) is value→values, generator-independent — it fits the backend's closure seam directly. Wiring it once in the kit benefits every PropertyLawKit consumer (now *two*: SwiftInfer verify stubs **and** SwiftIdempotencyPropertyBased — see W6) and avoids N copies of shrink logic across `RoundTripStubEmitter` + siblings. Alternative (inline shrink loop per stub) rejected unless v2.6 slips.
- **D2 — `--emit-regression` defaults ON for single `--suggestion` runs, OFF for `--all-from-index` survey runs** (avoids flooding `Tests/Generated/`). Overridable both ways.
- **D3 — SwiftIdempotency gets property-based assertions via a new opt-in product**, mirroring the `SwiftIdempotencyFluent` pattern, *not* by adding deps to its lean core. Detailed in the companion design note (see W6).

## Prerequisite — SwiftPropertyLaws v2.6 (Backend Shrinking)

**Status: v2.6.A SHIPPED (2026-06-26).** Implemented driver-level — cleaner than this section's original backend-level sketch. Files changed: `.../Internal/PerLawDriver.swift`, `.../Internal/LawCheckBuilders.swift`, `.../Model/CheckResult.swift`, `.../Public/EquatableLaws.swift`. `PropertyBackend`, `BackendCheckResult`, and `SwiftPropertyBasedBackend` are **untouched** (zero public/cross-repo break). Full design + rationale: `SwiftPropertyLaws/docs/v2.6 Backend Shrinking plan.md`.

- **v2.6.A — driver-level shrinking (done).** Shrinking is a `PerLawDriver` policy (it already owns result assembly, suppression, env fingerprinting). On first failure, `minimize` greedily descends a value-level `shrink: (Input) -> [Input]` closure, re-running the **same `property`** as the failure oracle, to fixpoint (cap 1000). Shrink metadata rides on `CheckResult` as additive `shrunkFrom: String?` + `shrinkSteps: Int` fields (the `nearMisses`/`coverageHints` precedent) — `Outcome.failed` is unchanged, so zero switch ripple. The shrinker enters via an optional `shrink:` on the unary/binary/ternary builders + `checkEquatablePropertyLaws` (proving family). Built from swift-property-based's `FixedWidthInteger.shrink(towards:)`. **Closure seam preserved — no `Generator` threading.** Ternary shrinking lifts the per-element shrinker one position at a time via a `liftToTripleShrinker` helper whose intrinsic `(Value, Value, Value)` type is the kit's only `large_tuple` site — waived with a **scoped `swiftlint:disable/enable` around that one helper**, not a config change.
- **Deferred:** the other 21 families (per-law opt-in rollout — unary/binary/ternary all supported by the builders; each family just threads a defaulted `shrink:`).
- **v2.6.B — tag a release;** SwiftInfer bumps its `SwiftPropertyLaws` pin.
- **Risk R3 (float) — resolved by construction.** The shrink oracle reuses the law's own `property` (its exact comparison, e.g. `isApproximatelyEqual`), so a candidate is accepted only if it *also* genuinely fails.

Tests shipped (`BackendShrinkingTests`, 5, all green; total 486→491): minimal still fails and ≤ original; non-shrinkable → `shrinkSteps == 0` identical to pre-v2.6; deterministic replay under a fixed seed; ternary per-component shrinking; end-to-end through the public Equatable suite.

## What v1.141 ships

> **Correction (2026-06-26, post-v2.6.0).** Earlier drafts said V1.141.B would have stubs "call the kit's now-shrinking backend (`check(…, shrink:)` / `checkXxxPropertyLaws`)." Both are wrong: (a) v2.6 put shrinking in `PerLawDriver`, reachable **only** via `checkXxxPropertyLaws`, not `PropertyBackend.check`; (b) verify stubs are hand-rolled programs that test **arbitrary inferred properties** (`inverse(forward(x)) ≈ x` via `isApproximatelyEqual`, idempotence, …) — **not protocol-law conformances** — so the kit's law functions don't apply. The minimal counterexample must come from a shrink loop the stub itself runs over its concrete carrier. **Open decision D4 below.**

- **D4 — where the verify-stub shrink loop lives.** (a) **Hand-roll in the emitted stub** — after the first failing `value` (e.g. `RoundTripStubEmitter` line ~245), emit a carrier-native shrink loop using swift-property-based's `Shrink.*` on the concrete carrier (`Complex<Double>` → shrink real/imag `Double`s; `Int`/`Double` directly), keep the first still-failing candidate to fixpoint, print the minimal. Self-contained, no new kit API, fits the existing "stub hand-rolls everything" pattern; the generated code is throwaway so the ~15 duplicated lines are cheap. (b) **Expose a public kit primitive** (`PropertyLawKit.shrinkToMinimal(_:shrink:stillFails:)`, refactor `PerLawDriver.minimize` onto it) that both verify stubs **and** `SwiftIdempotencyPropertyBased` (W6) import — DRY across the two non-law consumers, but costs another kit release + a generated-stub import. *Recommendation: (a) now (YAGNI), extract (b) when W6 actually lands and the second consumer is real.*

1. **V1.141.A** — Per the chosen D4 option, the carrier-specific stub emitters (`RoundTripStubEmitter` + `IdempotenceStubEmitter` / `CommutativityStubEmitter` / `AssociativityStubEmitter` / `StrategistDispatchEmitter`) gain a **shrink phase** that minimizes the failing input over the concrete carrier. Replaces the "report first failure" behavior.
2. **V1.141.B** — Stubs print `VERIFY_DEFAULT_SHRUNK` + `VERIFY_SHRINK_STEPS` alongside `VERIFY_DEFAULT_INPUT` on failure.
3. **V1.141.C** — the verify-output parser captures shrunk value + step count into `VerifyOutcome`; the renderer shows `shrank 14 steps: Complex(8.3e5, -2.1e5) → Complex(0.0, 0.0)`.
4. **V1.141.R2 (graceful degrade)** — when a carrier has no `Shrink.*` (or `DerivationStrategist` can't supply one), pass `nil`/`Shrink.None` → today's behavior (report first failure, `shrinkSteps: 0`, renderer note) rather than erroring.

## Downstream cycles (summary — own plans at activation)

- **v1.142 — auto-bridge (shipped).** On `measuredDefaultFails`, the verify pipeline auto-invokes `ConvertCounterexampleEngine` with the *shrunk* counterexample → `Tests/Generated/SwiftInfer/<template>/<callee>_regression_<hash>.swift`. `--emit-regression` (D2), wired into single + survey runs. `VerifyEvidence` records `counterexample` / `shrunkCounterexample` / `seed` / `regressionTestPath` as **additive optionals — no schema bump** (synthesized `decodeIfPresent`, per the `excludedActionCount` precedent; supersedes the earlier "schema v2 + migration" plan). **The `#assertIdempotent` adopter-idiom emission was rejected as a category error** — see the deliverables note: `#assertIdempotent` is retry-idempotence (`body() == body()`), not the template's algebraic `f(f(x)) == f(x)`, so it can't capture an algebraic counterexample.
- **v1.143 — replay corpus (shipped A+B).** `Sources/SwiftInferCore/VerifyCorpus.swift` + `VerifyCorpusStore` at `.swiftinfer/verify-corpus.json`; entry `{ identityHash, template, counterexample, shrunkCounterexample, seed, capturedAt, swiftInferVersion }`. **Note vs. original plan:** the verify path uses its own deterministic `SeedHex` (derived from the identity hash), *not* SwiftPropertyLaws' `Seed`/`Environment` — so the corpus stores the colon-joined `SeedHex` hex and there's no `Environment` field / `ReplayEnvironmentMismatch` guard (the seed is reproducible by construction). **A:** the corpus *accumulates* distinct counterexamples (dedup by identity+counterexample, first-seen wins; never dropped), populated from the single-suggestion pipeline on `.defaultFails`. **B:** `verify --replay-only` re-verifies each recorded identity (deterministic seed reproduces it), classifies still-failing / now-holding / inconclusive / skipped, prints a summary, and **exits non-zero while any recorded counterexample still fails** (CI regression gate; the "now-holds" classification *is* the "property behavior changed" signal). **Deferred:** survey-mode (`--all-from-index`) batch corpus population; replaying the corpus *inline* at the start of a normal verify run (before fresh trials) rather than only via `--replay-only`.
- **v1.144 — reporting + drift (shipped).** **A:** the verify counterexample output is one unified block — shrink trace (from the renderer), regression-test path, replayable seed, and corpus count + re-verify/`--replay-only` hints. (Tier transition is a discover-side concern — the single-suggestion verify pipeline doesn't mutate tiers — so it's not in this block.) **B:** `drift` emits a corpus-summary diagnostic (count + `--replay-only` pointer). **Reframed from the original plan:** "new corpus entries since baseline" isn't computable — `baseline.json` snapshots *suggestions*, not the corpus, and verify-disproven picks are vetoed out of the drift set (v1.68) so their counterexamples live only in the corpus; the regression check is `verify --replay-only`, which drift now points at.

## SwiftIdempotency tie-in (W6 — companion doc)

SwiftIdempotency is the **adopter-facing** layer: `#assertIdempotent` / `@IdempotencyTests` run `f` twice on *one fixed value* (`__idempotencyAssertRunTwice`) — no generation, no shrinking, no replay. **Crucially, that's *retry* idempotence (`body() == body()`), not the SwiftInfer idempotence template's *algebraic* `f(f(x)) == f(x)`** — the two are different properties. So the auto-bridge does **not** emit `#assertIdempotent` from a SwiftInfer idempotence counterexample (it would pass and miss the bug). This epic's remaining SwiftIdempotency work:

1. **W6 — `SwiftIdempotencyPropertyBased`** (SwiftIdempotency v0.4.0), an opt-in product giving property-based, shrinking, replay-corpus-backed *retry*-idempotency assertions — its own concern, serving SwiftIdempotency's notion (effects don't accumulate across retries), reusing v2.6 shrinking + the v1.143 corpus types. Depends only on v2.6 + v1.143 corpus → can proceed in parallel with v1.142–v1.144.

Full design: `swiftIdempotency/docs/property-based-assertions-design.md`.

## Adjacent gap (fast-follow, NOT this epic)

RefactorBridge emits kit-protocol conformances (`Semigroup`, etc.) but **does not emit `@Idempotent` / `@ExternallyIdempotent`** when an idempotence suggestion verifies — so a `verified` idempotence property never becomes a durable marker that SwiftEffectInference / SwiftProjectLint would then enforce. "verified ⇒ emit the marker macro" connects the inference loop to the lint loop. Track separately.

## Scope boundaries

- **In scope**: shrinking (kit, closure-seam), shrinkable carrier generators, verify→regression auto-bridge, replay corpus, opt-in SwiftIdempotency PBT product, unified reporting.
- **Deferred**: stateful / model-based testing beyond the W6.C effect-sequence seed (that is epic #2); coverage/classification of generator distribution (`classify`/`collect`); RefactorBridge marker emission (fast-follow above); non-tuple structural shrinkers beyond what `Shrink.*` + `DerivationStrategist` already cover.
