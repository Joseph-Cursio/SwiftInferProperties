# v1.44 Calibration Plan — Phase 1 step 3 (template + carrier coverage)

Working doc for the v1.44 cut (forty-first calibration cycle). **Third of 3 cycles** delivering the test-execution-evidence architectural shift. Where v1.42 shipped the minimum-viable verify pipeline (round-trip × `Complex<Double>`) and v1.43 layered the two-pass edge-case-biased design, v1.44 widens **template coverage** beyond round-trip and **carrier coverage** beyond `Complex<Double>`, then takes the **first calibration measurement** of the verify-mode lift over cycle-27's 72.4% baseline.

> **Motivation.** The v1.42+v1.43 verify pipeline closes the cycle-27 finding ("name + type heuristics asymptote at 72.4%") only for round-trip on `Complex<Double>` — a thin slice of the 109-surface. The 8 residual reject picks on that surface include false-positives on idempotence (`formIntersection`-shaped methods that pattern-match idempotence but mutate state non-idempotently) and commutativity (`binomial(n:k:)` matching by `(T,T) -> T` shape but failing `C(n,k) == C(k,n)`). To validate the verify-mode design at scale, v1.44 widens coverage and runs the first empirical measurement of the lift.

## Scope-of-cycle decision: idempotence + Double/Int, not the kitchen sink

The CLAUDE.md v1.44 priority entry reads "extend verify to idempotence / commutativity / dual-style; widen carrier set beyond `Complex<Double>`." All three templates plus all four+ new carriers in one cycle is **~4-5 cycles of work realistically scoped** — each template needs its own stub emitter (paralleling `RoundTripStubEmitter`), and each carrier needs its own generator-selection path. v1.44 scopes to the highest-leverage subset:

**Template choice — idempotence first.** Closest shape to round-trip (single function `f: T -> T`, single equality check `f(f(x)) == f(x)`). Reuses the round-trip subprocess + parser + renderer scaffolding with minimal change. Commutativity needs two-argument generation harness (`Gen<T>.zip(...)`); dual-style needs *two functions* (mutating + value-returning) plus result-equality threading — both warrant their own cycles.

**Carrier choice — `Double` + `Int`.** Highest-volume in real codebases. The kit already ships `Gen<Double>` + `Gen<Double>.doubleWithNaN()` (which is the natural edge-case-biased analog for the FP case) + `Gen<Int>.int(in:)` + `Gen<Int>.boundedForArithmetic()` (which prevents overflow on triple multiplication). No new kit-side work required. Deferred for v1.45+: `Float`, `String`, `Complex<Float>`, custom user types.

## What v1.44 ships

Five workstreams + closeout:

1. **V1.44.A — `IdempotenceStubEmitter`**. Mirrors `RoundTripStubEmitter`: a pure-function source emitter that produces a SwiftPM-runnable property test for `f: T -> T`. Property: `f(f(x)).isApproximatelyEqual(to: f(x))` for FP carriers, `f(f(x)) == f(x)` for non-FP. Two-pass design carried forward from v1.43 (default + edge-case-biased on FP; default-only on Int since the kit doesn't ship an edge-case-biased integer generator and integer arithmetic doesn't have the NaN/Inf semantic). New marker namespaces (same shape — `VERIFY_DEFAULT_*` / `VERIFY_EDGE_*` — so the V1.43.C parser is reusable).

   **Supported carriers in V1.44.A**: `Complex<Double>` (kit's `edgeCaseBiased()`), `Double` (kit's `doubleWithNaN()`), `Int` (kit's `boundedForArithmetic()`; single-pass — no edge-case-biased generator yet).

2. **V1.44.B — Carrier dispatch in `RoundTripStubEmitter`**. The v1.42/v1.43 emitter hard-codes `Complex<Double>`. V1.44.B extends it to dispatch on the carrier type:
   - `Complex<Double>`: existing v1.43 two-pass shape (default + edge-case-biased).
   - `Double`: two-pass with `Gen<Double>` (default) + `Gen<Double>.doubleWithNaN()` (edge pass — already injects NaN at ~5% per the kit's M4 convention).
   - `Int`: single-pass with `Gen<Int>.boundedForArithmetic()` (no edge pass — integers don't have NaN/Inf; the v1.42 default-only outcome shape applies). Result outcome enum sees `.bothPass` collapse to `.defaultPass` when the edge pass is unsupported; renderer text reflects single-pass.

   `RoundTripStubEmitter.supportedCarriers` expands from `["Complex<Double>"]` to `["Complex<Double>", "Double", "Int"]`. Carrier-out-of-set still throws `unsupportedCarrier(carrier:expected:)` — the surface contract is stable, the supported set just grows.

3. **V1.44.C — `IdempotenceStubEmitter` carrier dispatch**. Same carrier set as V1.44.B (`Complex<Double>`, `Double`, `Int`) with the same generator-selection logic. Emitted stub shape:

   ```swift
   for trial in 0 ..< trials {
       let value = defaultGenerator.run(using: &rng)
       let onceResult = <funcCall>(value)
       let twiceResult = <funcCall>(onceResult)
       if !twiceResult.isApproximatelyEqual(to: onceResult) {  // or == for Int
           print("VERIFY_DEFAULT_RESULT: FAIL")
           // ... per-trial markers, exit 1
       }
   }
   ```

   The property text differs from round-trip (`f(f(x))` instead of `reverse(forward(x))`), but the marker contract is identical so `VerifyResultParser` + `VerifyResultRenderer` from v1.43 work unchanged.

4. **V1.44.D — `Verify` CLI dispatch by template**. The v1.42 `Verify.runPipeline` always calls `RoundTripStubEmitter`; v1.44 looks at the suggestion's `templateName` and dispatches:

   ```
   templateName == "round-trip"   → RoundTripStubEmitter
   templateName == "idempotence"  → IdempotenceStubEmitter
   otherwise                      → VerifyError.unsupportedTemplate(name:expected:)
   ```

   The `IdempotencePairResolver` (analog of `RoundTripPairResolver`) maps a suggestion's evidence to the single-function call expression. Idempotence is single-function so no curated pair list — the resolver just reads `evidence[0].displayName` and emits the qualified call expression.

5. **V1.44.E — Tests** (similar shape to v1.43.E):
   - **E.1** Unit tests of `IdempotenceStubEmitter` (~12 tests) — mirrors `RoundTripStubEmitterTests` covering marker emission, carrier validation, seed rendering, trial-budget literals.
   - **E.2** Unit tests of `RoundTripStubEmitter` carrier dispatch (~6 tests) — assert `Complex<Double>` emits the v1.43 two-pass shape, `Double` emits the doubleWithNaN-driven edge pass, `Int` emits the single-pass shape.
   - **E.3** Integration tests (subprocess-based; `.integration` tag):
     - **E.3.a** Idempotence × `Complex<Double>` × `bothPass` — `Complex.exp(Complex.exp(z))` does NOT equal `Complex.exp(z)` so this picks a passing case instead: `forward = { z in z * Complex(0, 0) }` is idempotent on all finite inputs (returns 0; the edge-case-biased pass surfaces the NaN/Inf-input failure as `.edgeCaseAdvisory`). Pin `.edgeCaseAdvisory`.
     - **E.3.b** Idempotence × `Double` × `defaultFails` — `{ x in x * 2 }` is non-idempotent on all non-zero inputs; pin `.defaultFails` at trial 0.
     - **E.3.c** Idempotence × `Int` × `bothPass` — identity `{ x in x }` over Int; pin `.bothPass` (single-pass collapse — edge pass not run for Int).

   Aggregate: ~21 new tests. Test count goal: 2178 → ~2200.

6. **V1.44.F — Calibration measurement protocol + findings doc**. The v1.44 deliverable that *closes the test-execution-evidence Phase 1 narrative*. Protocol:

   - **Sample**: the cycle-27 corpus (29 Possible-tier picks; baseline 21 Accept / 8 Reject = 72.4%). Already captured in `docs/calibration-cycle-27-data/`.
   - **Per-pick verify invocation**: for each pick where templateName ∈ {`round-trip`, `idempotence`} AND carrier ∈ {`Complex<Double>`, `Double`, `Int`}, run `swift-infer verify --suggestion <hash> --budget standard`. (Budget=standard for higher confidence on the calibration measurement; v1.45+ accept-flow integration will use this.)
   - **Per-pick outcome classification**:
     - `.bothPass`: re-classifies the pick as **HIGH-CONFIDENCE ACCEPT** (strong evidence the property holds).
     - `.edgeCaseAdvisory`: the pick is accept *with an advisory* — still counts toward "Accept" in the calibration tally but the advisory note becomes part of the suggestion's caveat.
     - `.defaultFails`: re-classifies the pick as **VERIFY-CONFIRMED REJECT** — even if name+type heuristics said Accept, the property doesn't hold. Verify-mode catches the false-positive.
     - `.error` or carrier/template out of v1.44 scope: pick is **unverified** — falls back to the cycle-27 manual verdict.
   - **Tally**: re-compute the acceptance rate using the verify outcomes where available. The expected lift comes from `.defaultFails` outcomes converting prior Accepts into Rejects (sharpening precision) AND from `.bothPass` outcomes adding evidentiary weight to prior Accepts (no rate change but increases confidence). The expected suppression comes from any prior Rejects that verify-confirms.
   - **Findings doc**: `docs/calibration-cycle-28-findings.md` — first verify-mode measurement, table per template × carrier, projected lift, residual gaps.

   **Out-of-scope for the measurement protocol**: re-running verify against the *full* 109-surface (only the 29 sampled). That's a v1.45+ exercise once the accept-flow integration ships.

7. **V1.44.G–I** bundled closeout — perf baseline (`docs/perf-baseline-v1.44.md`), version → 1.44.0, CLAUDE.md "Repository state" pointer, tag + push.

## Why idempotence first, not commutativity or dual-style

Three reasons in order of weight:

1. **Shape similarity.** Idempotence `f: T -> T` is structurally identical to round-trip's single-function side — the stub emitter is a near-copy of `RoundTripStubEmitter` with a different equality check. Commutativity needs two-argument generation (`Gen<T>.zip(Gen<T>, Gen<T>)` pattern), and dual-style needs two functions plus result-equality threading. Both are real work but their scaffolding cost is higher per template.

2. **Volume on the cycle-27 surface.** Cycle-27 sampled 9 idempotence Accept + 1 idempotence Reject, vs 7 commutativity Accept + 0 Reject and 1 dual-style Accept + 0 Reject. Idempotence has the most picks where verify-mode could move the needle.

3. **Carrier overlap.** Idempotence picks on the cycle-27 surface use `Complex<Double>`, `Double`, and a few `String` carriers — the first two are exactly the V1.44.B set, so V1.44.E.3.a/b's integration coverage matches the calibration corpus. Commutativity picks include `Int` for `binomial`-shaped cases (the canonical false-positive) but also user types we can't verify without semantic resolution.

## Carrier choice rationale: Double + Int over Float / String / Complex<Float>

- **`Double`**: most common FP type in user code; kit ships `Gen<Double>.doubleWithNaN()` (the natural edge-case-biased analog — NaN at ~5% per trial); 0 new kit work.
- **`Int`**: most common integer type; kit ships `Gen<Int>.boundedForArithmetic()` (handles overflow safely); single-pass since integer arithmetic doesn't have NaN/Inf so the two-pass design's value diminishes.
- **Deferred to v1.45+**:
  - **`Float`**: kit ships `Gen<Float>.floatWithNaN()` — adding it is a one-line change but doesn't expand the calibration corpus meaningfully (Float is rare in real Swift code outside graphics APIs).
  - **`String`**: kit ships `Gen<String>` but the edge-case semantics differ (no NaN/Inf; the edges are empty string, single grapheme cluster, max-length Unicode-rich, etc.). Would need its own analog of `edgeCaseBiased()` from the kit — non-trivial.
  - **`Complex<Float>`**: the kit's `complexEdgeCases` is `Complex<Double>` only; extending requires `Gen<Complex<Float>>.edgeCaseBiased()` upstream.
  - **User types**: requires `DerivationStrategist`-driven generator inference at verify-time, which the v1.42 pipeline doesn't support (the kit's strategist is consulted only at writeout time today).

## Determinism caveat

Inherited from v1.43 — the kit's `doubleWithNaN()` uses non-seeded `Double.random(in:)` for the finite-path slice (same partial-determinism story as `edgeCaseBiased()`). The v1.44 Int carrier path is fully deterministic since `Gen<Int>.boundedForArithmetic()` uses only seeded `Gen<Int>.int(in:)`. The findings doc per V1.44.F notes any seed-dependent outcomes when the calibration corpus surfaces them.

## Scope boundaries

- **In scope (v1.44):** Idempotence template support; carrier extension to `Double` + `Int`; CLI dispatch by template; first calibration measurement on the cycle-27 corpus.
- **Out of scope, deferred to v1.45:** Commutativity template support; dual-style-consistency template support; `Float` / `String` / `Complex<Float>` carriers; user-type carriers via `DerivationStrategist`; accept-flow integration (verify outcomes flowing into `decisions.json`); verification cache.
- **Out of scope, deferred indefinitely:** Distributed verify, parallel verify, the "Verified" first-class tier as a signal class on Suggestion, sqlite-backed verification cache (Phases 2+).
- **Not changing:** Any v1.43-and-earlier verify pipeline behavior for `round-trip` × `Complex<Double>`. The V1.44.B carrier dispatch preserves the v1.43 two-pass shape for `Complex<Double>` bit-for-bit (same generator, same markers, same outcomes).

## Risks

1. **`doubleWithNaN()` doesn't preserve the edge-case-classification surface.** The kit's `Gen<Double>.doubleWithNaN()` injects NaN at ~5% but doesn't expose a `complexEdgeCases`-style curated list. So the `VERIFY_EDGE_INDEX` field can only resolve to `0` (NaN matched) or `-1` (non-curated value). **Mitigation**: V1.44.B emits a per-carrier curated list (just `[Double.nan]` for Double — single entry); the renderer's edge-case-#N classification falls back to "edge case #0 (NaN)" or "non-curated value". Acceptable for v1.44 — v1.45 can extend if the calibration motivates.

2. **`Int` carriers don't have a meaningful edge pass.** Integer arithmetic on bounded values can still overflow (`Int.max + 1` traps), but `Gen<Int>.boundedForArithmetic()` already bounds the magnitude to prevent that. There's no "boundary class" analogous to NaN/Inf for integers. **Mitigation**: V1.44.B's Int path is single-pass — the renderer reports `.defaultPass` (a new VerifyOutcome case? or recycle `.bothPass` with `edgeTrials == 0`?). **Sub-decision** (TBD): recycle `.bothPass` with `edgeTrials = 0`, `edgeSampled = 0` to keep the enum shape stable. Renderer reads `edgeTrials == 0` and emits "(integer carrier — edge pass not applicable)" instead of the "(N / 12 curated edge cases sampled)" line.

3. **Cycle-28 measurement may produce a small lift.** If the cycle-27 sample's 8 Reject picks include only 1-2 that fall in the {idempotence, round-trip} × {Complex<Double>, Double, Int} scope, the measured lift may be 1-2 picks → ~3-7pp on n=29. **Mitigation**: V1.44.F's findings doc reports the lift *and* the verifiable-fraction of the corpus. If verifiable fraction < 50%, the measurement is informational; if ≥ 80%, the lift estimate is high-confidence. The doc explicitly states the precision interval.

4. **Calibration measurement requires manual verify invocation per pick.** v1.44 doesn't ship batch / scripted verify — each of the ~20+ picks needs `swift-infer verify --suggestion <hash>` individually. **Mitigation**: V1.44.F includes a shell-script harness (`scripts/cycle-28-verify-sweep.sh`) that loops over the corpus and aggregates outcomes. The script is throwaway tooling, not part of the shipped CLI surface.
