# v1.33 Calibration Plan — SemanticIndex (PRD §20.1)

Working doc for the SwiftInferProperties v1.33 cut. v1.33 is the **thirtieth calibration cycle** and the **third design-completion release** (after v1.31 FP approximate-equality + v1.32 Domain Template Packs). Closes **PRD §20.1 SemanticIndex** — a persistent, queryable index of inferred properties across runs.

> **v1.33 ships JSON-backed storage as the v1 form.** The PRD sketches SQLite (`.swiftinfer/index.sqlite`) but the storage format is implementation detail. JSON keeps scope contained (no new dependency, no schema-migration design upfront), reuses the project's existing JSON-Codable infrastructure, and the SQLite upgrade is a non-breaking format-swap for a future cycle when query complexity warrants it.

**Ephemeral** — archive to `docs/archive/v1.33 Calibration Plan.md` once tagged.

## Why this work

Per PRD §20.1, the SemanticIndex is "a semantic lens over a Swift codebase that reveals latent algebraic structure" — useful for:
- API design feedback ("you have three monoids; consider unifying them under a custom Monoid protocol")
- Refactoring suggestions ("this function would slot into the existing semigroup chain")
- Documentation generation ("inferred properties of this type, exported as DocC")
- Integration with project linting

Today the per-run `.swiftinfer/decisions.json` records every accept/reject/skip but doesn't persist the inferred-property metadata that surfaced the decision. The SemanticIndex bridges that gap: every accepted suggestion populates an index row that survives across runs and is queryable.

## V1.33.A — `SemanticIndexEntry` data model

New file `Sources/SwiftInferCore/SemanticIndexEntry.swift`. Pure value type, `Codable` + `Sendable` + `Equatable`.

**Schema** (per PRD §20.1 with sensible additions):

```swift
public struct SemanticIndexEntry: Codable, Sendable, Equatable {
    public let identityHash: String        // suggestion identity hex (key)
    public let templateName: String        // e.g. "round-trip"
    public let typeName: String?           // carrier type; nil for free functions
    public let score: Int                  // suggestion total score
    public let tier: String                // "Strong" | "Likely" | "Possible"
    public let primaryFunctionName: String // first-evidence function display name
    public let location: String            // "<file>:<line>"
    public let decision: String?           // "accept" | "reject" | "skip" | nil (untriaged)
    public let decisionAt: String?         // ISO8601 from decisions.json
    public let firstSeenAt: String         // ISO8601 — when indexer first saw this signature
    public let lastSeenAt: String          // ISO8601 — most recent index run
}
```

The PRD sketches `(typeId, templateId, score, evidenceJson, decisionAt, lastSeenAt)`. The schema above expands `typeId → typeName`, adds `primaryFunctionName` + `location` for human-readable output, splits `evidenceJson` into the structured columns the rest of the codebase already produces, and adds `firstSeenAt` for "when did this signature appear" queries.

**Tests**: `SemanticIndexEntryTests.swift` — verify JSON round-trip, key-equality on identityHash, encoding stability.

## V1.33.B — `IndexStore` JSON-backed persistence

New file `Sources/SwiftInferCLI/IndexStore.swift`. Pure I/O wrapper over `.swiftinfer/index.json`.

**API**:

```swift
public enum IndexStore {
    public static let conventionalRelativePath = ".swiftinfer/index.json"

    public struct Index: Codable, Sendable, Equatable {
        public var entries: [SemanticIndexEntry]   // keyed by identityHash; deduplicated on upsert
        public var schemaVersion: Int              // 1 in v1.33
        public var updatedAt: String               // ISO8601
    }

    public static func load(from path: URL) throws -> Index
    public static func save(_ index: Index, to path: URL) throws
    public static func upsert(
        _ entries: [SemanticIndexEntry],
        into existing: Index,
        atRunTimestamp: String
    ) -> Index
}
```

**Upsert semantics**: existing entries (matched on `identityHash`) update `score`/`tier`/`decision`/`decisionAt`/`lastSeenAt` in-place but preserve `firstSeenAt`. New entries get `firstSeenAt = lastSeenAt = atRunTimestamp`. Entries no longer present in the current discover state are NOT pruned — historical entries are valuable for "what disappeared since last index" queries (v1.34+ feature).

**Schema versioning**: explicit `schemaVersion: Int = 1` so v1.34+ can detect older index files and migrate. Missing schemaVersion in an older index file (none exist today) implicitly maps to v1.

**Tests**: `IndexStoreTests.swift` — round-trip JSON, upsert preserves firstSeenAt, missing file returns empty index, malformed file emits warning + returns empty (per the ConfigLoader convention).

## V1.33.C — `swift-infer index` subcommand

New subcommand parallel to `Discover`, `Drift`, `Metrics`. Reuses pipeline infrastructure.

**Behavior**:
1. Resolve package root + `.swiftinfer/decisions.json` + `.swiftinfer/index.json`.
2. Run discover via `collectVisibleSuggestions` (same path as `swift-infer discover`).
3. Read `decisions.json` if present; build identityHash → DecisionRecord map.
4. For each visible suggestion: build a `SemanticIndexEntry`. Decision fields are nil when no decision recorded.
5. Load existing index, upsert, save.
6. Emit a one-line summary: "Indexed N suggestions (M new, K updated)."

**CLI flags**:
- `--target X` (same as discover)
- `--dry-run` (don't write the index; report counts)
- `--config / --vocabulary / --packs / --test-dir / --include-possible` (inherit from discover pipeline)

**No incremental analysis in v1.33.** The index rebuilds from a full discover each run. PRD §20.1 mentions incremental as a future optimization; deferred until profiling shows it's needed.

**Tests**: integration test that runs `index` against a synthetic corpus, verifies file written, re-runs `index` and verifies upsert preserves firstSeenAt.

## V1.33.D — `swift-infer query` subcommand

Basic-filter form of the PRD's "human-readable queries" concept. No natural-language DSL in v1.33; that's a future cycle once we have field experience with what queries matter.

**Behavior**:
1. Load `.swiftinfer/index.json` (or error if absent — user must run `index` first).
2. Apply filters from CLI flags.
3. Render the matching entries to stdout (one entry per line; sortable by score descending).

**CLI flags**:
- `--template <name>` — filter by templateName equality
- `--type <name>` — filter by typeName equality (or `"none"` for free functions)
- `--tier <Strong|Likely|Possible>` — filter by tier
- `--decision <accept|reject|skip|untriaged>` — filter by decision (untriaged = decision is nil)
- `--min-score <N>` — filter by score lower bound
- `--limit <N>` — cap output

Flag combinations AND together. No `--query` natural-language flag in v1.33.

**Tests**: query against a fixture index; verify each filter narrows correctly; verify AND combination.

## Scope boundaries

- **In scope**: JSON-backed persistence at `.swiftinfer/index.json`; `index` build subcommand; `query` filter subcommand; full-rebuild semantics (no incremental).
- **Out of scope**: SQLite backend (non-breaking upgrade later when query complexity warrants).
- **Out of scope**: Incremental analysis (only re-analyze files changed since last index). The full rebuild is fast enough on cycle-1..14 corpora (<1s per discover).
- **Out of scope**: Natural-language query DSL. The PRD sketches `swift-infer query 'monoids in MyApp'`; v1.33 ships structured flag-based filters; the parser is a future cycle informed by field experience.
- **Out of scope**: Cross-pack pruning, deduplication-across-runs heuristics, DocC export. Each is a discrete v1.34+ feature.

## Risk + mitigation

- **Risk**: JSON file grows unbounded as historical entries accumulate. **Mitigation**: explicit non-pruning is documented as intentional; v1.34+ can add `swift-infer index --prune` for the user-controlled lifecycle.
- **Risk**: schemaVersion drift makes old indexes unreadable. **Mitigation**: schemaVersion is explicit from v1.33; future versions detect-and-migrate.
- **Risk**: index file conflicts with `decisions.json` on the same key. **Mitigation**: identityHash is the join key; both files reference the same hash space. No conflict — index ≈ decisions ⊕ template metadata.
- **Risk**: PRD specified SQLite; users may expect it. **Mitigation**: CHANGELOG calls out the JSON-first form explicitly; migration path is documented as non-breaking.
