# v1.42 Calibration Plan — Test-Execution-Evidence Phase 1, Step 1

Working doc for the v1.42 cut (thirty-ninth calibration cycle). **First of ~3 cycles** delivering the test-execution-evidence architectural shift the user raised back at v1.25 (now finally landing — see `docs/calibration-cycle-38-findings.md` priority #4 and the upstream proposal at `docs/ideas/Edge-Case-Biased Generators Kit Proposal.md`).

> **Motivation.** Cycle-27's 72.4% acceptance rate is the asymptote of pure name-+-type heuristics. The remaining ~8 reject picks on the 109-surface are type-pattern false-positives that name-based reasoning structurally can't close (`binomial(n:k:)` matches commutativity by `(T, T) -> T` shape but fails `C(n,k) == C(k,n)`; etc.). **Test-execution evidence** — compile + run the candidate property test on N random inputs, treat result as a high-confidence signal — punches through the ceiling. v1.42 ships the minimum viable verify pipeline; v1.43 adds the edge-case-biased second pass; v1.44 extends template coverage.

**Ephemeral** — archive to `docs/archive/v1.42 Calibration Plan.md` once tagged.

## What v1.42 ships

Five workstreams + bundled closeout:

1. **V1.42.A — `Package.swift` kit pin bump** `from: "2.0.0"` → `from: "2.1.0"`. Standalone change, lowest risk; lands first so subsequent workstreams can `import PropertyLawComplex` if they want to. *(The v1.42 verify pipeline only uses the default generator; `PropertyLawComplex` becomes load-bearing in v1.43. Pin-bumping early avoids interleaving the dep update with verify wiring.)*

2. **V1.42.B — `swift-infer verify` subcommand surface**. Add to the existing `SwiftInferCommand` subcommand list (alongside `Discover`, `Drift`, `ConvertCounterexample`, `Metrics`, `Index`, `Query`, `SuggestRefactors`). Surface:

   ```
   swift-infer verify --suggestion <id> [--target <name>] [--budget <small|standard>]
   ```

   - `--suggestion <id>`: required. Matches the **prefix** of `SuggestionIdentity.hash` (the §7.5 stable hash). Prefix-match because users will copy from the explainability block in `discover` output, which renders a short prefix.
   - `--target <name>`: optional; defaults to `--target` resolution from `.swiftinfer/config.toml` then walks up for `Package.swift` (mirroring `discover`).
   - `--budget`: `small` (N=100) for fast iteration, `standard` (N=1000) for the higher-confidence verify decision. **Default `small`** for v1.42 — keeps a single verify call under ~5s for the round-trip-only scope, matches the opt-in / human-exploration posture of the verify gesture. `standard` is the budget the v1.45+ accept-flow integration will use, where slower-but-higher-confidence makes sense.

   The subcommand is **opt-in** (PRD §3.5 conservative-inference posture). Nothing in `discover` / `drift` / `accept` changes — verify is a separate human gesture.

3. **V1.42.C — Verify pipeline + harness**. The actual subprocess-based compile + run. Decomposes into:

   - **C.1** — `VerifyHarness.swift` in `SwiftInferCLI`. Looks up the suggestion by hash prefix using the existing `IndexStore` (v1.33). Hash-prefix matching is case-insensitive and tolerates / strips a leading `0x`. Failure modes surface specific `VerifyError` cases: `.indexMissing` (file absent), `.indexEmpty` (zero entries), `.suggestionNotFound(prefix:closest:)` (with nearest hashes named), `.ambiguousPrefix(prefix:matches:)` (with all matched hashes named). Stale-index detection (any tracked source file's mtime is newer than the index file's) emits a warning but proceeds with the lookup.
   - **C.5 (deferred)** — implicit reindex on demand. **Not landing in V1.42.C.1** because it requires hoisting `IndexCommand.run`'s discover → upsert → save pipeline out of `AsyncParsableCommand.run`'s instance method into a callable static. That `IndexCommand`-side refactor is its own load-bearing change and would interleave with the verify-pipeline wiring otherwise. Until V1.42.C.5 lands, the `.indexMissing` error directs the user to `swift-infer index --target <X>` to bootstrap.
   - **C.2** — Property-test stub synthesis. For round-trip only in v1.42: emit a `PropertyLawKit`-flavored `Gen<Complex<Double>>` round-trip property using the suggestion's `(forward, reverse)` pair from `evidence`. Reuses the existing `roundTripTestEmitter`'s output shape (M3.3) but writes to a *temp directory*, not `Tests/Generated/`. **Important:** v1.42 only supports `Complex<Double>` carriers (matches the kit's Phase 1 generator scope). Suggestions whose carrier isn't `Complex<Double>` get an explicit `unsupportedCarrier` error — no half-baked attempts.
   - **C.3** — Subprocess spawn. Create a throwaway SwiftPM verifier package in `.swiftinfer/verify-workdir/<hash-prefix>/` containing: (a) the synthesized stub, (b) a `Package.swift` depending on the user's package + `swift-numerics` (`PropertyLawKit` is already available via the user's existing pin), (c) a runner main that invokes the property check at the chosen budget and exits 0 / 1 / 2 for pass / fail / error. Build via `swift build` in that subdir; run the resulting binary; capture stdout/stderr.
   - **C.4** — Result rendering. Three outcomes for v1.42 (the four-outcome two-pass table lands in v1.43):

     ```
     ✓ verify holds: round-trip foo/bar over Complex<Double>, 100 trials, all pass
     ✗ verify fails: round-trip foo/bar over Complex<Double>, counterexample at trial 47:
         input  = Complex(0.0042, -1.7e6)
         forward(input)  = ...
         reverse(forward(input)) = ...
         expected ≈ input (within Complex.==)
     ! verify error: <description> [build error | runner crash | unsupported carrier]
     ```

4. **V1.42.D — Tests**. Three areas:

   - **D.1** Unit tests of `VerifyHarness.lookupSuggestion(prefix:)` — exact match, prefix match, ambiguous-prefix error, no-match error, missing-index-file error.
   - **D.2** Integration test against a known-good round-trip pair (e.g. `Complex.exp` / `Complex.log` from `swift-numerics`; well-formed and finite-domain pass). Exercises the full subprocess pipeline. Marked `.integration` so CI can disable if it's slow.

## Subprocess strategy — why not in-process?

The v1.42 candidate-property test executes **the user's own type code** (the function the suggestion is about). Running that in-process inside `swift-infer` would require:

- Dynamic-loading the user's `.dylib` (fragile on cross-platform; relies on the user having built their package first; SwiftPM's build products live in version-specific paths).
- Compiling the synthesized stub *and* the user's source against a shared runtime — basically reimplementing `swift run` poorly.

Subprocess-based execution side-steps all of that: we synthesize a self-contained SwiftPM package that depends on the user's package as a local path, invoke `swift build`, and run the binary. The cost is per-verify-call startup (~1–2s for an incremental `swift build` after the first one). For v1.42's opt-in `verify` gesture that's acceptable; v1.43+ may layer caching on top.

## Why round-trip only

Round-trip is the highest-value template for verify mode in cycle-23/27 measurement: cycle-27 round-trip accept-rate was 85.7% (the highest of any template that round); the residual 14% rejects are exactly the type-pattern false-positives verify is designed to close. Round-trip also has the cleanest property shape (one-input, one-output, `reverse(forward(x)) == x`), so subprocess synthesis is straightforward. Other templates need more nuance:

- **Idempotence non-lifted** needs `f(f(x)) == f(x)` — also straightforward, but pulls in non-`Complex<Double>` carriers (`Array.sort`, etc.) v1.42's generator scope doesn't cover.
- **Commutativity** needs two-argument generation — `Gen<Complex<Double>>.zip(...)` pattern, fine but more harness scaffolding.
- **Dual-style-consistency** needs *two functions* (mutating + value-returning) verified to produce the same result — multi-argument harness + result-equality threading.

v1.44 picks these up in priority order once the v1.42 harness scaffolding is in place.

## Why default-pass only (not two-pass)

The two-pass design (default generator pass + `edgeCaseBiased` pass) lands in v1.43 because:

1. The subprocess harness in v1.42 is already a significant new surface. Stabilizing it with a single-pass design first reduces risk.
2. The 4-outcome reporting table needs result aggregation across two subprocess runs — better to add once the single-run path is proven.
3. v1.43's edge-case pass uses the freshly-shipped `PropertyLawComplex` product, which itself just landed (`SwiftPropertyLaws v2.1.0`, commit `ba19ab7`, 2026-05-11). Letting it bake on its own for one cycle before depending on it from a verify-mode harness is the conservative move.

Result: v1.42 verify outcomes are 3-way (`pass` / `fail` / `error`); v1.43 expands to the 4-way two-pass table from the proposal §2.2.

## Scope boundaries

- **In scope (v1.42):** opt-in `verify` subcommand, subprocess-based round-trip verify on `Complex<Double>` carriers, default generator only, hash-prefix suggestion lookup, kit pin bump, 3-way result reporting.
- **In scope soon (v1.43):** two-pass design, `Gen<Complex<Double>>.edgeCaseBiased()` second pass, 4-outcome reporting, counterexample edge-case classification.
- **In scope soon (v1.44):** template extension (idempotence + commutativity + dual-style), carrier extension beyond `Complex<Double>`, Phase 1 calibration measurement.
- **Out of scope (deferred to Phase 2+):** sqlite-backed verification cache, accept-flow integration (currently verify is a separate gesture; verified suggestions don't flow into `decisions.json` yet), the "Verified" tier as a first-class signal class, distributed verify, parallel verify.
- **Not changing:** any v1.41-and-earlier suggestion ranking, scoring, or surface composition. v1.42 adds a *separate* gesture; cycle-27's 72.4% measurement still holds for the unverified surface.

## Risks

1. **SwiftPM subprocess fragility.** Local-path dependencies + dynamic Package.swift synthesis may surface unexpected interactions with the user's Package.resolved / SwiftPM cache. Mitigation: V1.42.C.3 spawns SwiftPM in an isolated temp dir; V1.42.D integration tests pin a known good shape.
2. **Build time per verify.** First-call cold `swift build` of a synthesized verifier package may be 30s+. Mitigation: V1.42.C.3 designs the workdir layout so SwiftPM's incremental build cache persists across runs; subsequent calls should be 1–3s. Phase 3 (cache) tightens this further but isn't required for v1.42 utility.
3. **Carrier-detection edge cases.** "Is this suggestion's carrier `Complex<Double>`?" needs to be a robust check. Mitigation: V1.42.C.2 uses the suggestion's `carrier` field (V1.34.A) directly — already a verified string in the SemanticIndex.
