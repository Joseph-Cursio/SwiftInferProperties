These are the cycle-6 picks #45-#47 sub-pattern (3 of 5 inverse-pair rejections), expanded to 6 by cross-file pairing across `OrderedSet+Partial SetAlgebra intersection.swift` × `OrderedSet+Partial SetAlgebra subtracting.swift` × `OrderedSet+UnorderedView.swift`.

The mathematical truth: `intersection` then `subtracting` doesn't recover the original input. Neither does any pair drawn from the SetAlgebra binary-op surface. These are *operations* on a SetAlgebra, not *inverses*.

### Why the existing V1.5.2 protocol-coverage veto misses these

Per `Sources/SwiftInferTemplates/InversePairTemplate.swift:205-214`:

```swift
private static func protocolCoverageVeto(
    for pair: FunctionPair,
    inheritedTypesByName: [String: Set<String>]
) -> Signal? {
    ProtocolCoverageMap.coverageVetoSignal(
        forTypeText: pair.forward.parameters.first?.typeText,
        inheritedTypesByName: inheritedTypesByName,
        candidateProperties: [.additiveInverse, .groupInverse]
    )
}
```

Three reasons the existing veto misses:
1. **`pair.forward.parameters.first?.typeText` is `"Self"`** for `intersection(_:) (Self) -> Self`. `inheritedTypesByName["Self"]` returns nothing.
2. Even if the veto walked through `containingTypeName` instead, **`OrderedSet` doesn't declare `: SetAlgebra` directly** — it has `extension OrderedSet : Partial SetAlgebra { ... }`-style extension splits that satisfy the SetAlgebra method surface without declaring conformance to the protocol. The textual `inheritedTypesByName["OrderedSet"]` doesn't contain `SetAlgebra`.
3. The candidate properties `additiveInverse` / `groupInverse` aren't the right shape — they're about kit-published inverse laws on Numeric / Group, not about SetAlgebra binary ops.

### Why a shape-only mechanism is correct here

The mathematical fact "any pair drawn from `{union, intersection, symmetricDifference, subtracting}` is not an inverse pair" holds **regardless of whether the carrier formally conforms to SetAlgebra**. The function names alone identify the SetAlgebra binary-op shape; carrier conformance isn't needed.

Curated-set-only check is sufficient for the cycle-11 deliverable. A future cycle could broaden to "any `Self -> Self` binary op on a SetAlgebra-conforming carrier" once SemanticIndex-style conformance resolution lands (PRD §20 v1.1+ trajectory).

## What v1.14 ships

Two deliverables (mirroring the v1.10/v1.11/v1.12 calibration-cycle shape):

1. **`InversePairTemplate.setAlgebraShapeVeto(for:)` extension method.** New helper in `Sources/SwiftInferTemplates/InversePairSetAlgebraShapeGate.swift` (file-split per the V1.6.1/V1.8.1/V1.10.1/V1.11.1/V1.12.1 file-length precedent — `InversePairTemplate.swift` is currently at the swiftlint type_body_length boundary). Returns a `Signal` with weight `-25` (calibrated for inverse-pair's `+25` typeSymmetry baseline; cleanly drops bare-shape SetAlgebra-shape pairs into Suppressed) when:
   - Forward function's first-parameter type and return type are both `"Self"`, AND
   - Reverse function's first-parameter type and return type are both `"Self"`, AND
   - Both forward and reverse function names are in a curated `setAlgebraBinaryOps` set: `{union, intersection, symmetricDifference, subtracting}`.

   New file `Sources/SwiftInferCore/SetAlgebraShape.swift` hosts the curated set as `public enum SetAlgebraShape { public static let binaryOps: Set<String> = [...] }`, mirroring the V1.13.1 `DirectionLabels` factoring posture so that future templates (round-trip, idempotence) can consume the same set without cross-template reach.

   Score arithmetic for inverse-pair (baseline `+25` typeSymmetry):
   - typeSymmetry alone: `+25` → Possible tier.
   - typeSymmetry + SetAlgebra-shape veto (`-25`): `+25 − 25 = 0` → Suppressed tier (clean margin from `+20`).
   - typeSymmetry + curated/project name (`+10`) + SetAlgebra-shape veto: `+25 + 10 − 25 = +10` → Suppressed (curated names like `parse/format` are unlikely to coincide with SetAlgebra ops; if they do, the structural argument still wins).

2. **Cycle-11 calibration run + findings writeup.** Re-run `swift-infer discover --include-possible` against the four cycle-1+...+9 corpora at the v1.14 commit; capture post-veto snapshots to `docs/calibration-cycle-11-data/`. Compare suppression deltas vs the cycle-9 post-roundtrip-direction-counter baseline (the 257-surface). Findings doc at `docs/calibration-cycle-11-findings.md` documents per-corpus delta + the **first cycle to ship a function-name + type-shape composite mechanism** framing + cycle-12 priority list.

PRD §19 sets the long-term acceptance-rate target at ≥ 70% after 6 months of dogfooding; v1.14 is cycle 11 of that loop.

### v1.14-only mechanical changes the cycle didn't cover

- **No new `Signal.Kind` case.** The `protocolCoveredProperty` case (V1.5.2) is the right factoring for veto-class signals; this veto reuses it. (Open decision #4 below considers whether to add a dedicated `Signal.Kind.setAlgebraShapeVeto` case for explainability clarity — defaulted no.)
- **No new `KnownProperty`.** The mechanism is shape-only, not protocol-coverage-based; no `ProtocolCoverageMap` table changes.
- **Other six templates untouched.** Round-trip / idempotence / commutativity / associativity / monotonicity / identity-element are byte-identical to v1.13. The SetAlgebra-shape pattern doesn't fire on round-trip (round-trip's existing veto already catches Codable-shape cases; SetAlgebra ops don't match) or idempotence (idempotence is `(T) -> T` single-function, not pair).
- **No curated-set extension.** `formUnion` / `formIntersection` / `formSymmetricDifference` / `formSubtract` are mutating Void-returning methods; they don't match the `(Self) -> Self` shape and won't enter inverse-pair pairing. Boolean ops like `isSubset` / `isSuperset` return `Bool`, also don't match. The 4-element curated set is sufficient.

## Open decisions

1. **Veto weight: -25 vs -1000 (full veto).** Two choices: (a) **-25 (this plan's default)** — non-veto counter that drops Score 25 → 0 cleanly; preserves recall on edge cases (custom `intersection`-named methods on non-SetAlgebra types that happen to be inverses for some weird domain) when curated/project name match adds `+10` (`+25 + 10 - 25 = +10` still Suppressed). (b) -1000 (Signal.vetoWeight) — full veto; structurally certain that SetAlgebra ops aren't inverses, so no need for recall preservation. Default proposal: **(a) -25.** Rationale: matches the v1.4.3b cross-type round-trip counter weight; conservative-precision posture (PRD §3.5) prefers softer signals when the rule's false-positive risk is non-zero. -1000 would also work because the curated names `{union, intersection, symmetricDifference, subtracting}` are well-defined SetAlgebra terms that would be unusual choices for non-SetAlgebra inverse pairs, but -25 leaves room for surprise.

2. **Curated set scope: 4 elements (`{union, intersection, symmetricDifference, subtracting}`) vs broader.** Two choices: (a) **4 elements (this plan's default)** — only the `(Self) -> Self`-shaped SetAlgebra methods. (b) Include `formUnion` / `formIntersection` / etc. — the mutating siblings. Default proposal: **(a) 4 elements.** Rationale: the form-mutating siblings are Void-returning and don't match `(Self) -> Self` shape; they won't even enter `FunctionPairing`'s candidate generation. Adding them to the curated set is harmless but adds line noise. Keep the set focused on what the structural shape gate actually accepts.

3. **Conformance check vs shape-only.** Two choices: (a) **shape-only (this plan's default)** — match function names + Self typing, no `inheritedTypesByName` lookup. (b) Combined check — names match curated set AND `containingTypeName` is in `inheritedTypesByName` with `SetAlgebra` conformance. Default proposal: **(a) shape-only.** Rationale: `OrderedSet` itself doesn't declare `: SetAlgebra` (only `Partial SetAlgebra` extensions); a conformance check would miss the 4 OrderedSet-side cycle-9 survivors. The `OrderedSet.UnorderedView` does declare `: SetAlgebra`, so a conformance check would catch the 2 UnorderedView-side survivors but miss the 4 OrderedSet-side ones (= 33% of the test bed). Shape-only catches all 6. The structural argument "intersection ∘ subtracting ≠ identity" is independent of conformance declaration.

4. **Whether to add a dedicated `Signal.Kind.setAlgebraShapeVeto` case.** Two choices: (a) **reuse `Signal.Kind.protocolCoveredProperty` (this plan's default)** — the existing veto-class case suffices. (b) Add a new case for explainability clarity. Default proposal: **(a) reuse.** Rationale: the explainability detail string can fully describe the structural argument ("Pair shape: `intersection ↔ subtracting` on Self-typed binary ops; SetAlgebra ops do not form an inverse pair"); `Signal.Kind` granularity isn't user-facing in the rendered output (the detail string is). Mirrors the cycle-7 → cycle-8 → cycle-9 reuse of `Signal.Kind.directionLabel` for three consumers.

5. **File split: pre-emptive vs reactive.** Default proposal: **pre-emptive split** mirroring the v1.10-v1.13 precedent. New file `Sources/SwiftInferTemplates/InversePairSetAlgebraShapeGate.swift` hosts the helper. The parent file is currently fine line-count-wise, but the file-split posture keeps each calibration mechanism in a self-contained file for attribution clarity (V1.6.1/V1.8.1/V1.10.1/V1.11.1/V1.12.1 precedent).

6. **`SetAlgebraShape.binaryOps` lives in core from cycle 1.** The v1.13 hoist of `directionLabels` provides the precedent: shared template-agnostic curated sets live in `SwiftInferCore.<Namespace>.curated` (or similar). v1.14 starts at the canonical home from day one rather than waiting for a third consumer to trigger a hoist. Future cycles that consume `SetAlgebraShape.binaryOps` (e.g., a hypothetical round-trip arm that recognizes `(Self) -> Self` binary ops on SetAlgebra carriers) consume directly from core.

7. **Whether to commit cycle-11 decisions data.** Default: **yes** — `docs/calibration-cycle-11-data/` mirroring cycles 7-9.

8. **Cycle numbering: skip 10 or count it.** v1.13 was cycle 10 (refactor character). v1.14 is cycle 11 (next mechanism cycle). Numbering is contiguous; the cycle-10 hoist is a real cycle in the trajectory chart, just one with a no-op surface delta.

## v1.14 trajectory framing

PRD v1.0 §17.3 names calibration as "the operationalization of high precision, low recall."

Cycle 7 (v1.10) introduced direction-label counter on `IdempotenceTemplate`.
Cycle 8 (v1.11) replicated on `InversePairTemplate`.
Cycle 9 (v1.12) completed the family on `RoundTripTemplate`.
Cycle 10 (v1.13) hoisted the abstraction.
Cycle 11 (v1.14, this release) ships the **first function-name + type-shape composite mechanism** — distinct from cycles 7-9's parameter-label-based class.

This expands the calibration loop's mechanism vocabulary. Earlier cycles surfaced these mechanism classes:
- **Cycle 1 (V1.4.3)**: textual type-name counter (FP-storage detection); cross-type pair counter.
- **Cycle 2 (V1.5.2)**: protocol-coverage veto via curated `KnownProperty` × conformance map.
- **Cycle 3 (V1.6.1)**: pair-formation skip-list (op-name × identity-name × type-shape filter).
- **Cycle 4 (V1.7.1)**: stdlib-conformance bake-in (extends V1.5.2's protocol-coverage substrate).
- **Cycle 5 (V1.8.1)**: shape-gated veto (combines V1.5.2's protocol-coverage with type-shape detection).
- **Cycles 7-9 (V1.10.1-V1.12.1)**: parameter-label counter (curated label set × first-param-label match).
- **Cycle 11 (V1.14.1, this release)**: function-name + type-shape composite (curated name set × Self-typed shape).

The cycle-12 priority list rotates post-v1.14 to drop the shipped SetAlgebra mechanism, promoting the domain-mismatch family on idempotence + inverse-pair + round-trip simultaneously to priority #1. That mechanism would be the **first cycle to ship a single rule applied to three templates simultaneously**, paralleling the direction-counter family's three-cycle deployment cadence but compressed into a single release.
