## V1.29.A — Inverse-pair asymmetric-pair full-veto (closes -2)

**Target rejects:** cycle-25 picks #28 + #29:
- `bucket(after:) × firstOccupiedBucketInChain(with:)`
- `bucket(before:) × firstOccupiedBucketInChain(with:)`

**Diagnosis (cycle-25 finding 1):** V1.27.B's name-prefix gate requires BOTH pair sides to match the `["index", "bucket", "word"]` prefix list — closes symmetric direction-pair noise (`bucket(after:) × bucket(before:)`) but doesn't fire on asymmetric (direction-op × search-op) shapes where one side is a non-direction lookup function.

**Mechanism:** extend `InversePairDirectionLabelCounter.directionLabelCounterSignal` with an asymmetric branch — when ONE side is direction-prefix-matched + direction-labeled AND the OTHER side is not direction-labeled, fire `Signal.vetoWeight`. The "true inverse pair" pattern is structurally symmetric (both sides direction-labeled cursors OR both sides non-direction transforms); the asymmetric shape is not.

**Score arithmetic:** current cycle-25 score is 20 (`+25 typeSymmetry + 10 value-semantic + -10 either-side counter == 25`... actually the cycle-25 score is 20 per discover output, meaning the +25 typeSymmetry + -10 counter + ? — let the implementation flow through `Score`). After full-veto: tier = `.suppressed`, suggestion drops.

**Risk:** false-negatives if a legitimate cursor-direction × value-search inverse pair exists. The empirical evidence (4-cycle reject on the bucket × firstOccupied pattern) plus the cycle-6 / cycle-14 / cycle-17 single-runner triage history (no asymmetric-cursor inverse pair ever ACCEPT) makes this low-risk.

**Tests:** extend `InversePairDirectionLabelCounterTests.swift` with the asymmetric pattern. Verify the symmetric V1.27.B path remains intact.

## V1.29.B — Identity-element algebraic-family narrow (closes -1)

**Target reject:** cycle-25 pick #30:
- `rescaledDivide(_:_:) (Complex, Complex) -> Complex` paired with `Complex.zero`.

**Diagnosis (cycle-25 finding 2):** `IdentityElementTemplate.identityNamingSignal` fires `+40` whenever a curated identity-name constant (`zero`/`one`/`empty`/`identity`) is bound to the op type-shape. The `protocolCoverageVeto` already op-class-narrows (`(zero, +)` → `additiveIdentityZero` coverage), but the +40 naming signal fires unconditionally. For non-additive operators (`/`, `rescaledDivide`, `pow`), `T.zero` is a type-shape false-positive — zero is not the identity of division.

**Mechanism:** introduce an **algebraic-family gate** on the +40 naming signal. When the identity name is `zero` (or `one`), check that the operator name is in a corresponding additive (or multiplicative) curated verb set. If the operator doesn't match the algebraic family, demote `+40` → `+10` (or full-veto).

**Curated sets** (new in `SwiftInferCore`):
- `AdditiveOperatorNames`: `{"+", "add", "addingProduct", "_relaxedAdd", "plus"}`
- `MultiplicativeOperatorNames`: `{"*", "multiply", "_relaxedMul", "times"}`

The `.empty` and `.identity` constants already have op-class-aware coverage paths in `identityCoverageCandidate`; they don't need the gate.

**Score arithmetic:** cycle-25 pick #30 is score 70 (`+30 typeShape + +40 naming = 70` Likely). After narrowing: `+30 + +10 (demoted naming) = 40` → still Likely tier, or apply a `-30` counter for explicit incompatible-family signal → `+30 + +40 + -30 = 40` Likely. To fully suppress, need `-50+` counter to drop below 20. Use full-veto on incompatible (zero, non-additive) or (one, non-multiplicative) pairs.

**Risk:** false-negatives if a non-standard additive op exists with a non-standard name. The curated sets can be extended in future cycles; conservative initial coverage is safer.

**Tests:** extend `IdentityElementTemplateTests.swift` with the rescaledDivide + zero case; verify the canonical (zero + plus, one + multiply) paths still ACCEPT.

## V1.29.C — Composition-lifted monotone-bounded full-veto promotion (closes -1)

**Target reject:** cycle-25 pick #36:
- `_HashTable.BucketIterator.advance(until:) mutating (Int) -> Void`.

**Diagnosis (cycle-25 finding carried-forward):** V1.21.B's `monotoneBoundedLabels` set (`{until, to, at, upTo, before, through}`) currently fires a `-25` counter (Strong → Likely demote). The cycle-17 plan noted this as a deliberate "Likely tier preservation" at small-n; cycle-19 + cycle-20 + cycle-23 + cycle-25 all measured REJECT on the `advance(until:)` representative. **4-cycle stable reject** justifies promotion to full veto.

**Mechanism:** in `CompositionTemplate.monotoneBoundedLabelSignal`, change `weight: -25` to `weight: Signal.vetoWeight`. Drop the "Likely" preservation comment in the docstring.

**Score arithmetic:** cycle-25 pick #36 is score 60 (`+30 typeShape + +40 verb + +5 carrier + +10 lift + -25 monotone-bounded = 60` Likely). After veto: tier = `.suppressed`, suggestion drops.

**Risk:** false-negatives if a Sequence-style additive API uses `to:` or `through:` additively. The labels are conservative (excludes `by:`, the canonical additive label). 4-cycle empirical 0% accept rate.

**Tests:** update `CompositionTemplateTests.swift` to assert veto rather than counter; cycle-23 + cycle-25 corpus evidence justifies.
