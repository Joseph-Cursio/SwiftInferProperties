# TestLifter M5 Execution Plan

Working doc for the TestLifter M5 milestone defined in `SwiftInferProperties PRD v0.4.md` §7.9 + §7.3 + §7.4 (rung 5). Decomposes M5 into seven sub-milestones so progress is checkable session-by-session. **Ephemeral** — archive to `docs/archive/TestLifter M5 Plan.md` once M5 ships and the §7.9 + §7.3 + §7.4 acceptance bars are met (mirroring TestLifter M1 + M2 + M3 + M4 + TemplateEngine M1–M8).

> **TestLifter M5 lights up the cross-validation `+20` for the remaining template column AND closes the §7.4 generator-fallback ladder.** TestLifter M1 + M2 + M3 wired the `+20` cross-validation seam for **round-trip / idempotence / commutativity** (three of the eight shipped templates). M4 closed the §7.4 mock-inferred generator rung. M5 widens the `+20` seam to **monotonicity** (Assert-Ordering-Preserved, PRD §7.3), **invariant-preservation** (Assert-Count-Change), and **associativity** (Assert-Reduce-Equivalence) — and adds the §7.4 rung-5 **Codable round-trip generator** as a third pass on top of M4's mock fallback. After M5, six of the eight shipped templates have a TestLifter cross-validation source, and the §7.4 ladder reads CaseIterable → RawRepresentable → memberwise → mock-inferred → **Codable round-trip** → `.todo` end-to-end. The two templates without a TestLifter source (identity-element, inverse-pair) stay signature-only — neither has a v1-shaped test-body pattern that maps to a single template's cross-validation key (open decision #6 below).

## What M5 ships (PRD v0.4 §7.9 + §7.3 + §7.4)

> **M5 (TestLifter).** Ordering, count-change, reduce-equivalence pattern detection. *(Verbatim from PRD §7.9.)*

> **§7.3 pattern → template mapping (verbatim from PRD v0.2 §7.3, unchanged in v0.4).** Assert-Ordering-Preserved → monotonicity; Assert-Count-Change → invariant preservation; Assert-Reduce-Equivalence → associativity.

> **§7.4 ladder (verbatim).** CaseIterable → RawRepresentable → memberwise (delegated to shared `DerivationStrategist`) → mock-inferred from observed test construction → **Codable round-trip** → `.todo`. Generator confidence (`high` / `medium` / `low`) flows into the explainability block.

Five concrete deliverables, all additive on top of M1 + M2 + M3 + M4's library:

1. **`AssertOrderingPreservedDetector` — monotonicity pattern.** New `Sources/SwiftInferTestLifter/AssertOrderingPreservedDetector.swift`. Public entry: `static func detect(in slice: SlicedTestBody) -> [DetectedMonotonicity]`. Recognizes (a) **explicit two-assert form** — `XCTAssertLessThan(a, b)` (or `<`) followed within the property region by `XCTAssertLessThanOrEqual(f(a), f(b))` (or `<=`) where `f` is the same single callee on both sides; (b) **collapsed `#expect`** — `#expect(a < b); #expect(f(a) <= f(b))` two-statement form (Swift Testing macros separate the two preconditions). Returns `DetectedMonotonicity { calleeName, leftArgName, rightArgName, assertionLocation }`. Tautology rejected (`a == b` precondition + `f(a) == f(b)` body trivially holds — not monotonicity); strict-inequality result `f(a) < f(b)` *also* recognized (strict monotonicity is a special case of non-strict; both produce the same template-side `+20` signal).

2. **`AssertCountChangeDetector` — count-invariant pattern.** New `Sources/SwiftInferTestLifter/AssertCountChangeDetector.swift`. Public entry: `static func detect(in slice: SlicedTestBody) -> [DetectedCountInvariance]`. Recognizes (a) **collapsed** — `XCTAssertEqual(f(xs).count, xs.count)` / `#expect(f(xs).count == xs.count)`; (b) **explicit** — `let result = f(xs); XCTAssertEqual(result.count, xs.count)`. Single-callee invariant: only one `f`, only one input collection binding. `keyPath` is hard-coded to `\.count` for M5 (open decision #2 below covers generalization to `\.<arbitrary keyPath>`). Returns `DetectedCountInvariance { calleeName, inputBindingName, assertionLocation }`. Tautology (`xs.count == xs.count`) and different-callee (`f(xs).count == g(xs).count`) shapes rejected.

3. **`AssertReduceEquivalenceDetector` — associativity pattern.** New `Sources/SwiftInferTestLifter/AssertReduceEquivalenceDetector.swift`. Public entry: `static func detect(in slice: SlicedTestBody) -> [DetectedReduceEquivalence]`. Recognizes (a) **collapsed reorder-equivalence** — `XCTAssertEqual(xs.reduce(seed, op), xs.reversed().reduce(seed, op))` and `#expect(xs.reduce(seed, op) == xs.reversed().reduce(seed, op))`; (b) **explicit two-binding** — `let lhs = xs.reduce(seed, op); let rhs = xs.reversed().reduce(seed, op); XCTAssertEqual(lhs, rhs)`. Method-chain `xs.reduce(_:_:)` only — free-function `reduce(xs, seed, op)` is out of scope per open decision #3 default. Single-`op` invariant: both sides must reference the same `op` callee. Single-seed invariant: both sides must use the same seed expression (textual equality on the seed expression source). Returns `DetectedReduceEquivalence { opCalleeName, seedSource: String, collectionBindingName, assertionLocation }`. Tautology (`xs.reduce(0, +) == xs.reduce(0, +)`) and differing-callee (`xs.reduce(0, +) == xs.reversed().reduce(0, *)`) rejected.

4. **Codable round-trip generator (§7.4 rung 5) + `GeneratorSelection.applyCodableRoundTripFallback(...)`.** Extend `Sources/SwiftInferTemplates/GeneratorSelection.swift` with a third pass that runs **after** M4's `applyMockInferredFallback(...)`. For every Suggestion whose `generator.source == .notYetComputed` after the mock pass, look up the recovered type in `corpus.typeDecls` and check `inheritedTypes` for `Codable` / `Encodable` + `Decodable` (string match — same posture as M3's strategist-driven derivation, no semantic resolution). On hit, rebuild the Suggestion with `generator.source = .derivedCodableRoundTrip`, `generator.confidence = .medium` (open decision #4 default), and a fixture-seeded encode/decode round-trip generator body produced by a new `CodableRoundTripGeneratorRenderer` helper. The `derivedCodableRoundTrip` `Source` enum case is **already reserved** at `Sources/SwiftInferCore/Suggestion.swift:106`; M5 wires the path, doesn't grow the type. Mock-inferred + Codable both fail → suggestion stays `.notYetComputed` and falls through to the existing `.todo<?>()` rendering path.

5. **`LiftedSuggestion` factories + `TestLifter.discover` six-detector fan-out + accept-flow rendering.** Extend `DetectedPattern` enum (M2.0 surface) with three new cases: `.monotonicity(DetectedMonotonicity)`, `.countInvariance(DetectedCountInvariance)`, `.reduceEquivalence(DetectedReduceEquivalence)`. Extend `LiftedSuggestion` with three new factories: `static func monotonicity(from:) -> LiftedSuggestion`, `static func countInvariance(from:) -> LiftedSuggestion`, `static func reduceEquivalence(from:) -> LiftedSuggestion`. Each builds a `CrossValidationKey` matching the production-side template name: `("monotonicity", [calleeName])`, `("invariant-preservation", [calleeName])`, `("associativity", [opCalleeName])`. Extend `TestLifter.discover(in:)`'s per-summary loop to fan out to all six detectors. Extend `LiftedTestEmitter` with three new arms (one per pattern) so accepted lifted-only suggestions render compiling stubs. Extend `InteractiveTriage+Accept`'s lifted-stub renderer dispatch to dispatch on the three new `DetectedPattern` cases. Extend `LiftedTestEmitter+Generators.swift`'s `chooseGenerator(for:typeName:)` helper to recognize `.derivedCodableRoundTrip` source and emit the round-trip body.

### Important scope clarifications

- **InvariantPreservationTemplate is annotation-only — `+20` lights up only when the user has tagged the function.** `InvariantPreservationTemplate.suggest(_:)` (`Sources/SwiftInferTemplates/InvariantPreservationTemplate.swift:39`) returns `nil` unless `summary.invariantKeypath != nil` (the `@CheckProperty(.preservesInvariant(\.foo))` annotation). The PRD §5.2 caveat — "structurally too easy to misinfer from signature shape alone" — is the reason. M5's `AssertCountChangeDetector` produces a `LiftedSuggestion` with `crossValidationKey = ("invariant-preservation", [calleeName])`; that key only matches a TemplateEngine-side suggestion when the user has the `@CheckProperty(.preservesInvariant(\.count))` annotation on the same callee. **When the annotation is absent**, the LiftedSuggestion enters the visible discover stream the same way M3 wired round-trip / idempotence / commutativity (M3's promote → suppress-or-emit pipeline). The lifted suggestion ships at `.likely` tier with the +50 `.testBodyPattern` signal (per M3.0's per-pattern signal-table extension); the user sees a freestanding count-invariant suggestion with provenance pointing at the test body. **Open decision #1** below covers the alternative (suppress lifted-only emission to preserve §5.2's annotation-only posture for the production-side template).

- **MonotonicityTemplate ships at `Possible` tier by default — `+20` escalates to `Likely`.** `MonotonicityTemplate.suggest(_:)` lands at +25 (codomain) + ≤+10 (curated name) + ≤+15 (accumulator body) ≈ +35–50 without TestLifter, which falls in `.possible` (20..<40) or low `.likely` (40..<75) per `Tier.swift:22`. Adding `+20` cross-validation pushes the codomain-only case from 25 → 45 (`.possible` → `.likely`); the codomain + name case from 35 → 55 (`.possible` → `.likely`); the codomain + name + accumulator case from 50 → 70 (`.likely` → still `.likely`, just stronger). The `.possible` → `.likely` escalation is the user-visible payoff for monotonicity since `.possible` is hidden behind `--include-possible` (`Tier.swift:47`). M5.4's integration test must assert the post-cross-validation tier is `.likely` (i.e., `score.total >= 40`), not just that the signal is present.

- **`MonotonicityTemplate.swift:244`'s "TestLifter corroboration not yet wired (gated on TestLifter M1)" caveat closes at M5.** The comment is the explicit dormant-hook note from TemplateEngine M7.1; M5.4's wiring lights it up. Update the docstring at the same time the explainability caveat list is regenerated.

- **No kit-side coordination commit needed.** `GeneratorMetadata.Source.derivedCodableRoundTrip` is already in the `Sources/SwiftInferCore/Suggestion.swift:102-111` enum (reserved at M1, untouched through M4). The Codable round-trip metadata is fully describable in the existing enum + confidence shape (`.medium` confidence per open decision #4 default); no new fields, no new variants. `Package.swift` stays at `from: "1.9.0"` for M5. No kit PR, no version bump.

- **Codable round-trip runs *after* mock-inferred fallback, not in place of it.** §7.4's ladder is ordered: CaseIterable → RawRepresentable → memberwise → mock → Codable → `.todo`. M5.3 wires this ordering by running `GeneratorSelection.applyCodableRoundTripFallback(...)` *third* on suggestions whose `generator.source == .notYetComputed` after both M3's strategist pass *and* M4's mock-inferred pass. Suggestions whose mock synthesis succeeded never see Codable fallback even if their type also conforms to `Codable`. The mock rung (test-corpus-grounded) wins over the Codable rung (Codable-grounded but corpus-blind) per the §7.4 ladder ordering.

- **Codable detection is by `inheritedTypes` string match, no semantic resolution.** `TypeDecl.inheritedTypes: [String]` (`Sources/SwiftInferCore/TypeDecl.swift:47`) carries inherited-clause source text. `T: Codable` matches `["Codable"]`; `T: Encodable, Decodable` matches `["Encodable", "Decodable"]`; `T: Equatable, Codable, Sendable` matches `["Equatable", "Codable", "Sendable"]` (substring). Conditional conformances (`extension T: Codable where Element: Codable`) and protocol-composition (`T: SomeProto where Self: Codable`) are out of scope — same posture as M3's `derivedCaseIterable` / `derivedRawRepresentable` recognition, which is also string-match. Open decision #5 below covers a tighter alternative.

- **Codable round-trip generator body is fixture-seeded, not random-encoded.** `CodableRoundTripGeneratorRenderer` emits `Gen<T> { /* let data = try? JSONEncoder().encode(<seed>); return try? JSONDecoder().decode(T.self, from: data ?? Data()) ?? <fallback> */ }` where `<seed>` is the canonical zero-value the strategist would memberwise-derive (one literal per stored property if the type is in `corpus.typeDecls`, else `?` sentinel). The "round-trip" is on the *serialization shape*, not on the *value space* — the generator returns a single deterministic value per call. Property tests against this generator catch encode/decode breakage but don't randomize; the user's responsibility is to widen the seed if they want broader coverage. The `// Codable round-trip generator (medium confidence — fixture-seeded; widen the seed for broader coverage)` provenance comment makes this explicit. Open decision #4 below covers a richer alternative (multiple seeds / shrinking).

- **Reduce-equivalence detection covers `xs.reversed().reduce(...)` only — not `xs.shuffled()` or arbitrary permutations.** `xs.reversed()` is deterministic; `xs.shuffled()` is non-deterministic (re-runs produce different orderings) and would need M7-style counter-signal scanning to handle correctly. The PRD §7.3 example is `xs.reduce(seed, op) == xs.reversed().reduce(seed, op)` — M5 mirrors that example verbatim. Open decision #3 below covers shuffled / arbitrary-permutation extension.

- **All three new detectors run against the *property region*, not the raw body.** Mirrors M1.3 / M2.1 / M2.2's `SlicedTestBody.propertyRegion` walking. Setup-region statements (the `let xs = [1, 2, 3]` literal-collection bindings the patterns reference) are already classified by the M1 slicer; the detector consumes the property region directly.

- **Stream-entry behavior matches M3 exactly — three new patterns enter the discover stream when no production-side cross-validation match exists.** No new architectural decision needed; `LiftedSuggestionPipeline.promote(...)` already handles the suppression / emission split. M5's wiring just adds three more `LiftedSuggestion` factory calls into the same fan-out.

- **§13 perf budget compliance with M5 active.** The 100-test-file budget already includes the M1 slicer + three M2 detectors + M3 promotion + M4 annotation/construction scanners. M5 adds three more detectors (each linear in property-region size) + one extra `GeneratorSelection` pass (linear in suggestion count × `corpus.typeDecls` lookup). Targeted overhead: <200ms on the synthetic 100-file corpus. The §13 row stays at `< 3s` wall.

- **No new SwiftPM dependencies.** Same as M2 / M3 / M4. Intra-package, `SwiftInferTestLifter` already depends on `SwiftInferCore`; `SwiftInferTemplates` already depends on `SwiftInferCore`. SwiftProtocolLaws stays at `from: "1.9.0"`.

## M5 acceptance bar

Mirroring PRD §7.9 + §7.3 + §7.4 + the v0.4 §5.8 acceptance-bar pattern + the M1 / M2 / M3 / M4 / TemplateEngine M1–M8 cadence, M5 is not done until:

a. **`DetectedPattern` enum carries six cases** — the M2 three (`.roundTrip`, `.idempotence`, `.commutativity`) plus M5's three (`.monotonicity`, `.countInvariance`, `.reduceEquivalence`).

b. **`AssertOrderingPreservedDetector` recognizes the monotonicity shape in property regions.** Both explicit two-assert (`XCTAssertLessThan(a, b); XCTAssertLessThanOrEqual(f(a), f(b))`) and collapsed two-`#expect` forms detect. Tautology and different-callee shapes are correctly rejected. Strict-inequality result variants (`<`) detect alongside non-strict (`<=`).

c. **`AssertCountChangeDetector` recognizes the count-invariant shape in property regions.** Both collapsed (`XCTAssertEqual(f(xs).count, xs.count)`) and explicit two-binding forms detect. `keyPath` is hard-coded to `\.count`. Tautology and different-callee shapes rejected.

d. **`AssertReduceEquivalenceDetector` recognizes the reduce-equivalence shape in property regions.** Both collapsed and explicit forms detect, scoped to method-chain `xs.reduce(_:_:)` paired with `xs.reversed().reduce(_:_:)`. Same-`op` and same-seed invariants enforced.

## Open decisions to make in-flight

1. **Count-invariance LiftedSuggestion when `@CheckProperty(.preservesInvariant(\.count))` annotation is absent — emit lifted-only or suppress entirely?**
   - **(a) Emit lifted-only.** When no production-side `InvariantPreservationTemplate` Suggestion exists (annotation absent), the LiftedSuggestion enters the visible discover stream the same way M3 wired round-trip / idempotence / commutativity. The user sees a freestanding count-invariant suggestion with provenance pointing at the test body; provides cross-pattern parity.
   - **(b) Suppress entirely.** Preserve §5.2's annotation-only posture for invariant-preservation across both production and lifted paths. The cross-validation `+20` only lights up when the user has the annotation; otherwise the detection is silent. More conservative; respects the §5.2 misinference-risk reasoning verbatim.
   - **Default unless reason emerges:** **(a) emit lifted-only.** §5.2's annotation-only posture is about *signature-side misinference* (the function's name + parameters look invariant-preserving but aren't); a *test body* asserting count-invariance is direct evidence — fundamentally different from name + signature heuristics. The lifted suggestion still ships at `.likely` tier (not `.strong`) because the +50 `.testBodyPattern` signal alone clears the 40-threshold; the user gets a visible suggestion grounded in their own test code.

2. **`AssertCountChangeDetector` keyPath generalization — hard-code `\.count`, or recognize arbitrary `\.foo` keypaths from test bodies?**
   - **(a) Hard-code `\.count`.** The PRD §7.3 row is "Assert-Count-Change → invariant preservation"; `count` is the canonical example and the most common shape in idiomatic Swift tests. M9's expanded-outputs work is the natural home for cross-test keyPath inference.
   - **(b) Recognize `f(xs).<keyPath> == xs.<keyPath>`.** Catches `XCTAssertEqual(normalize(s).hashValue, s.hashValue)` (hash-invariant), `XCTAssertEqual(canonicalize(d).id, d.id)` (id-preserving), etc. Doubles the surface area; expands the cross-validation key shape to carry the keyPath alongside the callee name, which then doesn't match `InvariantPreservationTemplate`'s annotation-keyed shape (the production-side template's identity *includes* the keyPath per `InvariantPreservationTemplate.swift:67-71`).
   - **Default unless reason emerges:** **(a) hard-code `\.count`.** Cross-validation keys must match the production-side template's identity shape; `InvariantPreservationTemplate.swift:67-71` includes the keyPath in `SuggestionIdentity` but the `CrossValidationKey` (templateName + sortedCalleeNames) doesn't. Any mismatch in the keyPath dimension would silently break the +20 signal. M5 keeps the count-invariance shape narrow; M9's per-test correlation pass owns the keyPath generalization story.

3. **`AssertReduceEquivalenceDetector` reordering operations — `xs.reversed()` only, or `xs.shuffled()` / arbitrary permutations too?**
   - **(a) `xs.reversed()` only.** Deterministic; the PRD §7.3 example uses `reversed()` verbatim. Property tests against shuffled-reduce would be unreliable (re-runs produce different orderings); recognizing them risks false-positive associativity claims.
   - **(b) Both `xs.reversed()` and `xs.shuffled()`.** Catches more associativity tests in the wild. `xs.shuffled()` non-determinism is a TestLifter concern, not an SwiftInfer-side concern (the test author chose `shuffled()` knowingly).
   - **Default unless reason emerges:** **(a) `xs.reversed()` only.** Matches the PRD §7.3 example. M7's counter-signal scan is the right home for non-determinism handling; bolting shuffled-recognition onto M5 conflates two milestones' concerns.

4. **Codable round-trip generator confidence — `.medium`, `.low`, or `.high`?**
   - **(a) `.medium`.** Between M3's `.high` (CaseIterable / RawRepresentable / memberwise — all corpus-grounded) and M4's `.low` (mock-inferred — observation-grounded but corpus-blind on value range). Codable round-trip is structurally sound (the encoder/decoder exists and round-trips by construction) but value-blind (always returns the same fixture-seeded value). `.medium` reflects that.
   - **(b) `.low`.** Same as M4 mock-inferred. Honest about the fixture-seeded shape — not actually random.
   - **(c) `.high`.** The encode/decode round-trip is structurally guaranteed by the protocol conformance; if `T: Codable` then `T(from: encoder.encode(t))` is `t` by construction.
   - **Default unless reason emerges:** **(a) `.medium`.** Slots cleanly between M3's corpus-grounded `.high` strategist-derived rungs and M4's observation-grounded `.low` mock rung — the §7.4 ladder ordering already implies a confidence ordering, and `.medium` matches the rung's between-memberwise-and-mock position. The encode/decode round-trip is structurally guaranteed by the protocol conformance (so `.low` understates it); the fixture-seeded value-space is genuinely narrow (so `.high` overstates it). The provenance comment "Codable round-trip generator (medium confidence — fixture-seeded; widen the seed for broader coverage)" carries the value-space caveat for the user.

5. **Codable conformance detection — `inheritedTypes` string match only, or include conditional + extension conformances?**
   - **(a) Direct `inheritedTypes` string match only.** `T: Codable` matches; `T: Encodable, Decodable` matches; `extension T: Codable` matches via `TypeDecl`'s extension-conformance handling (`TypeDecl.swift:11-14` describes the per-extension `TypeDecl`). Conditional conformances (`extension T: Codable where Element: Codable`) and protocol-composition conformances are ambiguous — `extension Pair: Codable where T: Codable` may or may not hold for a specific `Pair<U>`, and we can't statically resolve it.
   - **(b) Add conditional-conformance scoring.** Recognize `extension T: Codable where ...` and emit a `.low` confidence (vs `.medium` for direct, per OD #4 default) reflecting the partiality.
   - **Default unless reason emerges:** **(a) direct match only.** Conservative; matches M3's `derivedCaseIterable` / `derivedRawRepresentable` posture (those are also direct-match). Conditional conformances are rare for `Codable` in practice (most types either fully conform or don't).

6. **Identity-element + inverse-pair test-body patterns — slot in M5 as a fourth/fifth detector, or stay out of v1?**
   - **(a) Stay out of v1.** Identity-element's test shape (`f(x, identity) == x`) is recognizable but its cross-validation key would need to carry the *identity value* (which `IdentityElementTemplate` derives signature-side from naming heuristics, not from a literal) — the dimensional mismatch is the same as OD #2. Inverse-pair's test shape (`g(f(x)) == x`) is the round-trip pattern's already-detected by M1's `AssertAfterTransformDetector`; the difference is direction-of-derivation (round-trip is two callees on a value; inverse-pair is two callees on T's domain). The cross-validation key for inverse-pair would need to disambiguate by direction, which requires more semantic context than M5's syntactic detection has.
   - **(b) Slot in M5 as fourth/fifth detectors.** Doubles the M5 surface area; risks the same dimensional-mismatch silent breakage as OD #2.
   - **Default unless reason emerges:** **(a) stay out of v1.** PRD §7.9's M5 row spells out three patterns (ordering / count-change / reduce-equivalence); identity-element + inverse-pair aren't on the M5 list. Out of scope; revisit at v1.1+ alongside M9 expanded outputs.

7. **Detector entry-point shape — keep the M2.4 OD #4 default of free `static func detect(in:)` per detector, or adopt a shared `LiftedDetector` protocol now that we have six?**
   - **(a) Keep free `static func detect(in:)`.** Mirrors M1.3 / M2.1 / M2.2 / M5.1 / M5.2 / M5.3. Six fan-outs in `TestLifter.discover(in:)` is still tractable by hand.
   - **(b) Introduce `protocol LiftedDetector { static func detect(in: SlicedTestBody) -> [DetectedPattern] }` and iterate over a `[LiftedDetector.Type]` array.** Lets future detectors (TestLifter M9 expanded outputs, etc.) plug in without touching the discover loop.
   - **Default unless reason emerges:** **(a) keep free `static func detect(in:)`.** Same reasoning as M2.4 OD #4 — six is still small enough to fan out by hand. The protocol abstraction would be premature; introduce when there are nine detectors (M5 + M7 counter-signal + M9 expanded outputs).
