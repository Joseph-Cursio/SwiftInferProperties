# v1.0 Release Plan

Working doc for the SwiftInferProperties v1.0 cut. With TemplateEngine M1–M8 and TestLifter M1 closed on `main`, the v1 surface is feature-complete; this milestone closes the release-blocking guarantees in PRD v0.4 §13 + §16, refreshes perf baselines, bumps the version string, and tags `v1.0.0`. **Ephemeral** — archive to `docs/archive/v1.0 Release Plan.md` once the tag is pushed and the §11 + §16 release-blocking criteria are met (mirroring TemplateEngine M1–M8 + TestLifter M1).

> **v1.0 is a posture cut, not a scope cut.** The shipped templates do not change. What changes is that every release-blocking guarantee in the PRD has an explicit, named regression test backing it; the §13 budgets have a recorded baseline that future regressions can be measured against; and the CLI advertises a real version that downstream consumers can pin against.

## What v1.0 ships (PRD v0.4 §11 + §13 + §14 + §16 + §19)

The PRD calls out three release-blocking categories. Each category's status is the entry condition for v1.0:

1. **§13 perf budgets** — five rows; failure mode is "regression test fails; release blocked" (rows 1, 5) or "regression test fails" (rows 2, 3, 4).
2. **§16 hard guarantees** — six numbered guarantees; "violation is a release-blocking bug."
3. **§14 privacy guarantees** — runtime no-network for any subcommand, no telemetry artifacts.

## v1.0 acceptance bar

Mirroring PRD §11 (release blockers) + §19 (success criteria), v1.0 is not done until:

a. **All §13 perf budgets have an explicit regression test in CI.** Rows 1, 2 (already covered) plus rows 3, 4, 5 (R1.1.a, R1.1.b, R1.1.c). Each test asserts the documented threshold; budget overruns fail the build.

b. **All §16 hard guarantees have an explicit regression test in CI.** Rows #1, #6 (already covered) plus #2, #3, #4, #5 (R1.1.d, R1.1.e, R1.1.f, R1.1.g). Each test asserts the documented contract; violations fail the build.

c. **§14 privacy guarantees have a runtime test.** R1.1.h. The static code-grep stays as defense-in-depth; the runtime assertion satisfies the PRD §19 explicit success criterion.

d. **Performance baseline recorded.** `docs/perf-baseline-v1.0.md` lists wall numbers for every §13 row at the v1.0 commit so future regressions have a comparison point. (R1.2.)

e. **CLI version reports `1.0.0`.** `swift-infer --version` no longer emits the `0.0.0-scaffold` placeholder. (R1.3.)

f. **`CHANGELOG.md` exists at repo root.** v1.0 entry summarizes the shipped surface (M1–M8 + TestLifter M1 + the §13/§16 release-gate closure). (R1.3.)

g. **`docs/archive/v1.0 Release Plan.md` exists.** This plan is archived (R1.4) and `CLAUDE.md` repo-state reflects v1.0 cut.

h. **`v1.0.0` tag exists locally.** Pushed by the user (R1.5). The tag points at the R1.4 commit (or the last R-series commit that didn't introduce stale state — the R1.4 commit is the canonical v1.0 commit because it captures both the version bump and the doc archive).

## Out of scope

- **TestLifter M2** (idempotence + commutativity in test bodies) — not a v1.0 blocker; see PRD §7.9 M2+.
- **Generator inference for opaque types beyond `.todo`** — PRD §16 #4 explicitly mandates `.todo`; no v1.0 work.
- **`--seed-override` / `--show-suppressed`** — explicitly v1.1+ per PRD §16 #6.
- **Adoption tracking dashboard / `swift-infer metrics`** — v1.1+ per PRD §17 header.
- **Calibration loop** — six-month dogfooding window per PRD §17.3; can't be done at the v1.0 cut.

## Open decisions

1. **Memory test sampling point.** Sample resident memory at end-of-discover, or peak across the run? PRD §13 row 4 says "memory ceiling" — peak is the conservative read. Resolution: sample peak via a background thread polling every 100ms; assert the maximum sample. (R1.1.b.)

2. **`--interactive` first-prompt timer endpoint.** From `Discover.run` entry to first `PromptInput.prompt(...)` invocation, or from process start to first prompt? PRD §13 row 5 says "first-prompt latency < 1 second after process start" — the latter is the user-visible measurement. Resolution: time from `Discover.run` entry; the `main` entry overhead is negligible and untestable from inside the test target. Document the gap in the test header. (R1.1.c.)

3. **Where does `CHANGELOG.md` live and what format?** `Keep a Changelog` (https://keepachangelog.com) is the de facto Swift convention. Resolution: repo root, Keep-a-Changelog format, single `## [1.0.0] — 2026-05-03` entry summarizing the M1–M8 + TestLifter M1 + R1.1 closure. (R1.3.)

4. **Tag message.** Annotated or lightweight? Annotated tags carry a message and are SemVer convention for releases. Resolution: annotated, with the v1.0 highlights as the message body. (R1.5.)

5. **`R1.1.h` interception mechanism.** `URLProtocol`-based interception only catches `URLSession`-routed traffic; raw `Network.framework` / `nw_path_monitor` paths slip through. Resolution: combine — `URLProtocol` interceptor for the URL-stack path AND a DNS-resolution sniff via `getaddrinfo` interception (or a simpler approach: assert that no `URLSession`, `URLRequest`, `NWConnection`, or `getaddrinfo`-prefixed symbol is reachable from the discover/drift call graph). Pragmatic version: the static grep already exists; the runtime test focuses on the URLSession path and documents the `Network.framework` gap as covered by the static check. Close that gap if the static check ever loosens. (R1.1.h.)
