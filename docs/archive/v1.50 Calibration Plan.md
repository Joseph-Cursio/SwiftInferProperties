# v1.50 Calibration Plan — full 109-surface verify (Phase 2 opening)

Working doc for the v1.50 cut (forty-seventh calibration cycle). **First cycle of Phase 2** — extending verify-mode measurement from the cycle-27 stratified 32-pick sample (cycles 41–46) to the **full 109-surface** that v1.29 froze. The first full-coverage verify measurement in this loop's history.

> **Motivation.** V1.49.G's cycle-46 measurement closed Phase 1.5 at **30/30 = 100% per-pick agreement** on the 32-pick sample with **8/8 = 100% verifier-mode REJECT lift**. The 32 picks were a stratified subset of v1.29's 109-surface; **77 picks have never been verify-measured**. Cycle-46 explicitly framed this as the Phase 2 opening question: *does the agreement-rate signal generalize?* If yes, the verify pipeline is a confidence-amplification tool worth integrating into the accept-flow. If no, the first disagreement is the load-bearing finding that reshapes Phase 2's priorities.

## What v1.50 ships

Eight workstreams + bundled closeout. The cycle pivots from **architecture-building** (v1.42–v1.49) to **measurement-instrumentation**. No new emitter, template, strategy, or resolver. All four risks below center on measurement infrastructure.

1. **V1.50.A — Cycle-27 source-package indexing fixture**. Create a developer-only SwiftPM workspace at `fixtures/cycle27-surface/Package.swift` with dependencies on swift-numerics + OrderedCollections + swift-algorithms + SwiftPropertyLaws (re-using the kit pin from the main `Package.swift`). The fixture's source target contains test-only `@testable import`s for OrderedCollections + `import` for the other three. Running `swift-infer index` against the fixture's source target produces a 109-pick `SemanticIndex` mirroring the cycle-27 surface composition.

   **Sub-decision — `@testable` vs `import` for OrderedCollections.** The OC surface includes 30+ internal-typed picks (`_Bucket`, `_HashTable.UnsafeHandle`, etc.). v1.50 ships with `@testable import OrderedCollections` in the fixture's source target — exposes the internals to the indexer. If `@testable` causes SwiftPM build issues (only valid in test targets, not source targets), fall back to a single-pick coverage strategy: index only public-API picks, mark internal-typed picks as `architectural-coverage-pending` in cycle-47 findings.

2. **V1.50.B — `verify --all-from-index` flag**. Extend `SwiftInferCommand.Verify` with an `--all-from-index` flag that loads the SemanticIndex (default path or via `--index-path`) and iterates every entry, running the verify pipeline per-entry. Outputs a JSON record per entry to stdout (one line each) with shape:

   ```json
   {
     "identityHash": "0xBC43...",
     "templateName": "round-trip",
     "primaryFunctionName": "exp(_:)",
     "carrier": "Complex<Double>",
     "outcome": "bothPass",
     "outcomeDetails": { "defaultTrials": 100, "edgeTrials": 100, "edgeSampled": 7 }
   }
   ```

   Parallelizes per-entry builds via a `TaskGroup` (concurrent verify workdirs; 4-8 simultaneous builds within macOS file-descriptor / RAM constraints).

   **Sub-decision — parallelism level**. 109 sequential verify runs × ~13s = ~24 min. Parallelizing to 8 concurrent builds drops this to ~3-5 min — but each build resolves SwiftPM dependencies, hitting disk + network. v1.50 ships with a `--max-parallel` flag (default 4) for tunability. Cycle-47 records the actual wall-clock + flake rate.

3. **V1.50.C — Per-pick coverage classification**. After V1.50.B's `verify --all-from-index` run completes, a post-processing pass classifies each of the 109 picks against one of five categories:

   - `measured-bothPass`: verify ran and reported `.bothPass`.
   - `measured-edgeCaseAdvisory`: verify ran and reported `.edgeCaseAdvisory`.
   - `measured-defaultFails`: verify ran and reported `.defaultFails`.
   - `measured-error`: verify ran but the subprocess errored (build failure, missing dependency, etc.). Recorded with the error reason.
   - `architectural-coverage-pending`: V1.49 emitter would handle the pick, but the source-package indexing missed the type (e.g., a user-package type not in the cycle-27 fixture). Pre-v1.50 the cycle-46 #10/#11 picks fall here.

   The classification output JSON lives at `docs/calibration-cycle-47-data/full-surface-outcomes.json` (versioned alongside cycle-27 data).

4. **V1.50.D — Aggregate metrics**. From the classification, compute:

   - **Full-surface verifiable-fraction**: `measured / 109` (across all `measured-*` categories).
   - **Full-surface verifier-mode REJECT lift**: `measured-defaultFails matches cycle-27-REJECT / cycle-27-REJECT count`. Target: stays at 100% (8/8) if no new REJECTs surface or all new REJECTs verify-confirm; otherwise the v1.50 finding documents the gap.
   - **Per-template agreement-rate**: for each of the 7 templates, count how many picks have `(measured outcome) ↔ (cycle-27 verdict)` agreement.
   - **Per-carrier agreement-rate**: same breakdown per carrier.
   - **Disagreements**: the load-bearing measurement. Each disagreement is recorded with full evidence (pick identity, cycle-27 verdict, measured outcome, candidate explanation).

5. **V1.50.E — Tests**. Three areas:

   - **E.1** Unit tests for `verify --all-from-index`'s argument parsing (~3 tests).
   - **E.2** Unit tests for the per-entry-iteration loop on a synthetic 3-entry index (~5 tests).
   - **E.3** Subprocess integration test: a 3-entry synthetic fixture (each pick a known-good closure) → `verify --all-from-index` produces 3 lines of `.bothPass` JSON (~1 test). Tagged `.subprocess`.

   Aggregate: ~9 new tests. Test count goal: 2393 → ~2402.

6. **V1.50.F — Cycle-47 calibration measurement**. Per the cycle-46 protocol but full-surface:

   - Run `verify --all-from-index` against the V1.50.A fixture's index. Record outcomes to `docs/calibration-cycle-47-data/full-surface-outcomes.json`.
   - Classify per V1.50.C. Compute metrics per V1.50.D.
   - Compare against cycle-27's verdict map (held in `docs/calibration-cycle-27-data/triage-decisions.json`). Each disagreement (measured outcome ≠ predicted cycle-27 outcome) is the load-bearing finding.
   - Write up as `docs/calibration-cycle-47-findings.md`. The headline is the **per-pick agreement-rate signal generalized to 109 picks**.

   **Out-of-scope for v1.50**: re-running the 32-pick measurement (cycle-46's results are stable; only the 77 unmeasured picks are new). Accept-flow integration (verify outcomes → `decisions.json`) — that's v1.51+ if cycle-47 reports good agreement. Verification cache.

7. **V1.50.G — Source-of-truth survey output**. `docs/calibration-cycle-47-data/full-surface-outcomes.json` is the first full-surface measurement artifact. It should:

   - Be checked into the repo (cycle-data is canonical).
   - Be sorted by `identityHash` for stable diffs across runs.
   - Include a top-level `surveyVersion: 1` field for v1.51+ format evolution.
   - Have a sibling `full-surface-summary.md` that human-summarizes the aggregate metrics (similar to cycle-27's `surface-counts.md`).

8. **V1.50.H–J** bundled closeout — perf baseline (`docs/perf-baseline-v1.50.md`), version → 1.50.0, CLAUDE.md "Repository state" pointer + Phase 2 opening note, tag + push.

## Scope boundaries

- **In scope (v1.50):** Cycle-27 source-package indexing fixture; `verify --all-from-index` CLI flag with per-entry parallelism; per-pick classification scheme (5 categories); full-surface verifiable-fraction + verifier-mode REJECT lift + per-template/per-carrier agreement-rate metrics; cycle-47 findings doc as the first full-surface measurement; survey output JSON + summary md as cycle-data artifacts.
- **Out of scope, deferred to v1.51+:** Accept-flow integration (verify outcomes → `decisions.json`); verification cache; "Verified" first-class tier; PRD §20.2's higher-order property composition; full re-derivation of the 109-surface against the v1.49 emitter (cycle-27's heuristic verdicts stay as ground truth for cycle-47 comparison); any architectural extension beyond v1.49 (no new templates, carriers, strategies, or emitters).
- **Not changing:** Any v1.42–v1.49 verify pipeline behavior stays bit-stable. The new `--all-from-index` flag is strictly additive to the `verify` subcommand. Existing tests stay green.

## Risks

1. **`@testable` access for OrderedCollections internals**. SwiftPM's `@testable import` is only valid in test targets, not source targets. The V1.50.A fixture may need to be a test target rather than a source target — which changes the indexer's reach. If `swift-infer index` can't run against a test target's sources, the fallback is to mark all OC internal-typed picks as `architectural-coverage-pending` (cycle-47 records the gap). **Mitigation**: V1.50.A.1 (first implementation pass) probes this immediately; if blocked, the fallback is documented in cycle-47.

2. **109 verify-runs × ~13s parallelized = wall-clock load**. 4-8 concurrent SwiftPM builds will spike disk + network during the survey. The §13 perf-flake mitigation from V1.49.D (the `.subprocess` tag + `--skip` CI pattern) keeps the survey out of `swift test`'s default suite — but a developer running the survey locally still pays the full ~5-min cost. **Mitigation**: the survey is a one-shot developer tool, not a CI gate. v1.51+ can add caching (verification cache priority candidate) if the survey runs become a recurring cost.

3. **First verify-vs-heuristic disagreement may surface**. Five cycles of 100% agreement on 30 picks is a strong signal but bounded — 77 new picks could break the streak. Each disagreement is **methodologically valuable** (informs Phase 2 priorities) but also **destabilizing** to the Phase 1.5 close-out narrative. **Mitigation**: cycle-47 findings doc frames disagreements as expected-surface-discovery, not regression. If the rate is high (e.g., >5% disagreement), v1.51 priorities shift to closing the gap.

4. **Internal-typed picks may error at verify-time even when the indexer sees them**. Even with `@testable`, the verifier workdir is a separate SwiftPM build; accessing internals there requires either the workdir to also be `@testable`-built, or for the verify stub to compile against the internal type's visibility. Some picks (#10 `_Bucket` etc.) may need both `@testable` indexing AND the workdir's stub source built with internals visible. **Mitigation**: V1.50.B's `verify --all-from-index` propagates the build-error to the `measured-error` classification; cycle-47 records the per-pick error reasons.

5. **The `triage-decisions.json` cycle-27 source-of-truth may not have all 109 entries.** If cycle-27 only triaged the 32-pick sample (not the full 109), there's no ground-truth verdict for the 77 unmeasured picks to compare against. **Mitigation**: V1.50.F's findings doc establishes the cycle-47 measurement as the *first* verdict source for the 77; comparisons against cycle-27 only apply to the overlapping 32.

6. **Parallelism contention may exceed file-descriptor limits**. macOS default soft limit is 256 file descriptors per process; SwiftPM builds open many concurrent files. 8 parallel builds × per-build FD count may exceed the limit. **Mitigation**: V1.50.B's default `--max-parallel 4` keeps headroom; users who hit FD-limit errors can lower it; cycle-47 records observed FD pressure.

7. **`verify --all-from-index` output JSON shape stability**. The per-line JSON format will be persisted to cycle-data artifacts; future cycles depending on this shape will break if the field set changes. **Mitigation**: V1.50.G's top-level `surveyVersion: 1` field lets v1.51+ extend the format without breaking parsers. The output schema is documented in `verify --help` and in the cycle-47 findings.

8. **Cycle-47 may not actually run all 109 picks**. Real-package access constraints (internal types, missing test-fixture entry points) could cap measurement at, say, 60-80 picks. The remainder fall to `architectural-coverage-pending`. **Mitigation**: cycle-47 findings frame the measurement as "v1.50's measured-coverage subset within the full 109"; the agreement-rate signal applies to that subset; v1.51+ closes the residual gap.
