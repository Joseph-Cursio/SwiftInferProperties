# v1.36 Calibration Plan — Constraint Engine Foundation (PRD §20.2, part 1)

Working doc for the SwiftInferProperties v1.36 cut. v1.36 is the **thirty-third calibration cycle** and the **first cycle of the multi-cycle Constraint Engine refactor** (PRD §20.2).

> **v1.36 = foundation-laying cycle.** PRD §20.2 calls for replacing "templates as patterns over signatures" with "constraints over a function graph + types + usage." This is a multi-version architectural shift; v1.36 introduces the `Constraint` abstraction + a runner + migrates **one template (CommutativityTemplate)** as proof-of-concept. The other 9 templates continue to use their existing bespoke implementations in v1.36 — incremental migration is v1.37+. **Calibration data stays stable** because the runner produces bit-for-bit-equivalent suggestions to the bespoke matcher (verified via equivalence tests against a fixture corpus).

> **Why CommutativityTemplate first.** It's the simplest unary-summary template with no exotic dependencies (no `LiftedTransformation`, no `FunctionPair`, no `IdentityElementPair`, no `DualStylePair`). Its score arithmetic + signals are stable across cycles 17/20/23/25/27 at 33% accept rate. Migrating it first validates the abstraction's API against a representative-but-simple shape before tackling the more complex pair / lifted templates in subsequent cycles.

## V1.36.A — `Constraint<Subject>` data model

New file `Sources/SwiftInferCore/Constraint.swift`. The Constraint is a **data description** of how to match + score a property pattern against an input subject.

```swift
/// V1.36.A — Constraint Engine (PRD §20.2). A `Constraint<Subject>`
/// describes a property pattern as data: subject-shape gate, signal
/// accumulator, evidence builder, identity-key builder. Templates
/// express themselves as Constraint values; the ConstraintRunner
/// (V1.36.B) orchestrates them to produce Suggestion outputs.
public struct Constraint<Subject>: Sendable {

    /// Stable template-id string (e.g. "commutativity"). Used as the
    /// `Suggestion.templateName` value.
    public let templateName: String

    /// Subject-shape gate. Returns `true` iff the constraint applies to
    /// this subject (e.g., for commutativity: "is this a binary op
    /// `(T, T) -> T`?"). Pure function — no side effects.
    public let appliesTo: @Sendable (Subject) -> Bool

    /// Signal accumulator. Returns the list of signals to score against.
    /// Includes both positive signals (type shape, name match) and
    /// counter-signals (direction labels, asymmetric markers). Vetoes
    /// surface here too — the runner detects `Signal.isVeto` and
    /// collapses to `.suppressed`.
    public let signals: @Sendable (Subject) -> [Signal]

    /// Evidence builder for the §4.5 explainability block.
    public let evidence: @Sendable (Subject) -> [Evidence]

    /// Identity-key builder for the PRD §7.5 canonical-input hash.
    public let identity: @Sendable (Subject) -> SuggestionIdentity

    /// Carrier-type accessor for V1.34.A's `Suggestion.carrier` field.
    /// Returns `nil` for templates whose subjects don't expose a
    /// natural carrier (free functions, etc.).
    public let carrier: @Sendable (Subject) -> String?

    public init(
        templateName: String,
        appliesTo: @Sendable @escaping (Subject) -> Bool,
        signals: @Sendable @escaping (Subject) -> [Signal],
        evidence: @Sendable @escaping (Subject) -> [Evidence],
        identity: @Sendable @escaping (Subject) -> SuggestionIdentity,
        carrier: @Sendable @escaping (Subject) -> String? = { _ in nil }
    ) {
        self.templateName = templateName
        self.appliesTo = appliesTo
        self.signals = signals
        self.evidence = evidence
        self.identity = identity
        self.carrier = carrier
    }
}
```

**Notes**:
- The generic `Subject` lets unary templates use `Constraint<FunctionSummary>`, pair templates use `Constraint<FunctionPair>`, lifted templates use `Constraint<LiftedTransformation>`, etc. No type erasure needed at the data-model layer.
- The closures are `@Sendable` so a Constraint can be passed across actor boundaries (future-proofing for the v1.37+ multi-template registry).
- `carrier` defaults to `nil` so simple templates don't have to wire it.
- Explainability is built by the runner from `evidence + signals`; not a separate Constraint field (keeps surface narrow).

**Tests**: `ConstraintTests.swift` — verify field defaults (carrier returns nil), basic invocation of each closure against a fixture subject.

## V1.36.B — `ConstraintRunner.suggest(constraint:subject:)`

New file `Sources/SwiftInferCore/ConstraintRunner.swift`. Orchestrates the constraint:

```swift
public enum ConstraintRunner {

    /// Build a `Suggestion` from a `Constraint<Subject>` applied to a
    /// specific `subject`. Returns `nil` when:
    ///   1. `constraint.appliesTo(subject)` returns `false` (gate fails)
    ///   2. The accumulated `Score` lands in `.suppressed` tier (veto
    ///      present, or signal sum below the §4.2 threshold)
    public static func suggest<Subject>(
        constraint: Constraint<Subject>,
        subject: Subject
    ) -> Suggestion? {
        guard constraint.appliesTo(subject) else { return nil }
        let signals = constraint.signals(subject)
        let score = Score(signals: signals)
        guard score.tier != .suppressed else { return nil }
        let evidence = constraint.evidence(subject)
        return Suggestion(
            templateName: constraint.templateName,
            evidence: evidence,
            score: score,
            generator: .m1Placeholder,
            explainability: makeExplainability(evidence: evidence, signals: signals),
            identity: constraint.identity(subject),
            carrier: constraint.carrier(subject)
        )
    }

    /// Default explainability assembly — concatenates evidence-display
    /// strings + each signal's `formattedLine`. Templates that need a
    /// custom block can wrap the runner output and override
    /// `suggestion.explainability` (kept narrow for v1.36).
    private static func makeExplainability(
        evidence: [Evidence],
        signals: [Signal]
    ) -> ExplainabilityBlock {
        // ...
    }
}
```

## V1.36.C — `CommutativityTemplate` migration

The current template's `suggest(for:)` is a hand-rolled function that:
1. Checks the binary-op type shape
2. Accumulates type-shape + name-match signals (+ counter-signals)
3. Computes score, checks tier
4. Builds Suggestion + Evidence + identity

The v1.36.C migration expresses the same logic as `commutativityConstraint: Constraint<FunctionSummary>` data, and re-implements `CommutativityTemplate.suggest(for:)` as:

```swift
public static func suggest(for summary: FunctionSummary, ...) -> Suggestion? {
    ConstraintRunner.suggest(
        constraint: commutativityConstraint(/* params */),
        subject: summary
    )
}
```

**Critical invariant**: the migrated implementation must produce **bit-for-bit identical** `Suggestion` outputs as the old one on every input. Verified via:
- All ~30 existing `CommutativityTemplateTests` continuing to pass without modification.
- A new `CommutativityConstraintEquivalenceTests` suite that generates 50+ synthetic FunctionSummaries and asserts `oldSuggestNoConstraint(_:) == newSuggestViaConstraint(_:)`.

The migration is intentionally a refactor without behavior change. If the equivalence tests pass, the v1.36 plan-vs-actual is clean and v1.37+ can migrate the next template safely.

**Risk mitigation**: keep a `// V1.36.C — Constraint-based implementation; see commutativityConstraint below for the data form. Behavior preserved bit-for-bit per the V1.36.D equivalence tests.` comment at `suggest(for:)` so a future reader sees the design intent.

## Scope boundaries

- **In scope**: Constraint abstraction, runner, one template migrated, equivalence tests against fixture corpus.
- **Out of scope (v1.37+)**: migrate the remaining 9 templates (round-trip, idempotence, monotonicity, associativity, inverse-pair, identity-element, dual-style-consistency, composition, invariant-preservation). Each is a discrete future cycle.
- **Out of scope (v1.38+)**: cross-constraint composition ("a Group constraint composes a Semigroup constraint + an identity-element constraint + an inverse-pair constraint"). The v1 architecture is described as constraint-engine-ready; v1.36 lays groundwork but doesn't yet implement composition.
- **Out of scope (v1.39+)**: project-vocabulary constraint registration (user-defined constraints loaded from `.swiftinfer/constraints.swift` or similar).
- **Out of scope this cycle**: changing the calibrated signal weights / thresholds. The Constraint abstraction is a refactor of the matcher, not a recalibration of the scoring.

## Risk + mitigation

- **Risk**: the equivalence tests miss a subtle output difference (e.g., signal ordering, identity-hash drift). **Mitigation**: assert structural equality on the whole `Suggestion`, not just `score.total`. Run the full pre-migration test suite to verify no regressions in downstream consumers.
- **Risk**: `@Sendable` closure capture in the Constraint introduces compile-time issues (e.g., capturing non-Sendable `Vocabulary`). **Mitigation**: keep V1.36.A simple — only the constraint's data fields are stored; per-call inputs come from the subject. Vocabulary plumbing (when needed) is via the existing per-template private helpers.
- **Risk**: scope creep into "migrate all 10 templates this cycle." **Mitigation**: explicit "one template, equivalence tests, done" gate. v1.37+ continues incrementally.

No active milestone plan opens with v1.36.
