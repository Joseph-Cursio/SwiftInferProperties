### TestLifter M16 — General consumer-producer chain detection

PRD v1.0 §7.8 second example, generalized. Closes the M10 plan's deferred Option A: where M10 narrowed inferred-domain detection to known M5 round-trip pairs, M16 lifts that filter to **any** (consumer, producer) chain meeting a five-criterion narrow scope:

1. **Threshold:** consumer is observed on `≥ 3` test sites (mirrors M4.3 / M9 / M10).
2. **Homogeneity:** every site, post-identifier-resolution, classifies as `.callOutput(producerName: P)` for the same `P`. One outlier kills the chain (PRD §3.5).
3. **Producer-existence:** `P` resolves to a `FunctionSummary` from the source target's scanned function set. Stdlib initializers (`String("hello")`, `Int(s)`) and unknown identifiers fail this check.
4. **Type-alignment:** `producerSummary.returnTypeText == consumer.parameters[0].typeText` (textual `==`).
5. **Anti-double-fire:** no M5 round-trip pair `(forwardName: P, reverseName: consumer)` exists. M10 owns the round-trip case; M16 stays out of its lane.

M16.0 added `HintOrigin` (`.roundTripPair` / `.consumerProducerChain`) on `DomainHint` with default-back-compat (every M10 call site keeps compiling). M16.1 shipped `ConsumerProducerChainDetector` enforcing all five criteria; reuses M10's `ProducerVetoReason` + `DomainInferrer.computeVeto` verbatim for the four producer-veto checks (throws / async / multi-arg / non-generatable). M16.2 wired the detector through `LiftedSuggestionPipeline.promote(...)` as a sibling to the M11 advisory union; promoted suggestions enter the discover stream with `templateName == "consumer-producer-chain"` + `Tier.advisory` (per PRD §7.8 — documentation, not a runnable property). M16.3 added the accept-flow renderer arm — comment-only writeout to `Tests/Generated/SwiftInfer/consumer-producer/<consumer>_<producer>.swift` via the M11-shaped out-of-band `consumerProducerChainHintsByIdentity` side-map (preserves the §13 row 4 memory ceiling — no inline storage on every `Suggestion`).

Plan deviation: M16.3 surfaced an existing M10 gap — `DomainCorpusScanner.classify(_:)` treated `try`/`await`-wrapped calls as `.other`, which killed homogeneity for any throws/async producer chain. The M16.3 commit extended `classify(_:)` to peel `try`/`try!`/`try?`/`await` wrappers; M10 + M16 both now surface vetoed throws/async producer hints with the matching advisory comment, where they previously fell silent.

Sub-milestones M16.0–M16.3 closed in `docs/archive/TestLifter M16 Plan.md`. After M16, the only remaining narrow follow-up that could open as a non-§20 v1.x milestone is cross-test data-flow correlation (`let x = format(t)` in `testA` and `validate(x)` in `testB`) — explicitly deferred from M10 and M16, natural sequencing is post-SemanticIndex.

- **Multi-predicate equivalence classes (M13's deferred axis 3).** Different predicates per bucket where they're not negations of each other. Still deferred — same SemanticIndex-sequencing constraint as M12.
- **Cross-target enum coverage for N-class `coversDomain` (M14's deferred bit).** Extends axis 4 beyond same-target enums. Still deferred — same SemanticIndex-sequencing constraint as M12.
- **Multi-producer disambiguation.** When the corpus has `validate(formatJSON(t))` AND `validate(formatXML(t))` sites, M16's homogeneity criterion fails by definition. A future Option-A-deeper expansion could surface "validate's domain is `formatJSON ∪ formatXML` output" but is out of v1.3 scope.

- **PRD §20 v1.1+ trajectory.** SemanticIndex, IDE integration, `swift-infer apply`, `swift-infer metrics`. Carries forward as the post-v1.3 trajectory; SemanticIndex is the largest single lift.

- **Calibration loop.** Same posture as prior cuts — six-month dogfooding window per PRD §17.3.

## Open decisions

1. **CHANGELOG entry granularity for the M10 `try`/`await` follow-up.** The `DomainCorpusScanner.classify(_:)` peeling change is technically a M10 widening (vetoed throws/async round-trip-pair hints now render where they previously fell silent), discovered during M16.3 testing. Should the CHANGELOG mention this as a separate bullet under "TestLifter" or fold it into the M16 line? Default proposal: **(a) fold into the M16 line as a parenthetical** (mirrors v1.2's posture on M13's MarkerTable lift — the user-facing surface is "M16 ships chain detection"; the scanner widening is the implementation that makes the throws/async veto surface). Rationale: from the user's POV, the visible behavior is "throws/async producer chains now show up with a veto comment" — the consumer doesn't care which scanner widened to make it work.

2. **Perf baseline diff against v1.2.** The v1.3 baseline file should either (a) replace the v1.2 baseline as the canonical regression anchor or (b) sit alongside as `docs/perf-baseline-v1.3.md` with v1.2 retained for forensic comparison. Mirrors the v1.1 → v1.2 cut's posture; the v1.2 baseline retains forensic value for understanding the M16 trajectory's perf delta over the same six-detector-fan-out + N-class detector run.

After v1.3 ships, the natural next contributions are all SemanticIndex-blocked (or release-cut work):

- **PRD §20 SemanticIndex** — biggest v1.x lift. Was originally deferred from v1.0; the constraint-engine + index unlock the IDE integration + `apply`/`metrics` items downstream and cleanly enables M12 / M13.+ / M14 cross-target / cross-test data-flow correlation.
- **PRD §20 IDE integration** — depends on SemanticIndex.
- **PRD §20 `swift-infer apply`** — moves from "all output is opt-in via `--interactive`" to "single command applies a baseline set of accepted suggestions." Depends on a stable v1.1+ baseline-decisions surface (which already exists at v0.1.0).
- **PRD §20 `swift-infer metrics`** — surfaces the `.swiftinfer/decisions.json` aggregation per PRD §17.
- **Multi-predicate equivalence classes** (M13's deferred axis 3) — extends §7.8 example 3 beyond predicate-negation buckets. SemanticIndex-blocked (sibling to M12).
- **Cross-target enum coverage** (M14's deferred bit) — extends axis 4 beyond same-target enums. SemanticIndex-blocked.
- **Cross-test data-flow correlation** (M10 + M16's deferred surface) — `let x = format(t)` in `testA` and `validate(x)` in `testB` correlation. Narrow but complex; natural sequencing is post-SemanticIndex.
