# v1.41 Calibration Plan — Dominant-Pattern Cluster Classification

Working doc for the v1.41 cut (thirty-eighth calibration cycle). Closes the **v1.35 carrier-aware-refactor finding**: OrderedSet's 29-suggestion cluster classifies as `algebraicStructure` despite its dominant pattern being dual-style-consistency (12 of 29 entries).

> **Diagnosis.** The current `RefactorClusterAnalyzer.classify` fires `algebraicStructure` as soon as 2+ distinct algebraic templates (commutativity / associativity / identity-element) have ≥1 entry — without checking whether those algebraic templates are actually the dominant pattern. For OrderedSet (29 total: dual-style 12 + idempotence 5 + monotonicity 5 + round-trip 3 + associativity 2 + commutativity 2), the 4 algebraic entries = 14% of the total. The dual-style 12 entries = 41% are the real story. The current `suggest-refactors` output points the user at Semigroup/Monoid conformance when SetAlgebra is the more useful target.

> **Fix.** Add a count-weighted check to the algebraicStructure path: 2+ distinct algebraic templates AND their *sum* must be ≥50% of the cluster total. Otherwise fall through to per-template thresholds (priority 2–5 unchanged). This preserves Complex's algebraicStructure classification (algebraic sum 12 / 20 = 60% ≥50%) while reclassifying OrderedSet to dualStyleCluster.

## New `classify` logic

```swift
// Priority 1: algebraicStructure (2+ distinct algebraic templates,
// collectively ≥50% of total).
let algebraicTemplates: Set<String> = [
    "commutativity", "associativity", "identity-element"
]
let algebraicSum = algebraicTemplates.reduce(0) { sum, name in
    sum + (perTemplateCounts[name] ?? 0)
}
let presentAlgebraicCount = algebraicTemplates.filter { (perTemplateCounts[$0] ?? 0) > 0 }.count
if presentAlgebraicCount >= 2,
   algebraicSum * 2 >= total {
    return .algebraicStructure
}
// Priority 2–5: unchanged.
...
```

## Scope boundaries

- **In scope**: algebraicStructure path only. Priority 2–5 thresholds (idempotenceCluster / dualStyleCluster / roundTripCluster / generalCluster) are unchanged — they're already single-template thresholds.
- **Out of scope**: other dominant-pattern refinements. e.g., a cluster with 4 idempotence + 8 monotonicity would currently fire `idempotenceCluster` (≥3 idempotence) despite monotonicity dominating; we'd need a `monotonicityCluster` shape to address it. The 5 current shapes cover the known cases; expansion is a future cycle if a real corpus surfaces a miss.
