# v1.18 Calibration Plan — Value Semantics workstream

**Status:** Draft / proposal — pre-empirical, framing only.
**Date:** 2026-05-09.
**Origin:** ChatGPT conversations on value semantics + property-based testing in `~/Library/Mobile Documents/iCloud~md~obsidian/Documents/LLM inspired Projects/SwiftLint Rule Studio/SwiftProtocolLaws/Value Semantics/_{1,2,3}_*.md`.

Working doc for the SwiftInferProperties v1.18 cut. v1.18 is the **fifteenth calibration cycle** and the first cycle organised around a *single conceptual axis* (value semantics) rather than a single template-class. Three workstreams, sequenced across two mechanism cycles + one empirical cycle.

> v1.18 is a **mechanism cycle** (workstreams A + C); v1.19 is a follow-on mechanism cycle (workstream B); v1.20 is empirical-only measurement on the post-v1.19 surface. The split exists because workstream B (the lift) is large enough on its own and depends on workstream A landing first.

**Ephemeral** — archive to `docs/archive/v1.18 Calibration Plan.md` once the v1.18 tag is pushed.

## 1. Why this cycle is different

PRD §17.3's empirical-tuning loop has run 14 cycles to date. Every prior cycle has tuned an existing template's signal stack — direction labels, domain markers, SetAlgebra shape gates, op-name extensions — within the gate of `!summary.isMutating`. Every template gates on this (`IdempotenceTemplate.swift:124`, `RoundTripTemplate` via `FunctionPairing.swift:69`, `AssociativityTemplate.swift:113`, `CommutativityTemplate.swift:119`, `MonotonicityTemplate.swift:127`, `IdentityElementPairing.swift:154`, `InverseElementPairing.swift:103`).

**Consequence:** every `mutating func` in scanned source is invisible to the engine. Stdlib's `Array.append`, `Set.formUnion`, `Dictionary.merge`, `String.append`; protocol-extension `formUnion` / `formIntersection` siblings on `SetAlgebra`; every domain `mutating func normalize()` / `mutating func add(_:)`. None of these can earn an idempotence, identity, or composition suggestion today.

The ChatGPT `_3_Mutating API to Property Tests.md` conversation is the unlock: lift `mutating func op(self, x)` into the pure shadow `func op'(_ s: Self, x: X) -> Self { var c = s; c.op(x); return c }`, then run all the same algebraic templates against the lifted form. **Value semantics on the carrier is what makes the lift sound** — without it, the `var c = s` copy aliases shared state and the laws don't hold.

This reframes two existing items on the roadmap:
- **"Reference-type carrier counter-signal"** (`docs/calibration-cycle-14-findings.md:142`, carried forward 4 cycles, projected small empirical effect) — was scored as a precision micro-tweak; reframes as a **necessary precondition** for the lift.
- **`docs/ideas/ValueSemantic Kit Proposal.md` M-VS-1** (structural value-semantic detector) — was scoped as a downstream consumer of a future kit-side `ValueSemantic` protocol; reframes as a **standalone v1.18 mechanism** that doesn't need the kit work to land first.

## 2. The three workstreams

Each workstream is independently mergeable. v1.18 ships A + C; v1.19 ships B (depends on A).

### Workstream A — Carrier value-semantics signal

**What:** New `Signal.Kind` cases consumed by `IdempotenceTemplate`, `RoundTripTemplate`, `InversePairTemplate`, `IdentityElementPairing`, `InverseElementPairing`:

- `Signal.Kind.referenceTypeCarrier` (counter, `−10`) — carrier resolves via `TypeDecl.kind == .class || .actor`.
- `Signal.Kind.valueSemanticCarrier` (positive, `+5`) — carrier resolves to `kind == .struct || .enum` AND every entry in `storedMembers` has a `typeName` that:
  - is in the curated stdlib value-type allow-list (`Int`, `Int8/16/32/64`, `UInt*`, `Bool`, `Double`, `Float`, `String`, `Substring`, `Character`, `Data`, `Date`, `URL`, `UUID`, `Decimal`, `Range`, `ClosedRange`, `Optional<…>`, `Result<…, …>`, `Array<…>`, `Dictionary<…, …>`, `Set<…>`, tuple), OR
  - resolves via same-corpus `TypeDecl` lookup to another `kind == .struct || .enum` whose own `storedMembers` recursively pass (depth-bounded; 3 levels to avoid cycles), OR
  - is a generic parameter (treated as value-semantic by assumption — stdlib convention).

- Mixed (struct with one class-typed stored property, or one closure-typed stored property) → **no signal** (conservative; the kit proposal's §2.2 worked examples 2 and 3 are both bugs that look value-semantic structurally and would falsely score positive otherwise).

**Carrier resolution scope:** same-file only initially, matching `TypeShapeBuilder`'s current posture per `MarkerTable.swift` lookup conventions. (Open decision #1.)

**Why the magnitudes:** the negative is intentionally larger than the positive — false positives on `class` carriers are sharper bugs than missed value-semantic positives. Both magnitudes are starting points subject to v1.20 calibration tuning.

**Cost:** ~1 day. New `CarrierKindResolver` in `SwiftInferCore/`, `Signal.Kind` extension, threading through five template scoring stacks, ~30 unit tests covering the worked examples from the kit proposal.

**Mechanism-class taxonomy entry:** new class **9 — carrier-kind structural counter/positive** (currently 8 classes per CLAUDE.md repo-state).

### Workstream B — Mutating-method lift *(v1.19, depends on A)*

**What:** A new `LiftedTransformation` value type in `SwiftInferCore/`, derived during `FunctionScanner`'s second pass from any `FunctionSummary` matching:

```
isMutating == true
&& containingTypeName != nil
&& Workstream-A's valueSemanticCarrier signal is positive
```

The lift is **purely metadata** — no codegen, no source rewrite. It exposes the function as if it had been declared:

```swift
func <name>'(_ self: T, <params>) -> T  // shadow form
```

Templates updated to consume `LiftedTransformation` alongside `FunctionSummary`:

- **IdempotenceTemplate** — admits the lift if the original is `mutating func op() -> Void` (no params, lifted shape `(T) -> T`) or `mutating func op(x: T) -> Void` with `x` matching the carrier (lifted shape `(T) -> T` via curry).
- **IdentityElementPairing** — admits the lift for `mutating func op(by: X) -> Void` looking for an `X` value where `op'(s, x) == s` for all `s` (the ChatGPT example: `incremented(c, by: 0) == c`).
- **NEW: CompositionTemplate** — admits the lift for `mutating func op(by: X) -> Void` where `X: AdditiveArithmetic` (or a curated additive-monoid set), checking `op'(op'(s, a), b) == op'(s, a + b)`. This is the "increments compose additively" example from `_3_Mutating API to Property Tests.md` step 3.2.
- **InversePairTemplate** — admits dual-lifts: if `mutating func add(_:)` and `mutating func remove(_:)` both lift, propose `add(remove(s, x), x) == s` (inverse on lifted shadows).

Suggestions render against the lifted form with an explicit rationale line:

```
// Lifted from `mutating func MyType.increment(by:)`.
// Property holds iff `MyType` has value semantics — `var copy = original; copy.increment(by: x)` does not aliase original's state.
```

**Why this is high-leverage:** doubles the addressable function surface. Conservative estimate based on hand-counting `mutating func` declarations across the four cycle-1..14 corpora (Algorithms, Collections, OrderedCollections, ChartMath): 80-120 new candidates entering the scoring pipeline.

**Cost:** ~1 calibration cycle (3-5 days). New `LiftedTransformation` summary, new `CompositionTemplate`, fan-out updates to 4 existing templates, new mechanism-class entry **10 — mutation-lift admission**, ~80 unit tests.

**Why it depends on A:** the lift is sound only on value-semantic carriers. Admitting it on class carriers (which Swift forbids `mutating func` on, but where the equivalent shape is just `func` mutating an internal `var`) would generate properties that pass on the test runner but mask aliasing bugs in production. (Open decision #2.)

### Workstream C — Dual-style pair detection *(v1.18, independent)*

**What:** Detect canonical Swift dual-style pairs — `mutating func add(_:)` + `func adding(_:) -> Self`, `mutating func sort()` + `func sorted() -> Self`, `mutating func formUnion(_:)` + `func union(_:) -> Self`, etc. — via name pairing rules + type-shape match.

Pairing rules (curated, extensible via `vocabulary.json`):
- `X` ↔ `Xing` (Swift API Design Guidelines convention: `add` / `adding`, `append` / `appending`, `insert` / `inserting`)
- `formX` ↔ `X` (the `SetAlgebra` and `Sequence` sibling convention: `formUnion` / `union`, `formIntersection` / `intersection`)
- `X` ↔ `Xed` (the past-participle convention: `sort` / `sorted`, `reverse` / `reversed`, `normalize` / `normalized`)

Type-shape match:
- mutating member: `mutating func op(<params>) -> Void`, returns `Void` or `@discardableResult` something
- non-mutating member: `func op'(<same params>) -> Self` on the same containing type

When both are present, emit a **consistency** suggestion (new template `DualStyleConsistencyTemplate`):

```swift
let original = a
var mutated = a
mutated.<mutating-name>(<args>)
return mutated == a.<non-mutating-name>(<args>)
```

This is conversation `_2_Designing APIs Around Transformations.md`'s "sweet spot" pattern verbatim. Stdlib follows it religiously across `SetAlgebra`, `Sequence`, `Collection`. Failures are real bugs — the two implementations *should* agree, and divergence is one of the highest-signal-to-noise findings PBT can surface.

**Cost:** ~2-3 days. Reuses `FunctionPairing` infrastructure. New mechanism-class entry **11 — dual-style pair consistency**, ~25 unit tests.

**Why high precision:** this template *requires* both members to exist on the same type — there is no "single-function" fallback. The pairing constraint means false positives only fire when a developer uses one of the curated pair names for *non-paired* purposes (rare; the conventions are deeply embedded in Swift API Design Guidelines).

## 3. What v1.18 ships

Six deliverables (mirroring the v1.16 mechanism-cycle shape):

1. **Workstream A implementation.** New `SwiftInferCore/CarrierKindResolver.swift`, extension to `Signal.Kind`, threading through `IdempotenceTemplate` + `RoundTripTemplate` + `InversePairTemplate` + `IdentityElementPairing` + `InverseElementPairing`. ~30 unit tests.

2. **Workstream C implementation.** New `SwiftInferTemplates/DualStyleConsistencyTemplate.swift`, new `SwiftInferTemplates/DualStylePairing.swift` (pairing rules + matcher), curated pair-name list with hooks for `vocabulary.json` extension. ~25 unit tests.

3. **Pre-mechanism rate measurement.** Re-run the v1.17 229-surface harness with workstream-A signals enabled and workstream-C template enabled. Snapshot before/after suggestion counts per cell (per-template × per-corpus matrix). Confirms that A is precision-positive (suppresses class-carrier picks that were marginal) and C is recall-positive (introduces new picks that didn't exist before).

4. **Cycle-15 findings writeup.** New `docs/calibration-cycle-15-findings.md` documenting:
   - Workstream A counter-signal hits per corpus (which class/actor carriers got demoted).
   - Workstream A positive-signal hits per corpus (which struct carriers cleared the value-semantic check).
   - Workstream C new-pick count per corpus + sample of paired suggestions emitted.
   - Mechanism-class taxonomy update: 8 → 11 classes.
   - v1.19 pre-implementation surface for workstream B (function count of `mutating func` declarations across the four corpora that pass workstream A's value-semantic gate).
   - Cycle-16 priority list rotated post-v1.18.

5. **Perf baseline.** New `docs/perf-baseline-v1.18.md` measuring impact of the carrier resolver pass + dual-style pairing on the four cycle-1..14 corpora. Expected delta: ≤+5% wall time (carrier resolver is single-pass over `TypeDecl` cache; dual-style pairing reuses `FunctionPairing` candidate index).

6. **Version + CHANGELOG + README.** CLI version `1.17.0` → `1.18.0`; CHANGELOG `[1.18.0]` entry covering workstreams A + C + the new mechanism-class entries; README status block + version pin + cycle-15 findings link refresh.

## 6. Coordination with the kit-side ValueSemantic proposal

`docs/ideas/ValueSemantic Kit Proposal.md` defines four downstream M-VS milestones (M-VS-1 through M-VS-4). The mapping:

| Kit proposal milestone | v1.18-19 workstream | Status |
|---|---|---|
| M-VS-1 (structural detector) | **Workstream A** | Same work; engine-side ships first; doesn't wait on kit. |
| M-VS-2 (CoW opt-in suppression) | Future v1.21+ | Requires kit-side `ValueSemantic` protocol to land. Becomes a *positive override* on workstream A's negative signal: when `inheritedTypes.contains("ValueSemantic")`, suppress the structural negative. |
| M-VS-3 (RefactorBridge "Add `ValueSemantic` conformance" writeout) | Future v1.21+ | Requires kit-side protocol. New writeout target in `RefactorBridgeOrchestrator`. |

**v1.18-19 do not block on the kit work.** Workstreams A + B + C are pure engine-side mechanisms. The kit proposal is an additive future enhancement.

## 7. Open decisions

1. **Carrier-resolution scope (Workstream A).** Same-file only (matches `TypeShapeBuilder`'s current posture; cheap; misses cross-file structs) or corpus-wide (matches `EquatableResolver`'s posture; more expensive; complete)? **Lean: same-file only for v1.18 + revisit at v1.20 if the empirical data shows a meaningful corpus of cross-file value-semantic struct carriers being missed.**

2. **Lift conservatism (Workstream B).** Require workstream-A `valueSemanticCarrier` signal `> 0` to admit the lift (strict; misses unannotated value structs whose stored properties don't all resolve), or admit on `kind == .struct` alone and let the negative signal demote downstream (permissive; relies on scoring to suppress)? **Lean: strict for v1.19 → if v1.20 shows recall too low, relax to permissive for v1.21.**

3. **Composition template scope (Workstream B).** Ship just `f(f(x, a), b) == f(x, a+b)` for numeric params (`AdditiveArithmetic` curated set: `Int`, `Double`, `Float`, `Decimal`, `Duration`, `Measurement`), or generalize to monoid-shaped accumulators on day one (any `T` where the project corpus also has a `combine` or `+` op identifying `T` as monoidal)? **Lean: numeric-only for v1.19; monoid-shaped extension is a v1.21+ candidate after the numeric-only acceptance rate is measured.**

4. **Coexistence with the kit proposal naming.** Treat workstreams A+B+C as standalone v1.18-19 calibration cycles, *or* fold them into the proposal's M-VS-1 / M-VS-4 nomenclature so engine + kit work share a series name? **Lean: standalone v1.18-19 — the empirical loop's per-cycle naming convention is load-bearing for the data continuity (cycle-N findings cross-reference each other) and the kit proposal's milestones are a separate axis.**

5. **`@ValueSemantic` opt-in honoring (Workstream A, future).** When the kit-side `ValueSemantic` protocol lands, should workstream A's signal:
   - (a) treat `inheritedTypes.contains("ValueSemantic")` as an **override** that flips the structural negative to positive (developer asserts CoW-correct), OR
   - (b) treat it as a **boost** on the structural positive only (developer assertion + structural check both required), OR
   - (c) emit a separate signal type so scoring can weight differently?
   
   Defer this decision to the v1.21 plan when the kit work materializes. The choice depends on how much trust is placed in opt-in conformance vs. structural verification.

6. **Pair-name rule extensibility (Workstream C).** The curated `X`/`Xing`, `formX`/`X`, `X`/`Xed` rules are project-extensible via `vocabulary.json` — same posture as `idempotenceVerbs`. Should the extension surface allow custom regex patterns, or only literal name pairs (`{ "addItem": "addingItem" }`)? **Lean: literal pairs only for v1.18 (matches `vocabulary.json`'s current schema simplicity); regex-pattern extension is a v1.21+ candidate if vocabulary-side demand surfaces.**

7. **Multi-step / sequenced-mutation testing (out of v1.18-19 scope).** ChatGPT `_3_Mutating API to Property Tests.md` step 6 ("Sequences of mutations / stateful testing") is a richer property family that catches the closure-capture bug class from the kit proposal §2.2 example 3. Scoping into v1.21+ as workstream **D — sequenced-mutation property template** (depends on workstream B + a new `[Op]` array generator in `SwiftPropertyLaws`).

## 8. Release-blocking criteria

v1.18 cuts the tag when:
- All workstream-A unit tests pass.
- All workstream-C unit tests pass.
- Existing 1618-test suite still passes (no regression on the v1.16 surface that *isn't* explained by the new signals).
- Cycle-15 findings doc complete with per-corpus signal-hit table.
- Perf baseline `≤+5%` wall time on all four cycle-1..14 corpora.
- CHANGELOG + README + version-string updates committed.

v1.19 cuts the tag when:
- All workstream-B unit tests pass.
- Lifted-suggestion surface measurable per corpus (cycle-16 findings doc has the count).
- Existing test suite still passes.
- Perf baseline `≤+10%` wall time (lift admission adds a per-mutating-func resolver call per template; budget is intentionally looser than v1.18).

v1.20 cuts the tag when:
- 50-decision triage complete on the post-v1.19 surface.
- Cycle-17 findings doc reports the per-template + per-corpus + aggregate acceptance rate, comparable to cycles 6 + 14.
- Cycle-18 priority list rotated post-v1.20.

## 9. Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Workstream A's curated stdlib value-type allow-list misses commonly-used types, causing false negatives on the positive signal | Medium | Allow-list extensible via `vocabulary.json`; the cycle-15 findings doc will surface "struct carriers that didn't earn the positive signal" as a diagnostic line — easy to inspect and extend. |
| Workstream B's lift admits properties on types that are *technically* value-semantic but where the mutating method has unstated preconditions (e.g. `mutating func append(_:)` only valid when `count < capacity`) | Medium | The composition template's failure mode is precondition violation in the runner; suggestions ship with the explicit "requires value semantics on T" rationale and the kit's existing precondition-hint signal already gates these. |
| Workstream C's name-pair rules conflict with the existing `IdempotenceTemplate`'s `curatedVerbs` list — `sort` / `sorted` is an idempotence pair *and* a dual-style consistency pair | Low | Both templates emit; the cross-validation key already deduplicates suggestion identity. The two suggestions express different properties (`sort` is idempotent vs. `sort` and `sorted` agree) so dual emission is correct. |
| Perf budget exceeded on the cycle-1..14 corpora | Low | `CarrierKindResolver` is a single pass over `TypeDecl` records already in memory; `DualStylePairing` is O(N) over `FunctionSummary` records; both reuse existing scanner caches. |
| The empirical v1.20 measurement reveals workstream B's lifted-suggestion acceptance rate is low (e.g. < 30%), suggesting the lift is admitting too much noise | Medium | This is the empirical-loop's job — a low rate informs cycle-18 suppression mechanisms. The risk is *not* that the data comes back negative; it's that the data comes back *flat*, indicating workstream B is admitting noise *and* signal in equal measure. v1.20 findings will explicitly call this out as a methodology check. |
| Scope creep — three workstreams across two cycles is the largest single-axis batch since the M8 algebraic family | High | Workstream C is independently mergeable; workstream A is independently mergeable. v1.18 ships A + C even if workstream B's design surfaces an issue. v1.19 (workstream B) can slip without blocking v1.18. |

## 10. What this plan is *not*

- **Not** a kit-side change. SwiftPropertyLaws stays at `from: "2.0.0"`. The kit-side `ValueSemantic` protocol from `docs/ideas/ValueSemantic Kit Proposal.md` is a future independent workstream.
- **Not** a TestLifter change. M-VS-4 from the kit proposal (TestLifter detector for `let copy = a; copy.x = ...; XCTAssertEqual` shapes) stays out of scope.
- **Not** a SemanticIndex prerequisite. The textual approximations workstream A and C use are good enough for the four cycle-1..14 corpora; SemanticIndex (PRD §20.1) would tighten them but isn't blocking. Workstream B's lift is a *behavioral* change, not a textual approximation, and benefits less from SemanticIndex than from runtime kit features.
- **Not** an IDE-integration change. PRD §20 v1.1+ work stays deferred.
