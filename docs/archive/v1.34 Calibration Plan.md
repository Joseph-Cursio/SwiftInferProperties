# v1.34 Calibration Plan — typeName Enrichment (post-§20.1 follow-up)

Working doc for the SwiftInferProperties v1.34 cut. v1.34 is the **thirty-first calibration cycle** and a **focused follow-up** to v1.33's PRD §20.1 SemanticIndex. Closes the v1.33-deferred `typeName` field so `swift-infer query --type Foo` works end-to-end.

> **Why this cycle exists.** v1.33 shipped the SemanticIndex with `typeName: String?` in the schema but every v1.33-emitted entry has `typeName == nil` because `IndexCommand.buildEntry` didn't have access to the suggestion's carrier type. The fix is to widen the `Suggestion` data model so each template surfaces its carrier alongside the other suggestion metadata, then thread that through to the index emitter.

> **Sequence:** v1.34 is "a quick fix before v1.35 Constraint Engine." The 3-cycle design-completion sequence agreed on (v1.32 Domain Template Packs → v1.33 SemanticIndex → Constraint Engine upgrade) becomes a 4-cycle sequence with the Constraint Engine now landing at v1.35.

**Ephemeral** — archive to `docs/archive/v1.34 Calibration Plan.md` once tagged.

## What v1.34 ships

Eight deliverables (standard A–H cadence):

1. **V1.34.A**: Add `carrier: String?` field to `Suggestion` in SwiftInferCore. Optional + defaults `nil` for backward compatibility.
2. **V1.34.B**: Thread carrier through every template's Suggestion construction (~12 sites). Each template populates from whichever input field is natural:
   - Unary templates (`IdempotenceTemplate`, `MonotonicityTemplate`, `InvariantPreservationTemplate`, `DualStyleConsistencyTemplate`): `summary.containingTypeName`
   - Pair templates (`RoundTripTemplate`, `InversePairTemplate`, `CommutativityTemplate`, `AssociativityTemplate`): `pair.forward.containingTypeName`
   - Identity-element (`IdentityElementTemplate`): `pair.operation.containingTypeName`
   - Lifted templates (`+Lifted` variants, `CompositionTemplate`): `lifted.carrier`
3. **V1.34.C**: Update `IndexCommand.buildEntry` to use `suggestion.carrier` instead of returning nil. End-to-end verification against ComplexModule + OC corpora.
4. **V1.34.D**: Cycle-31 findings.
5. **V1.34.E**: Perf re-measurement.
6. **V1.34.F**: Version bump.
7. **V1.34.G**: Closeout.
8. **V1.34.H**: Tag.

## V1.34.A — Suggestion data model widening

Add a single field to `Sources/SwiftInferCore/Suggestion.swift`:

```swift
public struct Suggestion: Sendable, Equatable {
    // ... existing 8 fields ...

    /// V1.34.A — carrier type name for `swift-infer query --type Foo`
    /// filtering. Populated from `FunctionSummary.containingTypeName`
    /// (unary + pair templates) or `LiftedTransformation.carrier`
    /// (lifted templates). `nil` for free functions and templates
    /// that don't have a meaningful carrier.
    public let carrier: String?
}
```

The `init` gains the parameter with default `nil` so existing call sites compile unchanged.

**Tests**: extend an existing Suggestion-construction test to verify the new field, plus a unit test that round-trips Suggestion through Equatable with the new field.

## V1.34.B — thread carrier through every template

Each of the ~12 suggestion-construction sites adds a `carrier: ...` argument to the `Suggestion(templateName:..., carrier: ...)` call.

| Template | File | Carrier source |
|---|---|---|
| Idempotence non-lifted | `IdempotenceTemplate.swift` | `summary.containingTypeName` |
| Idempotence lifted | `IdempotenceTemplate+Lifted.swift` | `lifted.carrier` |
| RoundTrip | `RoundTripTemplate.swift` | `pair.forward.containingTypeName` |
| InversePair non-lifted | `InversePairTemplate.swift` | `pair.forward.containingTypeName` |
| InversePair lifted | `InversePairTemplate+Lifted.swift` | `lifted.carrier` |
| Monotonicity | `MonotonicityTemplate.swift` | `summary.containingTypeName` |
| Commutativity | `CommutativityTemplate.swift` | `summary.containingTypeName` |
| Associativity | `AssociativityTemplate.swift` | `summary.containingTypeName` |
| IdentityElement non-lifted | `IdentityElementTemplate.swift` | `pair.operation.containingTypeName` |
| IdentityElement lifted | `IdentityElementTemplate+Lifted.swift` | `lifted.carrier` |
| DualStyleConsistency | `DualStyleConsistencyTemplate.swift` | `summary.containingTypeName` |
| Composition (lifted only) | `CompositionTemplate.swift` | `lifted.carrier` |
| InvariantPreservation | `InvariantPreservationTemplate.swift` | `summary.containingTypeName` |

All ~13 sites are single-line additions. Tests that construct `Suggestion` directly (none-to-few in templates tests — most use `TemplateRegistry.discover` and verify the output) need no changes if they use the default-nil parameter; the value flows through transparently.

**Risk**: a template surfaces a suggestion where `containingTypeName` is set but doesn't match the user's mental model of "the carrier" (e.g., a free function inside an enum where containingTypeName is the enum but the function is logically a static helper). v1.34 accepts the FunctionSummary's containingTypeName at face value; v1.35+ can refine if user feedback identifies misclassifications.

## V1.34.C — `IndexCommand.buildEntry` consumption

Update `carrierType(for:)` in `Sources/SwiftInferCLI/IndexCommand.swift`:

```swift
// V1.34.C — consume the suggestion.carrier the templates now populate.
// Prior v1.33 behavior was to always return nil; this closes the gap
// so `swift-infer query --type Foo` works end-to-end.
private static func carrierType(for suggestion: Suggestion) -> String? {
    suggestion.carrier
}
```

**End-to-end verification**:
- ComplexModule: `swift-infer index --target ComplexModule` then `swift-infer query --type Complex` should match `_relaxedAdd`/`_relaxedMul` + math forward suggestions.
- OrderedCollections: `swift-infer query --type OrderedDictionary` should match the sort lifts.
- Free functions: `swift-infer query --type none` continues to match (no regression on the v1.33 behavior).

## Scope boundaries

- **In scope**: data-model widening + thread-through + index emitter consumption. End-to-end `--type` query.
- **Out of scope**: lifted-suggestion carrier-vs-original-containingTypeName disambiguation. For lifted suggestions, the carrier IS the value-semantic struct the mutating method is lifted onto; the original's containingTypeName might differ (e.g., for nested types). v1.34 uses `lifted.carrier` (the canonical lifted carrier) for lifted templates.
- **Out of scope**: persistent carrier-resolution against `inheritedTypesByName` for cross-module type lookup. SemanticIndex integration (PRD §20.4 SourceKit) is the right place for that; v1.34 stays within the v1 scope.
- **Out of scope**: query-side carrier normalization (e.g., matching `OrderedDictionary` vs `OrderedCollections.OrderedDictionary`). Exact-string match for v1.34; future cycle if users report friction.

## Risk + mitigation

- **Risk**: a template I miss leaves its carrier as nil, silently broken behind the `--type` filter. **Mitigation**: end-to-end run against the 4 corpora at V1.34.C verifies each template surfaces carriers correctly. Any template that should have a carrier but emits nil shows up as a query gap.
- **Risk**: existing Equatable assertions on `Suggestion` break because the new field changes equality. **Mitigation**: all existing tests use default-nil for the new field; carrier values are surfaced only via fresh discover passes. Test count delta should be small.
- **Risk**: re-running `swift-infer index` against an existing index produces a `firstSeenAt` collision for previously-nil entries that now have a carrier. **Mitigation**: `identityHash` is the upsert key, not `(identityHash, typeName)`. Adding a carrier value doesn't change the hash (it's computed from the canonical-input string per V1.7.5, which doesn't include carrier). Upsert preserves `firstSeenAt`.
