## What v1.46 ships

Six workstreams + bundled closeout, mirroring v1.45's structure exactly:

1. **V1.46.A — `AssociativityStubEmitter`**. New emitter mirroring `CommutativityStubEmitter` but for the three-argument property `f(f(a, b), c) ≈ f(a, f(b, c))`. Three-carrier dispatch from the start (`Complex<Double>` / `Double` / `Int`). The stub:

   - Generates three values per trial — three independent `defaultGenerator.run(using: &rng)` calls (same pattern as v1.45's two-value commutativity; no `Gen<T>.zip` dependency).
   - Computes `lhsResult = f(f(a, b), c)` (left-associated) and `rhsResult = f(a, f(b, c))` (right-associated).
   - Checks equality: `isApproximatelyEqual` for FP, `==` for Int.
   - On failure: emit `VERIFY_DEFAULT_RESULT: FAIL` + `VERIFY_DEFAULT_TRIAL` + `VERIFY_DEFAULT_INPUT` (the `(a, b, c)` triple as a single string for parser compatibility) + `VERIFY_DEFAULT_FORWARD` (= `lhsResult`) + `VERIFY_DEFAULT_INVERSE` (= `rhsResult`) + exit 1.

   Same V1.43.C marker contract — `VerifyResultParser` consumes associativity output identically to round-trip / idempotence / commutativity.

   **Two-pass design.** Mirrors V1.45.A: Complex<Double> two-pass (default finite + `Gen<Complex<Double>>.edgeCaseBiased()` per-slot rotation), Double two-pass (with the inlined `doubleWithNaN` per-slot rotation), Int single-pass with zero-edge sentinel.

   **Sub-decision — edge bias slot (per-slot rotation).** For the three-value edge pass, **rotate the edge-biased slot across trials** — trial `t` draws the edge value into slot `t % 3` (0 = `a`, 1 = `b`, 2 = `c`) and the other two slots from the default finite generator. Over 100 trials this gives ~34 / 33 / 33 trials with the edge value in slot `a` / `b` / `c` respectively, surfacing associativity breaks regardless of which operand position triggers them. This costs more design surface than v1.45's static lhs-edge convention but is a strictly better advisory signal for associativity specifically — `(edge ⊕ b) ⊕ c` vs `edge ⊕ (b ⊕ c)`, `(a ⊕ edge) ⊕ c` vs `a ⊕ (edge ⊕ c)`, and `(a ⊕ b) ⊕ edge` vs `a ⊕ (b ⊕ edge)` exercise the three-fold nesting differently and a single-slot bias would miss two of them.

   **Sub-decision — `matchEdgeCaseIndex` plumbing under per-slot rotation.** The match runs against the slot's edge value (the one drawn from `edgeGenerator`), not against a fixed slot. The stub computes `edgeSlot = trial % 3` per iteration and feeds `edgeSlot == 0 ? a : (edgeSlot == 1 ? b : c)` into `matchEdgeCaseIndex`. The failing-input string still carries all three values (`(a, b, c)`) for context, plus a `VERIFY_EDGE_SLOT: <0|1|2>` marker line so the renderer can phrase the edge-case attribution correctly. The marker is purely informational — `VerifyResultParser` ignores unknown marker lines, so the parser stays unchanged.

   **Sub-decision — file split.** The Complex<Double> implementation alone will likely exceed the 400-line `VerifyResult.swift` precedent. Mirror v1.45's split: `AssociativityStubEmitter.swift` (Complex<Double> + shared section helpers) + `AssociativityStubEmitter+NonComplex.swift` (Double + Int). Keeps both files under 280 lines.

2. **V1.46.B — `AssociativityPairResolver`**. Resolves a `SemanticIndexEntry` to a single-function three-argument call expression. Analog of `CommutativityPairResolver`:

   - Validates `templateName == "associativity"` + carrier in `AssociativityStubEmitter.supportedCarriers`.
   - Constructs `<bareCarrier>.<funcName>` from `primaryFunctionName`.
   - Stub calls `<funcName>(<funcName>(a, b), c)` and `<funcName>(a, <funcName>(b, c))` (the stub composes the nesting; the resolver just returns the bare call).

3. **V1.46.C — `Verify` CLI template dispatch extension**. Add `"associativity"` to the `buildStubBundle` switch in `VerifyCommand+TemplateDispatch.swift`. Update the `unsupportedTemplate` `expected:` list to `["round-trip", "idempotence", "commutativity", "associativity"]`. Extend `RenderShape` (in `VerifyResult.swift`) with the `.associativity` case:

   - `subjectLine`: `"associativity on \(forwardName) over \(carrierType)"`.
   - `forwardExpression`: `"\(forwardName)(\(forwardName)(a, b), c)"`.
   - `inverseExpression`: `"\(forwardName)(a, \(forwardName)(b, c))"`.
   - `expectedExpression`: `"\(forwardName)(a, \(forwardName)(b, c))"`.

## Why associativity, not DerivationStrategist (or both)

Three reasons in order of weight:

1. **Lower-risk continuation of v1.45's pattern.** v1.45's commutativity machinery (`{StubEmitter, PairResolver, RenderShape}` triple + 3-carrier dispatch + 3 integration tests) is the proximate template for v1.46. Three-value generation extends the two-value pattern with one more independent `defaultGenerator.run(using: &rng)` call — no architectural lift. DerivationStrategist integration, by contrast, requires a non-trivial decision on how to plumb kit-side carrier derivation into the verify workdir + how to import / link the user's domain types; that's better as v1.47 after associativity is a clean baseline.

2. **Matched cycle-42 measurement predictions.** Cycle-42's per-pick predictions for #25 and #27 (`.defaultFails`) are derivable from the same mechanism that V1.45.F validated for #22 / #23 (asymmetric two-arg → asymmetric three-arg via the same algebraic identity). v1.46 is the smallest cycle that surfaces a **load-bearing prediction check** on the agreement-rate signal — if #25 or #27 doesn't verify-confirm `.defaultFails`, the cycle-42 prediction model has a gap worth investigating.

3. **FP-associativity-tolerance is the design risk that needs in-cycle resolution.** Floating-point associativity is *not* strictly preserved — `(a + b) + c ≠ a + (b + c)` for FP rounding reasons. The verifier relies on `isApproximatelyEqual` (relative tolerance ≈ `sqrt(.ulpOfOne)`) to accommodate this. Cycle-43 measurement will tell us whether the tolerance is generous enough on the cycle-27 corpus's three-arg shapes (specifically #26 `_relaxedMul`). If `_relaxedMul` verify-fails on Pass 1 due to FP-rounding-only divergence, the cycle-43 finding records the gap and the v1.47 plan decides between (a) widening the tolerance or (b) restricting associativity to algebraic (Int-only or curated function-name) carriers.

The v1.47 plan picks up DerivationStrategist (and the chunked-Index REJECT cluster #7–#9) using v1.46's now-established two- + three-value design as the precedent for any further multi-value templates.

## Why not extend the curated round-trip pair list this cycle

The v1.45 plan bundled "commutativity verify + curated pair-list expansion" because both targeted the cycle-41 in-scope-expansion question. Cycle-42's pair-list expansion (the 6 hyperbolic entries) covered all canonical math inverse pairs visible in the cycle-27 corpus; the residual non-curated pairs (#1 `_value`/`_bucketContents`, #2 `_minimumCapacity`/`_scale`) need the SemanticIndex schema bump to persist both halves — that's its own architectural decision, deferred to v1.46+ (cycle-42 trajectory table's last row).

v1.46 is therefore a single-template-extension cycle, not a bundled cycle. Cleaner deliverable boundary; easier to measure cycle-43's verifiable-fraction lift attributable solely to associativity.

- **Out of scope, deferred to v1.47:** DerivationStrategist integration at verify-time for non-FP/Int carriers (`Base.Index`, `_Bucket`, user types); idempotence-lifted template; monotonicity template; dual-style-consistency template; non-curated round-trip pair derivation (`Suggestion.evidence`-based pair lookup); `Float` / `Complex<Float>` / `String` carriers; accept-flow integration (verify outcomes → `decisions.json`); verification cache; "Verified" first-class tier; widening `isApproximatelyEqual` tolerance for associativity-specific FP-rounding edge.
- **Not changing:** Any v1.45-and-earlier verify pipeline behavior for `round-trip` / `idempotence` / `commutativity` × all 3 carriers. The V1.46.C template-dispatch extension is strictly additive; the `RenderShape` change only adds a new enum case (existing call sites keep their `roundTrip` / `idempotence` / `commutativity` dispatch unchanged).

## Risks

1. **FP associativity false-positives.** Real floating-point addition / multiplication is non-associative in the IEEE 754 rounding sense — `(a + b) + c ≠ a + (b + c)` when intermediate results land on different rounding boundaries. The verifier's `isApproximatelyEqual` (relative tolerance ≈ `sqrt(.ulpOfOne)` ≈ `1.5e-8`) should absorb single-rounding divergence in practice, but adversarial inputs (e.g. `a = 1e16, b = 1, c = -1e16`) can produce relative errors above tolerance. **Mitigation**: D.4.a's `{ (a, b) in a + b }` test pins the canonical FP-associative case at `.bothPass`. If V1.46.D.4.a flakes intermittently, the cycle-43 findings record the failure rate; widening the tolerance is a v1.47 follow-up (not a v1.46 scope creep).

2. **`_relaxedMul` semantic gap.** Cycle-42's prediction for #26 is `.bothPass(100, 100, 0..12)` on the assumption that the kit's `_relaxedMul` is associative within `isApproximatelyEqual`. If the relaxed-FMA implementation chooses to associate slightly differently (e.g. one path fuses the multiply-add and the other doesn't, surfacing a 1-ulp diff that exceeds the relative tolerance), the verify outcome could be `.defaultFails`. **Mitigation**: this is a load-bearing prediction worth verifying; if cycle-43 surfaces a disagreement (predicted `bothPass`, measured `defaultFails`), the cycle-43 findings flag it as the first verify-vs-heuristic disagreement and the v1.47 priority list adds a kit-side inquiry.

3. **Per-slot rotation reduces per-slot edge density.** Under per-slot rotation (V1.46.A's chosen design), each individual slot sees the edge generator on only ~33% of trials — slot `a` gets ~34 edge draws per 100-trial Pass 2 vs. v1.45 commutativity's ~100 lhs-edge draws. Functions whose associativity break requires *the same* slot to be edge-biased multiple times in a row would be sampled less densely than under static single-slot bias. **Mitigation**: increase the Pass 2 trial count under per-slot rotation if the cycle-43 signal shows undersampling, or surface the per-slot sampled-index breakdown in the `VERIFY_EDGE_SAMPLED` marker (split into `VERIFY_EDGE_SAMPLED_A`, `_B`, `_C`). v1.46 ships the simpler aggregate-count marker; the per-slot breakdown is a v1.47+ refinement if needed.

   This risk replaces v1.45's "first-slot-only misses some edge breaks" gap — the trade-off is acknowledged: per-slot rotation gains broader coverage at the cost of per-slot density. For associativity specifically, breadth across the three nesting positions is the higher-value choice.

4. **Cycle-43 measurement plateaus below the cycle-42 trajectory.** Cycle-42's projected verifiable-fraction after v1.46 is 31.3% (10/32). If any of #25, #26, #27 fails to materialize as predicted — e.g., `Int.distance` isn't reachable as a member function (it's a global on `BinaryInteger`, indexable but harder to call as `Int.distance`), or `_relaxedMul` requires a special build flag — the actual lift could be smaller. **Mitigation**: V1.46.E.3's findings doc reports the actual verifiable-fraction; an unexpected drop relative to the 31.3% projection becomes an explicit v1.47 thread to investigate. The integration test fixtures (D.4.a–c) use synthetic closures, so the CI signal is independent of cycle-27 indexing reachability.

5. **Three-value seeded RNG determinism.** Each trial draws three values; the Xoshiro seeded RNG should produce identical triples for identical seeds across runs (and across the v1.45 two-value pattern), but the draw-three-per-trial pattern is new and worth a unit-test smoke check. **Mitigation**: V1.46.D.1 includes a "same-seed reproducibility" test asserting that two emitter invocations with identical inputs produce byte-identical source (and downstream, identical triples).
