# v1.35 Calibration Plan — Carrier-Aware Refactoring Suggestions

Working doc for the SwiftInferProperties v1.35 cut. v1.35 is the **thirty-second calibration cycle** and a **carrier-aware refactor surface** built directly on v1.34's enriched SemanticIndex.

> **Why this cycle exists.** PRD §20.1 motivated SemanticIndex with "API design feedback ('you have three monoids; consider unifying them under a custom Monoid protocol')" and "Refactoring suggestions ('this function would slot into the existing semigroup chain')." v1.33 shipped the index; v1.34 populated `typeName` end-to-end. v1.35 ships the **first user-facing refactor-suggestion surface** that queries the index for carrier-level patterns the per-suggestion view can't see.

> **Sequence:** v1.35 = carrier-aware refactors (focused single-cycle); **v1.36 = Constraint Engine upgrade (PRD §20.2)** — the multi-cycle architectural refactor we discussed. v1.35 is intentionally narrow so the Constraint Engine effort opens with a clean v1.35.0 boundary.

## V1.35.A — `RefactorCluster` data model + analyzer

**`RefactorCluster`** — one row per (typeName, cluster-shape) pair:

```swift
public struct RefactorCluster: Sendable, Equatable {
    public let typeName: String
    public let totalSuggestionCount: Int
    public let perTemplateCounts: [String: Int]   // templateName → count
    public let shape: ClusterShape
    public let representativeFunctions: [String]  // up to 5 names for the user
}

public enum ClusterShape: String, Sendable, Equatable {
    /// 2+ algebraic-structure templates on the same type — suggests
    /// the type already behaves like a kit-defined algebra (Semigroup
    /// / Monoid / Semilattice / Semiring) and could conform formally.
    case algebraicStructure

    /// 3+ idempotence (lifted + non-lifted combined) on the same type
    /// — suggests a cluster of "normalize this value" mutators worth
    /// documenting + treating uniformly.
    case idempotenceCluster

    /// 3+ form/non-form dual-style pairs on the same type — suggests
    /// the type is a SetAlgebra / similar mutating-pair-rich abstraction.
    case dualStyleCluster

    /// 3+ round-trip pairs on the same type — suggests the type is a
    /// codec / serialization-bearing structure worth surfacing via a
    /// Codec-shaped abstraction.
    case roundTripCluster

    /// Catch-all: 4+ suggestions of any mix on the same type but
    /// outside the above named shapes. Worth surfacing as "this type
    /// has a lot of inferred properties" without prescribing a
    /// specific refactor.
    case generalCluster
}
```

**`RefactorClusterAnalyzer`** — single static function:

```swift
public enum RefactorClusterAnalyzer {
    public static func analyze(
        _ entries: [SemanticIndexEntry]
    ) -> [RefactorCluster]
}
```

**Algorithm**:
1. Filter out entries with `typeName == nil` (free functions don't cluster by carrier).
2. Group by `typeName`.
3. For each type, build `perTemplateCounts: [templateName: Int]`.
4. Classify the shape using priority order (more-specific shapes win):
   - `algebraicStructure` if 2+ distinct templates in `{commutativity, associativity, identity-element}` are present.
   - `idempotenceCluster` if `idempotence` count ≥ 3.
   - `dualStyleCluster` if `dual-style-consistency` count ≥ 3.
   - `roundTripCluster` if `round-trip` count ≥ 3.
   - `generalCluster` if total ≥ 4 and none of the above fired.
   - Otherwise skip (not a "cluster" worth surfacing).
5. Pick up to 5 representative function names from the first evidence column for human-readable output.
6. Return sorted by `totalSuggestionCount` descending.

**Tests**: `RefactorClusterAnalyzerTests.swift` — verify each classification rule on hand-crafted fixtures, including (a) the priority order (algebraicStructure beats generalCluster when both apply), (b) the threshold edges (idempotence count = 2 doesn't cluster; = 3 does), (c) typeName filtering (nil entries excluded), (d) ordering by size.

## V1.35.B — `swift-infer suggest-refactors` subcommand

New AsyncParsableCommand registered alongside `discover` / `drift` / `convert-counterexample` / `metrics` / `index` / `query`.

**Behavior**:
1. Resolve `.swiftinfer/index.json` (mirrors `query` subcommand).
2. Load via `IndexStore.load(...)`.
3. Run `RefactorClusterAnalyzer.analyze(...)`.
4. Render to stdout sorted by cluster size descending.

**CLI flags**:
- `--directory <path>` — override package root for walk-up.
- `--index-path <path>` — explicit index file path.
- `--min-suggestions <N>` — filter clusters below this size (default 3).
- `--shape <name>` — filter by `ClusterShape` (algebraicStructure / idempotenceCluster / etc.).
- `--limit <N>` — cap output count.

**Rendered output** (per cluster):

```
[OrderedSet] 7 inferred properties — idempotence cluster
  templates: idempotence ×7
  representatives: sort(), _regenerateHashTable(), _regenerateExistingHashTable(), _isUnique(), _ensureUnique()
  suggestion: This type has several idempotent mutators that look CoW-stable.
              Consider documenting the idempotence invariant on the type's
              API contract. If multiple types share this shape, an
              "IdempotentNormalize" protocol may capture the pattern.

[Complex] 12 inferred properties — algebraic structure cluster
  templates: commutativity ×6, associativity ×6
  representatives: +(z:w:), *(z:w:), _relaxedAdd(_:_:), _relaxedMul(_:_:), ...
  suggestion: This type has multiple commutativity + associativity
              suggestions. If your kit publishes Semigroup / Monoid /
              CommutativeMonoid, formal conformance lets SwiftPropertyLaws
              verify the laws on every CI run.
```

The per-shape "suggestion" text is a curated string per `ClusterShape`. Stable wording so users can grep for it in PR descriptions.

## Scope boundaries

- **In scope**: index → cluster analysis → render. Carrier-aware cross-template grouping. Curated per-shape suggestion text.
- **Out of scope**: writing markers / annotations / code changes (PRD §16 #1 — v1 doesn't auto-modify source; `swift-infer suggest-refactors` is read-only render).
- **Out of scope**: cross-type abstraction discovery ("Foo and Bar both have the same shape — extract a protocol"). v1.35 surfaces single-type clusters; cross-type pattern matching is a future cycle.
- **Out of scope**: hierarchy resolution (when `OrderedSet` and `OrderedSet.UnorderedView` cluster similarly). v1.35 treats nested type names as distinct carriers; future cycle can compose.
- **Out of scope**: confidence scoring or rejection (e.g., "this cluster looks like a Monoid but the empty-set element is missing — escalate to Semigroup only"). The PRD §20.2 Constraint Engine is the right home for that logic.

## Risk + mitigation

- **Risk**: noisy output on a corpus with many small clusters. **Mitigation**: `--min-suggestions 3` default threshold; user can tune.
- **Risk**: cluster shapes don't match how users think about the data. **Mitigation**: the 5 shapes are a starting taxonomy; user feedback in v1.36+ can refine. Curated text is stable enough that users can grep + filter.
- **Risk**: a future template introduces a new shape that doesn't fit the 5-shape taxonomy. **Mitigation**: `generalCluster` catch-all already exists; the priority-order classification is forward-stable.
