## Why bundle all four

Three reasons the user picked "bundle all four" over the three-alternatives split:

1. **Single tagged release reaches the cycle-27 endpoint.** v1.42–v1.48 collectively built the verify pipeline from 6.25% → 87.5%. v1.49 finishes the arc at 100%. Splitting across v1.49 + v1.50 would push the close-out signal across two cycles for no architectural benefit — the four workstreams touch different subsystems and don't share design risk.

2. **Each workstream is independently scoped.** (A) stub-preamble: ~50 LoC across emitter input structs + setup composer. (B) `.memberwiseArbitrary`: ~30 LoC of zip-composition emit. (C) non-curated pair: ~80 LoC across SemanticIndexEntry + Suggestion-side persistence + resolver fallback. (D) perf-flake mitigation: ~20 LoC of test-tag refactor + CI doc. Total ~180 LoC new code + tests; comparable to v1.47's lift.

3. **Risk surface is bounded.** None of the four workstreams require new architectural concepts — each is a mechanical extension of v1.46/v1.47/v1.48 patterns. The largest cycle measured in terms of test-count delta isn't necessarily the largest measured in risk; v1.49 distributes risk across four small bets rather than concentrating it.

## What v1.49 ships

Workstream order chosen to surface architectural risk early. The plan suggests tackling (A) stub-preamble first because it's the simplest and its API shape (the new `preamble` field on emitter Inputs) blocks the dual-style integration test that the cycle-45 open thread carried forward.

1. **V1.49.A — Stub-preamble channel**. Add `preamble: String` to every emitter's `Inputs` struct (default `""`). The `setupSection` composer renders the preamble after the imports + before the `var rng = ...` declaration. Mechanically:

   - `RoundTripStubEmitter.Inputs` + `IdempotenceStubEmitter.Inputs` + `CommutativityStubEmitter.Inputs` + `AssociativityStubEmitter.Inputs` + `StrategistDispatchEmitter.Inputs` all gain the optional preamble field.
   - `StrategistDispatchEmitter.setupSection` emits the preamble verbatim when non-empty.
   - All v1.46/v1.47/v1.48 hardcoded emitters mirror the same.
   - Pair resolvers don't change — the preamble lives at the emit-time `Inputs` layer.
   - VerifyCommand+TemplateDispatch threads an empty preamble by default; v1.49 doesn't surface a preamble from the SemanticIndex (that's a v1.50+ extension if needed).

   **Sub-decision**: should the preamble field be `String` or `[String]` (multi-line)? `String` with embedded newlines is simpler and matches `extraImports`'s pattern. Going with `String`. Documents in the docstring that callers can use multi-line `"""` literals.

   **Unblocks**: V1.48.H.2 dual-style integration test can now extend the carrier with a real mutating method inside the stub via preamble: `"extension Int { mutating func bumpInPlace() { self += 1 } }"`. V1.48.H.2 unskipped in this cycle's V1.49.F tests.

2. **V1.49.B — `.memberwiseArbitrary` strategy emit**. Extend `StrategistDispatchEmitter.recipe(for:carrier:)` to handle the `.memberwiseArbitrary(members: [MemberSpec])` strategy case. Emit logic:

   - For 1-member structs: `Gen<RawType>.exprFor(member.rawType).map { T(memberName: $0) }`.
   - For N-member structs (N ≥ 2): `zip(Gen<Raw1>.exprFor(specs[0].rawType), Gen<Raw2>.exprFor(specs[1].rawType), ...).map { (m1, m2, ...) in T(name1: m1, name2: m2, ...) }`.
   - `swift-property-based` ships `zip` overloads for arities 2–10 (per kit-side `Generator+Zip.swift`); arity 11+ falls through to `.todo` (matching the strategist's `memberwiseArityLimit = 10`).

   **Sub-decision**: how to handle structs without a synthesized memberwise init (the kit's `hasUserInit == true` case)? The strategist already filters these out into `.todo` — v1.49.B doesn't need to re-check. But the recipe should pin the carrier-type-qualifier: `T(name1: m1, ...)` requires `T` to be a real type name accessible at the verify workdir. v1.49 uses the bound carrier name from the recipe (already correct).

   **Imports**: `swift-property-based` is already in every strategist-routed stub via `PropertyBased`. No new imports.

   **Unblocks**: cycle-27 picks #10 + #11 if their stored-member shapes are raw-typed. Cycle-46 measurement settles which (if either) materializes.

3. **V1.49.C — Non-curated round-trip pair derivation**. Three sub-workstreams:

   - **C.1** — Add `secondaryFunctionName: String?` to `SemanticIndexEntry` (backward-compat via `decodeIfPresent`; nil for non-round-trip templates and for round-trip entries whose suggestion didn't surface a second half). JSON format version bumps 2 → 3.
   - **C.2** — Discover-side population: `IndexCommand.buildEntry` looks at the round-trip Suggestion's evidence array (the second `Evidence.displayName` is the inverse half by the discover-side convention) and populates `secondaryFunctionName`. For other templates, leave nil.
   - **C.3** — `RoundTripPairResolver.resolve` (and the inlined version in `VerifyCommand+TemplateDispatch`'s strategist path) first tries the curated lookup; on miss, falls back to constructing the pair from `entry.primaryFunctionName` + `entry.secondaryFunctionName` (when present). The `.unsupportedPair` error fires only when *both* the curated lookup and the secondary-name fallback fail.

   **Sub-decision**: should we widen `RoundTripPairResolver.curated` instead? No — the cycle-44 + cycle-45 findings ratified the schema-bump path; cycle-27 has two non-curated pairs (#1 + #2), neither of which is a canonical-math-inverse to add to the curated list. The general mechanism is more durable.

   **Unblocks**: cycle-27 picks #1 (ACCEPT) + #2 (REJECT — **8th verify-confirmed REJECT, brings verifier-mode REJECT lift to 8/8 = 100%**).

4. **V1.49.D — §13 perf-test flake mitigation**. Three options the cycle-45 perf baseline outlined:

   - **(a)** Split subprocess integration tests to a separate `swift test --filter` invocation outside the default suite. Requires a Swift Testing tag (e.g., `@Test(.tags(.subprocess))`) and a CI documentation note.
   - **(b)** Widen the §13 budget on the perf tests when contention is detected. Brittle — no clean way to detect contention without OS-level counters.
   - **(c)** Serialize the subprocess builders via a test-time semaphore. Risks new flake patterns in the seek-and-yield logic.

   v1.49 ships **(a)**: introduce a `.subprocess` Swift Testing tag (defined in a new `Tests/SwiftInferIntegrationTests/Tags.swift` file), apply it to the 18 V1.42.D / V1.44.E / V1.45.E / V1.46.D / V1.47.G / V1.48.H integration tests, and document the CI command pattern (`swift test --filter='!subprocess'` for the perf-only run; default `swift test` keeps everything for completeness).

   **Mitigation**: cycle-46 measurement confirms whether the flake recurrence drops post-tag. If still flaky, v1.50 picks (c) the semaphore option.

## Risks

1. **`.memberwiseArbitrary` zip-arity edge cases.** swift-property-based ships zip overloads for arities 2–10; 1-arity uses `.map` directly. v1.49.B emits the correct shape for each arity. If a future kit version changes the zip-arity ceiling, v1.49.B's emit needs updating. **Mitigation**: explicit arity check against `DerivationStrategist.memberwiseArityLimit` (10); emit `.todo` fallback for arity 11+ (the strategist already does this; v1.49.B just respects it).

2. **`secondaryFunctionName` discover-side population**. Round-trip suggestion's evidence array is the canonical source for the pair's inverse half. v1.49.C.2 reads `evidence[1]` (if present) when populating. If the Suggestion struct doesn't actually carry the inverse half in evidence[1] — e.g., if the template emits the pair structurally elsewhere — this discover-side path returns nil and the resolver falls back to `.unsupportedPair`. **Mitigation**: V1.49.C.2 implementation starts by tracing the discover-side flow (where does the round-trip template currently store the inverse half?); if `evidence[1]` is the wrong field, switch to whichever field actually carries the second half.

3. **`.memberwiseArbitrary` carrier with non-stdlib members.** If a struct's stored member is itself a struct (not a raw type), the strategist falls through to `.todo` via the `RawType(typeName:)` check. v1.49.B doesn't help with this case — multi-level memberwise composition is a v1.50+ recursive design. **Mitigation**: cycle-46 findings record whether cycle-27 picks #10 / #11 actually hit this gap (likely _Bucket does, given OrderedCollections' internal struct shape).

4. **Stub-preamble correctness depends on caller.** V1.49.A's preamble is emitted verbatim; if a caller injects malformed Swift (mismatched braces, etc.), the verify subprocess fails to build with a confusing error. **Mitigation**: V1.49.A's docstring warns callers; V1.49.F.1's test uses a known-good preamble. Future syntactic validation is out-of-scope.

5. **Perf-flake tag may not eliminate flake**. v1.49.D ships option (a) — split via tag. If CI doesn't actually use `--filter='!subprocess'` for the perf-only run, the flake recurs. **Mitigation**: cycle-46 records flake frequency; if unchanged, v1.50 picks (c) the semaphore option. The tag is still a net improvement (gives downstream consumers + CI a clean way to split when they need it).

6. **Cycle-46 may land at 93.8% (pessimistic), not 100%.** If both #10 and #11 hit `.memberwiseArbitrary` gaps that v1.49.B doesn't cover (multi-level composition or non-derivable members), the verifiable-fraction stops at 30/32. **Mitigation**: cycle-46 findings record the actual landing; v1.50 plan picks up from there. The verifier-mode REJECT lift hits 8/8 = 100% regardless of #10/#11 — that's the cleaner load-bearing measurement.

7. **Test wall-clock impact at 21 parallel subprocess builds.** v1.49.F adds 3 new subprocess tests (21 total). Under contention this may push wall-clock past the §13 budget more frequently. v1.49.D's mitigation is the primary defense; if it doesn't drop contention, v1.50 should serialize the subprocess builders.

8. **The "100% verifiable-fraction" claim is on cycle-27, not the full 109-surface.** Cycle-46's measurement remains corpus-limited. The Phase 1.5 arc closes at 100% *for the cycle-27 sample*; running verify across all 109 surface picks is Phase 2 work that requires the user's domain-package indexing. **Mitigation**: cycle-46 findings frame the close-out as "Phase 1.5 corpus-limited 100%" rather than "v1.49 achieves full verify-mode coverage". The framing matters for stakeholder expectations on Phase 2.
