## What v1.58 ships

Three workstreams + bundled closeout.

Mirrors V1.51.A's `Complex → Complex<Double>` pattern: cycle-27's discover layer captured bare `OrderedSet`; v1.58's binding canonicalizes to `OrderedSet<Int>` (Int is the canonical Element for cycle-27 alignment, per V1.47.D).    **Scope**: cycle-55 measurement confirms the binding fires (carrier becomes `OrderedSet<Int>` after V1.47.F's `bound(_:)` check); the downstream pipeline (strategist instance generation + mutating-method emission for `sort()` etc.) is v1.59+ work. 2. **V1.58.B — Methodology guard for `GenericBindingResolver.curatedBindings`**. New test suite `V1_58MethodologyGuardTests` at `Tests/SwiftInferCLITests/`. Two tests:

   - `everyBindingMatchesAFixtureCarrier` — loads the cycle-27 fixture, collects all `typeName` + `typeShape.storedMembers[].typeName` values, asserts every `curatedBindings` key is in the resulting set (or in `intentionallyUnmatchedKeys`). Surfaced 4 latent V1.47.D bindings on first run; resolution: add to `intentionallyUnmatchedKeys` with a documented rationale (V1.47.D protocol-extension anticipation).
   - `intentionalEscapeHatchesAreActualBindings` — asserts every entry in `intentionallyUnmatchedKeys` is actually a `curatedBindings` key (no orphan escape-hatch entries).

   **Why this matters**: cycles 50 and 53 surfaced V1.52.C + V1.51.B both shipped with dead binding keys (V1.52.C's `<Type>.Index` keys never matched cycle-27 carriers; V1.51.B's dual-style pairs latent). The guard would have caught both pre-merge. v1.58 forward — any new binding key that doesn't match cycle-27 fails this test, forcing either removal or explicit escape-hatch entry with rationale.
