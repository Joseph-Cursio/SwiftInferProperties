# v1.54 Calibration Plan — V1.52.A free-function revert + V1.52.C dead-binding cleanup + RealModule import for FP strategist recipes

Working doc for the v1.54 cut (fifty-first calibration cycle). **Fourth Phase 2 gap-closing cycle** — closes three cycle-50 follow-up items: the V1.52.A free-function regression on round-trip Complex EF surface (8 picks), the V1.52.C dead bindings (zero picks moved), and the V1.54.A-revealed `RealModule` missing import in the strategist's FP recipe (would regress 2 monotonicity-on-Double picks without this).

> **Motivation.** Cycle-50 measurement (`docs/calibration-cycle-50-findings.md`) showed: (1) V1.52.A's free-function shape regressed 8 round-trip Complex picks (`exp`, `log`, `sin`, `cos`, `tan`, `sinh`, `cosh`, `tanh`) — bare `exp(value)` doesn't resolve from a workdir that imports only `ComplexModule` + `RealModule`; (2) V1.52.C's `<Type>.Index` bindings never fired because the indexer outputs bare type names (`ChunkedByCollection`, `OrderedSet`), not the `.Index` form. v1.54 reverts both — restoring `.staticMethod` shape for the EF surface (closes the 8 picks) and removing the dead bindings (cleanup, zero picks moved).

**Methodology note from V1.54 smoke test**: the round-trip `exp/log` pick (cycle-46-predicted `.bothPass`) actually produces `.defaultFails` because the v1.42 RoundTripStubEmitter's default generator (`Double.random(in: -1e6 ... 1e6)`) is too wide for `exp` — large inputs overflow to `inf`. This is the **first prediction-vs-actual disagreement** in the project's history, but it surfaces a generator-tuning issue rather than a verifier-correctness issue. The structural cycle-46 prediction is still valid; the v1.42 generator just exceeds `exp`'s stable domain. Worth noting for v1.55+ generator-range refinement.

## What v1.54 ships

Five workstreams + bundled closeout. Smaller than v1.52, larger than v1.53.

1. **V1.54.A — V1.52.A free-function revert**. Empty out `CallExpressionShape.freeFunctionMap` to `[:]`. The map stays as an extension point — v1.55+ may revive free-function classification by adding `import _Numerics` to the V1.49.A preamble, at which point the EF surface entries can be restored. All 14 EF-surface names on Complex/Double/Float route back to `.staticMethod`. Test updates: restore `Complex.exp`-style assertions in `RoundTripPairResolverTests` + `V1_49PreambleTests` + the V1.52.D classify tests.

2. **V1.54.B — V1.52.C dead-binding cleanup**. Remove the 4 `<Type>.Index` entries (`ChunkedByCollection.Index`, `ChunkedOnCollection.Index`, `ChunkedByLazyCollection.Index`, `OrderedSet.Index`) from `GenericBindingResolver.curatedBindings`. Cycle-50 evidence: these keys never matched because the indexer outputs bare type names. Replace the V1_52GenericBindingExpansionTests suite with a V1_54 suite that asserts the dead keys return nil + the existing V1.47.D / V1.51.A bindings still resolve.

   **Why not just fix the keys**: bare `ChunkedByCollection → Int` would let resolution pass but the synthesized stub still won't compile — the chunked picks are *instance methods* on the wrapper, not free or static. Genuine fix requires v1.55+ TypeShape-driven generic instantiation + instance-method emission.

3. **V1.54.C — Add `RealModule` import for FP strategist recipes**. Discovered during cycle-51 smoke-testing: the V1.54.A revert restored `Double.log(_:)`-style calls, but the strategist's RawType recipe only imports `Foundation` + `PropertyBased` — and the static `Double.log` declared by `ElementaryFunctions` lives in `RealModule`. Pre-V1.54.A the bare `log(value)` form resolved to Darwin's C `log()` (always in scope via Foundation); post-V1.54.A the `Double.log(value)` form needs `RealModule` explicitly imported. Fix: extend the RawType branch to add `RealModule` when the carrier is `Double` or `Float`. The kit's `Package.swift` already pulls in swift-numerics → `RealModule` is available as a dep. Closes the 2 monotonicity-on-Double picks that V1.54.A alone would have regressed.

4. **V1.54.E — Cycle-51 calibration measurement**. Per the cycle-50 protocol:

   - Re-run `verify --all-from-index` against the cycle-27 fixture.
   - Compute the new full-surface verifiable-fraction. Projected: **~18-22/109 ≈ 17-20% measured-execution**.
   - **Headline target**: +8 measured outcomes (V1.54.A revert closing the 8 regressions).
   - Compare against cycle-46 predictions; note the `exp/log` overflow case (first prediction-vs-actual disagreement, generator-tuning origin).

5. **V1.54.F — Cycle-51 findings + roadmap doc**. Standard cycle-N-findings format. §"v1.55+ roadmap" reordered around: (a) TypeShape-driven OC instantiation (60+ picks; dominant remaining category); (b) `_relaxedMul`/`rescaledDivide` build-failure investigation; (c) FP generator range refinement (cycle-51's `exp/log` overflow finding); (d) instance-method emission for chunked-Index picks; (e) Phase 2 accept-flow integration.

## Risks

1. **The 8 newly-reaching-runtime picks may not all produce `.bothPass`**. Cycle-46 predicted `.bothPass` for all 8 (all are EF-surface round-trips). The smoke test on `exp/log` (#3) showed `.defaultFails` due to generator overflow — the same may apply to `sinh`, `cosh`, `tanh` (also overflow-prone) but probably not `sin`, `cos`, `tan` (bounded by ±1). **Mitigation**: cycle-51 findings honestly report each pick's outcome; the disagreement-vs-prediction count goes in the doc.

2. **V1.54.B's binding removal could break a future cycle that adds back the right-format keys**. The 4 removed entries are documented in the inline comment as v1.55+ TypeShape-work targets. **Mitigation**: the comment captures the why; v1.55's TypeShape work will redesign anyway.

3. **Removing the 4 V1.52.D.4 tests reduces coverage of `GenericBindingResolver`**. **Mitigation**: V1_54GenericBindingExpansionTests retains the V1.47.D + V1.51.A regression guard; the dead-keys-removed assertion is a new test.
