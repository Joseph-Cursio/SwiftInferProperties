# v1.59 Calibration Plan — strategist OC recipe + double-qualifier bug fix + instance-method-shape reclassification

Working doc for the v1.59 cut (fifty-sixth calibration cycle). **Ninth Phase 2 gap-closing cycle and second TypeShape scaffold step** — V1.58.A's binding-resolver layer plus V1.59.A's strategist-recipe layer get 23 OS picks past the resolution layer into `swift build`. They compile-fail at the next layer (instance-method shape); a new V1.56.A-style pattern reclassifies them so the `.measured-error = 0` baseline is preserved.

> **Motivation.** Cycle-55 confirmed V1.58.A's OrderedSet binding fires but stops at the strategist's `resolveRecipe` (no curated generator for `OrderedSet<Int>`). v1.59 adds the curated recipe. **A bug surfaces along the way**: the resolver was building `OrderedSet.OrderedSet.sort` (double-qualified) because some indexer-produced `primaryFunctionName` values include a type-qualifier prefix (e.g. `OrderedSet.sort()` vs Complex's bare `exp(_:)`). Fix at `CallExpressionShape.render` strips the prefix when it matches the qualifier. After both fixes, OS picks reach `swift build`; they fail with "instance member ... cannot be used on type" (mutating-method shape mismatch). V1.59 also extends V1.56.A's pattern matcher to recognize this error class, preserving the `.measured-error = 0` baseline from cycle-53.

## What v1.59 ships

Three workstreams (bundled into a single feat commit) + closeout.

1. **V1.59.A — Curated `OrderedSet<Int>` strategist recipe + double-qualifier fix + instance-member reclassification**:

   - **Strategist recipe** (in `StrategistDispatchEmitter`): new private `curatedOCRecipe(carrier:)` returns a `GeneratorRecipe` for `OrderedSet<Int>` with deterministic expression `Gen<Int>.int(in: 0 ... 100).map { OrderedSet([$0, $0 + 1, $0 + 2, $0 + 3]) }`. Inserted between the RawType branch and the typeShape branch in `resolveRecipe`. Imports include `OrderedCollections`.

   - **Verifier workdir Package.swift update**: `VerifierWorkdir.renderDependenciesBlock` adds `.package(url: "https://github.com/apple/swift-collections.git", from: "1.0.0")`; `renderTargetDependenciesBlock` adds `.product(name: "OrderedCollections", package: "swift-collections")`. Future cycles can conditionalize on carrier (currently every workdir gets swift-collections, even non-OC ones — costs are negligible at SwiftPM resolve time given the kit-side caching).

   - **Double-qualifier strip in `CallExpressionShape.render`**: surfaces during V1.59.A smoke-test. The indexer's `primaryFunctionName` is bare for some picks (Complex `exp(_:)`) and qualified for others (`OrderedSet.sort()`). The resolver builds `<typeQualifier>.<funcName>` assuming bare — producing `OrderedSet.OrderedSet.sort` for the qualified case. Fix: in `CallExpressionShape.render`, if `bareFunctionName` starts with `<typeQualifier>.`, strip the prefix. No-op for already-bare names (Complex picks unchanged).

   - **`instance-method-shape-not-supported` reclassification** (in `VerifyCommand+AllFromIndex.architecturalPendingDetail`): pattern-matches `"instance member"` + `"cannot be used on type"` in build output; returns detail string `"instance-method-shape-not-supported"`; the surveyRecord becomes `.architectural-coverage-pending` instead of `.measured-error`. Preserves the v1.56 `.measured-error = 0` invariant.

   - `matchesInstanceMemberError` + `matchesInstanceMemberOnStderr` + `internalAccessPrecedesInstanceMember` tests for `architecturalPendingDetail`. (Precedence rule: when both patterns match, `internal-api-not-accessible` wins — it's the more semantically meaningful gap.)

## Risks

1. **Adding swift-collections to every workdir's Package.swift is slightly wasteful** for non-OC picks. **Mitigation**: SwiftPM caches packages; resolve cost ~0. Verifier workdirs already have 3 SwiftPM deps; adding a 4th is negligible. v1.60+ may conditionalize.

2. **The double-qualifier strip in `render` could over-eagerly trim** if a function genuinely starts with the type-qualifier prefix (e.g. `OrderedSet.staticHelper` where `staticHelper` is the real name). **Mitigation**: vanishingly unlikely in practice (function names don't usually contain their owning type's name); the strip applies to the FIRST occurrence so only one level is stripped; cycle-56 evidence shows no false positives on the cycle-27 surface.

3. **V1.59.A's reclassification could mask real `.measured-error` cases** that share the "instance member" pattern. **Mitigation**: the pattern is Swift-compiler-specific and unambiguous; V1.59.B's test for precedence (internal-access > instance-member) handles the case where both patterns appear; V1.56.A's framing applies — measurement-tooling gaps vs architecture gaps.

4. **The deterministic `OrderedSet<Int>` generator has narrow coverage** (4-element sets `{n, n+1, n+2, n+3}` for n ∈ [0, 100]). **Mitigation**: v1.60+ can widen the generator once mutating-method emission lands; the current narrow generator is sufficient to surface the next-layer gap.
