## Design decisions confirmed before drafting

Three load-bearing scope decisions ratified by the user before this plan:

1. **All three templates in one cycle.** Confirmed via /ask — over a "one template per cycle" alternative. Trade-off accepted: ~2× v1.47's scope (~10 workstreams + ~50 tests) in exchange for closing the cycle-27 template-coverage matrix in a single tagged release. Cycle-45 measurement impact attribution becomes a bundle ("v1.48 = +15 picks"), not a per-template breakdown.

2. **Strategist-only routing for the new templates.** v1.46/v1.47's hardcoded `{Complex<Double>, Double}` carrier path stays in place for the existing four templates (round-trip / idempotence / commutativity / associativity). The new three templates route exclusively through `StrategistDispatchEmitter` — no per-template hardcoded emitter for FP carriers. Rationale: cycle-27 has zero Complex<Double>/Double picks for these three templates; building hardcoded paths now would be speculative-design overhead. If a future cycle has such a pick, the hardcoded path can be added then.

3. **No new strategist strategies (`.memberwiseArbitrary` still deferred).** v1.47.E's `.memberwiseArbitrary` rejection stays — that's a v1.49+ architectural addition, not a v1.48 deliverable. The 15 cycle-27 picks v1.48 unblocks all use carriers the existing strategist surface handles (mostly Int / collection-of-Int via lifted wrappers).

## What v1.48 ships

Ten workstreams + bundled closeout:

1. **V1.48.A — Extend `StrategistDispatchEmitter` with three new template composers**. Add to `StrategistDispatchEmitter+Templates.swift`:

   - `composeIdempotenceLiftedPass(inputs, recipe) -> String` — wraps the recipe's element generator in `Gen<[T]>.array(of: ...)` (or the kit-equivalent helper), then checks `f(f(xs)) == f(xs)` where `xs: [T]`. Single-pass with zero-edge sentinel.
   - `composeDualStyleConsistencyPass(inputs, recipe) -> String` — draws one value `x`, calls the non-mutating function `nonMutCall(x)`, then mutates a copy `var xCopy = x; mutCall(&xCopy)` and asserts `nonMutCall(x) == xCopy`. `inputs.functionCalls` carries `[nonMutCall, mutCall]`.
   - `composeMonotonicityPass(inputs, recipe) -> String` — draws two values, sorts so `a <= b`, applies `f` to each, asserts `f(a) <= f(b)`. Carrier must conform to `Comparable`; v1.48 trusts the strategist's surface (Int / String / Bool / fixed-width ints — all `Comparable`).

   Top-level `defaultPassSection` switch in `StrategistDispatchEmitter.swift` gains three new cases routing to the composers.

   **Sub-decision — collection generator for idempotence-lifted.** Pre-implementation check on swift-property-based 1.2.x revealed the kit ships `Generator<T>.array(of: ClosedRange<Int>)` natively (`Sources/PropertyBased/Gen+Collection.swift`). v1.48 uses the kit helper directly: `elementGenerator.array(of: 0 ... 8)`. Cleaner than the inline `flatMap` the plan originally drafted; no extra imports needed (already included in the strategist's default `PropertyBased` import). If cycle-45 measurement surfaces array-generation as too narrow (e.g. sort idempotence-passes trivially on 1-element arrays), v1.49 can widen the range.

   **Sub-decision — dual-style-consistency value-copy semantics.** Pure-value `var copy = x` is correct for value-semantic types (Int, String, Bool, fixed-width). For reference types (class instances), this would alias the same instance and the mutation would affect both. v1.48 documents this as a precondition: strategist-routed dual-style-consistency assumes value-semantic carriers. The strategist's current surface (raw types + structs + enums) is value-semantic by construction, so this isn't a load-bearing risk for v1.48.

2. **V1.48.B — Three new PairResolvers**. Mirror the v1.45/v1.46 pattern but adapted to each template's call shape:

   - `IdempotenceLiftedPairResolver` — single-function on lifted carrier. Same shape as `IdempotencePairResolver` but the carrier reference uses `Array<T>` form (or the lifted form already in `entry.typeName`).
   - `DualStyleConsistencyPairResolver` — pair of names: non-mutating function + mutating function. Single carrier. Pair lookup against a `curated` list (`sort` ↔ `sorted`, `reverse` ↔ `reversed`, `shuffle` ↔ `shuffled`, etc.).
   - `MonotonicityPairResolver` — single-function. Same shape as `IdempotencePairResolver`.

   Each is ~60 lines. PRD §14 file-length cap is generous.

3. **V1.48.C — `VerifyCommand+TemplateDispatch` extension**. Add three entries to `resolveFunctionCalls`:
   - `"idempotence-lifted"` → `IdempotenceLiftedPairResolver.resolve(entry)`
   - `"dual-style-consistency"` → `DualStyleConsistencyPairResolver.resolve(entry)`
   - `"monotonicity"` → `MonotonicityPairResolver.resolve(entry)`

   Update `supportedTemplates` list to seven entries. `v1_46HardcodedCarriers` set stays as `{Complex<Double>, Double}` — these three new templates have no v1.46-hardcoded path, so any carrier in that set with one of these template names routes to `strategistBundle` and surfaces a `.unsupportedCarrier` error if the strategist can't handle it (acceptable v1.48 behavior; cycle-27 has no such picks).

4. **V1.48.D — `RenderShape` extension**. Three new cases in `Sources/SwiftInferCLI/VerifyResult.swift`:

   - `.idempotenceLifted`: subjectLine `"idempotence-lifted on \(forwardName) over \(carrierType)"`, forwardExpression `"\(forwardName)(xs)"`, inverseExpression `"\(forwardName)(\(forwardName)(xs))"`, expectedExpression `"\(forwardName)(xs)"`.
   - `.dualStyleConsistency`: subjectLine `"dual-style-consistency on \(forwardName)/\(inverseName) over \(carrierType)"`, forwardExpression `"\(forwardName)(x)"`, inverseExpression `"{ var copy = x; copy.\(inverseName)(); return copy }()"`, expectedExpression `"\(forwardName)(x)"`.
   - `.monotonicity`: subjectLine `"monotonicity on \(forwardName) over \(carrierType)"`, forwardExpression `"\(forwardName)(a)"`, inverseExpression `"\(forwardName)(b)"`, expectedExpression `"f(a) ≤ f(b) when a ≤ b"`.

   `renderShape(for:)` gains three new cases mapping `templateName` to the new `Kind`.

## Why bundling, not sequencing

The user picked bundling over sequential (v1.48/v1.49/v1.50 one-per-template). Three rationales as recorded in the /ask response:

1. **Closes the cycle-27 template-coverage matrix in one tag.** Sequential cadence would have v1.48 + v1.49 + v1.50 reach the same endpoint (87.5% verifiable-fraction); bundling reaches it sooner. The trade-off is per-template measurement-impact attribution — cycle-45's headline becomes "+15 picks in one cycle" rather than "+6 in v1.48, +5 in v1.49, +4 in v1.50".

2. **Shared infrastructure makes bundling cheap.** All three templates route through the existing `StrategistDispatchEmitter`'s recipe-resolution layer. The per-template emit logic (V1.48.A) adds ~30 lines per template = ~90 lines total, vs ~300 lines if each template needed its own hardcoded emitter (v1.46/v1.47 pattern). The 3× scale on tests is the main cycle-effort multiplier.

3. **No new architecture in v1.48.** Every workstream is mechanical extension of v1.47 patterns. The plan is reviewable in one pass; the implementation should be too.

## Scope boundaries

- **In scope (v1.48):** Three new template emit composers (idempotence-lifted, dual-style-consistency, monotonicity) inside `StrategistDispatchEmitter+Templates.swift`; three new pair resolvers; CLI dispatch extension; `RenderShape` extension; cycle-45 calibration measurement.
- **Out of scope, deferred to v1.49+:** `.memberwiseArbitrary` strategy (still throws .unsupportedCarrier); non-curated round-trip pair derivation (the SemanticIndex schema bump for both-halves storage; unblocks the final cycle-27 REJECT); accept-flow integration (verify outcomes → `decisions.json`); verification cache; "Verified" first-class tier; Complex<Double>/Double hardcoded paths for the new three templates.
- **Not changing:** v1.42–v1.47 verify pipeline behavior for round-trip / idempotence / commutativity / associativity × all carriers stays bit-stable. The V1.48.A–D extensions are strictly additive — three new switch arms in the dispatch + render layers.

## Risks

1. **Array generator availability in `swift-property-based`.** Sub-decision V1.48.A picks inline `Gen<Int>.int(in: 0..<8).flatMap { ... }` to produce arrays. If `swift-property-based` ships a `Gen<Element>.array(...)` helper, using it would be cleaner; if not, the inline form is the only path. **Mitigation**: V1.48.A's first commit lands the inline form; if a kit-side helper exists and reads cleaner, swap in V1.48.A.2. Cycle-45 measurement is unaffected either way — the property check is the same.

2. **Idempotence-lifted may verify-pass trivially on small arrays.** A 0- or 1-element array is trivially idempotent under any deterministic function (`f([]) = []` and `f(f([])) = []`; same for 1-element). If the inline array generator emits mostly small arrays, the verify pass becomes a weak signal. **Mitigation**: V1.48.H.1's `sorted()` test uses an array generator with size 0..<8 — the expected sample includes arrays of 4–7 elements ~half the time. If cycle-45 surfaces a "sort idempotence passes trivially" concern, widen the array-size range or add a min-size floor.

3. **Dual-style-consistency carrier value-semantic precondition.** The `var copy = x` mutation idiom is correct only for value types. Reference-type carriers (class instances) would alias the mutation, breaking the equality check. **Mitigation**: the strategist's surface is value-semantic by construction (raw types + value-typed structs + enums), so this isn't load-bearing for v1.48. Documenting the precondition in the renderer's error path (when a reference-type carrier surfaces in v1.49+) is sufficient.

4. **Monotonicity on String carriers — lexicographic ordering.** `String: Comparable` uses Unicode-collation ordering, which is the expected semantic. Functions that should be monotone on string inputs are rare in practice; v1.48's monotonicity tests only exercise Int. **Mitigation**: V1.48.H.3 covers Int; cycle-45 measurement will record whether any cycle-27 picks fall on string monotonicity (current cycle-27 sample shows none).

5. **Test wall-clock impact**. v1.48 adds ~3 subprocess integration tests. 16 → 19 parallel integration tests. Cycle-44 already documented the §13 perf-test flake intensifying under contention; v1.48 may make this further-frequent. **Mitigation**: v1.48 doesn't address the flake (still deferred). Cycle-45 findings record any new flake patterns. If recurrent in CI, the v1.49 plan should prioritize the test-isolation mitigation cycle-47's notes outline.

6. **Dual-style-consistency curated-pair table small.** The initial curated list (sort/sorted, reverse/reversed, shuffle/shuffled) covers ~4 entries — cycle-27 has 5 picks, so at least one pick might fall outside the curated list. **Mitigation**: the resolver throws `.unsupportedPair` cleanly when no match; that pick stays out-of-scope. Cycle-45 records which picks (if any) need additional curated entries. The list is trivially extensible in v1.49+.

7. **The "no new architecture" claim is load-bearing.** v1.48's scope assumes the existing v1.47 `StrategistDispatchEmitter` + dispatch infrastructure absorbs all three templates without new abstractions. If implementation surfaces a need for a per-template carrier-handling layer (e.g. idempotence-lifted needing a `Gen<[T]>` wrapper that the strategist doesn't natively produce), the v1.48 plan's effort estimate is wrong. **Mitigation**: V1.48.A.1 (idempotence-lifted) is the most-likely-to-surface-this-issue workstream; tackle it first. If a new abstraction is needed, the plan can be revised inline.
