# v1.37 Calibration Plan — Constraint Engine Migration #2 (MonotonicityTemplate)

Working doc for the v1.37 cut. v1.37 is the **thirty-fourth calibration cycle** and the **second template migration** to the Constraint Engine (PRD §20.2). Applies the mechanical pattern established in V1.36.D to `MonotonicityTemplate`.

> **Why MonotonicityTemplate next.** Same unary-summary shape as Commutativity. Single runtime input (`vocabulary` only — no `inheritedTypesByName` since monotonicity has no protocol-coverage veto). Caveats are constant (no FP-storage-conditional caveat). Simpler than Commutativity in three of three dimensions. After v1.37 the migration pattern is validated against a second representative shape → v1.38+ can batch-migrate the remaining 8.

**Ephemeral** — archive to `docs/archive/v1.37 Calibration Plan.md` once tagged.

## What v1.37 ships

Eight deliverables (compressed cadence — A is the only Sources/ workstream; D–H bundled):

1. **V1.37.A**: `MonotonicityTemplate` migration. `suggest(for:)` becomes a 4-line `ConstraintRunner.suggest` wrapper; `makeConstraint(vocabulary:)` factory + signal-accumulator + caveats helpers.
2. **V1.37.B**: Equivalence tests against a 6+ fixture corpus covering accept (curated verb + suffix + vocabulary), gate fails (non-monotonic codomain, multi-param, mutating), non-deterministic veto.
3. **V1.37.C**: Cycle-34 findings doc.
4. **V1.37.D–H**: Perf + version + closeout + tag bundled.

## V1.37.A — `MonotonicityTemplate` migration

Apply the V1.36.D pattern:

1. Add `makeConstraint(vocabulary:) -> Constraint<FunctionSummary>` factory on `MonotonicityTemplate`.
2. Extract `accumulatedSignals(for:vocabulary:) -> [Signal]` package-internal helper (signal-accumulation order preserved).
3. Extract `makeCaveats(for:) -> [String]` package-internal helper (constant caveat list).
4. Rewrite `suggest(for:vocabulary:)` as a 4-line `ConstraintRunner.suggest` wrapper.

**Critical invariant**: existing MonotonicityTemplate test suite (`MonotonicityTemplateTests.swift` etc.) must pass without modification.

## Risks

- **Risk**: monotonicity has a `comparableCodomains` curated set + suffix-matching logic; the constraint gate must replicate this exactly. **Mitigation**: extract the existing private `orderedCodomainSignal(for:)` as the gate's `appliesTo` predicate (same function, return-non-nil check). The signal-accumulator helper re-invokes it for the actual signal value, since the gate doesn't expose the result.
- **Risk**: scope creep. **Mitigation**: hard cap at one template migration. Cycle-35+ continues incrementally OR batch-migrates if v1.37 confirms the pattern is robust.
