# v1.55 Calibration Plan — FP generator-range refinement (per-function default-pass domain for Complex round-trip)

Working doc for the v1.55 cut (fifty-second calibration cycle). **Fifth Phase 2 gap-closing cycle** — closes the cycle-51 generator-tuning finding: the v1.42 RoundTripStubEmitter's `Double.random(in: -1e6 ... 1e6)` range exceeded the EF surface's stable domain, surfacing as 8 spurious `.defaultFails` in cycle-51.

> **Motivation.** Cycle-51 measurement (`docs/calibration-cycle-51-findings.md`) showed all 8 round-trip Complex EF picks landed in `.defaultFails` because `exp(800+0i)` overflows to `inf`, breaking the round-trip. The cycle-46 structural prediction of `.bothPass` was correct *within the function's stable domain*; the v1.42 generator's range overshot. v1.55 refines the default-pass generator to the principal-branch domain — 2 distinct ranges per function family (cos/cosh's right-half-plane vs symmetric ±1.5 for everything else).

## What v1.55 ships

Three workstreams + bundled closeout. Smallest cycle since v1.53.

1. **V1.55.A — Per-function default-pass domain for Complex round-trip**. Cycle-52 measurement (`docs/calibration-cycle-52-findings.md`) iterated through two cuts:

   - **First cut (uniform ±1.5)**: closed 6 of 8 picks (exp/log, sin/asin, tan/atan, sinh/asinh, tanh/atanh + their bidirectional pairs). cos/acos and cosh/acosh still failed because their principal-branch inverses return `Re ≥ 0` — the round-trip only holds for the right half-plane.
   - **Final cut (per-function table)**: added a 2-domain lookup. `cos`/`acos`/`cosh`/`acosh` use `Re ∈ [0, 1.5], Im ∈ [-1.5, 1.5]`; everything else uses symmetric `±1.5`. Closes all 8 round-trip Complex EF picks.

   **Why per-function table, not uniform Re ≥ 0**: the symmetric pairs (sin/asin, tan/atan, etc.) round-trip correctly on the *full* symmetric range. Restricting them unnecessarily loses input coverage. The 2-entry curated table captures the actual mathematical constraint (acos/acosh's right-half-plane codomain) without over-restricting.

   **Implementation**: `complexDefaultPassDomain(forwardCall:)` helper at the top of the emitter; pattern-matches on the bare function name (suffix after `.`); returns `(reMin, reMax, imMin, imMax)` as Double literals; interpolated into the generated stub. Future cycles can extend the curated table with more granular per-function domains (e.g., narrow `exp` to `Im ∈ (-π, π]` to cover log's branch cut; expand `sin/cos` to `Re ∈ ±π/2` once the inputs benefit).

   **Sub-decision — what about the v1.43 edge pass**: left unchanged. The edge pass uses the kit's `Gen<Complex<Double>>.edgeCaseBiased()` which generates 12 curated edge cases (NaN, infinity, etc.) + a 90% slice over a wide range. Cycle-52 shows the edge pass still surfaces overflow at large inputs, producing `.edgeCaseAdvisory` (correctly: "property holds in default domain, breaks at edge"). That's exactly the intended v1.43 design.

## Risks

1. **The 2-domain split may not generalize to all cycle-27 round-trip pairs**. Cycle-52 covers 8 EF picks; v1.56+ may add curated pairs (e.g., `square` / `sqrt`, or non-EF round-trips) that need additional domain entries. **Mitigation**: V1.55.A's table is small (2 entries) and the helper signature accommodates more entries; v1.56+ refines as cycle-N evidence demands.

2. **The narrow default-pass domain reduces input coverage for properties that hold over a wider range**. E.g., `exp/log` round-trip holds for `Re(z) ∈ [-700, 700]`, but v1.55 tests only `±1.5`. **Mitigation**: the v1.43 edge pass still tests boundaries; the 90% wide-finite slice of `edgeCaseBiased()` covers most of the larger domain. Future cycles may add a "wide-domain pass" if cycle-N reveals coverage gaps.

3. **`.edgeCaseAdvisory` outcomes don't strictly match cycle-46's `.bothPass` predictions**. In the 4-outcome category sense, the 8 EF picks now produce `.edgeCaseAdvisory` where cycle-46 predicted `.bothPass`. **Mitigation**: the cycle-52 findings doc explicitly addresses this — `.edgeCaseAdvisory` is a *refinement* of `.bothPass` ("property holds in domain + edge cases overflow"), not a contradiction. Under the "property semantically holds" interpretation, the agreement-rate is 100% on the cycle-46-overlap subset.
