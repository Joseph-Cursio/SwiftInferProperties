## What v1.53 ships

Three workstreams (A + E + F) plus closeout. Smallest cycle since v1.41 (commits dominated by docs + measurement, code change ~80 LoC).

1. **V1.53.A â€” `DYLD_LIBRARY_PATH` injection on verifier subprocess**. In `VerifierSubprocess.runVerifierBinary`: detect the active Swift toolchain's testing-library directory; inject `DYLD_LIBRARY_PATH=<toolchain>/usr/lib/swift/macosx/testing` on the spawned subprocess env.

   **Detection**: `swift -print-target-info` â†’ JSON `paths.runtimeResourcePath` + `macosx/testing` suffix. **Not** `xcrun --find swift` â€” xcrun returns Xcode's default toolchain, which is the wrong one when the user has a swiftly-managed or `TOOLCHAINS`-overridden toolchain (cycle-49's host has both Xcode default *and* a custom swift-6.2.4-RELEASE toolchain; only the latter has libTesting installed).

   **Caching**: `static let cachedTestingLibraryDirectory: String?` so the 109-pick survey pays the lookup cost once instead of 109 times.

   **Graceful fallback**: if `swift -print-target-info` fails or returns an unexpected JSON shape, the helper returns nil and the subprocess inherits the parent's environment unchanged. v1.52 behavior preserved on machines without the right toolchain layout.

   **Sub-decision â€” DYLD_LIBRARY_PATH composition**. When the parent already has `DYLD_LIBRARY_PATH` set, the new entry is **prepended** so our libTesting fix takes precedence on conflict. User-set values in the rest of the path still resolve.

   **Alternative approaches considered** (in cycle-49 findings):
   - (a) Drop the PropertyBased dependency, inline what we need: significant work, ~500+ LoC, defeats the kit-side reuse.
   - (b) Add `-Xlinker -rpath` at swift-build time: bakes the fix into the binary but couples the workdir to a specific toolchain path; brittle when the toolchain moves.

   (V1.53.A's runtime DYLD injection is the most surgical fix â€” no link-time, no workdir-synthesis changes, just an env var on subprocess spawn.)

   **Smoke test** (already run): `swift-infer verify --suggestion 0EE1` (cycle-49 parse-dyld pick: `Complex.pow` associativity) now produces `.defaultFails` â€” the property check ran 100 trials, found a counterexample, and printed the expected expressions/inputs. **First non-error end-to-end measurement in the project's calibration history.**

2. **DYLD_LIBRARY_PATH may be scrubbed on macOS due to SIP**. Mitigation: cycle-50 smoke test confirms propagation on the host; cycle-51+ adds a binary-path check.

3. **Cycle-50's 12-outcome agreement-rate is computed on a small N**. 12 out of 109 = 11% of the full surface. Statistically thin; cycle-51+ should grow the sample. Mitigation: cycle-50 findings doc frames this honestly â€” the agreement-rate is informative but not yet load-bearing for v1.0 ship readiness.
