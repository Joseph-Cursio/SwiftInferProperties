# v1.52 Calibration Plan — close the cycle-48 call-expression-shape gap + first measurable `.bothPass` / `.defaultFails` signal

Working doc for the v1.52 cut (forty-ninth calibration cycle). **Second gap-closing cycle of Phase 2** — addresses the new gap cycle-48 surfaced below cycle-47's: the resolver's `<TypeQualifier>.<funcName>` call-expression shape is wrong for operator-named functions and free functions promoted to type namespaces. Cycle-48 landed 22/109 = 20.2% measured-execution but all 22 in `measured-error` (11 build-failed + 11 parse-error); v1.52's primary goal is to produce the **first measurable subset of `.bothPass` / `.defaultFails`** so the synthetic↔real-indexer per-pick agreement-rate becomes computable.

> **Motivation.** Cycle-48 established that closing the carrier-resolution gap (v1.51) doesn't get you to `.bothPass`-class outcomes — the call-expression-shape gap is the next load-bearing fix. 11 of the 22 measured picks build-failed because the resolver builds `Complex./` for `/(z:w:)` (Swift needs `(/)(z, w)`) and `Complex.exp` for `exp(_:)` indexed under Complex (the real Swift call is the free function `exp(value)`). The other 11 parse-error (subprocess exit 6 SIGABRT) with no diagnostic — v1.52 also captures stderr so the next cycle can tell whether parse-errors are a runtime version of the same shape gap or something else (NaN trap, missing import).

## What v1.52 ships

Six workstreams + bundled closeout. Smaller than v1.50 + v1.51; pure resolver-layer fix + a small infrastructure improvement + a forward-looking binding-table expansion.

1. **V1.52.A — Call-expression-shape resolver**. The current resolver builds the call-expression string at four sites (`RoundTripPairResolver.swift:105/111`, `VerifyCommand+TemplateDispatch.swift:219/228`, and parallel sites in the commutativity / associativity / monotonicity / idempotence pair resolvers). All four use the same pattern:

   ```swift
   forwardCall: "\(typeQualifier).\(stripParameterLabels(name))"
   ```

   The fix introduces a new `CallExpressionShape` enum with three cases:

   - `.staticMethod(qualifier:method:)` — emits `"\(qualifier).\(method)"` (current behavior).
   - `.operatorFunction(name:)` — emits `"(\(name))"` (e.g. `(+)`, `(/)`); Swift's binary-operator-as-function form. The synthesized stub passes arguments as a tuple: `(+)(a, b)`. The emitter's existing `\(inverseCall)(\(forwardCall)(value))` shape still composes correctly because `(+)(a, b)` and `Complex.exp(value)` are both expression forms that take parenthesized arguments.
   - `.freeFunction(name:)` — emits `"\(name)"` (no qualifier). For `exp(_:)` indexed under `Complex`, the call is `exp(value)`, not `Complex.exp(value)`. Swift resolves the free function via the `ElementaryFunctions` conformance on `Complex<Double>`.

   **Classification rule**:
   - If the function name (post-`stripParameterLabels`) is a sequence of one or more operator characters (`+`, `-`, `*`, `/`, `%`, `<`, `>`, `=`, `!`, `&`, `|`, `^`, `~`, `?`, `.`) — `.operatorFunction`.
   - Else if `(carrier, bareFunctionName)` is in a curated **free-function set** (initial entries: `Complex` × `{exp, log, sin, cos, tan, asin, acos, atan, sinh, cosh, tanh, asinh, acosh, atanh}` + `Double`/`Float` × `{log, exp, sqrt, sin, cos, tan, asin, acos, atan, sinh, cosh, tanh, asinh, acosh, atanh}` — the `ElementaryFunctions` surface) — `.freeFunction`.
   - Else — `.staticMethod` (current behavior; default).

   **Two design candidates**:

   - **(a) Inline classification at each call site**: 4 sites × ~10 LoC = ~40 LoC; keeps the new logic local to where forwardCall is built.
   - **(b) New `CallExpressionShape` type with a `classify(carrier:bareFunctionName:)` factory**: ~30 LoC for the type + 4 × 3 LoC at call sites = ~42 LoC; centralizes the rule, easier to unit-test, easier to extend in v1.53+ (e.g. instance methods on a value, `init`-as-function).

   Going with **(b)** — the classify rule is non-trivial and exists in 4 places; centralization avoids drift and gives V1.52.D a clean unit-test boundary.

   **Sub-decision — where the free-function set lives**. Two options:
   - **(b.i)** Hardcoded in `CallExpressionShape.swift` (~15 LoC for the curated set).
   - **(b.ii)** Threaded through `SemanticIndexEntry` as a new field (schema bump).

   Going with **(b.i)** — schema bumps are heavyweight and the ElementaryFunctions surface is well-defined. If v1.53+ needs to discover free-function-ness from the indexer, that's a separate decision.

   **Expected impact**: closes all 11 build-failed picks. Some of the 11 parse-error picks may also clear (if the SIGABRT comes from `Complex.exp(value)` resolving differently at runtime than `exp(value)`). Cycle-48 findings speculated this; V1.52.B will tell us.

2. **V1.52.B — Subprocess stderr capture for parse-error diagnostics**. Currently `VerifierSubprocess.swift` captures stdout and reports "stdout (last 5 lines, pipe-joined): ..." on parse-error; cycle-48 picks land with **empty** stdout because the subprocess crashes before printing. Adding stderr capture surfaces the actual trap reason (e.g. `Fatal error: NaN in normalization`, missing-import errors, precondition failures).

   **Implementation**:
   - `VerifierSubprocess.swift`: route the child process's stderr into a separate `Pipe`, drain in parallel with stdout, return both as `(stdout, stderr)`.
   - `VerifyResultParser`: when the outcome is `.error(parseFailureWithStderr)`, include the last 5 stderr lines in the detail string. Format: `"stdout (last 5 lines, pipe-joined): <...>; stderr (last 5 lines, pipe-joined): <...>"`.
   - `VerifyResultRenderer`: rendering carries forward unchanged — the detail string just gets longer.

   **Scope**: small. ~30 LoC plus 2 unit tests. **No CLI changes**; the new detail surfaces through the existing rendering path.

   **Sub-decision — line-cap and truncation**. Cycle-48 saw empty stdout; stderr could be very long if a `swift build` failure ends up routed there (a future cycle that misroutes builds, not v1.52's scope). v1.52 caps at the last 5 lines (mirrors stdout); if the line is longer than 200 chars, truncate with an ellipsis. Keeps the findings JSON tractable.

The v1.51.A canonicalizer landed `"Complex" → "Complex<Double>"`; v1.52.C adds the indexer-evidenced bindings.

   **Expected impact**: closes the 3 chunked-Index REJECTs cycle-46 predicted `.defaultFails` + ~7 other Index-aliased picks that hit `.unsupportedCarrier` in cycle-48. The 60+ OrderedSet/OrderedDictionary picks that need full TypeShape-driven Element binding stay deferred to v1.53.

Aligns with V1.51.B's forward-looking-fix pattern.

## Scope boundaries

- **In scope (v1.52):** Call-expression-shape classification (operator names + free functions promoted to type namespaces + static methods); subprocess stderr capture; `GenericBindingResolver.curatedBindings` expansion for chunked-Index + OrderedSet.Index placeholder; ~17 unit tests; cycle-49 measurement + findings doc.
- **Out of scope, deferred to v1.53+:** TypeShape-driven generic instantiation for `OrderedSet` / `OrderedDictionary` / etc. (Element-type binding from indexed TypeShape; closes ~60 OC picks); Phase 2 accept-flow integration (verify outcomes → `decisions.json`; gated on cycle-49's agreement-rate signal); call-expression-shape extensions beyond operator + free function (instance methods on a value, `init`-as-function, subscript-as-function); `Complex<Float>` emitter path; verification cache; "Verified" first-class tier.
- **Not changing:** No template additions, no new strategies, no new emitter Inputs fields (V1.49.A's `preamble` stays the universal hook), no SemanticIndex schema bump. All v1.42-v1.51 behavior on synthetic inputs stays bit-stable (V1.52.A's classify defaults to `.staticMethod` for non-Complex/non-Double-Float carriers, preserving the current shape for the existing test fixtures).

## Risks

1. **V1.52.A's free-function classification may be wrong for some Complex methods.** swift-numerics' `Complex` exposes `exp`/`log`/etc. through the `ElementaryFunctions` conformance, but it also has *some* static methods on the `Complex` type proper (e.g. `Complex.i`). The curated free-function set must not over-classify — if a future cycle indexes `Complex.i(_:)` and it gets `.freeFunction`-classified, the synthesized call `i(value)` fails to compile. **Mitigation**: V1.52.D.1 includes an explicit "Complex × non-ElementaryFunctions name falls back to `.staticMethod`" test; the curated set lists exact names from the ElementaryFunctions protocol (14 entries), not a wildcard.

2. **The 11 parse-error picks may not be call-expression-shape-related.** If the SIGABRT comes from a numerical trap (NaN normalization, divide-by-zero on `0/0`), V1.52.A's free-function fix won't help. **Mitigation**: V1.52.B's stderr capture surfaces the actual trap reason for cycle-49 analysis; the v1.52 plan accommodates this by not promising all 11 parse-errors clear — projected outcome 4-11 `.bothPass`/`.defaultFails`, not 22.

3. **Cycle-49's agreement-rate signal could land below the cycle-46 synthetic 100%.** Cycle-46 reported 100% per-pick agreement on synthetic-shape-class inputs; real-indexer measurement may show 80-95% (or lower) on the first non-zero subset. **Mitigation**: V1.52.F's findings doc frames the result honestly — the synthetic and real numbers measure different things; a divergence is methodologically interesting, not a failure. If the divergence is large (<70% real-indexer agreement), v1.53's roadmap rotates toward gap-finding rather than coverage expansion.

4. **The chunked-Index bindings in V1.52.C may name the wrong qualified path.** The cycle-48 survey JSON lists carriers but not always the canonical qualified form (e.g. `ChunkedByCollection<Base, Subject>.Index` vs `Base.SubSequence.Index` vs `Int`). The placeholders chosen here assume the simplest form. **Mitigation**: V1.52.D.4 pins the exact lookups; cycle-49 measurement reports which of the 5 new bindings surface and which don't. v1.53 refines based on evidence (mirrors V1.51.B's pattern).

5. **V1.52.B's stderr-capture infrastructure could introduce a race on subprocess exit.** Reading stdout and stderr from separate threads/Tasks and merging on subprocess termination is a known concurrency-correctness footgun (deadlock if one pipe fills). **Mitigation**: use the same `Pipe` + async-stream draining pattern that V1.42.C.3 uses for stdout (proven on 50+ subprocess tests); V1.52.D.3's tests deliberately exercise the dual-pipe path with non-trivial output volumes. If a flake surfaces, fall back to capturing stderr-only and dropping stdout (the cycle-48 evidence shows stdout is empty on parse-error anyway).
