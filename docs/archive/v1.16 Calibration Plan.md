## Context: the post-v1.15 OC SetAlgebra survivors

Per `docs/calibration-cycle-12-data/post-domain-marker-counter-swift-collections-OrderedCollections.discover.txt`, the 6 OC Possible-tier survivors that fit the SetAlgebra-shape pattern at v1.15:

```
round-trip claims surviving cycles 7-12 on OC with SetAlgebra shape (n=2):
  intersection(_:) (Self) -> Self  ↔  subtracting(_:) (Self) -> Self
    in  OrderedSet+Partial SetAlgebra intersection.swift:46
    × OrderedSet+Partial SetAlgebra subtracting.swift:45
  intersection(_:) (Self) -> Self  ↔  subtracting(_:) (Self) -> Self
    (cross-file pair via UnorderedView)

idempotence claims surviving cycles 7-12 on OC with SetAlgebra shape (n=4):
  intersection(_:) (Self) -> Self
    in  OrderedSet+Partial SetAlgebra intersection.swift:46
  subtracting(_:) (Self) -> Self
    in  OrderedSet+Partial SetAlgebra subtracting.swift:45
  intersection(_:) (Self) -> Self
    in  OrderedSet+UnorderedView.swift:416
  subtracting(_:) (Self) -> Self
    in  OrderedSet+UnorderedView.swift:583
```

(Sourced via per-suggestion `^Template:` extraction on the cycle-12 OC snapshot. **Note**: the cycle-12 findings table mis-stated the idempotence count as 2 — the narrative text correctly said "4 SetAlgebra ... idempotence claims"; only the summary table row was wrong. Plan-vs-actual for v1.16 should treat **6** as the projection target, not 4. V1.16.6 will fix the cycle-12 findings table alongside the v1.16 archive pass.)

The mathematical truth: applying `intersection(_:)` twice with the same argument is idempotent only if the second application sees the *same* `other` set — but for the discover-tier inferred property "applied twice equals applied once on the same input pair", `intersection(intersection(x, y), y) == intersection(x, y)` does hold, but the discover engine's idempotence claim is the strictly weaker `f(f(x)) == f(x)` for a unary `(T) -> T` shape. The `(Self) -> Self` shape on SetAlgebra binary ops is a *partial application* shape — `(other) -> result`, not a true idempotent unary. **Same structural argument as V1.14.1's inverse-pair veto**: any unary function drawn from the SetAlgebra binary-op surface is a partial application, not a self-mappable transformation.

For round-trip: the v1.14 plan's argument applies verbatim — `intersection ↔ subtracting` aren't inverse pairs, and they aren't round-trip pairs either (the discover engine's two surface `intersection(_:) ↔ subtracting(_:)` round-trip claims emerge from FunctionPairing pairing two SetAlgebra binary ops as a forward+reverse, which is the exact same false-positive pattern).

### Why the existing V1.14.1 veto misses these

V1.14.1's `setAlgebraShapeVeto(for:)` is wired into `InversePairTemplate.suggest(...)` only. The same structural argument applies to round-trip + idempotence's signal pipelines, but they don't currently consult `SetAlgebraShape.binaryOps`. v1.16 closes that.

### Why a single mechanism extended to two more templates is correct here

The structural argument "any function drawn from `{union, intersection, symmetricDifference, subtracting}` on `Self`-typed shape is a SetAlgebra partial-application, not a true self-map / inverse / round-trip" applies uniformly across all three templates that surface the false-positive shape:
- **Inverse-pair** (V1.14.1, cycle 11): pair-based; `intersection ↔ subtracting` isn't an inverse.
- **Round-trip** (v1.16): pair-based; same pair-pattern false-positive shape.
- **Idempotence** (v1.16): single-function; `intersection(_:)` applied twice as `(T) -> T` is the partial-application false-positive.

Mechanism-class continuity: function-name + type-shape composite (cycle-11 family) extended to two more templates. Reuses `SwiftInferCore.SetAlgebraShape.binaryOps` (canonical-from-day-one in V1.14.1) and the existing `Signal.Kind.protocolCoveredProperty` case (V1.14.1's reuse posture).

## What v1.16 ships

Two deliverables (mirroring v1.14/v1.15 calibration-cycle shape):

1. **Hoist + per-template SetAlgebra-shape veto wiring + tests.**

   (a) **Hoist `isSelfTypedBinaryOp(_:)` to `SwiftInferCore.SetAlgebraShape`** as a `public static func`. Currently lives `private` on `InversePairTemplate` extension in `InversePairSetAlgebraShapeGate.swift` (V1.14.1). Hoisting follows the v1.13 `DirectionLabels` hoist precedent: when a template-agnostic helper has more than one consumer, it lives in `SwiftInferCore.<Namespace>.<helper>`. Idempotence + round-trip become consumers in v1.16.

   (b) Two new files mirror the V1.14.1 file-split precedent:
   - `Sources/SwiftInferTemplates/RoundTripSetAlgebraShapeGate.swift` — `RoundTripTemplate.setAlgebraShapeVeto(for:)` extension method. Both pair sides must pass `SetAlgebraShape.isSelfTypedBinaryOp(_:)` AND both names in `SetAlgebraShape.binaryOps`.
   - `Sources/SwiftInferTemplates/IdempotenceSetAlgebraShapeGate.swift` — `IdempotenceTemplate.setAlgebraShapeVeto(for:)` extension method. Single function must pass `SetAlgebraShape.isSelfTypedBinaryOp(_:)` AND name in `SetAlgebraShape.binaryOps`.

## Open decisions

1. **Veto weight: -25 uniform vs per-template tuning.** Default proposal: **(a) -25 uniform** (matches V1.14.1's existing inverse-pair calibration; produces clean Suppressed for round-trip/idempotence at `+30 - 25 = +5`, well below `+20`). Per-template tuning (e.g., -30 to push round-trip's curated-name composition deeper Suppressed) adds calibration surface without empirical justification.

2. **Hoist `isSelfTypedBinaryOp` to core (default A) vs duplicate per-template (B) vs file-private cross-template (C).**
   - (a) **Hoist to `SwiftInferCore.SetAlgebraShape.isSelfTypedBinaryOp(_:)`** (this plan's default). Mirrors v1.13's `DirectionLabels` hoist precedent: template-agnostic helper with N > 1 consumers lives in `SwiftInferCore.<Namespace>.<helper>`.
   - (b) Duplicate the helper in each template's gate file. Three copies; deviates from v1.13 hoist precedent.
   - (c) Keep the helper `private` on `InversePairTemplate` extension; have round-trip + idempotence reach across files via cross-template helper functions. Doesn't fit the established pattern.

   **Default proposal: (a) hoist.** Cleanest factoring; consistent with v1.13 + v1.14 + v1.15 namespace-from-cycle-1 ergonomics.

3. **Whether to add a dedicated `Signal.Kind.setAlgebraShapeVeto` case.** Default proposal: **(a) reuse `Signal.Kind.protocolCoveredProperty`** (matches V1.14.1's posture). Mirrors v1.14 open decision #4.

4. **File split: 2 new files vs inline.** Default proposal: **2 new files** (`RoundTripSetAlgebraShapeGate.swift` + `IdempotenceSetAlgebraShapeGate.swift`) mirroring V1.14.1's file-split posture. Keeps each calibration mechanism in a self-contained file.

5. **Single-cycle delivery.** v1.16 ships round-trip + idempotence in a single commit (V1.16.1) — V1.14.1 already covered inverse-pair, so v1.16 is a 2-template extension, not a 3-template introduction. 6. **Whether to commit cycle-13 decisions data.** Default: **yes** — `docs/calibration-cycle-13-data/` mirroring cycles 7-9 + 11-12.

7. **Cycle numbering: v1.16 = cycle 13.** Contiguous with v1.15 = cycle 12, v1.14 = cycle 11; v1.13's hoist refactor was cycle 10 (no-op surface delta).

8. **Whether to amend the cycle-12 findings table during v1.16.** v1.15.0 is already tagged (the doc snapshot at the tag is frozen); main can be amended. ## v1.16 trajectory framing

Cycle 11 (v1.14) introduced function-name + type-shape composite mechanism on inverse-pair.
Cycle 12 (v1.15) introduced parameter-label semantic-intent counter on three templates simultaneously (compression).
