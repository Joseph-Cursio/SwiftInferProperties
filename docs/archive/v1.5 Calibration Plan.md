# v1.5 Calibration Plan

Working doc for the SwiftInferProperties v1.5 cut. v1.5 is the **second calibration cycle** — closes cycle-2 priority #1 from `docs/calibration-cycle-1-findings.md` (protocol-conformance-driven property suppression) and runs the §17.3 loop a second time against the cycle-1 corpora. Mirrors the v1.4 plan's V1.4.0–V1.4.8 cadence with two parallel work streams: (a) code work for the `protocolCoveredProperty` mechanism, (b) empirical work re-running discover on the cycle-1 corpora to measure suppression delta.

> **v1.5 is a calibration cut, not a feature cut.** It doesn't extend PRD §7.8 — it adds a single counter-signal driven by a curated `protocolName → [KnownProperty]` map, gating the algebraic templates (identity-element / commutativity / associativity / inverse-pair / round-trip) so suggestions whose property is already covered by the candidate type's existing conformance are suppressed. The mechanism is a strict generalization of cycle 1's framing of "operator-aware identity-element pairing" (`docs/calibration-cycle-1-findings.md`'s headline cycle-2 priority): operator-aware pairing falls out for free because `Numeric`'s covered-properties set contains `.additiveIdentityZero` but not `.multiplicativeIdentityOne`, so `.zero × *` is no longer mismatched-by-curation but covered-by-protocol.

**Ephemeral** — archive to `docs/archive/v1.5 Calibration Plan.md` once the tag is pushed and the §13 + §16 release-blocking criteria still hold.

## What v1.5 ships

Three deliverables:

1. **`ProtocolCoverageMap` curated table + `protocolCoveredProperty` veto.** New `Sources/SwiftInferCore/ProtocolCoverageMap.swift` (mirrors the `FloatingPointStorageNames.swift` curated-table pattern) holds:
   - A new `KnownProperty` enum enumerating the property surfaces SwiftInfer's templates can emit (`additiveAssociative`, `additiveCommutative`, `additiveIdentityZero`, `additiveInverse`, `multiplicativeAssociative`, `multiplicativeCommutative`, `multiplicativeIdentityOne`, `multiplicativeInverse`, `distributivity`, `setUnionAssociative`, `setUnionCommutative`, `setUnionEmptyIdentity`, `setIntersectionIdempotent`, `equatableReflexive`, `equatableSymmetric`, `equatableTransitive`, `comparableTotalOrder`, `hashableConsistency`, `codableRoundTrip`, `monoidIdentity`, `groupInverse`, `semilatticeIdempotence`).
   - A `protocolCoverage: [String: Set<KnownProperty>]` table covering the 14 stdlib + kit protocols whose laws PropertyLawKit publishes (`Equatable` / `Comparable` / `Hashable` / `AdditiveArithmetic` / `Numeric` / `SignedNumeric` / `SetAlgebra` / `Codable` / kit `Semigroup` / `Monoid` / `CommutativeMonoid` / `Group` / `Semilattice`). Transitive coverage is baked in (`Numeric ⊇ AdditiveArithmetic`'s set, `SignedNumeric ⊇ Numeric`'s set, etc.) so callers don't need to walk inheritance chains.
   - A new `Signal.Kind.protocolCoveredProperty` case using `vetoWeight` (mirrors `nonDeterministicBody` / `nonEquatableOutput` posture). Vetoed suggestions stay in the `Decisions` calibration record per the existing `Score` short-circuit so cycle-3 metrics can introspect "how many were suppressed by which conformance."

2. **Wire the veto through five templates.** Each algebraic template gains a `protocolCoverageVeto(for:typeShapes:)` helper that:
   - Resolves the candidate's primary type (already on `IdempotenceTemplate.canonicalSignature` etc.).
   - Looks up the type's merged `inheritedTypes` (already merged via `TypeShapeBuilder`).
   - Asks the template "what `KnownProperty` would I emit?" — a static per-template constant for `idempotence` / `commutativity` / `associativity` / `inverse-pair` / `round-trip`; identity-element uses the curated constant's name + the op's name to pick between `additiveIdentityZero` / `multiplicativeIdentityOne` / `setUnionEmptyIdentity` / `monoidIdentity`.
   - Returns a veto-weight signal when any conformance in `inheritedTypes` covers the property, with detail `"Property already covered by conformance to '<protocol>' — checked by PropertyLawKit's check<Protocol>PropertyLaws"`.

3. **Cycle-2 calibration run + findings writeup.** Re-run `swift-infer discover --include-possible` against the four cycle-1 corpora at the v1.5 commit; capture post-rule snapshots to `docs/calibration-cycle-2-data/`. Compare suppression deltas vs the cycle-1 post-tune baseline (358 total `--include-possible` hits across the four corpora). Findings doc at `docs/calibration-cycle-2-findings.md` documents per-corpus suppression counts + acceptance-rate movement on the surviving surface + cycle-3 hypotheses.

PRD §19 sets the long-term acceptance-rate target at ≥ 70% after 6 months of dogfooding; v1.5 is cycle 2 of that loop.

### v1.5-only mechanical changes the cycle didn't cover

- **`KnownProperty` lives in `SwiftInferCore`, not `SwiftInferTemplates`.** The catalog is engine-wide, not template-local — placing it next to `Signal.Kind` keeps the curated property-name list and the signal-emitting templates in lockstep when new arms ship (e.g., the cycle-2-deferred `KitFloatingPointTemplate` will emit `.codableRoundTrip`-shaped properties).
- **`Decisions` carries the protocol-coverage veto reason.** Existing `DecisionRecord` already records suppression cause via the `Score`'s vetoed signal list; no schema change needed. The cycle-3 metrics command will be able to show "X suggestions suppressed by `: AdditiveArithmetic`" without new fields.
- **Transitive coverage is hand-coded into the table values, not computed.** Each protocol's `Set<KnownProperty>` already includes its parent's properties (e.g., `Numeric` includes `additiveAssociative` from `AdditiveArithmetic`). Computing transitivity at lookup time would require modelling Swift's protocol inheritance graph, which is a v1.1 constraint-engine concern (PRD §20.2). Hand-coded values cost ~14 × 5 ≈ 70 lookup-table entries; the savings are zero textual-conformance-walk logic.

## v1.5 acceptance bar

Mirroring PRD v1.0 §11 + §17.3 + §19 + the v1.x release plans' acceptance bars, v1.5 is not done until:

a. **All §13 perf budgets still pass at the v1.5 commit.** Rows 1–5 hold. Per-suggestion table-lookup cost is O(|inheritedTypes|) ≈ 1–4; suppressed suggestions skip downstream allocation. (V1.5.5 confirms.)

b. **All §16 hard guarantees still pass.** The veto reads only `TypeDecl.inheritedTypes` (already populated) + the new static `protocolCoverage` table. No new file I/O, no new networking. (Implicit; not a separate sub.)

c. **§14 privacy guarantee still holds.** Curated table is in-source; no telemetry. (Implicit.)

d. **`ProtocolCoverageMap` + `KnownProperty` + `Signal.Kind.protocolCoveredProperty` exist.** Catalog ships at V1.5.1 with the documented protocols + transitive-coverage values + lookup helper.

e. **All five algebraic templates emit the veto.** `idempotence` / `commutativity` / `associativity` / `inverse-pair` / `identity-element` / `round-trip` all consult the map at suggest-time and suppress when covered. (V1.5.2.)

f. **Empirical effect measured on ≥ 3 of the 4 cycle-1 corpora.** Captured snapshots in `docs/calibration-cycle-2-data/`; per-template + per-protocol suppression breakdown in the findings doc. (V1.5.3 + V1.5.4.)

g. **Cycle-2 findings writeup committed.** `docs/calibration-cycle-2-findings.md` documents the cycle and is referenced from the CLAUDE.md repo-state pointer table. (V1.5.4 + V1.5.7.)

h. **Performance baseline recorded for v1.5.** `docs/perf-baseline-v1.5.md` lists wall numbers + memory delta for every §13 row. (V1.5.5.)

i. **CLI version reports `1.5.0`.** `swift-infer --version` no longer emits `1.4.0`. (V1.5.6.)

j. **`CHANGELOG.md` gains a `## [1.5.0]` entry.** Summary covers the protocol-coverage rule + cycle-2 run + perf re-baseline. (V1.5.6.)

## Out of scope

- **Cycle-2 priority #2 — approximate-equality template arm for FP types.** A real `KitFloatingPointTemplate` emitting `checkFloatingPointPropertyLaws(for: T.self, using: gen)` stubs is sequenced as a v1.6 candidate. Synergy with v1.5: once protocol-coverage suppresses Numeric/AdditiveArithmetic-covered suggestions on FP types, the FP-conforming-but-not-AdditiveArithmetic-covered surface (e.g., trigonometric identities, transcendental compositions) is the natural target for that template arm.
- **Cycle-2 priority #3 — Possible-tier sampling on the post-rule surface.** Empirical-only follow-up. Best run after V1.5.3 captures the post-rule baseline so triage targets are stable.
- **Cycle-2 priority #4 — `surfacedAt` plumbing.** Independent of the suppression rule. Unblocks PRD §17.2's time-to-adoption metric. Sequenced as a v1.6 mechanical addition.
- **RefactorBridge complement (proposing conformances when structural evidence + missing conformance).** The protocol-coverage table is symmetric — it could feed both directions. v1.5 ships only the suppression direction; the RefactorBridge integration is a natural follow-on but its UX (bidirectional suggestions) needs scoping.
- **Conditional conformance.** `Array<T>: Equatable where T: Equatable` is not modelled. PRD §20.2 v1.1+ constraint engine concern.
- **Protocol inheritance graph beyond curated transitive values.** A user-defined protocol `MyAlgebra: Numeric` whose conformers don't textually list `Numeric` won't get `Numeric`'s coverage. Same v1 textual-only limitation as `EquatableResolver`'s `knownEquatableConformance`.
- **Module-qualified protocol names.** `: Swift.Numeric` won't match the bare `"Numeric"` table key. Documented limitation; cycle 3 may add a normalization step that strips known module prefixes.
- **Custom kit conformances declared in the corpus.** A library that conforms its types to PropertyLawKit's `Monoid` / `Group` / `Semilattice` will get the coverage automatically (those keys are in the table). Custom user-defined algebraic protocols (not in the kit) need the user to extend the table — not in scope.
- **The cycle-2-deferred Possible-tier sampling triage round.** That's empirical work — not code work — and the post-V1.5.3 surface is the right target for it. v1.6 candidate.
- **Telemetry / hosted aggregation.** PRD §14 + §19 + §17 are explicit: aggregation is local, no telemetry. v1.5 preserves this.
- **PRD §20 v1.1+ trajectory items.** SemanticIndex, IDE integration, `swift-infer apply` stay deferred. Cycle-2 closes priority #1 only.
- **Kit-side coordination.** `Package.swift` stays at `from: "2.0.0"`. The kit's `check<Protocol>PropertyLaws` surface is read-only — v1.5 references existing kit functions in advisory text; the kit itself doesn't need a new release.

## Open decisions

1. **Single-priority release vs bundled.** Two choices: (a) v1.5 ships cycle-2 priority #1 only (this plan); (b) bundle priorities #1 + #2 (the FP approximate-equality template arm) into v1.5. Default proposal: **(a) priority #1 only.** Rationale: cycle 1's pattern was "one rule per release," which kept the empirical effect attributable. Bundling #2 ~doubles the surface to triage in V1.5.3, blurring the per-rule effect measurement. Priority #2 is its own well-scoped v1.6 plan.

2. **Where the `KnownProperty` enum lives.** Two choices: (a) `Sources/SwiftInferCore/ProtocolCoverageMap.swift` alongside the table; (b) `Sources/SwiftInferCore/KnownProperty.swift` standalone. Default proposal: **(a) co-located.** Rationale: the enum exists to be looked up against the table; splitting them adds an import hop without adding clarity. Mirrors `FloatingPointStorageNames`'s self-contained file.

3. **Veto vs counter-signal.** Two choices: (a) full veto (`Signal.vetoWeight`) when covered — collapses to suppressed; (b) heavy counter-signal (-25, similar to `crossTypeRoundTripPair`) — drops Score 30 → 5 = Suppressed, but still scores. Default proposal: **(a) full veto.** Rationale: `crossTypeRoundTripPair` used -25 because the rule is approximation-via-textual-`containingTypeName` (SemanticIndex-blocked). Protocol coverage is authoritative when it matches — the kit's `check<Protocol>PropertyLaws` *does* verify the property, so the suggestion is genuinely redundant, not "probably noise." Full veto matches the certainty.

4. **Whether round-trip should consult the map.** Two choices: (a) yes — `Codable` conformance covers `codableRoundTrip` per PropertyLawKit's `checkCodablePropertyLaws`; (b) no — round-trip is structurally distinct (cross-function pairing on test patterns, not single-type laws). Default proposal: **(a) yes, but only for `Codable`.** Rationale: the cycle-1 cross-type counter-signal already reduced round-trip Possible 990 → 181; the remaining 181 surface includes signature-only same-type round-trips that may overlap with `Codable`-covered round-trips. Suppressing the Codable-covered subset is consistent with the rule's spirit. Same-type non-Codable round-trips (e.g., custom encode/decode) stay surfaced.

5. **Whether idempotence participates.** Two choices: (a) yes, mapped to `setIntersectionIdempotent` (covered by `SetAlgebra`) + `semilatticeIdempotence` (covered by kit `Semilattice`); (b) no, idempotence is too generic — `f(f(x)) == f(x)` doesn't have a one-to-one protocol-law mapping outside set-shaped types. Default proposal: **(a) yes for set-shaped + semilattice ops only.** Rationale: idempotence on a `: SetAlgebra` type's `intersection` / `union` is genuinely covered by `checkSetAlgebraPropertyLaws`; idempotence on arbitrary functions (`normalize`, `simplify`) is template-only. The veto fires only when the type and op match a protocol-covered idempotence law.

6. **How to expose the corpus-wide `[String: TypeDecl]` map to templates.** Two choices: (a) add `typeShapes: [String: TypeDecl] = [:]` parameter to each template's `suggest(for:)` call (mirrors `opsWithIdentitySeed` / `reducerOps`); (b) introduce a `TemplateContext` struct carrying all the corpus-wide signals, refactor every template's signature to take it. Default proposal: **(a) parameter-by-parameter.** Rationale: cycle 1 added two such parameters (`opsWithIdentitySeed`, `reducerOps`) using the same posture; cycle 2 adds one more. The `TemplateContext` refactor is better motivated by a third-or-fourth corpus-wide signal addition, when the parameter list is genuinely unwieldy. Currently 3 parameters across 5 templates is fine.

7. **Whether to commit cycle-2 decisions data.** Two choices: (a) commit `docs/calibration-cycle-2-data/` mirroring cycle-1; (b) keep cycle-2 data in working copies only (suppression-only run; no new triage decisions). Default proposal: **(a) commit them.** Rationale: cycle-2 captures the *post-rule surface* (not new accept/reject decisions — that's cycle 3's work). The surface diff is the empirical record. Files are small.

8. **Re-run cadence after the rule lands.** Two choices: (a) capture only post-rule snapshots (cycle 1 had pre-tune + post-tune; cycle 2 has post-tune-of-cycle-1 + post-rule-of-cycle-2); (b) re-capture pre-tune-of-cycle-2 baselines too. Default proposal: **(a) post-rule only.** Rationale: cycle-1's post-tune snapshots are already committed and serve as the cycle-2 pre-rule baseline. Re-running pre-tune would just re-capture the same numbers.

9. **Whether the metrics command should call out protocol-coverage suppressions.** Two choices: (a) extend `swift-infer metrics`'s table to add a "suppressed-by-protocol-coverage" column; (b) leave the existing template-suppressed aggregate column unchanged. Default proposal: **(b) unchanged.** Rationale: the per-veto-reason breakdown is more valuable in the findings doc (one-time analysis) than in the recurring metrics report. Cycle 3 may revisit if protocol-coverage suppressions become the dominant suppression class.

## v1.5 trajectory framing

PRD v1.0 §17.3 names calibration as "the operationalization of high precision, low recall — without it, the philosophy is just a slogan." v1.5 is cycle 2 of that loop. After v1.5 ships:

- The protocol-coverage map is in-tree, so cycle 3 can extend the curated table without a refactor (add a key + property set).
- The cycle-2 post-rule snapshots in `docs/calibration-cycle-2-data/` are the diff target for cycle 3's perturbation analysis.
- The cycle-2 findings doc captures one-time observations (per-protocol suppression breakdown, operator-aware-pairing-as-fallout demonstration) that future cycles can build on.

The natural cadence stays at one cycle per release. Cycle 3 (~v1.6) likely picks up cycle-2 priority #2 (FP approximate-equality template arm) + #4 (`surfacedAt` plumbing) as paired mechanical additions, plus the empirical Possible-tier sampling triage that priority #3 names. After two more cycles, the §19 acceptance-rate target should be measurable.

After the calibration loop reaches steady state, the natural next contributions return to the §20 v1.1+ trajectory:

- **PRD §20 SemanticIndex** — biggest v1.x lift. The protocol-coverage map is the textual approximation of one slice of what SemanticIndex would deliver authoritatively (via `lookupConformance`); cycle-3+ may re-frame the table as a `SemanticIndex` consumer once that engine ships.
- **PRD §20 IDE integration** — depends on SemanticIndex.
- **PRD §20 `swift-infer apply`** — moves from "all output is opt-in via `--interactive`" to "single command applies a baseline set of accepted suggestions."
- **Multi-predicate equivalence classes** (M13 axis 3) — SemanticIndex-blocked.
- **Cross-target enum coverage** (M14 deferred bit) — SemanticIndex-blocked.
- **Cross-test data-flow correlation** (M10 + M16 deferred) — natural sequencing post-SemanticIndex.

No active milestone plan opens with the v1.5 commit. Subsequent work picks one of the above and opens a fresh `docs/<plan>.md`.
