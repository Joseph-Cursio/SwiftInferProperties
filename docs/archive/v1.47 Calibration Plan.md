# v1.47 Calibration Plan — DerivationStrategist verify-time integration (broad scope)

Working doc for the v1.47 cut (forty-fourth calibration cycle). **Third cycle of the Phase 1.5 verifiable-fraction expansion arc** and the **first carrier-arm expansion** after v1.45 + v1.46 closed the four-template arc (round-trip / idempotence / commutativity / associativity).

> **Motivation.** V1.46.E's cycle-43 measurement put the verifiable-fraction at 10/32 = 31.3% with **4/8 = 50% of cycle-27 REJECTs verify-confirmable** — but the remaining 22 out-of-scope picks are blocked on **carriers**, not templates. The dominant blocker (3 picks: #7 `endOfChunk`, #8 `startOfChunk`, #9 `sizeOfChunk`) is `Base.Index` — a generic associated type on `ChunkedByCollection<Base>` that the hardcoded `CarrierKind` enum can't admit. v1.47 lifts the carrier gate by **plumbing SwiftPropertyLaws' `DerivationStrategist` into the verify pipeline**, opening the carrier set to any type the strategist can derive a generator for, **plus** a curated `GenericBindingResolver` that pins canonical concrete instantiations for generic associated types (per the user's broad-scope decision). Cycle-44 projects verifiable-fraction **31.3% → 47.0%** with **+3 verify-confirmed REJECTs** (#7–#9).

## Cycle-43 findings carry-forward — design decisions confirmed

Two load-bearing scope decisions ratified before this plan:

1. **Generic associated types: pin canonical instantiation.** For carriers like `Base.Index` where the type alone is unresolvable, v1.47 introduces `GenericBindingResolver` — a curated table mapping generic associated-type carriers to canonical concrete bindings (`ChunkedByCollection.Base → Array<Int>` → `Base.Index = Int`). The verifier rewrites the carrier name to its bound form before consulting the strategist. Alternatives considered (skip generic associated types, or add a `SuggestedCarrierBinding` discover-side hint) were rejected — the curated pinning is the smallest broad-scope step that unblocks #7–#9.

2. **`Complex<Double>` keeps its v1.46 hardcoded path.** The 12-entry curated edge-case list (V1.43.C) + `Gen<Complex<Double>>.edgeCaseBiased()` + `PropertyLawComplex` import are deliberately *outside* the strategist — they encode FP-edge intelligence the strategist's raw-type list doesn't. v1.47 keeps the two-arm dispatch: `Complex<Double>` → v1.46 hardcoded; everything else → strategist. Unification (folding Complex into a curated strategist case) is a v1.48+ cleanup target.

## What v1.47 ships

Nine workstreams + bundled closeout. Bigger than v1.46's six because the SemanticIndex schema bump touches the discover side too.

1. **V1.47.A — `IndexedTypeShape` mirror in `SwiftInferCore`** (mirror plan re-instated). Compiler-level constraint: Swift's `Codable` synthesis doesn't cross module boundaries, so adding `Codable` to the kit's `TypeShape` via extension would require hand-written `init(from:)` + `encode(to:)` *plus* hand-written conformances on `TypeShape.Kind` and `StoredMember` *plus* `@retroactive` suppressions on each — strictly more code than a mirror value type. v1.47 ships a mirror value type in `Sources/SwiftInferCore/IndexedTypeShape.swift`: fields parallel the kit's `TypeShape` (`name`, `kind`, `inheritedTypes`, `hasUserGen`, `storedMembers`, `hasUserInit`). `IndexedTypeShape` is `Codable + Sendable + Equatable`. Converters live alongside the type (`init(from kitShape: PropertyLawCore.TypeShape)` + `toKitShape() -> PropertyLawCore.TypeShape`); SwiftInferCore already imports `PropertyLawCore` so the cross-type conversions are local. The mirror's *secondary* benefit (decoupling persisted format from kit-version evolution) is a bonus, not the load-bearing rationale.

2. **V1.47.B — `SemanticIndexEntry.typeShape: IndexedTypeShape?` field**. Backward-compatible schema bump — `IndexedTypeShape?` defaults to `nil` for v1.46-and-earlier index entries (which fall back to the v1.46 `CarrierKind` hardcoded path at verify time). JSON round-trip: `typeShape` is omitted when `nil`. The `SemanticIndexFormat` version in the JSON header bumps `2 → 3` (existing v2 readers tolerate the unknown field per the format's forward-compatibility clause).

3. **V1.47.C — Discover-side `typeShape` population**. `IndexCommand.buildEntry` (or wherever entries are constructed) populates `typeShape` for entries whose carrier is a concrete type visible during indexing. Mechanism: when constructing a `Suggestion`, if the carrier is a struct/class/enum/actor declared in the indexed source, parse its `MemberBlockSyntax` (already available via SwiftSyntax) to populate `storedMembers` and `inheritedTypes`. For carriers that are stdlib raw types (Int, String, Double, etc.) or `Complex<Double>`, leave `typeShape == nil` — the verify harness recognises these by name and handles them via the v1.46 hardcoded path.

4. **V1.47.D — `GenericBindingResolver`**. Curated table mapping generic associated-type carriers to canonical concrete bindings. Initial table (5 entries — covers cycle-27 + the immediate follow-ons):

   | Carrier name | Canonical binding | Resolved carrier |
   |---|---|---|
   | `Base.Index` (Chunked*) | `Base = Array<Int>` | `Int` |
   | `Base.Element` (Chunked*) | `Base = Array<Int>` | `Int` |
   | `Self.Index` | `Self = Array<Int>` | `Int` |
   | `Self.Element` | `Self = Array<Int>` | `Int` |
   | `Iterator.Element` | `Iterator = Array<Int>.Iterator` | `Int` |

   `resolve(_:) -> String?` returns the bound concrete carrier name when the input matches a curated key, else `nil`. The verify harness calls `resolve` *before* consulting the strategist — if `resolve` returns a non-nil bound carrier, that's what the strategist sees.

5. **V1.47.E — `StrategistDispatchEmitter`**. New emitter that consumes `(carrier: String, typeShape: IndexedTypeShape?, template: String)` → invokes `DerivationStrategist.strategy(for:)` → emits per-strategy generator code. The dispatch matrix is now **template × strategy** (not template × carrier):

   - `.rawRepresentable(.int)` → emit `Gen<Int>.int(in: ...)` (and per-template variation: 1 / 2 / 3 values per trial).
   - `.rawRepresentable(.double)` → emit `Double.random(in: -1e6...1e6)` for the default pass + inlined `doubleWithNaN` for the edge pass (existing v1.46 path).
   - `.rawRepresentable(.string)` → emit `Gen<String>.letterOrNumber.string(of: 0...8)`.
   - `.caseIterable` → emit `Gen<T>.element(of: T.allCases)`.
   - `.memberwiseArbitrary(members:)` → emit the per-member `zip` composition the kit's macros would. Capped at 10 members per `memberwiseArityLimit`.
   - `.userGen` → emit `T.gen()` (assumes the user's type declares `gen()`).
   - `.todo(reason:)` → fall back to v1.46 `CarrierKind` dispatch if the carrier name is in the v1.46 set; otherwise emit a verify-mode `.error(reason: "no derivation strategy for <T>: <reason>")`.

   The emitter is template-aware — round-trip emits 1-value per trial, commutativity emits 2-value per trial, associativity emits 3-value per trial — so per-template wrapping logic lives in the emitter (the strategy enum just yields the per-value generator expression).

   **File split**: `StrategistDispatchEmitter.swift` (orchestration + strategy dispatch) + `StrategistDispatchEmitter+RawTypes.swift` (the 13-entry `RawType → generator expression` mapping). Both should land under 280 lines.

6. **V1.47.F — Verify harness integration**. `VerifyCommand+TemplateDispatch.swift` adds a two-arm carrier router (per the §"design decisions confirmed" #2 ratification):

   ```
   carrier = entry.typeName ?? "(none)"
   carrier = GenericBindingResolver.resolve(carrier) ?? carrier
   if carrier == "Complex<Double>":
       use v1.46 hardcoded path (per-template-emitter)
   elif entry.typeShape != nil:
       use StrategistDispatchEmitter(carrier, typeShape, template)
   else:
       use v1.46 CarrierKind fallback (Double, Int) or fail
   ```

   The v1.46 emitters (`RoundTripStubEmitter`, `IdempotenceStubEmitter`, `CommutativityStubEmitter`, `AssociativityStubEmitter`) keep their existing 3-carrier dispatch for the fallback path. The new strategist path is the **primary** path post-v1.47; the v1.46 path is the **fallback**.

## Why broad scope over tight scope

Three reasons in order of weight:

1. **One architectural cycle vs three template-extension cycles.** Tight scope (one carrier — `Base.Index`) closes 3 cycle-27 picks. Broad scope closes the same 3 picks *plus* opens the door to any future carrier the strategist + binding resolver can resolve. v1.48+ template-arm work (idempotence-lifted, monotonicity, dual-style-consistency) operates over whatever carriers v1.47 unlocks; the broader the v1.47 carrier surface, the bigger the v1.48+ lift.

2. **SemanticIndex schema bump is incremental.** Adding `typeShape: IndexedTypeShape?` is backward-compatible at every level — JSON decoding (nil-default), API consumers (typeShape is optional), and verify-time fallback (v1.46 CarrierKind path remains). The v1.34 cycle's `carrier: String?` field was the precedent: also a backward-compatible optional, also threaded through 16 construction sites, also a non-breaking deploy.

3. **`DerivationStrategist` already exists; v1.47 only routes to it.** No kit-side change required. The strategist is the kit's canonical generator-derivation logic (already used on the discover side at `GeneratorSelection.swift:42`); v1.47 just connects the verify side to the same source-of-truth.

## Scope boundaries

- **In scope (v1.47):** `IndexedTypeShape` value type + JSON serialization; `SemanticIndexEntry.typeShape` field; discover-side population; `GenericBindingResolver` (5 curated entries); `StrategistDispatchEmitter` covering all 5 strategy cases (plus `.todo` fallback); verify harness two-arm router; cycle-44 calibration measurement.
- **Out of scope, deferred to v1.48+:** Folding `Complex<Double>` into the strategist (current: two-arm dispatch); folding `Float` / `Float80` / `CGFloat` carriers into the verify pipeline; user-domain types beyond the strategist's reach (i.e. types without a derivable generator and without a `gen()` declaration); accept-flow integration (verify outcomes → `decisions.json`); verification cache; "Verified" first-class tier; expanding the `GenericBindingResolver` table beyond cycle-27 + immediate follow-ons.
- **Not changing:** Any v1.42–v1.46 verify pipeline behavior for `Complex<Double>` × all 4 templates, or `Double` / `Int` × all 4 templates when entry's `typeShape == nil` (the v1.46 CarrierKind fallback). Existing tests stay green.

## Risks

1. **`IndexedTypeShape` ↔ kit's `TypeShape` drift.** If the kit ships a new `TypeShape` variant (e.g., adds a `protocol` `Kind`), v1.47's `IndexedTypeShape` mirror won't carry it across the JSON boundary. **Mitigation**: the converter (`init(from kitShape:)`) is the single coupling point — if a kit-side variant is unrepresentable in `IndexedTypeShape`, the converter falls back to `nil` (treat as "no typeShape information"; verify-time dispatch falls back to v1.46 `CarrierKind`). Add a one-line CI lint that flags new kit-side variants without an `IndexedTypeShape` mirror.

2. **`StrategistDispatchEmitter` × FP-edge gap.** The strategist's `.rawRepresentable(.double)` strategy emits a finite-domain `Double` generator with no inlined `doubleWithNaN` edge pass — v1.46's `IdempotenceStubEmitter+NonComplex.swift` does that work outside the strategist. v1.47's strategist path for `Double` carrier would therefore emit a *less rigorous* verify-time check than v1.46's hardcoded path. **Mitigation**: the verify harness detects `carrier == "Double"` and routes to the v1.46 path (same as the Complex<Double> special-case). Effectively v1.47 introduces the strategist for *new* carriers (Int via name-only path, String, plus the generic-bound + memberwise cases); the v1.46 hardcoded path stays in place for `{Complex<Double>, Double}`. **Sub-decision**: keep `Int` on the v1.46 path or move it to the strategist? Recommendation: move Int to the strategist — the v1.46 Int generator (`±2^(bitWidth/4)`) is functionally equivalent to the strategist's `Gen<Int>.int(...)` output, and the zero-edge sentinel pattern doesn't depend on the carrier.

3. **GenericBindingResolver curated-table bias.** The 5 curated bindings all use `Array<Int>` as the canonical container — a deliberate choice for cycle-27 alignment (`Array<Int>.Index == Int` is the simplest non-trivial Index type). But `ChunkedByCollection`'s actual chunking behavior may differ across `Base` types — a `String.Index` or `Set<Int>.Index` could surface chunking-specific bugs that `Array<Int>.Index` doesn't. **Mitigation**: cycle-44 records the chunked-Index outcomes; if `.bothPass` for all three (`endOfChunk` / `startOfChunk` / `sizeOfChunk`) under `Base = Array<Int>`, the binding is sufficient for cycle-27. If the cycle-27 verdict on these picks is REJECT but the v1.47 verify says `.bothPass`, that's a verify-vs-heuristic disagreement worth investigating — could indicate the heuristic was too conservative *or* the binding doesn't surface the bug.

4. **Cycle-44 verifiable-fraction may plateau at the pessimistic end.** Projection band 40.6%–46.9%. If `_Bucket` resolves to `.todo` (wraps a raw pointer or has a non-derivable member), and `[NearMiss]` is similarly unreachable, v1.47 lands at 40.6% — still meaningful (+9.3pp; +3 verify-confirmed REJECTs) but below the cycle-43 projection of 47.0%. **Mitigation**: cycle-44 findings doc records the actual outcomes for #10 + #11; a `_Bucket`-only blocker would surface as a v1.48 open thread (potentially expanding the GenericBindingResolver to bucket-specific bindings).

5. **Test wall-clock impact**. v1.47 adds ~50 unit tests + ~4 integration tests. Integration tests each cost ~10–15s (SwiftPM workdir build); 4 new ones push the full `swift test` wall-clock from ~160s (v1.46) to **~200–220s**. **Mitigation**: G.6.a–d cover orthogonal dispatch paths; bundling them into a single test that runs all four in one subprocess workdir would save ~30s. Recommendation: keep them separate for cycle-44's load-bearing prediction matrix (each integration test pins a specific strategy outcome).

6. **`memberwiseArbitrary` 10-member arity cap.** The strategist caps memberwise derivation at 10 stored members (kit's `memberwiseArityLimit`). Cycle-27's `_Bucket` and `[NearMiss]` are both narrow types (likely <5 members each) so this isn't a v1.47 blocker, but a wider user-domain type (e.g., a 20-field domain struct) would hit the cap. **Mitigation**: cycle-44 findings doc records any encountered arity-cap rejections as v1.48+ open threads; no v1.47 action.
