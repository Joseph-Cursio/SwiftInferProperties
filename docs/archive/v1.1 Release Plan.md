# v1.1 Release Plan

Working doc for the SwiftInferProperties v1.1 cut. With TestLifter M10 (round-trip-pair domain narrowing) and TestLifter M11 (predicate equivalence-class detection) closed on `main`, the §7.8 expanded-outputs row is fully closed (M9 preconditions + M10 inferred domains + M11 equivalence classes). v1.1 is the version cut that captures both milestones; the underlying work is already shipped, so this plan is mechanical (perf re-baseline + version + CHANGELOG + archive). **Ephemeral** — archive to `docs/archive/v1.1 Release Plan.md` once the tag is pushed and the §13 + §16 release-blocking criteria still hold.

> **v1.1 is a feature cut, not a posture cut.** v0.1.0 closed the release-blocking guarantees and tagged the M1–M8 + TestLifter M1 surface; v1.1 adds two TestLifter expanded-output features (§7.8 examples 2 + 3) that ship with the same hard-guarantee + perf-budget posture. The §16 hard guarantees are unchanged; the §13 numbers are re-verified rather than recalibrated.

## What v1.1 ships (PRD v1.0 §7.8 + §7.9 rows M10 + M11)

The v1.1 line was opened by M10 and closed by M11. Both shipped with their own M-series acceptance bars; v1.1 is the version that surfaces them to consumers.

### TestLifter M10 — Inferred domains (round-trip-pair scope)

PRD v1.0 §7.8 second example. Round-trip suggestions whose reverse-side test corpus uniformly receives forward-side output get a `DomainHint` that overrides the generator with `Gen<T>.map(forward)` plus a `// Inferred domain:` provenance comment. Hard veto on throws / async / multi-arg / non-generatable producers (comment-only fallback names the veto reason).

Sub-milestones M10.0–M10.3 closed in `docs/archive/TestLifter M10 Plan.md`. The Option A general consumer-producer chain detection is explicitly deferred to a future v1.x.

### TestLifter M11 — Predicate equivalence-class detection

PRD v1.0 §7.8 third example. Test corpora that partition on `Valid`/`Invalid` method-name markers — with both buckets reaching the M4.3 ≥3 site threshold and homogeneous predicate + polarity — surface as advisory `equivalence-class` suggestions. Comment-only writeout to `Tests/Generated/SwiftInfer/equivalence-class/EquivalenceClasses_<predicate>.swift` on accept; suggested filter generators (`Gen<T>.gen().filter(predicate)`, `Gen<T>.gen().filter { !predicate($0) }`) or veto reason rendered in the documentation block.

Sub-milestones M11.0–M11.2 closed in `docs/archive/TestLifter M11 Plan.md`. The general partition surface (arbitrary marker labels, N-class, multi-predicate, cross-class relations) is explicitly deferred to a future v1.x.

### v1.1-only mechanical changes the milestones didn't cover

- `Tier.advisory` (TestLifter M11.0) is a new tier value rendered as `[Advisory]`. Distinct from `Strong` / `Likely` / `Possible` so consumers can tell documentation surfaces apart from runnable property suggestions. `init(score:)` never returns `.advisory` — the surfacing pipeline sets it explicitly via `Score(advisorySignals:)`.
- `AssertionInvocation.Kind.xctAssertFalse` (M11.0) is now first-class. The Slicer recognizes `XCTAssertFalse(...)` calls and routes them to the new kind; the M11.1 detector relies on this for the polarity-homogeneity check.
- The accept-flow renderer dispatches on `templateName == "equivalence-class"` ahead of the runnable-stub switch and writes a comment-only documentation block. The hint travels via `InteractiveTriage.Context.equivalenceClassHintsByIdentity` rather than inline on `Suggestion` (the §13 row 4 memory-ceiling regression test caught a 65MB delta when the hint was inlined as an optional struct field — Optional<EquivalenceClassHint> reserves the full ~240-byte struct inline even when nil, and the side-map shape avoids that per-Suggestion cost).

## Open decisions

1. **CHANGELOG entry granularity for the side-map carrier fix.** The §13 row 4 regression caught at M11.2 was a real bug in the first-pass design (inline `EquivalenceClassHint?` on `Suggestion`); the fix is the carrier-via-`Context` shape. Should the CHANGELOG mention this as a separate bullet or fold it into the M11 line? Default proposal: **(a) fold into M11 line as a parenthetical** ("M11 — predicate equivalence-class detection (hint travels via the `InteractiveTriage.Context` side map to keep the §13 row 4 memory budget)"). Rationale: the side-map shape is implementation detail; the consumer-facing surface is "M11 ships". The release-plan doc and archive plan record the discovery + fix for future contributors.

2. **Perf baseline diff against v0.1.0.** The v1.1 baseline file should either (a) replace the v0.1.0 baseline as the canonical regression anchor or (b) sit alongside as `docs/perf-baseline-v1.1.md` with v0.1.0 retained for historical comparison. Default proposal: **(b) keep both, document v1.1 as the canonical anchor**. The v0.1.0 baseline retains forensic value for understanding the M9/M10/M11 trajectory's perf delta.

3. **Whether to pin the v1.1 tag at V1.1.3 or V1.1.4.** v0.1.0 pinned at R1.4 (the doc-archive commit) rather than R1.5 (the tag commit) — same posture here. **Default: V1.1.3.** Rationale: tagging the doc-archive commit means subsequent commits can land before the tag-push without rewriting tag history.

4. **Whether to bump the kit dep to a v2.x patch release.** SwiftPropertyLaws v2.0.0 is the current pin; no kit changes were needed for M10 or M11. **Default: stay at `from: "2.0.0"`.** Rationale: no v1.1 SwiftInfer change consumes anything not in v2.0.0; bumping for cosmetic reasons widens the test surface without buying anything.

## v1.1 trajectory framing

PRD v1.0 §20 ("v1.1+ trajectory") describes a list of post-v0.1.0 work items: TestLifter M10 (inferred domains), TestLifter M11 (equivalence-class detection), SemanticIndex, IDE integration, `swift-infer apply`, `swift-infer metrics`. Closing M10 + M11 closes the §7.8 row but leaves the §20 surface largely open — v1.1 doesn't promise the §20 items, only the §7.8 close-out.

After v1.1 ships, the natural next contributions are:
- **General consumer-producer chain detection** (M10's deferred Option A) — extends §7.8 example 2 beyond round-trip pairs. Not §20 per se; a v1.x narrow follow-up.
- **General partition surface** (M11's deferred Option A) — extends §7.8 example 3 beyond `Valid`/`Invalid`. Not §20 per se; a v1.x narrow follow-up.
- **PRD §20 SemanticIndex** — biggest v1.x lift. Was originally deferred from v1.0; the constraint-engine + index unlock the IDE integration + `apply`/`metrics` items downstream.
- **PRD §20 IDE integration** — depends on SemanticIndex.
- **PRD §20 `swift-infer apply`** — moves from "all output is opt-in via `--interactive`" to "single command applies a baseline set of accepted suggestions." Depends on a stable v1.1 baseline-decisions surface (which already exists at v0.1.0).
- **PRD §20 `swift-infer metrics`** — surfaces the `.swiftinfer/decisions.json` aggregation per PRD §17.

No active milestone plan opens with this commit. Subsequent work picks one of the above and opens a fresh `docs/<plan>.md`.
