# v1.43 Calibration Plan — Test-Execution-Evidence Phase 1, Step 2

Working doc for the v1.43 cut (fortieth calibration cycle). **Second of ~3 cycles** delivering the test-execution-evidence architectural shift (v1.42 shipped step 1 — the minimum-viable verify pipeline). v1.43 layers the **edge-case-biased second pass** on top of v1.42's single-pass round-trip verifier and expands the user-facing result to the four-outcome two-pass table from the upstream proposal at `docs/ideas/Edge-Case-Biased Generators Kit Proposal.md` §2.2.

> **Motivation.** v1.42 proved out the subprocess-based verify gesture: synthesize a SwiftPM workdir, build + run a candidate property test on N random inputs, surface pass / fail / error. The single default pass uses an inline finite-domain generator (`Double.random(in: -1e6 ... 1e6)`), which by construction never surfaces IEEE 754 boundary bugs (NaN, ±Inf, ±0, subnormals). The kit-side prerequisite — `PropertyLawComplex` shipping `Gen<Complex<Double>>.edgeCaseBiased()` with a curated 12-entry edge-case set — landed at `SwiftPropertyLaws v2.1.0` (commit `ba19ab7`, 2026-05-11) and is already pinned in `Package.swift` (V1.42.A). v1.43 wires it through.

## What v1.43 ships

Four workstreams + bundled closeout:

1. **V1.43.A — Verifier workdir dep wiring + `PropertyLawComplex` import**. Edit `VerifierWorkdir`'s synthesized `Package.swift` template so the generated verifier package declares `.product(name: "PropertyLawComplex", package: "SwiftPropertyLaws")` alongside the existing `PropertyLawKit` import. The user-side kit pin is already at `2.1.0` (V1.42.A); this workstream adds the *consumer-side* dep declaration in the throwaway workdir's `Package.swift`. Runner main gets `import PropertyLawComplex` so `Gen<Complex<Double>>.edgeCaseBiased()` and `Gen<Complex<Double>>.complexEdgeCases` resolve.

   **Sub-decision** — keep `import PropertyLawKit` too (the existing pass already uses `Xoshiro`/`Gen<T>` types from `PropertyLawCore`/`PropertyLawKit`); this is a *strict additive* import, not a replacement.

2. **V1.43.B — Two-pass stub synthesis in `RoundTripStubEmitter`**. The v1.42 stub runs one loop of `trialBudget` trials over `Double.random(in: -1e6...1e6)`-derived `Complex<Double>` values. v1.43 splits this into two consecutive labelled loops:

   - **Default pass**: `trialBudget` trials, generator unchanged from v1.42 (inline finite-domain — preserves the existing pass's behavior bit-for-bit so cycle-27-era expectations don't shift).
   - **Edge-case pass**: only run if default pass *passed*; `trialBudget` trials drawn from `Gen<Complex<Double>>.edgeCaseBiased()` (90/10 mix; ~10 edge-case trials at N=100, ~100 at N=1000 — enough to hit each of the 12 curated entries at least once at both budgets).

   Stdout markers extend from v1.42's `VERIFY_RESULT: PASS|FAIL` to a per-pass shape:

   ```
   VERIFY_DEFAULT_RESULT: PASS
   VERIFY_DEFAULT_TRIALS: 100
   VERIFY_EDGE_RESULT: PASS
   VERIFY_EDGE_TRIALS: 100
   ```

   …or on edge-case-only failure:

   ```
   VERIFY_DEFAULT_RESULT: PASS
   VERIFY_DEFAULT_TRIALS: 100
   VERIFY_EDGE_RESULT: FAIL
   VERIFY_EDGE_TRIAL: 7
   VERIFY_EDGE_INPUT: Complex(.nan, 0.0)
   VERIFY_EDGE_FORWARD: ...
   VERIFY_EDGE_INVERSE: ...
   VERIFY_EDGE_INDEX: 2          # 0-based index into complexEdgeCases
   ```

   …or on default-pass failure (short-circuit; edge pass not run):

   ```
   VERIFY_DEFAULT_RESULT: FAIL
   VERIFY_DEFAULT_TRIAL: 47
   VERIFY_DEFAULT_INPUT: Complex(0.0042, -1.7e6)
   VERIFY_DEFAULT_FORWARD: ...
   VERIFY_DEFAULT_INVERSE: ...
   ```

   Marker namespacing (`VERIFY_DEFAULT_*` / `VERIFY_EDGE_*`) lets the parser unambiguously separate the two passes without ordering assumptions, mirroring the v1.42 marker convention.

3. **V1.43.C — `VerifyOutcome` extension to 4-outcome table + parser rewrite**.

   Today's enum (`VerifyResult.swift:24-28`):

   ```swift
   public enum VerifyOutcome: Equatable, Sendable {
       case pass(trials: Int)
       case fail(trial: Int, input: String, forwardResult: String, inverseResult: String)
       case error(reason: String)
   }
   ```

   …becomes:

   ```swift
   public enum VerifyOutcome: Equatable, Sendable {
       case bothPass(defaultTrials: Int, edgeTrials: Int)
       case edgeCaseAdvisory(
           defaultTrials: Int,
           edgeTrial: Int,
           edgeInput: String,
           edgeForward: String,
           edgeInverse: String,
           edgeCaseIndex: Int          // index into complexEdgeCases; -1 if no match
       )
       case defaultFails(trial: Int, input: String, forwardResult: String, inverseResult: String)
       case error(reason: String)
   }
   ```

   The four outcomes map to the proposal §2.2 table:

   | Default | Edge | Outcome case | User-facing intent |
   |---|---|---|---|
   | Pass | Pass | `.bothPass` | Strong evidence — property holds across full domain |
   | Pass | Fail | `.edgeCaseAdvisory` | Holds for normal inputs, breaks at boundary — advisory with failing edge value + curated-set index |
   | Fail | — (skipped) | `.defaultFails` | Property is wrong; finite-pass counterexample |
   | (any error) | — | `.error` | Build/run error |

   `VerifyResultParser.parse` is rewritten end-to-end: read both `VERIFY_DEFAULT_RESULT` and `VERIFY_EDGE_RESULT` markers; cross-reference the exit code (0 for `.bothPass`, 1 for `.edgeCaseAdvisory` *or* `.defaultFails`, anything else → `.error`); populate the appropriate case. Reuses v1.42's `value(forMarker:in:)` helper.

   **Edge-case index resolution.** The runner emits `VERIFY_EDGE_INDEX` directly by tracking which path the sampler took. Implementation note: the kit's `edgeCaseBiased()` does **not** today expose a "which entry was selected on this trial" hook. The runner sidesteps this by *re-sampling* against `Gen<Complex<Double>>.complexEdgeCases` after a failure — match the failing input against each public entry by `==` (NaN-aware: use `isApproximatelyEqual` or `.bitPattern` comparison for NaN). If no match (i.e., the failing value came from the 90% default-path slice, not the 10% edge slice), index is `-1` and the renderer prints the value verbatim without a "edge case #N" tag. The 10/90 split makes this case rare for genuinely-edge-case-triggered failures.

4. **V1.43.D — `VerifyResultRenderer` 4-outcome rendering**. Extends the v1.42 three-outcome renderer (`VerifyResult.swift:82-124`):

   ```
   ✓ verify holds (strong): round-trip foo/bar over Complex<Double>,
       100 default trials + 100 edge-case-biased trials, all pass
       (12 / 12 curated edge cases sampled)

   ⚠ verify holds for finite domain; edge-case advisory:
       round-trip foo/bar over Complex<Double>, default pass 100/100,
       edge pass failed at trial 7 on edge case #3 (Complex(NaN, 0)):
       input  = Complex(.nan, 0.0)
       foo(input)  = ...
       bar(foo(input)) = ...
       expected ≈ input (within Complex<Double>.isApproximatelyEqual)

   ✗ verify fails: round-trip foo/bar over Complex<Double>,
       counterexample at trial 47 (default pass):
       input  = Complex(0.0042, -1.7e6)
       foo(input)  = ...
       bar(foo(input)) = ...
       expected ≈ input (within Complex<Double>.isApproximatelyEqual)

   ! verify error: <reason>
   ```

   The edge-case advisory format names the specific curated entry (`#3 (Complex(NaN, 0))`) when `edgeCaseIndex >= 0`; otherwise it omits the `#N (...)` clause and prints only the failing value. The curated-entry names mirror the 12-row table in `docs/ideas/Edge-Case-Biased Generators Kit Proposal.md` §3.2.

   The `(12 / 12 curated edge cases sampled)` line on the strong-pass output is **best-effort**: emitted by the runner using the same re-sampling approach as V1.43.C's index resolution, counting distinct hits over the edge-pass trial budget. Not load-bearing for v1.43 correctness — if the count is wrong by ±1 due to noise, the user-facing outcome is unchanged.

5. **V1.43.E — Tests**. Three areas:

   - **E.1** Unit tests of `VerifyResultParser` for the four-outcome shape — one test per case + two malformed-marker error cases. ~6 new tests.
   - **E.2** Unit tests of `VerifyResultRenderer` for the four-outcome shape — golden-string per case, including the edge-case-index-omitted variant. ~5 new tests.
   - **E.3** Integration tests (subprocess; `.integration` tag, mirroring V1.42.D's posture):
     - **E.3.a** `bothPass` end-to-end: identity round-trip (the simplest pair the v1.42 D.2 integration uses) — assert exit 0, `.bothPass` outcome, both trial counts populated.
     - **E.3.b** `edgeCaseAdvisory` end-to-end: a synthesized round-trip pair that holds for finite inputs but explicitly rejects NaN (e.g., `forward = { Complex($0.real * 2, $0.imag) }` paired with `reverse = { Complex($0.real / 2, $0.imag) }`, with a NaN guard that flips the sign of `imag`) — assert exit 1, `.edgeCaseAdvisory`, `edgeCaseIndex` correctly resolved.
     - **E.3.c** `defaultFails` end-to-end: the V1.42.D.3 asymmetric pair — assert exit 1, `.defaultFails`, edge pass not executed (no `VERIFY_EDGE_*` markers in stdout).

     Aggregate: 3 integration tests. The v1.42 known-bad / known-good integration tests (`VerifyPipelineTests.swift`) are updated to match the new outcome enum but retain their structural shape.

   Test-count goal: 2171 → ~2190 (+~19).

## Why short-circuit the edge pass on default failure

Three reasons, in order of weight:

1. **The proposal §2.2 table makes it explicit.** Row 3 reads "Property is wrong; skip edge pass; surface finite-pass counterexample." The default-fail outcome already proves the property doesn't hold; an additional edge pass adds no information — at best it confirms; at worst it produces a noisy second counterexample that distracts from the real one.
2. **Build-cost amortization.** Both passes share the same compiled binary, so the cost saved by short-circuiting is only the runtime of one trial loop (~ms at N=100). But the *cognitive cost* of a two-counterexample output is non-trivial — the user has to mentally diff them to decide which to act on.
3. **Determinism story stays simple.** With short-circuit, the four outcomes are mutually exclusive on output: any stdout has either `VERIFY_DEFAULT_RESULT: FAIL` (no edge markers) OR `VERIFY_DEFAULT_RESULT: PASS` (then `VERIFY_EDGE_RESULT: PASS | FAIL`). The parser doesn't need to handle "both passes failed" because it's by construction impossible.

The opposite design — always run both — is recoverable if v1.44 calibration finds the strong-vs-default-only signal worth surfacing on `.defaultFails` outcomes. For v1.43 the short-circuit is the conservative choice.

## Determinism caveat (from the kit-side resolution)

The kit's `edgeCaseBiased()` ships with **partial determinism** (`docs/ideas/Edge-Case-Biased Generators Kit Proposal.md` §8 departure note):

- The edge-vs-finite branch decision and which-edge-case-entry is selected are fully reproducible from the Xoshiro seed.
- The finite-Complex components on the 90% slice use non-seeded `Double.random(in:)`, matching the kit's existing `Gen<Double>.doubleWithNaN()` convention.

For v1.43, this means:

- **`.edgeCaseAdvisory` outcomes** (the high-value row) are fully reproducible across reruns at the same seed — the failing trial's input is always a curated edge case, and the index resolution is deterministic.
- **`.bothPass` outcomes** are reproducible at the pass/fail level (no advisory upgrades on rerun) but not bit-for-bit on the 90% finite values. Acceptable — the user-facing outcome is unchanged.
- **`.defaultFails` outcomes** inherit v1.42's `Double.random(in:)` non-determinism (already a known caveat from V1.42.C.2). Rerunning may produce a different failing trial index for the same seed, which is a v1.42 pre-existing issue and out of scope for v1.43.

A future cycle can swap in a fully-seeded finite-Complex generator on the kit side; for v1.43 the partial-determinism is documented and accepted.

- **Out of scope, deferred to v1.44:** Template extension (idempotence-lifted / commutativity / dual-style); carrier extension beyond `Complex<Double>`; Phase 1 calibration measurement of the verify-mode lift over cycle-27's 72.4% baseline.

- **Not changing:** Any v1.42-and-earlier suggestion ranking, scoring, surface composition, discover/drift/index/query/suggest-refactors subcommands. v1.43 strictly extends `swift-infer verify` output; the surface composition still reflects cycle-27's 72.4% measurement.

## Risks

1. **Edge-case-index resolution may misclassify.** The runner re-matches the failing value against `complexEdgeCases` after the fact. NaN equality (`Complex(.nan, .nan) == Complex(.nan, .nan)` is `false` by IEEE 754) needs a NaN-aware comparison — the runner uses `value.real.isNaN ? complexEdgeCases[i].real.isNaN : value.real == complexEdgeCases[i].real` (and same for `imag`). **Mitigation**: V1.43.E.3.b integration test pins a NaN edge-case and asserts the index resolves to entry #3 specifically. Mismatch surfaces as `-1`, which the renderer falls back to value-only rendering — degrades gracefully.

2. **`VerifyResultParser` rewrite is a non-trivial surface change.** V1.42.C.4 ships ~80 LOC of parser + renderer; v1.43 roughly doubles it. **Mitigation**: V1.43.E.1 + E.2 cover all four cases + malformed-marker edge cases. The v1.42 parser stays as inspiration for the per-marker `value(forMarker:in:)` helper; the new parser is a clean replacement (no `VerifyOutcome.pass(trials:)` callers — the enum case is renamed not extended).

3. **Build-time inflation from `PropertyLawComplex` pulling `swift-numerics` into the verifier workdir.** The user's package already depends on `swift-numerics` (otherwise their `Complex<Double>` round-trip wouldn't exist), so the resolved-package set doesn't change. **Mitigation**: V1.43.A's `Package.swift` template adds only the `PropertyLawComplex` product line — the existing `swift-numerics` dep is already declared. Re-resolution may take ~1–2s on the first cold run but caches per the V1.42.C.3 workdir layout.

4. **Edge-pass false positives from the 90% finite slice.** A failure in the edge pass may technically come from the 90% finite-path slice (the bias is 90/10, not 0/100). When this happens, `edgeCaseIndex` resolves to `-1` and the user sees "edge pass failed on Complex(...)" without a curated-entry tag. This is correct (the property *does* fail on this input), but the "advisory" framing implies an edge case. **Mitigation**: V1.43.D's renderer falls back to a non-advisory phrasing when `edgeCaseIndex == -1`: "edge pass failed at trial N on a non-curated value: ..." — leaves the interpretation to the user. v1.44 calibration measurement reports the frequency of this fallback path; if it's nonzero noise we can tighten the bias ratio (proposal §3.3).
