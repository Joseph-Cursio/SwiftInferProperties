# v1.4 Calibration Plan

Working doc for the SwiftInferProperties v1.4 cut. v1.4 is the **first calibration cycle** — operationalizes PRD §17.3's empirical-tuning loop and closes PRD §17.2's `swift-infer metrics` deferred subcommand. Mirrors the v1.x release-plan cadence (V1.4.0–V1.4.8) but with two parallel work streams: (a) code work for the metrics command, (b) empirical work for the dogfooding run + weight tuning.

> **v1.4 is a posture cut, not a feature cut for the inference engine.** v0.1.0 closed the release-blocking guarantees; v1.1 added the §7.8 trio first cut; v1.2 widened the third example; v1.3 closed the second example with M16. v1.4 doesn't extend §7.8 — instead it operationalizes the §17.3 calibration loop that PRD §17.3 names as "the operationalization of high precision, low recall — without it, the philosophy is just a slogan." The signal weights in §4.1 have been "tunable starting points" since v0.3 and have never been empirically validated against real corpora; v1.4 is the first cycle that does so.

**Ephemeral** — archive to `docs/archive/v1.4 Calibration Plan.md` once the tag is pushed and the §13 + §16 release-blocking criteria still hold.

## What v1.4 ships

Three deliverables:

1. **`swift-infer metrics` subcommand** (PRD §17.2). Closes the v1.1+ deferred surface listed in PRD §20. Reads one or more `.swiftinfer/decisions.json` files, aggregates per-template acceptance / rejection / suppression rates + per-signal weight statistics, renders a tabular report to stdout. Three of PRD §17.2's five metrics are shippable from the existing `DecisionRecord` shape (acceptance rate, rejection rate as the false-positive proxy, suppression rate). The remaining two — time-to-adoption + post-acceptance failure rate — require new fields on `DecisionRecord` (`surfacedAt: Date` and `firstCommitPasses: Bool?`) and are deferred to v1.5+ alongside the `surfacedAt` plumbing.

2. **Calibration cycle #1 against four benchmark corpora.**
   - **swift-collections** (Apple) — algebra-heavy: `BitSet`, `OrderedSet`, `OrderedDictionary`, deque merging.
   - **swift-numerics** (Apple) — algebra-heavy: `Complex`, `RealFunctions`.
   - **swift-algorithms** (Apple) — algebra-heavy: combinatorics + folds.
   - **SwiftPropertyLaws** (sibling kit) — literally the testbed SwiftInfer was designed against.

   Total triage burden estimated at 50–150 suggestions across the four corpora. Each suggestion records a `DecisionRecord` per-corpus; the metrics command aggregates across them.

3. **Empirically-derived signal weight tuning + findings writeup.** Apply PRD §17.3 step 2 weight-perturbation analysis to the aggregated decisions data: which signal weights, if changed by ±10, would have improved the precision/recall trade-off without spiking the false-positive rate? Land tunings in a single commit; re-run discover on the corpora to verify projected improvements. Writeup at `docs/calibration-cycle-1-findings.md` documents corpus selection + raw metrics + perturbation hypotheses tested + final tunings shipped.

PRD §19 sets the long-term acceptance-rate target at ≥ 70% after 6 months of dogfooding; v1.4's one-shot cycle establishes the baseline + the first round of corrections, not the full target.

### v1.4-only mechanical changes the milestones didn't cover

- **Metrics-command target placement.** The new subcommand lives in `Sources/SwiftInferCLI/MetricsCommand.swift` (mirroring the M8 `ConvertCounterexampleCommand.swift` placement), wired into `SwiftInferCommand.subcommands` alongside `Discover` / `Drift` / `ConvertCounterexample`.
- **`Decisions.merge(_:)` helper.** The metrics command needs to fold multiple decisions files into one in-memory aggregate. Add a small `merge(_:)` helper on `Decisions` (de-dup by `identityHash` keeping the most recent decision when collisions occur — same posture as `Decisions.upserting(...)`).

## v1.4 acceptance bar

Mirroring PRD v1.0 §11 + §17.3 + §19 + the v1.x release plans' acceptance bars, v1.4 is not done until:

a. **All §13 perf budgets still pass at the v1.4 commit.** Rows 1–5 hold. The metrics command lives outside the discover hot path; signal-weight changes don't materially alter `Score` computation cost. (V1.4.5 confirms.)

b. **All §16 hard guarantees still pass.** The metrics command writes nothing — it reads `.swiftinfer/decisions.json` and renders to stdout. Weight-tuning changes touch `Signal.kind`'s default weights but never edit existing source. (Implicit; not a separate sub.)

c. **§14 privacy guarantee still holds.** Metrics aggregation is purely local — no networking-API touches. PRD §14 paragraph "the v1.1 Adoption Tracking dashboard … operates on `.swiftinfer/decisions.json` only and does not phone home" is preserved verbatim. `NoNetworkRuntimeTests` covers the runtime path. (Implicit.)

d. **`swift-infer metrics` subcommand exists and renders the three shippable §17.2 metrics.** Acceptance rate, rejection rate (FP proxy), suppression rate — all per template. (V1.4.1.)

e. **Calibration cycle #1 complete.** Decisions captured for ≥ 3 of the 4 benchmark corpora (PRD §19 ≥ 3-package criterion); aggregate decision count ≥ 50 (so per-template metrics have ≥ a few data points). (V1.4.2.)

f. **At least one weight tuning shipped, justified by §17.3 step-2 analysis.** Even if calibration data shows the v0.3 starting weights were broadly correct, ship at least one targeted tuning to validate the loop end-to-end. (V1.4.3.)

g. **Calibration findings writeup committed.** `docs/calibration-cycle-1-findings.md` documents the cycle. (V1.4.4.)

h. **Performance baseline recorded for v1.4.** `docs/perf-baseline-v1.4.md` lists wall numbers + memory delta for every §13 row at the v1.4 commit. (V1.4.5.)

i. **CLI version reports `1.4.0`.** `swift-infer --version` no longer emits `1.3.0`. (V1.4.6.)

j. **`CHANGELOG.md` gains a `## [1.4.0]` entry.** Summary covers the metrics command + calibration cycle + tunings + perf re-baseline. (V1.4.6.)

k. **`docs/archive/v1.4 Calibration Plan.md` exists.** This plan is archived (V1.4.7) and `CLAUDE.md` repo-state reflects v1.4 cut.

## Out of scope

- **Time-to-adoption metric.** PRD §17.2 row "Time-to-adoption (timestamp(accept) − timestamp(suggestion first surfaced))." Requires adding a `surfacedAt: Date` field to `DecisionRecord` + plumbing through `Discover.run`'s suggestion surfacing path. v1.5+.
- **Post-acceptance failure rate metric.** PRD §17.2 row "accepted suggestions whose test fails on commit / total accepted." Requires new `firstCommitPasses: Bool?` field + a CI-side hook to set it after the suggestion's accept-flow stub runs in the consuming repo. v1.5+ at earliest; depends on UX design for the hook.
- **`swift-infer metrics --output json`.** v1.4 ships table-only output. JSON output is straightforward to add later if calibration tooling needs it.
- **Per-corpus metrics breakdown.** v1.4 metrics aggregate across all `--decisions` paths into a single table. Per-corpus drill-down (which template fired well on swift-collections but poorly on SwiftPropertyLaws?) is a future refinement.
- **Cross-cycle calibration drift detection.** PRD §17.2 last row mentions "score-threshold drift across kit versions." Requires comparing v1.4 metrics against a future v1.5 metrics output. Out of scope for cycle 1.
- **Calibration-driven template retirement.** PRD §17.2 calls templates with < 50% acceptance after 20 suggestions "candidates for retirement." v1.4's threshold work is weight tuning only — actually deleting a template is a v1.5+ decision after the cycle 1 data is reviewed.
- **Telemetry / hosted aggregation.** PRD §14 + §19 + §17 are explicit: aggregation is local, no telemetry. v1.4 preserves this.
- **PRD §20 v1.1+ trajectory items besides metrics.** SemanticIndex, IDE integration, `swift-infer apply` stay deferred. v1.4 closes the metrics deferred surface but doesn't promise the others.
- **Kit-side coordination.** `Package.swift` stays at `from: "2.0.0"`. Calibration touches signal-weight defaults in this repo only.

## Open decisions

1. **Where does the metrics command read decisions from by default.** Two natural choices: (a) walk up to the package root and read `<root>/.swiftinfer/decisions.json` (mirrors `discover`'s walk-up posture); (b) require an explicit `--decisions <path>` flag (no default). Default proposal: **(a) walk-up** for the no-args case, plus accept zero-or-more `--decisions <path>` overrides for multi-corpus aggregation. Rationale: the no-args case (`swift-infer metrics` from a package root) is the most common — single-repo developers checking their own decisions over time. The repeatable flag handles the calibration use case where the user is aggregating across the four cloned repos.

2. **Signal-weight tuning surface — where do the new defaults live.** Two choices: (a) edit `Signal.weight` defaults at the call sites (each template / scorer); (b) introduce a `SignalDefaults.swift` central table that templates read from. Default proposal: **(a) edit at call sites.** Rationale: the v0.3 weights are scattered across templates today; a central table is a refactor that's better motivated by a second-cycle calibration that touches many weights at once. Cycle 1 may only need 1–3 weight changes, in which case the refactor is over-engineering.

3. **Whether to commit the cycle-1 decisions data into the repo.** Two choices: (a) commit `docs/calibration-cycle-1-data/{swift-collections,swift-numerics,swift-algorithms,SwiftPropertyLaws}.decisions.json` so future cycles can diff against them; (b) keep the decisions in their respective benchmark-repo working copies only; reference them from the findings doc but don't include them in this repo. Default proposal: **(a) commit them.** Rationale: PRD §17.3 step 1 is "aggregate decisions from a corpus of opt-in projects" — the corpus selection IS part of the calibration record; committing the raw decisions makes cycle 2's perturbation analysis reproducible. Each decisions.json is small (one entry per surfaced suggestion); even four of them is a few KB.

4. **How to handle a corpus that produces zero suggestions.** Possible outcomes during V1.4.2: a benchmark repo's discover surfaces 0 visible suggestions (e.g., suppression cuts everything to Possible). Default proposal: **note in the findings doc + count toward "≥ 3 of the 4 corpora" criterion as long as the discover ran without error.** Rationale: a zero-suggestion result is still a data point — it tells us either the suppression filter is calibrated correctly for that corpus or that the corpus has no relevant patterns; both are useful observations for cycle 1.

5. **Whether the cycle-1 weight tuning should target the most-rejected template aggressively or be spread across multiple signals.** Default proposal: **target the highest-impact single signal first** (highest absolute weight delta × number of contributing suggestions). Rationale: cycle 1's purpose is to validate the loop, not to perfectly tune every weight. A single targeted change is easier to reason about and easier to roll back if cycle 2's data shows it was wrong.

6. **Re-run cadence after tuning.** Two choices: (a) re-run discover on all four corpora after the tuning patch lands, capturing a "v1.4 post-tune" decisions snapshot for cycle 2's diff target; (b) just verify the tuning compiles + the existing test suite still passes; defer the re-run to cycle 2. Default proposal: **(b) compile + existing tests only**. Rationale: re-running the full triage on four corpora would double the human-time investment for a cycle 1 sanity check that the tuning didn't break anything. The full re-run is cycle 2's natural input.

7. **Whether to capture per-corpus discover output (the visible suggestion stream) alongside the decisions.** Default proposal: **(a) capture both.** The visible stream is what surfaced; the decisions are what was decided. Both forms together let cycle 2 diff "what changed in v1.4-tuned discover output" against cycle 1's baseline, even before any new triage happens.

8. **Whether the metrics command should warn when fewer than 20 decisions exist for a template.** PRD §17.2 names 20 as the cliff for "candidates for retirement." Default proposal: **(a) yes, surface a `note: <template> has fewer than 20 decisions; results are advisory only` line above the table.** Rationale: keeps the user from over-interpreting low-count statistics; mirrors the §3.5 conservative-bias posture.

## v1.4 trajectory framing

PRD v1.0 §17.3 names calibration as "the operationalization of high precision, low recall — without it, the philosophy is just a slogan." v1.4 is cycle 1 of that loop. After v1.4 ships:

- The `swift-infer metrics` command exists, so cycles 2+ don't need to re-derive the analysis tooling.
- The cycle-1 decisions corpus (committed at V1.4.2) is the diff target for cycle 2's perturbation analysis.
- The cycle-1 findings doc captures one-time observations (corpus selection, gaps in metric coverage, etc.) that future cycles can build on.

The natural cadence is one cycle per release — every 2–3 months — until the §19 acceptance-rate target stabilizes. Cycle 2 (~v1.5) widens the corpus (additional benchmark repos, possibly invited closed-source pilots per PRD §17.3 OD), runs the metrics command against the merged history, applies a second round of perturbations, and re-publishes the findings doc.

After the calibration loop reaches steady state, the natural next contributions return to the §20 v1.1+ trajectory:

- **PRD §20 SemanticIndex** — biggest v1.x lift. Was originally deferred from v1.0; the constraint-engine + index unlock the IDE integration + `apply`/`metrics` (already shipping in v1.4) items downstream and cleanly enables M12 / M13.+ / M14 cross-target / cross-test data-flow correlation.
- **PRD §20 IDE integration** — depends on SemanticIndex.
- **PRD §20 `swift-infer apply`** — moves from "all output is opt-in via `--interactive`" to "single command applies a baseline set of accepted suggestions."
- **Multi-predicate equivalence classes** (M13 axis 3) — SemanticIndex-blocked.
- **Cross-target enum coverage** (M14 deferred bit) — SemanticIndex-blocked.
- **Cross-test data-flow correlation** (M10 + M16 deferred) — natural sequencing post-SemanticIndex.

No active milestone plan opens with this commit. Subsequent work picks one of the above and opens a fresh `docs/<plan>.md`.
