## 2. The four workstreams

Each workstream is independently mergeable. v1.24 ships A + B + C + D in four focused commits; sub-commit ordering is small-to-large by code complexity.

### Workstream A — Asymmetric label class mismatch counter on round-trip

**What:** New counter signal on `RoundTripTemplate.suggest(for:)` that fires at `-25` (full veto magnitude) when the pair has **asymmetric label class mismatch**: forward side has `direction-label` (in `DirectionLabels.curated`) and reverse side has `domain-marker` (in `DomainMarkerLabels.curated`), or vice versa.

**Cycle-19 origin + cycle-20 reconfirmation:** the cycle-19 findings doc identified ~9 OC asymmetric round-trip cross-pairs (`index(after:) × _minimumCapacity(forScale:)`-shape) surviving V1.22.B's both-sides direction-counter at score 20 (Possible). V1.22.B fires at -25 only when **both** sides are direction-labeled; asymmetric pairs fall through to V1.12.1's single-side -15 path. Cycle-20 sample measured these 5/5 = 100% reject (cycle-20 #7-#10, #11) — first measurement of the asymmetric class.

**Implementation site:** new file `Sources/SwiftInferTemplates/RoundTripAsymmetricLabelCounter.swift` extending `RoundTripTemplate` with `asymmetricLabelClassMismatchCounterSignal(for:)`. Wired into `RoundTripTemplate.suggest(for:)` alongside the existing `directionLabelCounterSignal`, `domainMarkerCounterSignal`, `setAlgebraShapeVeto`, `mathForwardFunctionPairVeto` calls.

**Score posture:** veto fires at -25. Combined with V1.12.1 single-side direction-counter -15 (one side direction-labeled) + V1.15.1 domain-marker counter (only fires when BOTH sides domain-marker-labeled — not asymmetric), score becomes 30 + 5 (carrier) - 15 (V1.12.1 direction) - 25 (V1.24.A asymmetric) = -5 → Suppressed. Closes the asymmetric class.

**Mechanism class:** new sub-class of class 6 (parameter-label semantic-intent counter) — the "asymmetric label class mismatch" sub-class. Mirrors V1.22.B's both-sides direction-counter pattern but on a different shape.

**Cost:** ~half day. New file ~50 LOC, ~8 unit tests covering asymmetric pair detection (direction × domain-marker), single-class pair preservation (both direction = V1.22.B; both domain-marker = V1.15.1; both stride-style = V1.22.D), no-fire on neither side labeled.

**Plan-vs-actual projection:** **closes ~5-7 OC asymmetric cross-pair candidates** (cycle-20 sample 5 picks + 0-2 unsampled survivors).

### Workstream B — `reverse` / `removeFirst` / `removeLast` veto on idempotence-lifted for non-IteratorProtocol carriers

**What:** New veto signal on `IdempotenceTemplate.suggest(forLifted:)` that fires at `Signal.vetoWeight` when the lift's underlying mutating method name is in `MutatorBlockedFromIdempotence.curated` AND the carrier is value-semantic (per the lifted-admission gate). The new curated set:

```
{reverse, removeFirst, removeLast, popFirst, popLast,
 dropFirst, dropLast}
```

**Cycle-20 origin:** cycle-20 sample measured 4/4 reject on `OrderedDictionary.reverse()`, `OrderedDictionary.removeFirst()`, `OrderedDictionary.removeLast()`, `OrderedSet.reverse()`. These methods aren't IteratorProtocol-conforming carriers (V1.21.A doesn't catch them) but their lifted shadows aren't idempotent:
- `reverse()` lifted: `reverse(reverse(s)) = s ≠ reverse(s)` for non-palindromes.
- `removeFirst()` / `removeLast()` lifted: state advances per call (removes one element).
- `popFirst` / `popLast` / `dropFirst` / `dropLast` (added to curated set for future-corpora coverage): same shape as `removeFirst`/`removeLast`.

**Implementation site:** new file `Sources/SwiftInferTemplates/IdempotenceTemplate+MutatorBlocklistVeto.swift` extending `IdempotenceTemplate` with `mutatorBlocklistVeto(forLifted:)`. Wired into `IdempotenceTemplate.suggest(forLifted:)` via `liftedAuxiliarySignals(...)` alongside the existing V1.21.A `iteratorProtocolCarrierVeto`. New core file `Sources/SwiftInferCore/MutatorBlockedFromIdempotence.swift` for the curated set.

**Score posture:** full veto via `Signal.vetoWeight` collapses score to Suppressed. Calibration record preserved per V1.5.2 reuse posture.

**Mechanism class:** extension of V1.21.A's class 7 carrier-protocol-conformance veto sub-class — generalizes from "Iterator-method-name on Iterator-conforming carrier" to "explicit-non-idempotent-mutator-name on any value-semantic carrier" (no protocol-conformance requirement).

**Cost:** ~half day. New core file ~30 LOC, new template extension ~50 LOC, ~10 unit tests covering all 7 curated names + non-curated names + non-value-semantic carrier (no-fire).

**Plan-vs-actual projection:** **closes ~4-6 OC candidates** on the v1.22 surface (`OrderedDictionary.reverse()`, `OrderedDictionary.removeFirst()`, `OrderedDictionary.removeLast()`, `OrderedSet.reverse()` + 2 more from v1.22 surface that cycle-20 didn't sample).

### Workstream C — Non-deterministic shuffle veto extension

**What:** Extend `IdempotenceTemplate`'s `nonDeterministicVeto(...)` body-signal detection to fire on `mutating func shuffle()` lifted-idempotence claims. Cycle-20 sample measured 1 unknown on `OrderedDictionary.shuffle()` lifted-idempotence — the suggestion surfaced despite shuffling being non-deterministic; the existing `nonDeterministicVeto` body-signal detector didn't catch the OC stdlib RNG call pattern.

**Two implementation options (open decision #2 below):**

**Option A — name-fallback.** Add `shuffle` to a new curated `NonDeterministicMutatorNames.curated` set. Fires veto when `mutating func shuffle()` is detected on a value-semantic carrier (lifted path only — non-lifted path doesn't have this surface). Conservative (small curated set; future-corpora extension via project vocabulary).

**Option B — extended body-signal detection.** Extend the `bodySignals.hasNonDeterministicCall` detection to catch additional RNG patterns: `SystemRandomNumberGenerator()`, `Int.random(in:)`, `Bool.random()`, etc. Broader (catches RNG patterns regardless of method name).

**Lean: (a) name-fallback** for v1.24. Smaller surface; fewer false-positive risks; the name-fallback is symmetric with V1.22.A's name-based extensions to V1.21.A. The body-signal extension is a separate v1.25+ candidate.

**Implementation site:** new file `Sources/SwiftInferCore/NonDeterministicMutatorNames.swift` for the curated set + extension to `IdempotenceTemplate+IteratorVeto.swift` (or new file) for the veto helper. Wired into `IdempotenceTemplate.suggest(forLifted:)` alongside V1.21.A `iteratorProtocolCarrierVeto` + V1.24.B `mutatorBlocklistVeto`.

**Score posture:** full veto via `Signal.vetoWeight`.

**Mechanism class:** extension of class 7 carrier-protocol-conformance veto sub-class.

**Cost:** ~1-2 hours. ~5 unit tests covering `shuffle` + non-shuffle no-fire.

**Plan-vs-actual projection:** **closes ~2-4 OC candidates** (`OrderedDictionary.shuffle()` + `OrderedSet.shuffle()` + 0-2 more).

### Workstream D — Capacity-from-scale + formatter shape-disambiguation veto on idempotence non-lifted

**What:** Extend `IdempotenceTemplate.suggest(for:)` non-lifted path with a new veto helper that fires on:

1. **Capacity-from-scale shape:** function name matches `*Capacity*` or `*Count*` AND first-parameter label is `forScale:` or `forCapacity:` AND `(Int) -> Int` shape. Examples: `_minimumCapacity(forScale:)`, `_maximumCapacity(forScale:)`, `wordCount(forScale:)`, `_scale(forCapacity:)`. The idempotence claim `f(f(scale)) == f(scale)` is meaningless — capacity-of-capacity (or word-count-of-word-count) is a shape-coincidence, not a fixed-point operation.

2. **Formatter shape:** function name matches `_description*` or `format*` AND single-param `(T) -> String` shape (where T may be the type name itself, an enum, or a tuple). Examples: `_description(type:)`, `format(_:)`, `nearMissLines(_:)`. The idempotence claim `f(f(x))` doesn't type-check — `format` returns String; `format(format(x))` would require String input which doesn't match the original parameter type.

**Cycle-20 origin:** cycle-20 idempotence non-lifted sample measured 0/5 = 0% (cycle-20 #14 `_minimumCapacity`, #16 `wordCount`, #18 `format`, #12 `_description(type:)`, #15 `bucket(after:)`). The 5-cycle-flat 0% rate is dominated by these shape-coincidence patterns — the surviving 23-candidate pool's composition is ~10-15 such picks.

**Implementation site:** new file `Sources/SwiftInferTemplates/IdempotenceTemplate+ShapeDisambiguationVeto.swift` extending `IdempotenceTemplate` with `shapeDisambiguationVeto(for:)`. Wired into the non-lifted `IdempotenceTemplate.suggest(for:)` alongside existing vetoes (`setAlgebraShapeVeto`, `mathForwardFunctionVeto`, `protocolCoverageVeto`, etc.).

**Score posture:** full veto via `Signal.vetoWeight`.

**Mechanism class:** extension of class 7 (function-name + type-shape composite) — third extension in the lineage (V1.14.1 → V1.16.1 SetAlgebra + V1.21.C math-forward + V1.24.D capacity/formatter).

**Cost:** ~half day. New file ~80 LOC, ~12 unit tests covering:
- Capacity-from-scale: `_minimumCapacity(forScale:)`, `_maximumCapacity(forScale:)`, `_scale(forCapacity:)`, `wordCount(forScale:)`, `bucketCount(forCapacity:)` all veto.
- Formatter: `_description(type:)`, `format(_:)`, `nearMissLines(_:)`, `headerLine(_:)` all veto.
- Negative cases: `normalize(_:)` (curated verb; should NOT veto), `simplify(_:)` (FixedPointNames; should NOT veto), `clamp(_:)` (FixedPointNames; should NOT veto).

**Plan-vs-actual projection:** **closes ~10-15 of 23 idempotence non-lifted candidates** (the cycle-20-finding shape-coincidence class).

## 5. Open decisions

1. **Workstream B curated set extension scope.** Default proposal: **(a) include `popFirst` / `popLast` / `dropFirst` / `dropLast` for future-corpora coverage** (in addition to the cycle-20-measured `reverse` / `removeFirst` / `removeLast`). Rationale: these are canonical Swift mutator names that share the state-advancing semantics; cycle-20 didn't sample because v1.22 surface didn't include them, but future corpora may. Trade-off: 4 extra curated names; minimal false-positive risk because joint match (curated method name + value-semantic carrier) is required.

2. **Workstream C — name-fallback vs body-signal extension.** Default proposal: **(a) name-fallback only** (curated `shuffle` name on value-semantic carrier vetoes). Rationale: smaller surface; symmetric with V1.22.A's name-based extension. The body-signal extension is a separate cycle-22+ candidate after the v1.24 measurement validates the name-fallback approach.

3. **Workstream D — formatter pattern matching.** Default proposal: **(a) prefix match `_description*` + `format*`** (and `*Format*` for case-insensitive coverage). Conservative; future-cycle extension if user-code formatter naming patterns diverge.

4. **Whether to re-measure perf at V1.24.E.** Default proposal: **(a) yes, re-measure.** v1.24 ships 4 Sources/ changes; mirrors v1.21/v1.22 plan §"Open decisions" patterns. Expected delta vs v1.23 baseline: ≤+5% wall time (the new vetoes are O(1) hash-set lookups per template-suggest call).

5. **Whether to re-run the cycle-1..14 corpora at V1.24.E for plan-vs-actual closure.** Default proposal: **(a) yes, re-run all four** (mirrors V1.22.E posture).

6. **Cycle numbering: cycle 21 = v1.24.** Contiguous with v1.23 = cycle 20 (empirical), v1.22 = cycle 19, v1.21 = cycle 18.
