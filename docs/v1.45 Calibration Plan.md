Working doc for the v1.45 cut (forty-second calibration cycle). **First of the Phase 1.5 verifiable-fraction expansion arc** the cycle-41 findings (`docs/calibration-cycle-41-findings.md`) called out. Where v1.42–v1.44 completed Phase 1's architectural shift (verify pipeline + 4-outcome reporting + 2-template × 3-carrier matrix), v1.45 begins lifting the **verifiable-fraction** on the cycle-27 corpus from V1.44.F's 6.25% baseline toward the **+27.6pp aggregate ceiling** the findings projected.

## What v1.45 ships

Six workstreams + bundled closeout, mirroring the v1.44 structure:

1. **V1.45.A — `CommutativityStubEmitter`**. New emitter mirroring `IdempotenceStubEmitter` but for the two-argument property `f(a, b) ≈ f(b, a)`. Three-carrier dispatch from the start (`Complex<Double>` / `Double` / `Int`). The stub:

   - Generates two values per trial — `Gen<T>` `.zip(Gen<T>)` shape (or equivalent inline composition).
   - Computes `lhsResult = f(a, b)` and `rhsResult = f(b, a)`.
   - Checks equality: `isApproximatelyEqual` for FP, `==` for Int.
   - On failure: emit `VERIFY_DEFAULT_RESULT: FAIL` + `VERIFY_DEFAULT_TRIAL` + `VERIFY_DEFAULT_INPUT` (the `(a, b)` pair as a single string for parser compatibility) + `VERIFY_DEFAULT_FORWARD` (= `lhsResult`) + `VERIFY_DEFAULT_INVERSE` (= `rhsResult`) + exit 1.

   Same V1.43.C marker contract — `VerifyResultParser` consumes commutativity output identically to round-trip/idempotence.

   **Two-pass design.** Mirrors V1.44.A/C: Complex<Double> two-pass (default finite + `Gen<Complex<Double>>.edgeCaseBiased()`-derived two-value gen), Double two-pass (with the existing inlined `doubleWithNaN`), Int single-pass with zero-edge sentinel.

   **Sub-decision**: the two-value edge generator. For Complex<Double>, the edge cases are 12 single-value entries — pairing them yields 12 × 12 = 144 pair combinations, most of which are non-finite × non-finite (point-at-infinity collapse, structurally always commutative). Cleanest: bias the *first* value to the edge generator and draw the *second* from the default generator (so each trial gets one curated edge + one finite value). Symmetric pairing would surface the same advisory signal at half the cost. The runner's `matchEdgeCaseIndex` still resolves the edge-side value's curated index.

2. **V1.45.B — `CommutativityPairResolver`**. Resolves a `SemanticIndexEntry` to a single-function two-argument call expression. Analog of `IdempotencePairResolver`:

   - Validates `templateName == "commutativity"` + carrier in `CommutativityStubEmitter.supportedCarriers`.
   - Constructs `<bareCarrier>.<funcName>` from `primaryFunctionName`.
   - Stub calls `<funcName>(a, b)` and `<funcName>(b, a)` (the stub appends the arg pair; the resolver just returns the bare call).

3. **V1.45.C — `Verify` CLI template dispatch extension**. Add `"commutativity"` to the `buildStubBundle` switch in `VerifyCommand+TemplateDispatch.swift`. Update the `unsupportedTemplate` `expected:` list to `["round-trip", "idempotence", "commutativity"]`.

4. **V1.45.D — `RoundTripPairResolver.curated` expansion**. Append the hyperbolic pairs to the curated list:

   ```
   ("sinh(_:)", "asinh(_:)"), ("asinh(_:)", "sinh(_:)"),
   ("cosh(_:)", "acosh(_:)"), ("acosh(_:)", "cosh(_:)"),
   ("tanh(_:)", "atanh(_:)"), ("atanh(_:)", "tanh(_:)")
   ```

   Plus a permissive pair for `cos(_:)`/`acos(_:)` extension to match cycle-27's `tan/atan` orientation: this is already in the curated list, no change needed there. Verify cycle-27 picks #4 (`sinh/asinh`) and #5 (`tanh/atanh`) become in-scope ACCEPTs after this expansion. Zero new code beyond the list append; the existing round-trip stub + dispatch handles them.

   **Sub-decision**: should the resolver accept any pair that appears in the `Suggestion.evidence` array (the non-curated derivation path the cycle-41 findings discussed)? **Deferred to v1.46+** — that path requires a SemanticIndex schema bump to persist both pair halves and is its own architectural decision. v1.45 sticks with the curated-list expansion.

5. **V1.45.E — Tests**. Three areas:

   - **E.1** Unit tests for `CommutativityStubEmitter` — ~14 tests mirroring `IdempotenceStubEmitterTests` (carrier validation, imports, two-arg call shape, equality forms per carrier, seed hex, trial budget, marker contract, exit codes, edge generator references).
   - **E.2** Unit tests for `CommutativityPairResolver` — ~7 tests mirroring `IdempotencePairResolverTests`.
   - **E.3** Integration tests (subprocess; `.integration`):
     - **E.3.a** commutativity × `Complex<Double>` × `bothPass` — `{ (a, b) in a + b }` (Complex addition is commutative on the finite domain *and* on non-finite via point-at-infinity). Pin `.bothPass`.
     - **E.3.b** commutativity × `Double` × `defaultFails` — `{ (a, b) in a - b }` (subtraction is non-commutative for any unequal pair; `a - b ≠ b - a` whenever `a ≠ b`, which is overwhelmingly probable in 100 trials of `Double.random`). Pin `.defaultFails`.
     - **E.3.c** commutativity × `Int` × `bothPass` — `{ (a, b) in a + b }` (Int addition is commutative; single-pass zero-edge sentinel applies). Pin `.bothPass` with `edgeTrials: 0`.

## Why commutativity, not associativity (or both)

Three reasons in order of weight:

1. **Highest-leverage single addition per cycle-41 findings.** Commutativity directly unlocks 2 cycle-27 REJECTs (#22 `binomial`, #23 `distance`); associativity unlocks 2 more (#25 `distance`, #27 `-`). Doing only commutativity in v1.45 surfaces the **first measured verify-mode REJECT confirmation** as a discrete milestone, before complicating the cycle with a second template's design + integration work.

2. **Shape distinction — two-value vs three-value generation.** Commutativity needs `Gen<T>.zip(Gen<T>)`; associativity needs `Gen<T>.zip(Gen<T>, Gen<T>)` (or three-tuple equivalent). They share the multi-argument-generation infrastructure but differ in arity and per-trial cost. Building commutativity first establishes the two-value pattern cleanly; associativity in v1.46 builds on it.

3. **Determinism considerations.** The commutativity stub's `(a, b)` pair under the seeded Xoshiro is straightforward to reason about (each trial draws two values). Associativity's three-value pair across `f(f(a, b), c)` and `f(a, f(b, c))` has more interaction points where IEEE 754 rounding could surface false-positive failures on FP carriers. Doing them in separate cycles keeps the per-cycle design surface bounded.

The v1.46 plan picks up associativity using v1.45's two-value design as the precedent.

## Why expand the curated pair list now

Two reasons:

1. **Zero-code cost.** The expansion is a `RoundTripPairResolver.curated` array append. ~6 entries × 2 directions. No new emitter, no new stub, no new dispatch.

2. **First non-zero ACCEPT addition to the verifiable-fraction.** Picks #4 (`sinh/asinh`) and #5 (`tanh/atanh`) land in-scope after the expansion. They're predicted `.bothPass` (canonical math inverse pairs), adding evidentiary weight to the cycle-27 ACCEPT count. The cycle-42 measurement gets one more data point in the "verify confirms ACCEPT" column.

## Scope boundaries

- **In scope (v1.45):** Commutativity template (`f(a, b) ≈ f(b, a)`) × 3 carriers; CLI template dispatch extension; curated round-trip pair list expansion (6 hyperbolic pair entries); cycle-42 calibration measurement.
- **Out of scope, deferred to v1.46:** Associativity template; dual-style-consistency template; monotonicity template; non-curated round-trip pair derivation (`Suggestion.evidence`-based pair lookup); `DerivationStrategist` integration at verify-time for non-FP/Int carriers (`Base.Index`, `_Bucket`, user types); `Float` / `Complex<Float>` / `String` carriers; accept-flow integration (verify outcomes → `decisions.json`); verification cache; "Verified" first-class tier.
- **Not changing:** Any v1.44-and-earlier verify pipeline behavior for `round-trip` × all 3 carriers or `idempotence` × all 3 carriers. The V1.45.C template-dispatch extension is strictly additive.

## Risks

1. **Commutativity edge generator design — `(a, b)` pairing semantics.** Two-value generation has a design choice: bias both, bias one, or use independent generators. The proposed "bias first value to edge, draw second from default" gives ~10 curated-vs-default pairs per 100 trials. **Mitigation**: pin the choice with an integration test that's expected `.edgeCaseAdvisory` only if the property's commutativity breaks specifically on edge-paired inputs. If the kit-supported one-value edge generators don't naturally compose, inline the existing v1.44 pattern.

2. **Complex `(a, b)` point-at-infinity equality dominates.** swift-numerics' Complex `==` collapses all non-finite values to a single equivalence class, so `f(a_inf, b_inf) ≈ f(b_inf, a_inf)` is structurally always true for any function whose non-finite outputs stay non-finite. The advisory signal on Complex commutativity is therefore limited to **functions that map some non-finite inputs to finite outputs** (the same constraint as V1.44.E.3.a's idempotence test). **Mitigation**: pick the V1.45.E.3.a integration fixture carefully — a function that's commutative on finite domain but breaks on a specific NaN-finite pairing.

3. **Cycle-42 measurement may stay low even after v1.45.** Even with commutativity verify + 6 curated-pair-list entries, the verifiable-fraction goes from 6.25% to maybe 25%. That's progress but far from the 100% ceiling. **Mitigation**: the cycle-42 findings doc explicitly reports the verifiable-fraction trajectory and updates the v1.46+ roadmap table from V1.44.F. No false equivalence between "v1.45 added measurement" and "v1.45 closed the verify-mode-lift question".

4. **`binomial(n: k:)` might not actually `.defaultFails`.** The Int carrier's default generator (`boundedForArithmetic` at `±65,536`) samples `n` and `k` independently. For typical `(n, k)` pairs, `binomial(n, k) = n! / (k! (n-k)!)`. If `n < k`, the result is 0 (vacuous). `C(n, k) ≠ C(k, n)` when `n ≠ k`. For random `(n, k)`, the probability of `n = k` exactly is ~1/130000 — overwhelmingly the trial finds a counterexample. **Mitigation**: V1.45.F's findings doc reports the actual verify outcome; if `binomial` somehow doesn't verify-confirm (e.g., the function isn't reachable as `Int.binomial` in any user package the verifier can build against), the doc records the gap as a v1.46+ open thread.
