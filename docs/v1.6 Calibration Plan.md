# v1.6 Calibration Plan

Working doc for the SwiftInferProperties v1.6 cut. v1.6 is the **third calibration cycle** — closes cycle-3 priority #1 from `docs/calibration-cycle-2-findings.md` (op-class-aware identity-element pair-formation filter) and runs the §17.3 loop a third time against the cycle-1 corpora. Mirrors the v1.5 plan's V1.5.0–V1.5.8 cadence with two parallel work streams: (a) code work for the pair-formation filter, (b) empirical work re-running discover on the cycle-1+2 corpora to measure suppression delta vs cycle-2's post-rule baseline.

> **v1.6 is a calibration cut, not a feature cut.** Like v1.5 it doesn't extend PRD §7.8 — it adds a single pair-formation skip rule on `IdentityElementPairing`, gating cross-product `(constant, op)` pairs that don't bind to a kit-published identity law. The mechanism is the *complementary* mechanism to v1.5's coverage veto: v1.5 suppressed *what the kit already verifies*; v1.6 suppresses *what's structurally mismatched*. Combined, identity-element suggestions on swift-numerics/ComplexModule should drop to zero (5 surviving cross-product pairs from cycle-2 → 0).

**Ephemeral** — archive to `docs/archive/v1.6 Calibration Plan.md` once the tag is pushed and the §13 + §16 release-blocking criteria still hold.

## What v1.6 ships

Two deliverables:

1. **Pair-formation `(constant, op)` skip rule on `IdentityElementPairing`.** New `IdentityElementPairing.skipsKnownMismatched(identityName:opName:)` private helper checks the candidate's (constant-name, op-name) pair against a curated *known-mismatched* set:
   - **Constant in kit-blessed set `{zero, one, empty, identity}` AND (name, op) does not map to a `KnownProperty`** → skip. Examples: `(zero, *)`, `(zero, /)`, `(zero, pow)`, `(one, +)`, `(empty, *)`. These are the cycle-2 cross-product noise items that v1.5's coverage veto can't filter (no kit law applies, so no "covered" path exists).
   - **Constant NOT in kit-blessed set** (e.g., `none`, `default`, or any custom user constant) → emit normally. Preserves recall for custom monoid identities the engine doesn't yet model.
   - **(name, op) maps to a `KnownProperty`** → emit normally; v1.5's coverage veto handles the kit-covered subset.

   Reuses V1.5.2's `IdentityElementTemplate.identityCoverageCandidate(identityName:opName:)` directly (lifted from `private` to `internal` so the pairing layer can call it). No new mapping table; the v1.5 vocabulary is the v1.6 vocabulary.

2. **Cycle-3 calibration run + findings writeup.** Re-run `swift-infer discover --include-possible` against the four cycle-1 corpora at the v1.6 commit; capture post-filter snapshots to `docs/calibration-cycle-3-data/`. Compare suppression deltas vs the cycle-2 post-rule baseline (353 total `--include-possible` hits across the four corpora). Findings doc at `docs/calibration-cycle-3-findings.md` documents per-corpus suppression counts + the headline "ComplexModule identity-element 5 → 0" demonstration + cycle-4 hypotheses.

PRD §19 sets the long-term acceptance-rate target at ≥ 70% after 6 months of dogfooding; v1.6 is cycle 3 of that loop.

### v1.6-only mechanical changes the cycle didn't cover

- **Lift `identityCoverageCandidate` from `private` to `internal`.** The function lives on `IdentityElementTemplate` per V1.5.2's structure. v1.6 adds `IdentityElementPairing` as a second consumer; making it `internal` (rather than moving the function or duplicating the table) keeps the V1.5.2 origin point as the canonical authority for the (constant, op) → KnownProperty mapping. Future cycle-4 changes extend the table in one place.
- **`IdentityElementPairing.candidates(...)` signature is unchanged.** The filter is purely additive — same `(summaries, identities)` input, same `[IdentityElementPair]` output. Pair count goes down (cross-product noise filtered) but the API contract holds.
- **No `Signal.Kind` additions, no new `KnownProperty` cases.** v1.6's mechanism is a structural skip at pair-formation, not a new scoring signal. The v1.5 catalog stays exactly as-is.

- **Cycle-3 priority #3 — approximate-equality template arm for FP types.** Real `KitFloatingPointTemplate` emitting `checkFloatingPointPropertyLaws(for: T.self, using: gen)` stubs. Sequenced as a v1.7 or v1.8 candidate.

## Open decisions

1. **Allow-list vs skip-list for the filter.** Two choices: (a) skip-list (this plan) — emit by default, skip when `(name, op)` is structurally mismatched; (b) allow-list — emit only when `(name, op)` maps to a `KnownProperty`. Default proposal: **(a) skip-list.** Rationale: (b) would suppress `(none, +)`, `(default, *)`, and any custom user constant from emitting, which is too aggressive. The cycle-1 finding was that *cross-product noise* (kit-blessed constants paired with mismatched ops) is the problem; *unrecognized constants* might be legitimate user-defined identities the engine doesn't model. Skip-list is the conservative choice: filter only what we *know* is wrong.

2. **Where the filter helper lives.** Two choices: (a) `IdentityElementPairing.skipsKnownMismatched(identityName:opName:)` private helper (this plan) — filter logic lives in the pairing layer; (b) move `identityCoverageCandidate(...)` from `IdentityElementTemplate` into a shared `IdentityElementOpClass` enum used by both. Default proposal: **(a) private helper.** Rationale: V1.5.2's `identityCoverageCandidate(...)` is the canonical (constant, op) → KnownProperty authority. Moving it to a separate file adds an indirection without changing semantics. Promoting from `private` → `internal` is the minimal change.

3. **Which file the new tests live in.** Two choices: (a) extend the existing `Tests/SwiftInferTemplatesTests/IdentityElementTemplateTests.swift` — natural home given the function signature lives on `IdentityElementTemplate`; (b) new `IdentityElementPairingFilterTests.swift` — pair-formation-shaped concerns deserve their own file. Default proposal: **(a) extend `IdentityElementTemplateTests.swift` if it stays under the 400-line limit; otherwise (b).** The existing file is at 331 lines (V1.5.2 baseline); 5-7 new tests should land it ~390 lines. If the line limit is breached, split per the V1.5.2 ProtocolCoverageVeto-suite precedent.

4. **Whether to commit cycle-3 decisions data.** Two choices: (a) commit `docs/calibration-cycle-3-data/` mirroring cycle-1+2; (b) keep cycle-3 data in working copies only. Default proposal: **(a) commit them.** Rationale: cycle-3 captures the *post-filter surface*. The surface diff vs cycle-2 is the empirical record. Files are small.

5. **Re-run cadence after the filter lands.** Two choices: (a) capture only post-filter snapshots (cycle-3 has post-rule-of-cycle-2 + post-filter-of-cycle-3); (b) re-capture pre-filter-of-cycle-3 baselines too. Default proposal: **(a) post-filter only.** Rationale: cycle-2's post-rule snapshots are already committed and serve as the cycle-3 pre-filter baseline. Re-running pre-filter would just re-capture the same numbers.

6. **Whether the metrics command should distinguish pair-formation skips from coverage vetoes.** Two choices: (a) extend `swift-infer metrics`'s table to add a "filtered-at-pair-formation" column; (b) leave the existing template-suppressed aggregate column unchanged (pair-formation skips never produce a Suggestion, so they don't reach the Decisions record). Default proposal: **(b) unchanged.** Rationale: pair-formation skips are upstream of Suggestion construction; they don't have an Identity hash and never enter the Decisions record. The metrics command operates on Decisions records only. The cycle-3 findings doc captures the empirical effect (per-corpus pair-count delta) without needing a metrics-command change.

7. **Whether to bundle cycle-3 priority #6 (citation-determinism fix) into v1.6.** Two choices: (a) ship priority #1 only (this plan); (b) bundle priorities #1 + #6. Default proposal: **(a) priority #1 only.** Rationale: same "one rule per release" precedent as v1.5. Priority #6 is a 30-min orthogonal fix; saving it for v1.7's `surfacedAt` plumbing pass keeps v1.6's empirical effect attributable.

## v1.6 trajectory framing

PRD v1.0 §17.3 names calibration as "the operationalization of high precision, low recall — without it, the philosophy is just a slogan." v1.6 is cycle 3 of that loop. After v1.6 ships:

- The two complementary identity-element rules (v1.5's coverage veto + v1.6's pair-formation filter) are in-tree, so cycle-4 can extend either without a refactor.
- The cycle-3 post-filter snapshots in `docs/calibration-cycle-3-data/` are the diff target for cycle 4.
- The cycle-3 findings doc captures the v1.5 + v1.6 composition demonstration on ComplexModule (5 cycle-2 surviving identity-element pairs → 0).

The natural cadence stays at one cycle per release. Cycle 4 (~v1.7) likely picks up cycle-3 priority #2 (stdlib-conformance bake-in — the highest-empirical-leverage remaining item) + #5 (`surfacedAt` plumbing) + #6 (citation-determinism fix) as paired mechanical additions, plus the empirical Possible-tier sampling triage that priority #4 names. After two more cycles, the §19 acceptance-rate target should be measurable.

After the calibration loop reaches steady state, the natural next contributions return to the §20 v1.1+ trajectory:

- **PRD §20 SemanticIndex** — biggest v1.x lift. The cycle-3 stdlib-conformance bake-in (v1.7 candidate) is the textual approximation of one slice of what SemanticIndex would deliver authoritatively (via `lookupConformance`); cycle-4+ may re-frame the curated stdlib table as a `SemanticIndex` consumer once that engine ships.
- **PRD §20 IDE integration** — depends on SemanticIndex.
- **PRD §20 `swift-infer apply`** — moves from "all output is opt-in via `--interactive`" to "single command applies a baseline set of accepted suggestions."
- **Multi-predicate equivalence classes** (M13 axis 3) — SemanticIndex-blocked.
- **Cross-target enum coverage** (M14 deferred bit) — SemanticIndex-blocked.
- **Cross-test data-flow correlation** (M10 + M16 deferred) — natural sequencing post-SemanticIndex.

No active milestone plan opens with the v1.6 commit. Subsequent work picks one of the above and opens a fresh `docs/<plan>.md`.
