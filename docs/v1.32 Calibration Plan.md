# v1.32 Calibration Plan — Domain Template Packs (PRD §20.3)

Working doc for the SwiftInferProperties v1.32 cut. v1.32 is the **twenty-ninth calibration cycle** and the **second design-completion release** (after v1.31's FP approximate-equality template arm).

> **v1.32 splits the monolithic 10-template registry into named domain packs** per PRD §20.3. Cycles 1–28's empirical calibration data is the prerequisite — the PRD explicitly says *"v1 registry is monolithic by design — splitting requires benchmark data to know which signals fire too often outside their natural domain."* That benchmark data now exists.

**Ephemeral** — archive to `docs/archive/v1.32 Calibration Plan.md` once tagged.

## Why this work

The 10-template registry (`round-trip`, `idempotence`, `monotonicity`, `commutativity`, `associativity`, `inverse-pair`, `identity-element`, `dual-style-consistency`, `composition`, `invariant-preservation`) currently fires monolithically — every template runs on every corpus. Cycles 1–28 data shows:

- A **serialization-heavy corpus** (e.g., a protocol-buffer codebase) gets lots of `round-trip` + `inverse-pair` signal (useful) plus noise from `commutativity` + `associativity` on identifier types (not useful).
- A **math-heavy corpus** (e.g., swift-numerics) gets `round-trip` (canonical math inverses), `commutativity`, `associativity`, `identity-element` signal (useful) plus noise from `dual-style-consistency` shape-matching on non-collection types (not useful).
- A **collections-heavy corpus** (e.g., swift-collections OrderedSet) gets `idempotence` (sort lifts), `dual-style-consistency` (form/non-form pairs), `monotonicity` signal (useful) plus noise from `commutativity` on `index(_:offsetBy:)` (not useful — cycle-27 #23 REJECT).

Pack-level enable/disable lets users opt their codebase out of the templates that fire too often outside their natural domain. Doesn't change the inference precision per-template — addresses recall vs noise tradeoff at the call site.

## V1.32.A — `TemplatePack` enum

New file `Sources/SwiftInferCore/TemplatePack.swift`. Mirrors the `MathForwardFunctions` / `IdentityOperatorAlgebra` / `FloatingPointEquatableTypes` curated-set pattern.

**Pack identity** (per PRD §20.3 wording + cycle-1–28 surface evidence):

| Pack | Templates | Why this grouping |
|---|---|---|
| `numeric` | commutativity, associativity, identity-element, monotonicity | Algebraic ops on numeric types: distributivity, identities, ordering. Targets math-heavy corpora. |
| `serialization` | round-trip, inverse-pair | Forward/inverse pairs: encode/decode, parse/format, pack/unpack. Targets codec/serialization corpora. |
| `collections` | idempotence, monotonicity, dual-style-consistency, composition, invariant-preservation | Collection-shaped properties: sort/normalize idempotence, form/non-form dual-style, monotone iteration. Targets collection-library corpora. |
| `algebraic` | commutativity, associativity, identity-element, idempotence, composition | Algebraic-structure laws: semigroup, monoid, group, semilattice, semiring. Targets algebra-modeling corpora. |
| `concurrency` | (empty for now) | Aspirational per PRD §20.3 — no current templates target concurrency primitives. Pack reserved for future template additions. |

Templates can belong to multiple packs (e.g., `monotonicity` is in both `numeric` and `collections`; `commutativity` is in both `numeric` and `algebraic`). This is correct — pack membership describes "this template is useful for codebases of type X," not "this template *only* applies to X."

**Helper**: `TemplatePack.resolve(_:Set<TemplatePack>) -> Set<String>` returns the union of `templateNames` across the supplied packs.

**Tests**: `TemplatePackTests.swift` verifies (1) each pack's membership; (2) cross-pack overlap (monotonicity ∈ numeric ∩ collections); (3) `resolve` union semantics; (4) the `concurrency` pack is empty.

## V1.32.B — `templateFilter` parameter on `TemplateRegistry.discover`

Add optional parameter to both `discover` entry points and to `discoverArtifacts`:

```swift
public static func discover(
    in summaries: [FunctionSummary],
    identities: [IdentityCandidate] = [],
    // ... existing parameters ...
    templateFilter: Set<String>? = nil  // V1.32.B
) -> [Suggestion]
```

**Semantics**: when `nil`, no filtering (current behavior). When non-nil, post-filter `counterSignaled` to keep only suggestions whose `templateName` is in the set. Applied AFTER cross-validation + counter-signal passes so filtering doesn't affect those passes' inputs.

**Backward compatibility**: all existing call sites compile unchanged (default `nil`). All 1959 pre-existing tests pass bit-for-bit.

## V1.32.C — CLI `--packs` flag + config TOML

**Discover subcommand**:
```swift
@Option(
    name: .long,
    help: "Comma-separated list of template packs to enable: numeric, serialization, collections, algebraic, concurrency. Default: all packs."
)
public var packs: String?
```

**Config TOML**:
```toml
[discover]
packs = ["numeric", "serialization"]
```

**Precedence**: CLI > config > all (default). Mirrors `--include-possible` precedence (V1.20.A).

**Validation**: unknown pack names emit a diagnostic warning and are ignored (lenient). An empty effective set after parsing emits a warning ("no packs enabled; no suggestions will surface").

## Scope boundaries

- **In scope**: opt-in filter mechanism. Templates run unchanged when in scope; filtered out when out of scope.
- **Out of scope**: per-pack signal-weight overrides. PRD §20.3 mentions packs improving precision; v1.32 ships the registry split, not signal recalibration. If empirical evidence shows a template needs different weights per pack, that's a future cycle.
- **Out of scope**: pack-author API for third-party templates. Adding new templates is still a Sources/ change in `SwiftInferTemplates/`; v1.32 packs only group *existing* templates.
- **Out of scope**: pack-aware perf optimization (skipping work entirely when a template is filtered out). Current implementation runs all templates and filters post-collection — simpler, no per-template perf change. Future cycle can skip template invocation if it becomes a measurable cost.

## Risk + mitigation

- **Risk**: user configures `--packs numeric` on a collection-heavy codebase and gets zero suggestions, thinks SwiftInfer is broken. **Mitigation**: lenient-validation warning when post-filter surface is empty; clear `--help` text listing all packs.
- **Risk**: a future template fits multiple packs and the membership decision is contentious. **Mitigation**: multi-pack membership is supported (a template can be in 2+ packs). The grouping is a recommendation, not a constraint.
- **Risk**: removing a template from a pack in a future release breaks user configs. **Mitigation**: pack membership is a contract; document at PRD §20.3 update.
