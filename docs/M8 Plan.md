# M8 Execution Plan

Working doc for the M8 milestone defined in `SwiftInferProperties PRD v0.4.md` §5.4 + §5.8. Decomposes M8 into seven sub-milestones so progress is checkable session-by-session. **Ephemeral** — archive to `docs/archive/M8 Plan.md` once M8 ships and the §5.8 acceptance bar is met (mirroring M1–M7).

> **M8 closes out the v1 TemplateEngine surface.** It ships the multi-template signal-accumulation pass that PRD §5.4 has been promising since v0.2, retires the "no stub writeout available for template X in v1" diagnostic across the full template surface, lands the last unary template (`inverse-pair`), and lifts three structural-conformance arms (`CommutativeMonoid`, `Group`, `Semilattice`) from PRD §5.4's "claim-only" status to real kit-defined conformances by extending SwiftProtocolLaws v1.8.0's protocol cluster. After M8, every PRD §5.2 / §5.4 row is implemented; the v1 TemplateEngine is feature-complete and the focus moves to TestLifter M1 (still unstarted in this repo) plus the v1.0 release prep (§16 hard-guarantee re-audit, perf budget re-baseline).

## What M8 ships (PRD v0.4 §5.4 + §5.8)

> **M8 (was M7).** Algebraic-structure composition (§5.4) — semigroup / monoid / group / semilattice / ring claims accumulated from per-template signals on the same type; expanded identity-element detection (init-based + reduce-usage signals); `inverse-pair` template ships standalone for non-Equatable cases (suppressed per §16 #6 explainability).

Four concrete deliverables (one new vs the pre-revision plan — kit-side prereq):

1. **Kit-side prereq — SwiftProtocolLaws v1.9.0.** Lifts three new kit-defined protocols (`CommutativeMonoid`, `Group`, `Semilattice`) into `Sources/ProtocolLawKit/Public/`, mirroring the v1.8.0 four-commit `Semigroup` / `Monoid` arc (`362322e` / `88cc4a1` / `6c4ec2a` / `1b3af7d`). Each ships with its `combine*` Strict law(s), discovery-plugin hookup, and release docs. The natural inheritance shape per the v1.8.0 pattern is:
   ```
   Semigroup
     └─ Monoid
          ├─ CommutativeMonoid       (+ combineCommutativity)
          │    └─ Semilattice         (+ combineIdempotence)
          └─ Group                    (+ combineLeftInverse / combineRightInverse)
   ```
   v1.9.0 is additive vs v1.8.0 — no breaking changes; the existing Semigroup / Monoid downstream continues to work unchanged. SwiftInferProperties bumps the dep from `from: "1.8.0"` (`c143258`, 2026-05-02) to `from: "1.9.0"` once the kit ships v1.9.0. **Out of scope for v1.9.0:** kit-side `Ring` (two-op shape — see open decision #4 below; PRD §5.4's `Numeric`/stdlib target stays canonical), kit-side `CommutativeGroup` (rare in everyday Swift; defer), kit-side `Group acting on T` (function-space carrier doesn't fit the existing kit shape).
2. **Algebraic-structure composition orchestrator.** Extend `RefactorBridgeOrchestrator` (M7.5) to recognize the full PRD §5.4 table beyond M7.5's Semigroup / Monoid arms: **CommutativeMonoid** (associativity + identity + commutativity → kit-defined per M8.0), **Group** (associativity + identity + inverse → kit-defined per M8.0), **Semilattice** (associativity + commutativity + idempotence → kit-defined per M8.0), **Ring** (two compatible monoids on the same type, distributive → stdlib `Numeric`). The accumulator stays per-(type, op-set) per open decision #5 default `(b)`; M8.4 widens the `TemplateSignal` enum and adds promotion rules to `TypeAccumulator.proposal`.
3. **`InversePairTemplate`** — standalone non-Equatable inverse detection. Type pattern `T → U` and `U → T` with `T` *not* satisfying the §3.3 Equatable resolver. Round-trip (M1.4) vetos on non-Equatable; this template fills the gap as a Possible-tier informational claim, suppressed by default per §16 #6 explainability. Pairs with M8.4 indirectly — the inverse signal Group needs is the *unary* `T → T` shape (M8.3's inverse-element detection), not the *function-pair* `T → U` / `U → T` shape M8.1 surfaces. They're distinct.
4. **LiftedTestEmitter arms** for `commutativity` / `associativity` / `identity-element` / `inversePair`. Retires the M6.4 "no stub writeout available for template X in v1" diagnostic across the entire shipped template surface. Mirrors of M6.3's `idempotent` / `roundTrip` and M7.3's `monotonic` / `invariantPreserving` arms. Plus expanded identity-element + inverse-element detection (init-based + `reduce(into:)` accumulator-with-empty-seed signals + curated unary inverse names) per the PRD §5.8 M8 row + the M8.4 Group-arm signal need.

### Important scope clarifications

- **Three of M8's four new conformance arms are real kit-defined extensions, not claim-only.** v0.4 PRD §5.4 (drafted before v1.8.0 shipped) reads "(no kit protocol yet — claim only; M8)" for Group; M8.0 closes that gap by adding `Group` (plus `CommutativeMonoid` + `Semilattice`) to SwiftProtocolLaws v1.9.0. M8.5's `LiftedConformanceEmitter` emits real `extension TypeName: <Protocol> {}` writeouts for all three; the user's project compiles end-to-end against `import ProtocolLawKit`; the discovery plugin verifies the laws on every CI run. **PRD §5.4's table needs a corresponding refresh** — small docs PR after M8.0 lands; not a v1 blocker, just a stale-text cleanup. Tracked in M8.6's validation suite as a doc-consistency check.
- **The remaining claim-only writeout shape collapses to one edge case: "Group acting on T".** PRD §5.4 row 6 — `T → T` self-functions with inverse + identity (permutation groups, etc.). The carrier is the function space, not the type itself; doesn't fit the existing kit shape cleanly. Stays informational with a comment-block writeout. Deferred kit-side modeling to v1.1+ if real corpora justify it. Open decision #1 covers this single case rather than the previous-revision four cases.
- **Ring stays `Numeric` (stdlib), not kit-defined.** PRD §5.4: "Two monoids on same type, distributive → Ring → suggest `Numeric` (with caveats — see below)." Kit-side Ring requires designing for two distinct binary-op witnesses (`+` and `*`) with their own identities — significantly larger surface than the single-op v1.8.0/v1.9.0 protocol pattern. The kit-side Ring design is reasonable v1.1+ material; M8 stays with stdlib Numeric. M8.5's Numeric arm gates on TypeShape's numeric-shape classifier (open decision #4) so the writeout only fires for genuinely Numeric-shaped types.
- **Semilattice → kit `Semilattice` is the primary writeout; `SetAlgebra` (stdlib) drops to a secondary Option B.** Mirrors the PRD §5.4 row 2 pattern where Monoid is the primary kit conformance and `AdditiveArithmetic` is "still suggested as a secondary Option B when the `+` / `zero` shape fits". M8.5 surfaces SetAlgebra as a second `B`-arm alongside Semilattice when the type's curated naming includes set-shaped ops (`union` / `intersect` / `subtract` etc.); the user picks whichever conformance fits their type better. See open decision #3 (revised — was about whether SetAlgebra fires at all; now about how to surface the dual `B` choice).
- **Per-type prompt aggregation already lands in M7.5.** The `[A/B/s/n/?]` collapse-after-decided-type behaviour already works (M7.5b shipped). M8 doesn't touch the prompt UI — it widens which proposals the orchestrator emits, and the existing prompt machinery picks them up. The only prompt change is for the dual-secondary case (Semilattice + SetAlgebra on the same type) — open decision #3.
- **`inverse-pair` (M8.1) is suppressed by default.** PRD §5.8 M8: "`inverse-pair` template ships standalone for non-Equatable cases (suppressed per §16 #6 explainability)." Hidden behind `--include-possible` like monotonicity (M7.1) — the per-template caveat in the explainability block calls out the non-Equatable signal weakness. Strictly informational in v1; v1.1+'s SemanticIndex (PRD §20.1) would add the corroboration to lift it.
- **Inverse-element detection (M8.3) is a *signal source* for M8.4's Group arm, not a standalone template.** Distinct from M8.1's function-pair `inverse-pair`. M8.3 detects `T → T` unary functions named `negate` / `negated` / `inverse` / `reciprocal` / `complement` / `invert` on a type that already has a binary op `(T, T) → T` matching. Doesn't surface as its own suggestion — feeds the orchestrator's `inverseElement` signal. Same shape as identity-element detection (where a `static let zero` paired with a binary op contributes to the Monoid promotion).
- **Multi-claim disambiguation: incomparable arms emit separate proposals.** A type satisfying {associativity, identity, commutativity, inverse} sits at the DAG intersection of `CommutativeMonoid` and `Group` — neither strictly extends the other in v1.9.0's protocol shape (kit-side `CommutativeGroup` is out of scope). Open decision #6 covers the resolution: emit both, with the §4.5 block on each calling out the other arm's signals + a "v1.1+ kit-side CommutativeGroup would collapse these" forward-pointer.
- **TestLifter cross-validation `+20` still gated on TestLifter M1.** No change in M8 — the M3.5 seam stays dormant.
- **`apply` subcommand stays v1.1+** (PRD §20.6). M8's new conformance arms are reachable only through `--interactive`'s `B` route, same as M7.5.

## Out of scope for M8 (re-stated for clarity)

- **Kit-side `Ring` protocol.** Ring is two-op (`+` and `*`) with distributivity; the kit's existing single-op `static func combine(_:_:)` shape doesn't fit cleanly. Designing a dual-witness or layered `AdditiveGroup` + `MultiplicativeMonoid` shape is reasonable v1.1+ kit work; M8 stays with PRD §5.4's `Numeric`/stdlib target.
- **Kit-side `CommutativeGroup` protocol.** Rare in everyday Swift (most everyday Abelian groups in Swift code are integer/floating-point under `+`, already covered by stdlib `AdditiveArithmetic` / `Numeric`); the M8.4 orchestrator emits separate `CommutativeMonoid` + `Group` proposals on the same type when both fire (open decision #6 default), with a v1.1+ forward-pointer to "kit-side CommutativeGroup would collapse these." Defer until real corpora justify it.
- **Kit-side "Group acting on T" protocol.** Function-space carrier; awkward to express in the existing kit shape. Stays informational/claim-only as PRD §5.4 row 6 already says.
- **TestLifter cross-validation `+20` integration.** TestLifter M1 still hasn't started in this repo; M3.5's `crossValidationFromTestLifter` parameter stays dormant. M8.1's `inverse-pair` template's TestLifter-corroboration escalation path is wired but inert.
- **`@CheckProperty(.commutative)` / `.associative` / `.hasIdentity)` / `.hasInverse)` macro arms.** PRD §5.7 ships `@CheckProperty(.idempotent)` / `.roundTrip` / `.preservesInvariant`; the algebraic-structure annotation arms are a v1.1+ extension to `SwiftInferMacroImpl`. M8 ships the discover-side detection only.
- **`swift-infer apply --suggestion <hash>`** — v1.1+ per PRD §20.6. M8's new conformance arms are reachable only through `--interactive`'s `B` route.
- **SemanticIndex-driven escalation** for the inverse-pair template (lifting Possible → Likely when the SemanticIndex confirms the pair). v1.1+ per PRD §20.1.
- **Calibration corpus validation of M8's signal-accumulation precision** (PRD §17). M8's M1.6-style goldens cover byte-stability of the writeouts; the per-suggestion precision rate is a v1.1+ `swift-infer metrics` deliverable.

## Open decisions to make in-flight

1. **Claim-only writeout shape for "Group acting on T" (the only remaining no-target case).**
   - **(a) Emit a comment-block file** at `Tests/Generated/SwiftInferRefactors/<TypeName>/GroupActingOnT.swift` containing only the §4.5 explainability block — no `extension` line. The user gets a tracked record of the claim; drift suppression works the same way; v1.1+ can later add the kit protocol and the writeout becomes a real extension.
   - **(b) No writeout — these proposals don't surface the `B` arm.** The CLI renders the §4.5 block in the discover output and the prompt stays `[A/s/n/?]`. Cleaner UX (no dead extension files), but loses drift-tracking and the user has no persistent record beyond the suggestion's identity hash.
   - **Default unless reason emerges:** **(b) no writeout for the single remaining edge case**. With M8.0 lifting CommutativeMonoid / Group / Semilattice into kit-defined writeouts, the universe of claim-only cases shrinks to "Group acting on T" alone — rare enough that the drift-tracking justification doesn't carry the same weight. Keep the affordance simple; revisit if/when v1.1+ adds the kit-side carrier-function-space protocol.

2. **Inverse-pair tier: Possible by default vs Likely with curated name pair + non-Equatable T?**
   - **(a) Possible by default per the PRD §5.8 suppressed-default posture.** Hidden behind `--include-possible`. Mirrors monotonicity (M7.1 open decision #4 default `(a)`).
   - **(b) Likely when curated name pair + type pattern both fire.** Visible by default; downgrades to Possible if only one fires.
   - **Default unless reason emerges:** **(a) Possible by default**. PRD §5.8 explicitly calls out "suppressed per §16 #6 explainability" for the inverse-pair template. The non-Equatable signal is fundamentally weaker than RoundTrip's Equatable shape (we can't sample-verify the round-trip — we can only claim it from name + type signals). Conservative posture wins.

3. **Semilattice + SetAlgebra dual-arm UX: extend prompt to `[A/B/B'/s/n/?]`, or surface SetAlgebra only when the user rejects Semilattice?**
   - **(a) Extend prompt to `[A/B/B'/s/n/?]`.** Both `B` arms visible at the prompt; user picks either (or both). The dual-secondary case is rare (only triggers when Semilattice's signals fire AND the type also has set-named curated ops) but explicit. Mirrors how PRD §5.4 row 2 describes Monoid + AdditiveArithmetic-secondary: "still suggested as a secondary Option B".
   - **(b) Sequential — show Semilattice first, only surface SetAlgebra if user picks `n` on Semilattice.** Keeps the prompt simple `[A/B/s/n/?]`; user gets a follow-up prompt only if they reject the primary. Adds prompt-flow complexity that doesn't exist in M7's UI.
   - **(c) Drop SetAlgebra entirely — kit Semilattice covers the structural claim; SetAlgebra's wider requirement set is a separate decision the user can make outside SwiftInfer.** Conservative; respects the §5.4 caveat that SetAlgebra has more requirements than just bounded-join-semilattice.
   - **Default unless reason emerges:** **(a) extend to `[A/B/B'/s/n/?]`**. PRD §5.4 row 2 explicitly establishes the "primary kit + secondary stdlib" pattern (Monoid + AdditiveArithmetic); Semilattice + SetAlgebra mirrors it. The prompt extension is a small UI cost — the M7.5 prompt machinery already parses arbitrary-letter responses; widening from `[ABsn?]` to `[ABB'sn?]` is a small parser change. Option (b) hides the affordance; option (c) drops a useful PRD-promised secondary on the floor.

4. **Ring detection: gate on `+`/`*` curated naming, or on TypeShape-detected numeric shape, or both?**
   - **(a) Curated naming alone.** Two compatible monoids whose ops are named `+` / `add` / `plus` vs `*` / `multiply` / `times` (or vocabulary-equivalent) → Ring. Naming is fragile.
   - **(b) TypeShape-detected numeric shape.** The `TypeShape` from M4.1 already classifies stored members + raw types; extend it with a "could be Numeric" classifier that fires when the type stores a single Numeric-conforming value or has an `IntegerLiteralConvertible`-compatible init.
   - **(c) Both — Numeric-shape type AND two compatible monoids on it.** Tightest.
   - **Default unless reason emerges:** **(a) for the Ring claim; (c) for the Numeric-conformance writeout**. The Ring *claim* is a reasonable inference from two-monoid signals; the *Numeric conformance writeout* is more dangerous (commits the user to the full Numeric protocol with all its requirements). Splitting the bar lets us emit a Ring claim broadly (with §4.5 caveats) while reserving the conformance writeout for cases where the type already looks Numeric. The Numeric-shape classifier is a small extension to `TypeShape`; M8.4 invokes it only on the writeout path.

5. **Multi-structure types: when a single type accumulates Ring-eligible signals AND CommutativeMonoid-eligible signals on a *third* op, emit one Ring proposal or two (Ring + CommutativeMonoid)?**
   - **(a) One Ring proposal.** Strict-greatest promotion — the type's strongest claim is Ring; subordinate claims fold into Ring's explainability block.
   - **(b) One proposal per distinct structure.** A type with `+`-Ring + `*`-Ring + `unrelatedOp`-CommutativeMonoid emits two proposals — one Ring (covering `+`/`*`) and one CommutativeMonoid (covering `unrelatedOp`). User decides each independently in the prompt.
   - **Default unless reason emerges:** **(b) one proposal per distinct structure**. Two structures on the same op pair are folded into one (Ring); a third structure on a *different* op is a separate claim and deserves a separate `B` prompt. Matches the user's mental model: "this type has two algebraic structures" not "this type is Ring." The orchestrator's per-type accumulator becomes a per-(type, op-set) accumulator.

6. **Incomparable promotion arms on the same type (CommutativeMonoid AND Group both fire): pick one, or emit both?**
   - **(a) Emit both as separate proposals.** A type with {associativity, identity, commutativity, inverse} sits at the DAG intersection of CommutativeMonoid and Group; neither extends the other in v1.9.0's protocol shape. Each arm carries the full §4.5 explainability with a forward-pointer to "v1.1+ kit-side CommutativeGroup would collapse these." Two `B` prompts.
   - **(b) Pick by inheritance depth — Group has more requirements than CommutativeMonoid → Group wins.** Arbitrary tiebreaker; loses the commutativity claim.
   - **(c) Pick by signal-rarity — inverse is rarer than commutativity → Group wins.** Same tiebreaker, different justification.
   - **(d) Pick CommutativeMonoid — strictly more "Swift-idiomatic" since CommutativeGroup-shaped types in idiomatic Swift are usually already covered by stdlib `AdditiveArithmetic`.** Loses inverse claim.
   - **Default unless reason emerges:** **(a) emit both**. Strict-greatest works inside a chain branch (Semigroup → Monoid → CommutativeMonoid → Semilattice; Semigroup → Monoid → Group); incomparable arms across branches don't have a non-arbitrary tiebreaker. Emitting both is the only choice that doesn't discard signal information; the v1.1+ kit-side `CommutativeGroup` collapse is the natural future evolution. UX cost is one extra `B` prompt on the (rare) double-fire case.

7. **Reduce-usage signal: extend `IdentityElementTemplate` scoring or surface as a separate `ReducerTemplate`?**
   - **(a) Extend `IdentityElementTemplate` scoring.** Adds a `+20` signal from type-flow analysis when `reduce(into: <static-let>) { … <template-op> … }` is detected. Existing template gets richer; no new template surface.
   - **(b) New `ReducerTemplate` standalone.** Mirrors v0.2's separate "Reducer/Event-Application" template (folded into §5.4's algebraic-structure composition per the v0.2 → v0.3 sweep — see PRD §5.2 line 264). Surfaces reduce-usage as its own claim independent of identity-element.
   - **Default unless reason emerges:** **(a) extend identity-element scoring**. PRD §5.2 line 264 explicitly says v0.2's separate Reducer/Event-Application template was *folded into* §5.4's composition cluster — it's no longer standalone. The reduce-usage signal naturally feeds the identity-element template (the seed value used in the reduce IS the identity candidate); separate-template would re-litigate a v0.2→v0.3 design decision.

8. **Promotion ordering for ties.** A type with `{associativity, identity, commutativity, idempotence}` qualifies for both Semilattice (need: assoc + comm + idem) and CommutativeMonoid (need: assoc + comm + identity). Which wins?
   - **(a) Semilattice wins over CommutativeMonoid.** Idempotence is the "rarer" signal (most monoid ops aren't idempotent); detecting it is stronger evidence for the more specific claim. Mathematically a Semilattice IS a CommutativeMonoid where every element is its own product — `Semilattice: CommutativeMonoid` in v1.9.0's inheritance chain.
   - **(b) CommutativeMonoid wins over Semilattice.** CommutativeMonoid has wider applicability; users are more familiar with it. Conservative recommendation.
   - **(c) Emit both.** Two `B` prompts on the same type. User sees both and picks.
   - **Default unless reason emerges:** **(a) Semilattice wins**. Strict-greatest within the CommutativeMonoid → Semilattice chain branch (v1.9.0's `Semilattice: CommutativeMonoid`); same rule the M7.5 orchestrator already follows for Monoid > Semigroup. The §4.5 explainability block for the Semilattice proposal mentions the type also satisfies CommutativeMonoid for users who want the wider claim.

## Cross-cutting per-template requirement (PRD §5.8)

M8 adds one new template (`inverse-pair`). Its §4.5 explainability block ships with the per-template caveats:

- **Inverse-pair caveats.** "non-Equatable T means SwiftInfer cannot sample-verify the round-trip — the suggestion is name-and-type-pattern only"; "TestLifter corroboration not yet wired (gated on TestLifter M1)"; "Possible-tier by default — escalation requires a custom equality witness or annotation API extension"; "if T conforms to Equatable, RoundTripTemplate (M1.4) handles this case with stronger evidence — inverse-pair only fires when EquatableResolver returns `.unknown` or `.no` for T".

The four LiftedTestEmitter arms (M8.2) and five LiftedConformanceEmitter arms (M8.5) inherit their templates' existing explainability content — M2.1 / M2.2 / M2.3 already ship per-template caveats for commutativity / associativity / identity-element; M8.5's kit-arm explainabilities (CommutativeMonoid / Group / Semilattice) call out their per-protocol law caveats (`combineCommutativity` for CMon; `combineLeftInverse` / `combineRightInverse` for Group; `combineIdempotence` for Semilattice); the stdlib arms (SetAlgebra / Numeric) carry their existing caveat blocks (SetAlgebra requirements gap; IEEE-754 caveat for Numeric).

Inverse-element detection (M8.3) doesn't surface as a standalone suggestion, so it doesn't need its own §4.5 block — the inverse-element witness is a contributing-signal data record consumed by M8.4's Group-arm aggregator. The Group proposal's aggregated explainability block credits the inverse-element source: "from inverse-element pairing: <funcName>(_:) -> Self".
