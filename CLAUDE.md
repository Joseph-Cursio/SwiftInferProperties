## What this repo is

**SwiftInferProperties** вЂ” type-directed property inference for Swift. Surfaces idempotence, round-trip pairs, and algebraic-structure (semigroup / monoid / group / semilattice / ring) candidates from function signatures, cross-function pairs, and existing unit tests. All output is human-reviewed; nothing auto-executes.

One-way downstream of [SwiftPropertyLaws](https://github.com/Joseph-Cursio/SwiftPropertyLaws):

```
SwiftInferProperties в†’ SwiftPropertyLaws (PropertyBackend, DerivationStrategist) в†’ swift-property-based
```

## Design decisions baked in (follow rather than re-litigate)

- **Conservative inference вЂ” high precision, low recall.** PRD В§3.5. When in doubt, fewer suggestions.
- **Opt-in, human-reviewed output.** Never auto-applies/executes/commits. CI mode emits warnings, not failures.
- **Avoid the Daikon trap.** Too many suggestions в†’ raise thresholds, don't pile on filters.
- **Explainability is a first-class output.** Every suggestion ships "why suggested" + "why this might be wrong." PRD В§4.5.
- **Generator inference delegates to SwiftPropertyLaws.** Call `DerivationStrategist`; don't reimplement. PRD В§11.
- **Relaxed partial-exploration is allowed for `.tca` interaction verify (cycle 124).** A `measured-bothPass` may be established over the *constructible-action subset* (payload-free + raw cases), skipping non-derivable composition cases (binding / nested `X.Action` / delegate / `Result`) вЂ” accepting that a counterexample reachable only via a skipped action is missed, in exchange for ~4Г— reach. **Binding guardrail:** every partial verdict MUST disclose the excluded set (`verified over M of N action types (excluded: вЂ¦)`) in its `detail` + render; the witness itself must still be constructible. A partial `bothPass` promotes idempotence `.likely в†’ .verified` (with the annotation); a later full-action-space verification supersedes it. This is the one deliberate carve-out from the strict all-actions-or-reject posture вЂ” see `docs/calibration-cycle-124-findings.md`.
- **A measured `bothPass` overrules the Finding-G `.possible` pin (cardinality / biconditional) ONLY at full action-space coverage (cycle 135).** The gated families are pinned because the static smell holds only 33вЂ“50% at runtime (often UI-layer-enforced). A *full-coverage* measured bothPass (`excludedCaseNames.isEmpty`) is sound per-candidate proof the reducer enforces the invariant itself в†’ overrule the pin and promote `.possible в†’ .verified` (30 + 50 = 80 в†’ `.strong` в†’ promoted, via a measured-evidence-only carve-out in `InteractionVerifyEvidenceScoring`, NOT a change to `tier(forScore:)`). A **relaxed / partial** bothPass does **not** overrule вЂ” it stays `.possible` вЂ” because cardinality's failure mode lives in exactly the action types relaxed exploration excludes (`binding` / `PresentationAction` / nested `X.Action`), so a partial bothPass is systematically biased toward false-pass. **Binding guardrail:** a promoted gated verdict MUST disclose that the pin was overruled by full-coverage measured execution (0 excluded). Static score alone still never overrules. Ratified, not yet built вЂ” see `docs/calibration-cycle-135-findings.md`.

## Build & test

- `swift package clean && swift test` (per global `~/CLAUDE.md`) on session start.
- **Makefile** wraps the test orchestration: `make test-fast` (the regex-skip fast path), `make test` (fast suite + the four sequential subprocess batches, fail-fast), `make batch1`вЂ¦`batch4` (individual batches), `make clean-temp` (sweep leftover verifier/corpus build dirs after a killed run), `make help`. Prefer `make test` over a bare full `swift test` вЂ” the batches bound peak temp-disk + avoid the В§13 perf-budget contention flakes. (A killed-mid-run subprocess suite skips its cleanup `defer` and can leak tens of GB of TCA build workdirs to `$TMPDIR`; `make clean-temp` clears them.)
- Skeleton expects `../SwiftPropertyLaws` as a sibling checkout. CI checks both repos out side-by-side.
- Non-subprocess fast path: `swift test --skip 'MeasuredTests|MeasuredExecutionTests|VerifyPipeline'` (~6s, 3200 tests / 429 suites). `--skip` takes a **regex matched against the test ID**, so this one alternation covers all 16 `.tags(.subprocess)` suites вЂ” the 10 `*MeasuredTests`, `InteractionVerifyMeasuredExecutionTests`, and the 5 `VerifyPipeline*` (Integration / CommAssoc / Lifted / Strategist / Idempotence) вЂ” and is self-maintaining for future `*MeasuredTests` / `VerifyPipeline*` suites. (History: the old per-name `--skip` list silently MISSED the four non-`Integration`-named `VerifyPipeline*` subprocess suites, so the "fast" command actually ran them for ~90 min. Don't enumerate names; use the regex. It deliberately does NOT match the *fast* `MeasuredPromotionDeterminismTests` вЂ” "Measured" there is a prefix, not the `вЂ¦MeasuredTests` suffix.) Full `swift test` is dominated by parallel subprocess builds (these suites spawn real `swift build` + verifier runs; the `.tca` ones additionally resolve swift-composable-architecture + swift-syntax, ~60вЂ“160s cold). **A new subprocess suite is auto-covered only if named `*MeasuredTests` or `VerifyPipeline*`; otherwise widen the regex.**
